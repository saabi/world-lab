Best Diagram Set

The project is best explained as a set of related diagrams rather than one system map. One diagram should orient new contributors in the workspace, one should explain route-level ownership, and the rest should zoom into the live /scene renderer, the reusable planet renderer, and the graph/compiler stack that is growing beside it.

1. Repository Map

The active application is fe/. The workspace also contains a reusable procedural graph stack under packages/ and a standalone graph editor app under apps/graph-editor. The archived fe.old/ reference is intentionally outside the active workspace.

Reading notes
  • fe/ remains the active production app and owns the live renderer.
  • packages/ contains framework-neutral graph/compiler/runtime work that is not yet the primary planet renderer.
  • _docs/architecture/procedural-graph/README.md is the canonical ADR for the graph stack.

2. Routes and State Flow

The /scene/[...path] route owns the editable scene document and shared transport. It passes state into the editor shell and viewport panels; selection round-trips through the URL path, while persistent scene and viewport settings use localStorage.

3. Scene Frame Pipeline

This is the most important runtime diagram. The route supplies a scene and time; the viewport evaluates transforms, chooses LOD, converts bodies to sphere/dot instances, selects procedural planet/moon targets, and records everything into one scene engine.

4. Shared GPU Pass and Resource Map

SceneEngine is the pass host: it owns one WebGPU device, one shared depth target, and extra sampleable targets used by water and atmosphere. This is the renderer-unification boundary that lets spheres, terrain, water, and atmosphere agree on depth.

5. Reusable Planet Renderer Pipeline

PlanetRenderer is a headless host around RenderBackend. It turns per-frame inputs into a RenderFrame, schedules transient patches, and delegates GPU work to the backend. The same host supports the standalone planet path and the body-relative /scene terrain path.

6. Data Ownership Boundaries

This diagram is useful for code review because it separates persisted authoring data from transient runtime state and derived frame data. Most bugs in this app come from crossing these boundaries accidentally.

7. Procedural Graph Stack

The graph stack is an adjacent architecture track: it is intended to become the typed procedural language for terrain, materials, vegetation, resources, and future WebGPU pipelines. Today it lives mostly in packages/ and the standalone graph editor.

Current relationship to the renderer
  • The live renderer still uses hand-authored WGSL modules under fe/src/lib/planet/gpu/wgsl.
  • The graph stack is additive: it should first prove typed IR, compiler, and runtime slices before replacing the planet kernel.
  • The eventual migration target is one graph feeding multiple consumers: terrain, material, collision, water, vegetation, and previews.

8. Integration Gates

The wave model deserves its own small diagram because it is an architectural control, not just a process note. The contracts in params/, patches/, and render/ should advance one wave at a time.
