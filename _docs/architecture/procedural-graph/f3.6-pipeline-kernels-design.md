· **Depends on:** F3.1 (`4c28431`), F3.2 (`0c4b7d8`), F3.3 (`2dc6009`), F3.4 (`0b39700`), F3.5

## Why this needs its own design pass, not a follow-on contract

Every prior Foundation 3 milestone extended or consumed F3.1's model: a primitive's resource
bindings (`KernelBindingTemplate[]`) are declared **once, statically, at registration time** — a
fixed signature, like a function. F3.4's own pre-drafting research confirmed this model does not
fit the real render pipeline: `stage.vertex` and `stage.fragment` are generic primitives reused by
*every* pipeline graph, and each document wires a *different* set of host uniforms, texture
channels, and params into them. A primitive's declared bindings can't vary per document under
F3.1's model — that's not a missing feature, it's a structural mismatch, confirmed by direct read
of `packages/graph/src/registry.ts:79-81`, `:37-48`, `:66-78` (a `{kind:'kernel'}` primitive can't
even carry a compatibility `wgsl` field, let alone a per-document binding set).

So F3.6 is not "F3.4 but for `stage.fragment` too." It's six separate, only loosely coupled
problems, each requiring its own design decision — the last two (E, F) were added after a
pre-drafting review found the first four alone would let an implementer satisfy the prose while the
old hardcoded paths kept running underneath:

- **(A) Dynamic binding sets** — a kernel whose bindings vary per document, not fixed at
  registration.
- **(B) Real varying dataflow** — F3.2 built the WGSL-assembly capability (`VaryingDecl`,
  `varyingsStructWgsl`, `assertVaryingsMatch`); nothing produces or consumes real graph-computed
  varying data yet.
- **(C) Presentation-target compatibility** — `target.display`'s only mechanism for finding "the
  fragment stage" is a hardcoded primitive-id string check.
- **(D) What replaces `stage.vertex`/`stage.fragment`'s hardcodes** — if anything.
- **(E) Pipeline-stage-aware assembly and execution** — once a stage node is discovered (C) and a
  binding set derived (A), does the actual execution path branch on that, or silently keep running
  the old hardcoded ShaderToy/fixed-grid assembly regardless?
- **(F) Vertex kernel invocation model** — what can a graph-authored vertex kernel actually read
  (`vertex_index`, `instance_index`, geometry-source sampling)? Undefined today, and a prerequisite
  for (B) to mean anything on the vertex side.

This document investigates each precisely, lays out real options with tradeoffs, and recommends a
direction — but does not commit to a final contract. That's deliberate: this is exactly the kind of
decision the standing instruction to request "a design discussion, not a contract" exists for.

## (A) Dynamic binding sets

**The gap, precisely.** `KernelBindingTemplate[]` lives on `PrimitiveImplementation`'s
`{kind:'kernel'}` variant (`packages/graph/src/implementation.ts:88`) — one list per *primitive*,
validated once at `registerPrimitive` time (`registry.ts:79-81`). `Node` itself
(`packages/graph/src/types.ts:79-88`) has no field that could carry a per-*instance* binding set:

```ts
export interface Node {
	id: string;
	primitive: string;
	name?: string;
	params?: Record<string, unknown>;
	inputs: Port[];
	outputs: Port[];
	position?: { x: number; y: number };
}
```

No `metadata` bag, no arbitrary-extra-fields escape hatch (unlike `NodePrimitive`, which has
`metadata?: PrimitiveMetadata` at the *registration* level). "A node instance declares its own
`KernelBindingTemplate[]` distinct from what its primitive registered" is not structurally
expressible today in any form — confirmed by direct read, not assumed.

**Option A1 — author bindings directly, per node instance.** Add `Node.bindings?:
KernelBindingTemplate[]` (or similar), an instance-level override/addition to whatever the
primitive declares. Resolution functions (`resolveKernelBindings`, `buildKernelBindingDecls`) would
source their `templates` from the node when present, falling back to the primitive's own.

- *Pros:* reuses F3.1's exact type/validation machinery almost unchanged — only the *source* of the
  template list changes.
- *Cons:* touches `Node` itself, a type used everywhere (layout persistence, editor UI, validation,
  every existing sample's document shape). "Author a binding list on a node" is a genuinely new
  authoring concept with no existing editor UI — would need new UI, not just a type change. Real
  documents don't naturally want to hand-declare "texture channel 0, texture channel 1" as
  structured data; they want to *wire a texture into a channel input* and have the binding fall out
  of that.

**Option A2 — derive the binding set from graph structure, don't author it.** For a
`stage.fragment`-shaped kernel node, walk its upstream dependency graph (the same `upstreamNodeIds`
traversal `emitGraphEval` already performs) and compute the binding set from what's actually
present: `input.channel` nodes → texture+sampler bindings, `host.iResolution`/`host.iTime` usage →
uniform fields, promotable params → uniform struct fields. This is not a new idea — it's exactly
what `assembleFullscreenFragmentModuleAsync` already does **by hand, today**
(`fullscreenFragment.ts`'s `usesShaderToyHost`/`emitted.usedChannels`/`hasGraphParams` — all
computed dynamically per call). F3.6 would generalize that one-off inline logic into a reusable,
kernel-model-shaped function instead of leaving it ShaderToy-fragment-specific.

- *Pros:* zero new `Node` fields, zero new authoring UI — matches this codebase's own established
  idiom of *deriving* structural facts from a document rather than asking authors to redeclare them
  (`deriveBufferFeedbackTarget`, `deriveComputeBufferTarget`, `derivePipelinePresentations` are all
  this same shape: walk the document, don't ask for a parallel declaration).
- *Cons:* `KernelBindingTemplate`'s F3.1 shape (`resourceKind: 'buffer'|'texture'|'sampler'`, tied
  to `ResourceBinding`/`resourceId`/`ResourceRealizer`) doesn't map cleanly onto host-uniform reads
  — those are `HostBinding`-shaped (a simpler value-injection mechanism, no GPU resource
  realization involved), not resource bindings. A texture channel read fits F3.1's model directly;
  a `host.iTime` read does not without stretching the model.

**The real fork this surfaces:** should `HostBinding` (F1.3's value-injection mechanism, used by
`host.fragCoord`/`host.iResolution`/`host.iTime`/`input.channel` — the complete current list of six
host-input registrations, confirmed by grep) and `KernelBindingTemplate` (F3.1's resource-binding
mechanism) be unified into one model, or stay separate, coexisting mechanisms? Unifying is the more
"complete" answer but is a bigger, riskier lift touching a mechanism every existing sample already
depends on. Staying separate is smaller and lower-risk but leaves two binding vocabularies that a
kernel author has to reason about simultaneously.

**Recommendation:** Option A2 (derive, don't author), and **don't unify** `HostBinding`/
`KernelBindingTemplate` yet — keep host-value-injection as today's separate, simpler mechanism, and
scope F3.6's new binding-set derivation to true GPU resources (buffers/textures/samplers) that
actually need `ResourceRealizer` allocation. This is the "no premature abstraction" call: unifying
two mechanisms because they're *conceptually* related isn't justified until a real use case needs
one binding declared through both paths simultaneously — nothing does yet. Note this leaves
`HostBinding.stages`'s existing, pre-F3.1 gap (declared but never enforced — `emitGraphEval.ts`
never reads it) untouched; that's a separate, already-known issue, not created by this
recommendation.

## (B) Real varying dataflow

**The gap, precisely.** `emitGraphEval` (`packages/runtime-webgpu/src/emitGraphEval.ts:218-442`,
the shared function `emitGraphScalarEval`/`emitGraphVec4Eval`/`emitGraphVec3Eval` all wrap) has a
strict one-output contract: one `PortRef` in, one `resultExpr: string` out
(`emitGraphEval.ts:439`). There is no batch/multi-port entry point. `assembleStageEntry`'s vertex
branch (`packages/compiler/src/stageEntry.ts:218-237`, landed F3.2) *can* assemble a `vs_main` that
assigns `out.position` from one WGSL function and `out.<varyingName>` from N more — but every one of
those N+1 functions must already exist, fully built, before the call. Nothing produces more than
one such function per invocation today.

**Option B1 — call `emitGraphEval` once per output (position, plus once per varying).** Simple,
reuses existing machinery completely unchanged. The cost: if position and a varying share upstream
computation (e.g. both depend on the same expensive noise call), that computation gets emitted
*and evaluated* twice — once inside each independently-generated function. Not a correctness bug,
a redundant-work cost.

**Option B2 — a multi-output `emitGraphEval` variant.** Compute the *union* of upstream dependencies
across N requested output ports once, emit one shared `body`, and return N `resultExpr`s instead of
one. Avoids the redundant computation, but means restructuring a function every single existing
sample's codegen already depends on (`emitGraphScalarEval`/`emitGraphVec4Eval`/`emitGraphVec3Eval`
are thin wrappers around this one shared implementation) — real regression risk to the "every
existing sample stays pixel-identical" constraint every Foundation 3 milestone has held so far.

**Recommendation:** start with B1 for the first real vertex-displacement proof. It's low-risk,
proven, and the redundant-computation cost is real but not yet measured — treat B2 as a candidate
*optimization* for a later milestone once a real sample shows it's actually needed, not something
to build speculatively now on a theoretical concern.

## (C) Presentation-target compatibility

**The gap, precisely, and this one has a clean, low-risk fix.** `target.display`'s only mechanism
for identifying "the fragment stage" is `fieldOutputForDisplay`
(`packages/graph/src/pipeline.ts:29-39`):

```ts
function fieldOutputForDisplay(doc: GraphDocument, displayNode: Node): PortRef | null {
	const toDisplay = incomingEdge(doc, displayNode.id, 'color');
	if (!toDisplay) return null;
	const fragmentNode = doc.nodes.find((node) => node.id === toDisplay.from.node);
	if (!fragmentNode || fragmentNode.primitive !== 'stage.fragment') return null;
	// ...
}
```

A hardcoded `.primitive !== 'stage.fragment'` string check — no fallback, no role check. A
graph-authored kernel-based fragment stage under any other primitive id makes this return `null`
unconditionally, and every downstream consumer (`presentationForDisplay`,
`derivePipelinePresentations`, `effectiveOutputs`, `derivePipelineConsumers`) silently produces zero
presentations rather than erroring. The equivalent check exists three more times, identically
hardcoded — **not two, four sites total**, confirmed by direct read, an earlier draft of this
document undercounted them:

- `packages/runtime-webgpu/src/pipelineGraph.ts`, `resolveVertexForFragment` (~line 120):
  `vertex.primitive !== 'stage.vertex'`.
- `packages/runtime-webgpu/src/pipelineGraph.ts`, inside `planPipelineGraph` itself (~line 230):
  `fragment.primitive !== 'stage.fragment'`.
- `packages/runtime-webgpu/src/pipelineGraph.ts:188-190`, inside `resolvePipelineTarget`'s
  **legacy fallback branch** (reached only when `derivePipelinePresentations(doc)` finds nothing —
  i.e. when role-based discovery itself comes up empty):
  ```ts
  const display = discoverExecutionRoots(doc).find(
  	(node) => node.primitive === 'target.display'
  );
  ```
  This one is lower-stakes than the other three (it's already a fallback of a fallback), but it's
  the same kind of gap and deserves the same fix rather than being quietly left as a permanent
  legacy-only exception with no stated reason.

The *discovery mechanism* fix these four call sites need already has a proven precedent **in the
same file**: `isPipelineTarget` (`pipeline.ts:189-192`) already discovers `target.display` nodes
structurally, by `metadata.role`, not by id:

```ts
export function isPipelineTarget(node: Node): boolean {
	const primitive = getPrimitive(node.primitive);
	return primitive?.metadata?.role === PIPELINE_TARGET_ROLE;
}
```

**Correction to an earlier draft of this recommendation.** The original version of this section said
to "define two new role constants... have `stage.vertex`/`stage.fragment` additionally declare
them." That's imprecise in a way that matters: `PrimitiveMetadata.role` is a **single, optional
string** (`packages/graph/src/primitive.ts:19`, `role?: string` — not an array), and
`stage.vertex`/`stage.fragment` **already both declare `role: 'pipelineStage'`**
(`packages/graph/src/primitives/pipeline/index.ts:39,56`) — the *same* role, for *both* primitives.
That role is not free to repurpose: it's the exact key `swapFamily()`
(`packages/graph/src/contract.ts:31`) groups primitives by for the editor's node-palette/accent-
color UI (`packages/graph-editor/src/nodeAccentColor.ts`, `nodePaletteModel.ts`) — a *different*
concern (UI swap-grouping) from presentation-target discovery. A primitive can't declare three
roles simultaneously under the current type, and silently replacing `'pipelineStage'` with
something new would change `swapFamily()`'s grouping too — `stage.vertex`/`stage.fragment` would
stop being each other's swap family (arguably more *correct*, since their port signatures are
completely different and swapping between them today would already fail port-compatibility checks,
but that's a UI-behavior change this document hasn't investigated and shouldn't cause as a side
effect of a discovery-mechanism fix).

**Revised recommendation:** don't reuse or repurpose `metadata.role` for this. Add a new,
purpose-specific optional field to `PrimitiveMetadata` instead — e.g. `pipelineStageKind?: 'vertex'
| 'fragment'` — leaving `role: 'pipelineStage'` and `swapFamily()`'s existing grouping completely
untouched. `stage.vertex` declares `pipelineStageKind: 'vertex'`, `stage.fragment` declares
`pipelineStageKind: 'fragment'`, and any new graph-authored kernel-based stage primitive (D2)
declares the matching one.

**Correction: not all four sites take the same fix.** `pipelineStageKind` only applies to the
**three checks that identify a vertex- or fragment-*stage* node** — `pipeline.ts:34`,
`pipelineGraph.ts`'s `resolveVertexForFragment` (~line 120), and `planPipelineGraph`'s own
`fragment.primitive !== 'stage.fragment'` check (~line 230). The fourth site
(`pipelineGraph.ts:188-190`) checks `node.primitive === 'target.display'` — a *target/sink* node,
not a stage node at all, and `pipelineStageKind` would never match it (a display node is neither a
vertex nor a fragment stage). That site already has its own correct, proven, role-based fix
sitting one function away: `isPipelineTarget(node)` (`pipeline.ts:189-192`,
`metadata.role === 'pipelineTarget'`) is exactly what `resolvePipelineTarget`'s primary path
already uses (`derivePipelinePresentations`, line 175, is built on it) — the legacy fallback at
line 188 should call the same `isPipelineTarget` predicate instead of re-deriving its own
`node.primitive === 'target.display'` check, not grow a `pipelineStageKind` check it could never
satisfy.

So: three call sites move to `getPrimitive(node.primitive)?.metadata?.pipelineStageKind`; the
fourth moves to `isPipelineTarget(node)` (a call that already exists and is already correct,
just not reused at this one remaining site). Purely additive either way — zero behavior change to
any existing sample, and zero interaction with `swapFamily`/the node palette.

**A second, separate contract these role/kind fixes don't touch: port names.** Role-based discovery
answers "is this node a vertex/fragment/target stage," but `pipeline.ts` and `pipelineGraph.ts`
*also* hardcode the **port names** wired between stages, independent of the primitive-id/role
checks — confirmed by direct read, e.g. `planPipelineGraph`'s `requireEdge` calls
(`pipelineGraph.ts:234-241`):
```ts
requireEdge(doc, { node: geometry.id, port: 'mesh' }, { node: persist.id, port: 'in' });
requireEdge(doc, { node: persist.id, port: 'out' }, { node: vertex.id, port: 'mesh' });
requireEdge(doc, { node: vertex.id, port: 'varyings' }, { node: fragment.id, port: 'varyings' });
requireEdge(doc, { node: fragment.id, port: 'texture' }, { node: target.displayTargetNode, port: 'color' });
requireEdge(doc, target.fieldOutput, { node: fragment.id, port: 'color' });
```
and `pipeline.ts`'s `fieldOutputForDisplay`/`resolveVertexForFragment`, which check
`toDisplay.from.port !== 'texture'` and `incoming(doc, fragmentStageNode, 'varyings')` respectively.
**A new graph-authored kernel-based vertex/fragment primitive must declare its ports named
`mesh`/`varyings` (vertex) and `varyings`/`color`/`texture` (fragment) — the exact same ids
`stage.vertex`/`stage.fragment` already use — or `planPipelineGraph` throws "missing X edge" and
`fieldOutputForDisplay` returns `null`, even with `pipelineStageKind` declared correctly.** This
document does not resolve whether that's an acceptable fixed contract to inherit (it's a small,
stable vocabulary — five port names total) or something a later milestone should also generalize;
it only makes explicit that (C)'s fix alone does not remove this second, port-name-shaped
constraint, so nobody discovers it by surprise mid-implementation.

## (D) What replaces `stage.vertex`/`stage.fragment`'s hardcodes

The genuinely open question — no prior document in `_docs/architecture/procedural-graph/` proposes
a concrete answer (checked `elemental-webgpu-architecture-review.md`, `wgsl-parsing-and-codegen.md`,
`pipeline-as-graph.md`, `node-model-design-notes.md`, `pipeline-realignment-report.md`; the plan
document itself says plainly: "deliberately left as a named, not-yet-designed follow-on rather than
guessed at"). Worth noting: `pipeline-as-graph.md:66-118` sketched the *original* intended shape
years before F3.1 existed — `stage.*` nodes each carrying "an embedded field subgraph... for its
shading math" — which is closer in spirit to what F3.6 is now solving toward than anything in
between. That's context, not a ready answer.

**Option D1 — replace `stage.vertex`/`stage.fragment` in place.** Change what these exact primitive
ids *do*, converting them from today's fixed-grid/ShaderToy-assembly behavior into real
graph-driven kernels. Every existing sample uses these exact ids — this is only safe if the new
behavior is proven byte-identical for the "trivial" case (no extra varyings, no extra bindings
beyond what's already handled). High-stakes: this is close to what the *original*, pre-rescope F3.4
plan attempted before research found it didn't fit F3.1's model at all.

**Option D2 — add new primitives alongside, leave the old ones frozen.** New, separately-named
graph-driven vertex/fragment kernel primitives (exact naming TBD at contract time) coexist with
`stage.vertex`/`stage.fragment`, which stay exactly as they are — same fixed-grid path, zero
behavior change, matching the "purely additive" discipline F3.4 and F3.5 both used successfully.
Existing samples keep using the old primitives unmodified forever, or until a *separate*, later,
evidence-backed migration decision retires them — not decided here.

**Recommendation:** D2, clearly. This isn't a close call: every milestone in this Foundation that
tried to be purely additive (F3.1, F3.3, F3.4, F3.5) shipped cleanly on the first or second review
round; the one time this exact area was scoped as "replace in place" (F3.4's original framing), it
had to be rescoped down before any code was written, precisely because of this risk. D2 defers the
"should `stage.vertex`/`stage.fragment` ever actually be retired" question to a later decision made
with a working alternative already in hand — which is a strictly better position to decide from
than now.

## (E) Pipeline-stage-aware assembly and execution

**Added after a pre-drafting review of this document found (A)-(D) alone would let an implementer
satisfy a new-fragment-kernel milestone's prose — role discovery lands, a new fragment-kernel
primitive gets registered — while the real render path keeps running the old, hardcoded ShaderToy
assembly regardless.** This
is not a variation on (C); (C) is about *discovering which node is the fragment stage*, this is
about *whether anything downstream actually does something different once it knows*. Both are
required; neither implies the other.

**The gap, precisely.** `planPipelineGraph` (`packages/runtime-webgpu/src/pipelineGraph.ts:210-253`)
already computes `vertexStageNode`/`fragmentStageNode` and returns them as part of its
`PipelineGraphPlan`. But `PipelineGraphExecutor.execute` (`pipelineGraph.ts:258-274`) discards that
identity immediately — it calls `executeFullscreenFragment` with only
`output: plan.fieldOutput` (plus device/graph/resolver/width/height/host/target/channelTargets),
never `plan.fragmentStageNode` or `plan.vertexStageNode`:

```ts
async execute(input: PipelineGraphInput): Promise<FullscreenFragmentResult> {
	const plan = planPipelineGraph(input.graph, input.output ? { output: input.output } : {});
	this.cache.realizeGeometry(geometryCacheFingerprint(input.graph, plan));
	return executeFullscreenFragment({
		device: input.device,
		graph: input.graph,
		output: plan.fieldOutput,
		resolver: input.resolver,
		width: input.width,
		height: input.height,
		host: input.host,
		target: input.target,
		channelTargets: input.channelTargets
	});
}
```

`executeFullscreenFragment`/`assembleFullscreenFragmentModuleAsync`
(`packages/runtime-webgpu/src/consumers/fullscreenFragment.ts:124-218`) then unconditionally builds
a ShaderToy-style fragment module from the field output alone — hand-built `BindingDecl[]` for
uniform/params/channels, `emitGraphVec4Eval`, `assembleStageEntry`'s fragment branch with no
varyings — completely independent of whether the upstream stage node is `stage.fragment` or a new
kernel primitive. **Even with (C)'s role-based discovery landed and a new fragment-kernel primitive
registered, this execution path has no branch point that would ever choose differently.**

The same disconnect applies to `assemblePipelineVertexWgsl` (`pipelineVertex.ts`) on the vertex
side — it's driven entirely by `resolvePipelineGeometryParams` reading numeric grid params, with no
path for `plan.vertexStageNode`'s actual primitive/implementation to influence it either.

**What the fragment-kernel milestone (and any vertex-kernel milestone) actually needs, stated as a
requirement, not left implicit:** `plan.fragmentStageNode`/`plan.vertexStageNode` must be threaded
from `planPipelineGraph`'s return value through `PipelineGraphExecutor.execute` into whatever
assembles the module, and that assembly step must branch on the stage node's `pipelineStageKind`
((C)'s new metadata field) plus its actual `PrimitiveImplementation` — today's fixed
ShaderToy/fixed-grid assembly for a `stage.fragment`/`stage.vertex` node (unchanged, per D2), a new
kernel-driven assembly path for a new kernel-based stage primitive. Without this, discovery (C) and
a new primitive with derived bindings (A) could both land, individually correct, and the feature
still wouldn't work end-to-end — exactly the kind of gap F3.5's own manual visual-gate check caught
once already (`GraphEditor.svelte`'s separate, un-updated early-exit guard), just one layer deeper
in the stack this time, and not catchable by a visual check alone since nothing would even attempt
the new path.

**The binding-set half of this same gap.** (A)'s recommendation (derive bindings from graph
structure) describes *what WGSL binding declarations to emit*, but stops before *how those bindings
connect to real GPU resources at runtime* — and today there are **two separate, parallel mechanisms
doing that connection**, confirmed by direct read:

- `GraphFrameExecutor.execute` (`packages/runtime-webgpu/src/graphFrameExecutor.ts:114-120`) builds
  a `channelTargets: Map<number, GPUTexture>` from `resolveChannelDependencies`'s cross-*pass*
  analysis (which pass reads which other pass's rendered texture), resolved through
  `ResourceRealizer.resolve(...)` — this is pass-graph-level, document-wide.
- `assembleFullscreenFragmentModuleAsync` separately derives its *own* `channelBindings` map
  (`fullscreenFragment.ts:244-250`) from what the fragment's own WGSL slice references
  (`emitted.usedChannels`), and cross-checks it against the caller-supplied `channelTargets` —
  entirely disconnected from F3.1's `KernelBindingTemplate`/`resolveKernelBindings`/
  `buildComputeBindGroupEntries` chain that F3.3/F3.4 already built (and proved) for *compute*
  kernels.

The fragment-kernel milestone needs to decide, explicitly, whether a new fragment-kernel's derived
bindings route through F3.1's existing resolution chain (giving `buildComputeBindGroupEntries`-
shaped machinery its first real *render*-pipeline consumer, not just compute) or continue as a
third parallel mechanism matching `channelTargets`'s existing shape. This document doesn't resolve
which — both are plausible, and the right answer likely depends on how much that milestone's actual
binding-set derivation ends up looking like F3.1's model once it's built — but it must be answered
by whoever writes that milestone's own contract, not left implicit the way the original draft of
this document left it.

## (F) Vertex kernel invocation model

**Added after the same pre-drafting review**, which found (B)'s "call `emitGraphEval` once per
output" plan doesn't yet account for what a graph-authored *vertex* kernel can actually read as
input — a real gap distinct from (B)'s multi-output question, and one that has to be resolved
before the vertex-kernel milestone is scoped, not discovered partway through implementing it.

**The gap, precisely.** `assembleStageEntry`'s vertex template already exposes
`@builtin(vertex_index) vid: u32, @builtin(instance_index) iid: u32` and passes them to generated
functions via `opts.callArgs` (`packages/compiler/src/stageEntry.ts:218-237`, landed F3.2) — the
*assembly* layer is ready to receive vertex-invocation values. But `emitGraphEval`'s host-input
vocabulary (`emitHostInput`, `packages/runtime-webgpu/src/emitGraphEval.ts:79-119`) is entirely
fragment/field-oriented, confirmed by reading every branch: `uv` → procedural `vec2<f32>(u, v)`,
`metricPosition` → `positionExpr` (defaults to a 2D-parametric expression), `fragCoord` →
`position.xy` (a **fragment** builtin, not available in a vertex shader at all), `iResolution`,
`iTime`, `channel` → `textureSample(...)` keyed on `position.xy` (again fragment-only). There is
**no defined contract** for:

- Reading `vertex_index`/`instance_index` as a graph value (a host-input primitive analogous to
  `host.fragCoord` but for `vid`/`iid` doesn't exist).
- Reading the current vertex's mesh/geometry attributes — a graph-authored vertex kernel presumably
  wants to read *this vertex's grid position* to displace it, but nothing today exposes
  `geometry.plane`/`buffer.persist`'s per-vertex output as a graph value; `plane_grid_position(...)`
  is called directly inside `assemblePipelineVertexWgsl`'s hand-written template, never through
  `emitGraphEval`.
- "Grid UV" as a vertex-stage concept distinct from the fragment-stage procedural UV `uv` already
  means today.

**This is genuinely open and needs its own resolution, not a one-line fix like (C)'s.** At minimum,
whoever scopes the vertex-kernel milestone needs to answer: what new `HostBinding` context/key
pairs (or new mechanism entirely, if `HostBinding`'s vocabulary doesn't fit — c.f. (A)'s
already-noted `HostBinding`/
`KernelBindingTemplate` fit question) represent `vertex_index`/`instance_index` and geometry-source
sampling; whether a vertex kernel reads geometry the same way a fragment kernel reads a texture
channel (structurally similar — both are "sample an upstream resource at the current invocation"),
or needs its own distinct primitive family; and how `resolvePipelineGeometryParams`'s existing
numeric grid params (`resU`/`resV`/`width`/`height`/rotation) coexist with a graph-authored
displacement on top of that same grid, versus being fully superseded by it.

## Tentative sequencing

Not a commitment — a starting shape for whoever scopes the first F3.6 contract to accept, adjust, or
discard. **Revised after review: F3.6.2/F3.6.3 as originally sketched here were closer to
"implementation-ready" in prose than in substance** — an implementer could satisfy (A)'s and (B)'s
text while the old hardcoded fullscreen-fragment path kept running underneath, or while a vertex
kernel had no defined inputs to actually read. (E) and (F) are now explicit, required design pieces
placed *before* the milestones that depend on them, not left implicit inside F3.6.2/F3.6.3's own
prose:

1. **F3.6.1 — role-based pipeline-stage discovery.** Sub-problem (C)'s fix: a new
   `PrimitiveMetadata.pipelineStageKind?: 'vertex' | 'fragment'` field (not a reused/repurposed
   `role` — see (C)'s correction). Four call sites total, two different fixes: **three** stage
   checks (`pipeline.ts:34`, `pipelineGraph.ts`'s `resolveVertexForFragment`, and
   `planPipelineGraph`'s own `fragment.primitive` check) move to `pipelineStageKind`; the fourth
   (`resolvePipelineTarget`'s legacy `target.display` fallback) moves to `isPipelineTarget` instead
   — it identifies a target/sink node, not a stage node, so `pipelineStageKind` would never match
   it. `stage.vertex`/`stage.fragment` additionally declare the new field. Zero behavior change to
   any existing sample, zero interaction with `swapFamily`/the node palette — the lowest-risk
   possible first slice.
2. **F3.6.2 — pipeline-stage-aware assembly and execution wiring.** Sub-problem (E), in full: thread
   `plan.fragmentStageNode`/`plan.vertexStageNode` from `planPipelineGraph` through
   `PipelineGraphExecutor.execute` into whatever assembles the module, branch on
   `pipelineStageKind` + `PrimitiveImplementation`, and decide the binding-set resource-routing
   question (F3.1's `resolveKernelBindings` chain vs. a `channelTargets`-shaped parallel mechanism).
   The real derived-binding-set fragment kernel (F3.6.3) doesn't need to exist yet, but the new
   branch still needs something registered to prove it's actually reachable — this milestone's own
   contract should register a small **test-only fixture primitive** (`pipelineStageKind:
   'fragment'`, a `{kind:'kernel'}` implementation, correct port names per (C)'s port-name-contract
   note) and assert the assembly path routes to the new branch for a document using it, and to the
   unchanged ShaderToy path for a document using plain `stage.fragment` — not a vaguer "no-op/
   pass-through branch" with nothing exercising it, which wouldn't be meaningfully testable. This
   reordering (wiring before the real new primitive, not after) is the direct fix for the
   "prose-satisfied but not actually wired" risk this review found.
3. **F3.6.3 — a new, additive fragment-kernel primitive with a derived binding set.** Sub-problem
   (A) per the A2 recommendation, now landing on top of F3.6.2's real branch point: generalize
   `fullscreenFragment.ts`'s existing `usesShaderToyHost`/`usedChannels`/`hasGraphParams` inline
   logic into a reusable binding-set derivation function, applied to a genuinely new primitive (not
   `stage.fragment` itself, per D2).
4. **F3.6.4 — vertex kernel invocation model.** Sub-problem (F) resolved into a concrete contract:
   whatever new `HostBinding`s or mechanism represents `vertex_index`/`instance_index` and
   geometry-source sampling, decided and built *before* F3.6.5 needs to consume it — not discovered
   partway through building the vertex kernel itself.
5. **F3.6.5 — a new, additive vertex-kernel primitive with real varying output.** Sub-problem (B)
   per the B1 recommendation, now landing on top of F3.6.4's invocation model: one `emitGraphEval`
   call per output (position + each varying), wired through F3.2's existing `assembleStageEntry`
   vertex-branch/`varyingsStructWgsl` machinery unchanged. Paired with F3.6.3's fragment kernel via
   `assertVaryingsMatch` (F3.2, already built, still unused in any real path — this is its first
   real caller).
6. **F3.6.6 — proof.** A bundled, pickable sample rendering real graph-authored vertex
   displacement — mirroring F2.5's and F3.5's own proof-milestone shape and visual-gate convention.
   This is the sample F3.5's own Handoff deferred here.

Each of F3.6.1-6 would need its own full contract (file list, exact signatures, gate-as-failing-
tests), written immediately before it's routed — exactly like every F3.1-F3.5 contract was, and
exactly like this document does *not* attempt to be. F3.6.2 and F3.6.4 in particular should be
scoped and routed as their own explicit contracts, not folded into F3.6.3's/F3.6.5's prose the way
an earlier draft of this sequencing implied — that's precisely the gap this review round found.

## Open questions this document deliberately does not resolve

- **Unify `HostBinding` and `KernelBindingTemplate`, or keep them separate?** Recommended: keep
  separate for now (see (A)) — but this is a real fork, not a settled question, and revisiting it
  once F3.6.3/F3.6.5 land real usage may change the calculus. (F)'s vertex-invocation question may
  also push on this fork independently, if geometry-source sampling turns out to fit one model
  better than the other.
- **Does a new fragment-kernel's derived binding set route through F3.1's `resolveKernelBindings`
  chain, or stay a `channelTargets`-shaped parallel mechanism?** (E) states the question precisely;
  does not answer it.
- **Exact naming for the new vertex/fragment kernel primitives** (D2) — not decided here; a contract-
  time decision.
- **Whether/when to ever retire `stage.vertex`/`stage.fragment`** — explicitly not this document's
  question; D2 defers it.
- **Whether F3.6.3's binding-set derivation should also cover compute kernels' bindings going
  forward** (today's `target.computeBuffer` still uses F3.1's static model, which is correct for a
  purpose-built, fixed-signature kernel) — not addressed; F3.4/F3.5's compute path is not broken and
  doesn't need this.
- **Does a graph-authored vertex kernel read geometry the same way a fragment kernel reads a
  texture channel, or does it need its own distinct primitive family?** (F) raises this; does not
  answer it.

## What this document deliberately does not do

Write a contract — no Fix steps, no Gate items, nothing here should be routed to an implementer
as-is; each F3.6.N milestone gets its own contract at the point it's actually scoped and routed.
Decide the `HostBinding`/`KernelBindingTemplate` unification question. Touch Foundation 4's command
graph (`{kind:'command'}` stays exactly as inert as every prior Foundation 3 milestone left it).
Revisit storage textures, multiple bind groups, or any other deferral already made and re-confirmed
across F3.1-F3.5 — nothing here provides new reason to un-defer them. Estimate calendar time.
