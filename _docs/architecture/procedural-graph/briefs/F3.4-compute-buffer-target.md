## Context

Fourth of Foundation 3's sequence (F3.1 → F3.2 → F3.3 → **F3.4** → F3.5 → F3.6).

**This is not the milestone originally described in the plan document.** The plan's original F3.4
text said F3.4 would "replace `stage.vertex`'s `wgsl-function` hardcode and `stage.fragment`'s
`legacy-structural` marker with real, graph-authored kernel primitives." Pre-drafting research
found that framing doesn't fit F3.1's design and the plan document has been corrected accordingly
(see its F3.4/F3.5/F3.6 entries): `stage.vertex`/`stage.fragment` are generic primitives reused by
every pipeline graph, whose actual binding set (which host uniforms, how many texture channels) is
discovered dynamically per document — but `KernelBindingTemplate[]` is validated *statically*, once,
at primitive registration (`registerPrimitive`/`normalizeImplementation`,
`packages/graph/src/registry.ts:79-81`). A primitive's declared bindings can't vary per document.
Converting `stage.fragment` itself into a `{kind:'kernel'}` primitive is a structural mismatch, not
a missing feature.

**What F3.4 actually does instead**: proves the full F3.1→F3.3 chain end-to-end with real pixels,
by registering one new, self-contained, purpose-built compute-kernel primitive — `target.computeBuffer`
— whose binding set genuinely *is* fixed (one buffer, known at registration time, exactly the shape
F3.1's model fits), and wiring it into `GraphFrameExecutor` so a document containing this node
actually dispatches a real `GPUComputePipeline` against a real, `ResourceRealizer`-managed buffer
every frame. This is **purely additive**: it does not touch `stage.vertex`, `stage.fragment`,
`pipelineVertex.ts`, `fullscreenFragment.ts`, `pipeline.ts`'s presentation-derivation logic, or any
existing sample graph. The ShaderToy fullscreen case (`cosinePaletteEffectGraph`), F2.5's
cross-pass-texture sample (`crossPassTextureReadGraph`), and F2.5's buffer-feedback sample
(`bufferFeedbackGraph`) all stay on their exact current code paths — zero regression risk, because
nothing they depend on is modified.

The harder problem — a kernel whose binding set varies per document, typed varyings actually
carrying graph-computed data from a vertex kernel to a fragment kernel, and what (if anything)
should eventually replace `stage.vertex`/`stage.fragment` — is deferred to **F3.6**, a distinct,
not-yet-designed problem (see the plan document). F3.4 does not attempt it and does not pretend to
be a step toward it beyond "the resource-realization/dispatch machinery it builds will likely be
reusable there too."

This milestone needs a **bundled sample and a manual/visual gate**: unlike F3.1-F3.3, real pixels
now flow through `GraphFrameExecutor` for the first time via graph-authored kernel dispatch. Per
the project's standing instruction, Fix step 12 includes a bundled, pickable sample graph
demonstrating `target.computeBuffer` — and Fix steps 6-9 are what make that sample actually
dispatch and become visible in the editor, not just exist as a document (see Round 2 below). (F3.5,
per the corrected plan, was going to be *only* this proof — folding the sample directly into F3.4
here means F3.5 can be requested as a smaller polish/hardening pass, or skipped if this milestone's
own sample already satisfies the standing
instruction; that call is left to whoever reviews this brief before routing.)

**Round 2 (pre-routing review, this revision):** the first draft's last Fix step ("add a bundled
sample") turned out to be unbuildable as written — the runtime/graph core (this revision's Fix
steps 1-5) was correct, but the sample cannot actually run or become visible in the editor without
three more
changes the first draft never scoped:

1. **Blocking — the editor's preview loop never calls `GraphFrameExecutor.execute` for a
   compute-buffer-only graph.** `createPreviewFrameLoop`'s early-exit guard
   (`packages/graph-editor/src/previewFrameLoop.ts:64-70`) only proceeds past `'No pipeline preview
   targets'` when `planIndependentGraphFramePasses(...).length > 0` **or**
   `deriveBufferFeedbackTarget(...)` is non-null — `deriveComputeBufferTarget` is never consulted. A
   graph containing only a `target.computeBuffer` node has zero passes and no buffer-feedback
   target, so the loop returns before ever calling `execute`. Fixed: the guard now also checks
   `deriveComputeBufferTarget` (Fix step 5a).
2. **Blocking — even if dispatched, the result has nowhere to go.** `PreviewFrameSnapshot`
   (`previewFrameLoop.ts:11-18`) has exactly one data field, `targets: Readonly<Record<string,
   Uint8Array>>`, and the result-copy code (`previewFrameLoop.ts:97-113`) copies only
   `result.targets`/`result.width`/`result.height` — nothing else `GraphFrameExecuteResult` might
   carry reaches the snapshot. The first draft's `computeBuffers?: Record<string, Float32Array>`
   addition to `GraphFrameExecuteResult` (unlike buffer-feedback, which reuses the *same* `targets`
   map because it's pixel-shaped like everything else in it) is a structurally new, differently-typed
   field with no existing path to the UI. Fixed: `PreviewFrameSnapshot` gains its own
   `computeBuffers?: Readonly<Record<string, Float32Array>>` field, copied through explicitly
   (Fix step 5b).
3. **Blocking — no existing preview panel can display a raw `Float32Array`.** Confirmed by direct
   read of both realistic candidates: `CpuPreviewPanel.svelte` is a per-pixel `PortRef` scalar
   evaluator (`evaluateGraphOutput` from `@world-lab/runtime-cpu`) — it never talks to
   `GraphFrameExecutor`/the preview loop at all, wrong data source entirely.
   `EffectPreviewPanel.svelte` does talk to the preview loop, but only reads
   `snapshot.targets[targetId]` and blits it as RGBA8 pixel data (`blitPreviewPixels`,
   `previewFrameLoop.ts:157-170`) — structurally incompatible with a flat scalar buffer (no
   width/height framing, and `ImageData.data.set(floatArray)` would silently reinterpret float
   values as clamped bytes, not a real rendering). `enumeratePreviewBuffers`
   (`previewBuffers.ts:168`) also has no discovery block for `target.computeBuffer` at all today —
   it wouldn't even appear as a selectable buffer. Fixed: the preview-loop guard, a new
   `enumeratePreviewBuffers` block, a new `PreviewFamily`/`PreviewRenderer` value (`'buffer'`), and
   a new, deliberately minimal panel (Fix steps 6-9).

The runtime/graph core from Round 1 (role-based discovery mirroring `target.bufferFeedback`, the
static binding set fitting F3.1, a dedicated `ResourceRealizer` instance for persistent buffer
state, `stage.vertex`/`stage.fragment` left untouched) was confirmed correct and is unchanged below.

## Verified current state

- **`normalizeImplementation` genuinely cannot accept a `{kind:'kernel'}` primitive with a
  compatibility `wgsl` field** (`packages/graph/src/registry.ts:37-48,66-78`, confirmed by direct
  read) — `implementationWgsl` only derives a `wgsl` field for `'wgsl-function'`/`'group'` kinds; for
  any other kind (including `'kernel'`) `derivedWgsl` is `undefined`, and line 67-70 throws
  `"Non-callable primitive implementation cannot declare wgsl: ..."` if `input.wgsl` is supplied
  anyway. This means a kernel primitive's WGSL **body** cannot be resolved through the same
  `moduleId`/`WgslModuleResolver` path `stage.vertex` uses — confirmed this is not an oversight to
  route around cleverly, but a real, load-bearing exclusion. `target.computeBuffer`'s WGSL body is
  therefore a hand-written template string living in `runtime-webgpu`, matching the *majority*
  precedent already established by `planeScalarPreview.ts`/`meshGen.ts`/`vegetationCandidates.ts`
  (all three hand-inline their compute WGSL directly, not via the module resolver — only
  `stage.vertex`'s fixed grid position function uses the resolver, and it's the minority case).
  `compileGraph`/`sliceGraph`/`generateWgsl` are **not** touched by this milestone — no attempt is
  made to route a kernel's body through them (that door stays closed until F3.6, if ever).
- **`{kind:'resource'}` (`packages/graph/src/implementation.ts:87`) has zero registered primitives
  using it** — confirmed by grep, same situation `{kind:'kernel'}` was in before F3.1. This milestone
  does **not** give it a first consumer either: `target.computeBuffer` is self-contained (no inputs),
  so its buffer's `resourceId` is synthesized directly from its own node id, not derived from an
  upstream `{kind:'resource'}` node. Registering a real, wireable resource-declaring primitive stays
  open for later work.
- **`target.bufferFeedback`'s exact discovery pattern is the direct precedent this milestone
  mirrors** (`packages/graph/src/bufferFeedbackTarget.ts`, full file, 61 lines): `isBufferFeedbackTarget(node)`
  checks `getPrimitive(node.primitive)?.metadata?.role === 'bufferFeedbackTarget'` — a **role-based**
  check via `NodePrimitive.metadata.role` (`packages/graph/src/primitive.ts:19`, a free-form
  string), not an id-string match. `deriveBufferFeedbackTarget(doc)` filters `doc.nodes` by that
  role, throws if more than one match, and returns a descriptor built entirely from the matched
  node's own `params` — no upstream/downstream edge-walking at all, because `target.bufferFeedback`
  has no input ports (`packages/graph/src/primitives/pipeline/index.ts:95-111`, confirmed: no
  `inputs` beyond the empty default). `GraphFrameExecutor.execute` calls this function directly
  (`packages/runtime-webgpu/src/graphFrameExecutor.ts:66`) — **independently of** `SinkDefinition`/
  `discoverExecutionRoots`/`sinkConsumers`'s generic invocation machinery (`target.bufferFeedback`
  additionally registers a `BUFFER_FEEDBACK_SINK_DEFINITION` for that generic machinery's benefit,
  but `GraphFrameExecutor`'s real per-frame decision never consults it). `target.computeBuffer`
  copies this exact shape: role-based discovery, no `SinkDefinition` needed at all (since
  `target.computeBuffer`'s `implementation.kind` must be `'kernel'`, not `'sink'` — one kind per
  primitive — so it is intentionally **invisible** to `discoverExecutionRoots`/`effectiveConsumers`,
  exactly as intentional as `target.bufferFeedback` bypassing that machinery for its own real
  dispatch decision).
- **`GraphFrameExecutor.execute`'s exact current structure**
  (`packages/runtime-webgpu/src/graphFrameExecutor.ts:43-141`, full file read): fields
  `pipeline = new PipelineGraphExecutor()`, `bufferFeedback = new BufferFeedbackExecutor()`, lazy
  `realizer`/`realizerDevice`. `execute(input)` (65-140): (1) `deriveBufferFeedbackTarget(input.graph)`
  (66) with a dimension-match check against the viewport (67-75); (2) builds a `PassGraph` and
  `buildPassOrder`s it (76-82); (3) loops `order.order`, calling `this.pipeline.execute(...)` per
  consumer (92-128); (4) `realizer.advanceFrame()` (130); (5) **if** `bufferTarget` was found, calls
  `this.bufferFeedback.execute(device, bufferTarget.gridWidth, bufferTarget.gridHeight)` (132-136),
  storing the result keyed by `bufferTarget.sinkNodeId` (137). Step (5) is the **exact integration
  point this milestone mirrors** — a third, additive, independent step of the same shape, gated on
  `deriveComputeBufferTarget(input.graph)` instead, calling a new `ComputeBufferExecutor.execute(...)`,
  keyed by the target's `nodeId`. Steps (1)-(4) are **not modified** by this milestone.
- **`BufferResourceTarget`'s exact shape** (`packages/runtime-webgpu/src/frameGraph/types.ts:11-16`):
  `{ id: string; shape: Extract<ResourceShape,{kind:'buffer'}>; lifetime: ResourceLifetime; size:
  {kind:'element-count', count: number} }`. `ResourceRealizer.resolve(targetId): RealizedTarget`
  returns `{ write: GPUBuffer | GPUTexture; readPrevious?: ... }` (`readPrevious` only populated for
  `lifetime: {kind:'history'}` targets — irrelevant here). `ResourceRealizer.realizeAll(graph,
  viewport)` computes `resolveBufferSizes(graph)`/`resolveTargetSizes(graph, viewport)` purely from
  `graph.targets` (confirmed by direct read, `frameGraph/order.ts:261-295`) — it does not read
  `graph.passes` or `graph.display` for allocation purposes, so a minimal, single-target `PassGraph`
  with `passes: []` and a placeholder `display` value is a legitimate, safe way to realize one
  buffer through the existing caching/fingerprinting machinery without touching the *shared*
  `PassGraph` `GraphFrameExecutor` already builds for display passes (keeping this fully independent,
  same reasoning `BufferFeedbackExecutor` already uses for its own dedicated `ResourceRealizer`
  instance).
- **`BufferFeedbackExecutor`'s exact shape is the model for `ComputeBufferExecutor`**
  (`packages/runtime-webgpu/src/consumers/bufferFeedback.ts:103-222`, confirmed by direct read in an
  earlier session): private lazy `realizer?`/`device?`/dimension fields, an idempotent
  `ensureAllocated` that disposes-and-rebuilds only when device/size changes, a `seeded` boolean
  guarding one-time initial data (`device.queue.writeBuffer`, only on first `execute()` call), and a
  `dispose()` releasing everything. `ComputeBufferExecutor` mirrors this shape exactly — **not**
  because compute dispatch itself needs statefulness (F3.3 deliberately made `executeComputeKernel`
  a stateless plain function), but because *this specific target* needs its buffer's contents to
  survive across frames (seeded once, doubled every frame after — the same "seed once, evolve every
  frame" idiom `BufferFeedbackExecutor` already uses, chosen deliberately as the clearest possible
  proof that dispatch really runs every frame, not just once: Gate item 14 asserts the *second*
  frame's result is double the first's, not just that a single call produces a plausible value).
  `ResourceRealizer.realizeAll` reuses the same physical buffer across calls whenever the
  fingerprint (device + size + usage) is unchanged (confirmed by direct read of its caching logic in
  an earlier session) — this is what makes "seed once, mutate every frame" actually work: the same
  `GPUBuffer` object is returned on every call, not a fresh one.
- **Three sample graphs are the regression-protection anchors, confirmed by direct read of
  `packages/graph-editor/src/graphBuilders.ts`**: `cosinePaletteEffectGraph()` (191-249, ShaderToy

## Fix

### 1. `ComputeBufferTargetDescriptor` and structural discovery

New file `packages/graph/src/computeBufferTarget.ts`, mirroring `bufferFeedbackTarget.ts` exactly:

```ts
import type { KernelBindingTemplate, PrimitiveImplementation } from './implementation.js';
import { getPrimitive } from './registry.js';
import type { GraphDocument, Node } from './types.js';

const COMPUTE_BUFFER_ROLE = 'computeBufferTarget';

export interface ComputeBufferTargetDescriptor {
	nodeId: string;
	elementCount: number;
	bindings: readonly KernelBindingTemplate[];
}

function positiveInteger(value: unknown, name: string, nodeId: string): number {
	if (typeof value !== 'number' || !Number.isInteger(value) || value <= 0) {
		throw new Error(
			`target.computeBuffer node "${nodeId}" requires ${name} to be a positive integer`
		);
	}
	return value;
}

export function isComputeBufferTarget(node: Node): boolean {
	return getPrimitive(node.primitive)?.metadata?.role === COMPUTE_BUFFER_ROLE;
}

export function computeBufferTargetForNode(node: Node): ComputeBufferTargetDescriptor {
	const implementation: PrimitiveImplementation | undefined = getPrimitive(
		node.primitive
	)?.implementation;
	if (!implementation || implementation.kind !== 'kernel') {
		throw new Error(`target.computeBuffer node "${node.id}" has no kernel implementation`);
	}
	return {
		nodeId: node.id,
		elementCount: positiveInteger(node.params?.elementCount, 'elementCount', node.id),
		bindings: implementation.bindings
	};
}

export function deriveComputeBufferTarget(doc: GraphDocument): ComputeBufferTargetDescriptor | null {
	const nodes = doc.nodes.filter(isComputeBufferTarget);
	if (nodes.length > 1) {
		throw new Error(
			`A graph may contain at most one target.computeBuffer sink; found ${nodes.length}`
		);
	}
	const node = nodes[0];
	return node ? computeBufferTargetForNode(node) : null;
}

export { COMPUTE_BUFFER_ROLE };
```

`bindings` is pulled directly from the matched node's own registered primitive (not re-declared or
hardcoded a second time at the call site) — single source of truth between registration (Fix step 2)
and dispatch (Fix step 5).

### 4. `ComputeBufferExecutor`

Same file, mirroring `BufferFeedbackExecutor`'s shape (Verified current state). Imports match
`bufferFeedback.ts`'s own exact style (`consumers/bufferFeedback.ts:1-3`):

```ts
import { executeComputeKernel, type ComputeKernelResource } from '../computeKernel.js';
import { ResourceRealizer } from '../frameGraph/realize.js';
import type { PassGraph } from '../frameGraph/types.js';
import type { ComputeBufferTargetDescriptor } from '@world-lab/graph';
import type { ConsumerShader } from '@world-lab/compiler';

const VALUES_RESOURCE_ID = 'computeBufferTarget:values';

export class ComputeBufferExecutor {
	private realizer?: ResourceRealizer;
	private device?: GPUDevice;
	private elementCount?: number;
	private seeded = false;

	private ensureAllocated(device: GPUDevice, elementCount: number): void {
		if (this.device === device && this.elementCount === elementCount && this.realizer) {
			return;
		}
		this.dispose();
		this.realizer = new ResourceRealizer(device);
		this.device = device;
		this.elementCount = elementCount;
	}

	async execute(
		device: GPUDevice,
		target: ComputeBufferTargetDescriptor
	): Promise<{ values: Float32Array }> {
		this.ensureAllocated(device, target.elementCount);
		const realizer = this.realizer!;

		const passGraph: PassGraph = {
			display: VALUES_RESOURCE_ID,
			passes: [],
			targets: [
				{
					id: VALUES_RESOURCE_ID,
					shape: {
						kind: 'buffer',
						element: { kind: 'scalar', scalar: 'f32' },
						access: 'read-write',
						usages: ['storage', 'copy-src', 'copy-dst']
					},
					lifetime: { kind: 'transient' },
					size: { kind: 'element-count', count: target.elementCount }
				}
			]
		};
		realizer.realizeAll(passGraph, { width: 1, height: 1 });
		const { write } = realizer.resolve(VALUES_RESOURCE_ID);
		const valuesBuffer = write as GPUBuffer;

		if (!this.seeded) {
			const seed = Float32Array.from({ length: target.elementCount }, (_, index) => index + 1);
			device.queue.writeBuffer(valuesBuffer, 0, seed.buffer, seed.byteOffset, seed.byteLength);
			this.seeded = true;
		}

		await executeComputeKernel(device, {
			shader: computeBufferShader(target.nodeId),
			bindings: target.bindings,
			wgslTypes: new Map([['values', 'array<f32>']]),
			resourceIds: new Map([['values', VALUES_RESOURCE_ID]]),
			resources: new Map([[VALUES_RESOURCE_ID, { kind: 'buffer', buffer: valuesBuffer }]]),
			outputFns: { values: 'scale_values' },
			callArgs: ['gid'],
			workgroupSize: [8, 1, 1],
			dispatch: { kind: 'buffer', elementCount: target.elementCount }
		});

		realizer.advanceFrame();
		return { values: await readBackFloat32Buffer(device, valuesBuffer, target.elementCount) };
	}

	dispose(): void {
		this.realizer?.dispose();
		this.realizer = undefined;
		this.device = undefined;
		this.elementCount = undefined;
		this.seeded = false;
	}
}
```

`readBackFloat32Buffer` is a small new helper, same file, following the established readback
convention exactly (`planeScalarPreview.ts:144-155`, reused in F3.3's own device test):

```ts
async function readBackFloat32Buffer(
	device: GPUDevice,
	buffer: GPUBuffer,
	elementCount: number
): Promise<Float32Array> {
	const byteLength = elementCount * 4;
	const readback = device.createBuffer({
		label: 'compute-buffer-target-readback',
		size: byteLength,
		usage: GPUBufferUsage.COPY_DST | GPUBufferUsage.MAP_READ
	});
	const encoder = device.createCommandEncoder({ label: 'compute-buffer-target-readback' });
	encoder.copyBufferToBuffer(buffer, 0, readback, 0, byteLength);
	device.queue.submit([encoder.finish()]);

	await readback.mapAsync(GPUMapMode.READ);
	const values = new Float32Array(readback.getMappedRange()).slice();
	readback.unmap();
	readback.destroy();
	return values;
}
```

export function rendererForPreviewFamily(
	family: PreviewFamily,
	opts: { preferGpu?: boolean } = {}
): PreviewRenderer {
	switch (family) {
		case 'image':
			return opts.preferGpu ? 'gpu' : 'effect';
		case 'geometry':
			return 'mesh';
		case 'data':
			return opts.preferGpu ? 'gpu' : 'cpu';

			return 'audio';

## Out of scope

Any change to `stage.vertex`, `stage.fragment`, `pipeline.ts`'s presentation-derivation logic,
`pipelineVertex.ts`, `fullscreenFragment.ts`, or any existing sample graph — that's F3.6's problem,
not this one's, and this milestone's whole point is proving it can land without touching any of
them. A `{kind:'resource'}` primitive giving graph authors a way to *wire* a resource into a
kernel's binding (this milestone's kernel is self-contained; resourceIds are synthesized, not
derived from graph edges) — deferred, most naturally to F3.6 alongside dynamic bindings. Routing a
kernel's WGSL body through `compileGraph`/`sliceGraph`/`generateWgsl` (this milestone's body is
hand-written, matching the majority existing precedent) — also F3.6's territory if ever needed.
Multiple `target.computeBuffer` nodes in one document (at most one, mirroring
`target.bufferFeedback`'s own constraint exactly). A `factor`/scale-amount param wired from the
graph (hardcoded `* 2.0` — keeps the shader trivial so the milestone's real weight stays on the
resource-realization/dispatch wiring, not shader design). Texture-shaped compute targets (buffer
only; texture dispatch domains were already proven pure-function-only in F3.3 — wiring one into
`GraphFrameExecutor` for real is a natural, separate follow-on once a real use case exists).
Typed varyings (F3.2, unrelated). `{ kind: 'command'; command: GpuCommandKind }` (Foundation 4's
seam). Any change to Foundation 1's, Foundation 2's, F3.1's, F3.2's, or F3.3's landed contracts.
A real data-visualization panel (chart, sparkline, heatmap) for `'buffer'`-family results —
`DataBufferPreviewPanel` stays deliberately text-only (Fix step 9); a richer visualization is a
separate, later concern once a real use case needs one. Retrofitting `'buffer'`-family support
into `CpuPreviewPanel`/`EffectPreviewPanel` themselves, or generalizing either panel — a new,
narrow panel is the right size for what this milestone needs, not a refactor of existing panels.
