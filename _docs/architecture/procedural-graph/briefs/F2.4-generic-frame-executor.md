## Revision note

**Revision 2** fixed three confirmed defects in revision 1, caught before routing:

1. **Blocking: multi-target textures lacked `COPY_SRC`.** `inferTextureUsage` (F2.3, landed) grants
   `COPY_SRC` only to `graph.display`, but `buildIndependentPassGraph` (F2.2-era, landed) sets
   `display` to only the **first** target — every other independently-planned target gets no
   `COPY_SRC`. F2.4's executor reads back **every** target via `copyTextureToBuffer` (that's the
   whole point of returning `targets: Record<string, Uint8Array>` for a multi-buffer preview) —
   any second target would fail WebGPU validation the first time this milestone's own code path ran
   against a real, multi-target graph. This defect was latent in F2.2/F2.3's landed code (nothing
   used `inferTextureUsage`'s output for real multi-target readback before this milestone) and only
   became a real, observable bug once F2.4 wired the shared path through it. Fixed below: a new,
   explicit `PassGraph.readbackTargets?: readonly string[]` field, additive and optional (no
   existing F2.1–F2.3 test constructs it, so none break), decoupling "needs `COPY_SRC` for host
   readback" from "is the one canonical `display` target" — these were never actually the same
   concept once more than one target can be independently previewed.
2. **Moderate: pass-id/target-id conflation with a silent skip.** Revision 1's execution loop
   looked up `buildPassOrder(...).order`'s entries (which are `Pass.consumerId` values) in a map
   keyed by `GraphFramePass.targetId`, relying on `consumerId === writeTarget === targetId` holding
   coincidentally for every graph `buildIndependentPassGraph` produces today — true today, but the
   type system deliberately keeps `consumerId` and `writeTarget` distinct concepts, and the loop's
   `if (!pass) continue` would silently drop a pass rather than surface the break if that equality
   ever stopped holding. Fixed below: resolve `consumerId → Pass.writeTarget → GraphFramePass`
   explicitly, throwing on any lookup miss instead of continuing past it.
3. **Minor: the brief's own sample imported `PassGraph` from the wrong module.** `order.ts` only
   imports `PassGraph` from `types.ts` for its own use — it never re-exports it. Fixed below: import
   `PassGraph` from `./frameGraph/types.js` directly.

## Context

Fourth of Foundation 2's sequence (F2.1 ✅ → F2.2 ✅ → F2.3 ✅ → **F2.4** → proof). Per the plan:
"Replace `GraphFrameExecutor`'s independent-only bypass with real execution of F2.2's ordered DAG
over F2.3's realized resources. Incremental, not a flag-day rewrite: today's zero-dependency
preview case must keep working throughout."

**A scope boundary this milestone deliberately holds, flagged now rather than discovered during
the Foundation 2 proof step that follows it:** today, no primitive or WGSL emission path lets one
pass's fragment shader sample another pass's texture as an input (`HostBinding`'s `'read-resource'`
context is reserved but nothing uses it — confirmed by grep, zero hits outside its own type
declaration), and `graphFramePlan.ts`'s document→`PassGraph` derivation
(`buildIndependentPassGraph`) only ever produces **zero-read** passes, matching what's actually
authorable today. **This milestone does not add a channel-read primitive, WGSL emission for
sampling another pass's texture, or document-driven non-trivial `reads`.** It wires the
already-built generic ordering (`buildPassOrder`) and realization (`ResourceRealizer`) machinery
into the *real* per-frame execution path for the graphs that are actually constructible today —
replacing ad hoc per-frame texture allocation with cached, reused resources, and routing execution
through the same pass-order the generic algorithm produces (a no-op for today's all-independent
case, but the correct hook for when real cross-pass reads exist). **Consequence for sequencing:**
the Foundation 2 proof step's two-pass render-target sample cannot yet exist as a real,
document-driven graph after this milestone alone — it needs the channel-read primitive this
milestone deliberately doesn't build. Flagging this now so the proof step's own contract can
either absorb that minimal primitive or be resequenced, rather than silently discovering the gap
once F2.4 is "done." (The buffer-feedback proof sample has the same gap one level deeper: it needs
a compute dispatch, which needs Foundation 3's kernels — a second, separate reason the literal
plan-sequence "F2.4 → proof" needs revisiting before the proof step is briefed.)

## Verified current state

- **`GraphFrameExecutor.execute`** (`graphFrameExecutor.ts`, 59 lines) calls
  `planIndependentGraphFramePasses(doc)` and loops calling `this.pipeline.execute(...)`
  (`PipelineGraphExecutor`) independently per target — never calls `buildIndependentPassGraph`,
  `buildPassOrder`, or anything from F2.3. Confirmed directly: no import of `frameGraph/` anywhere
  in this file.
- **`executeFullscreenFragment`** (`consumers/fullscreenFragment.ts:260,310`) creates a fresh
  `GPUTexture` with `device.createTexture(...)` and calls `texture.destroy()` at the end of
  **every single call** — i.e., every rendered frame, for every target, a texture is allocated and
  destroyed. This is the concrete inefficiency F2.3's `ResourceRealizer` exists to fix, and today
  nothing routes through it. Its hardcoded usage (`RENDER_ATTACHMENT | COPY_SRC`, unconditional)
  is exactly why this defect stayed latent — every target always got `COPY_SRC` regardless of
  "display" status, because nothing derived usage from `PassGraph` structure yet.
- **`inferTextureUsage`** (`frameGraph/realize.ts:101`) grants `COPY_SRC` only when
  `graph.display === targetId`. **`buildIndependentPassGraph`** (`graphFramePlan.ts:20`) sets
  `display: passes[0]?.targetId ?? ''` — only the **first** independently-planned target. Confirmed
  directly: for any independent graph with two or more live preview targets (e.g.
  `graphFramePlan.test.ts`'s local `dualDisplayGraph()` fixture, two `target.display` sinks), every
  target after the first would realize with `RENDER_ATTACHMENT` only, no `COPY_SRC` — and this
  milestone's executor calls `copyTextureToBuffer` against every one of them for CPU readback.
- **`PipelineGraphExecutor.execute`** (`pipelineGraph.ts`) is a thin pass-through to
  `executeFullscreenFragment` plus its own, unrelated `PipelineGraphResourceCache` (geometry
  realization keyed by a content fingerprint, via `buffer.persist`) — untouched by this milestone,
  no naming or behavior overlap with `ResourceRealizer`.
- **Only one production caller exists**, confirmed by grep across the monorepo:
  `packages/graph-editor/src/previewFrameLoop.ts`. It constructs `let executor = new
  GraphFrameExecutor()` **once** per preview-loop session and reuses it across every
  `requestAnimationFrame` tick; `device` is likewise requested once
  (`requestGpuDevice()`) and reused across ticks — confirmed by reading the tick loop directly.
  This is exactly the object lifetime a stateful, cross-frame `ResourceRealizer` needs to live
  inside.
- **`previewFrameLoop.ts`'s `destroy()`** currently calls `device?.destroy()` and discards the
  `executor` reference (`executor = new GraphFrameExecutor()`) — it has no way to dispose whatever
  GPU resources a stateful executor might be holding, because none exist yet. Introducing a
  persistent `ResourceRealizer` inside `GraphFrameExecutor` makes this a real gap this milestone
  must close, not a pre-existing one it can ignore.
- **`graphFrameExecutor.test.ts`'s first test** ("passes the same iTime and iFrame to every pass in
  a frame") uses `device: {} as GPUDevice` (a bare object, no methods) and mocks
  `PipelineGraphExecutor.prototype.execute` directly, bypassing the real render path entirely. Once
  `GraphFrameExecutor.execute` also calls `ResourceRealizer.realizeAll` (which calls real
  `device.createTexture`), this fake device needs `createBuffer`/`createTexture` mocks — matching
  `realize.test.ts`'s own `mockDevice()` helper pattern — or the test throws. Its second test
  (`it.skipIf(!hasWebGPU)`) already uses a real device and needs no change.
- **`fullscreenFragment.test.ts`'s two `it.skipIf(!hasWebGPU)` tests** call
  `executeFullscreenFragment` directly with a real device and no `target` — they need updating to
  create their own texture and pass it in, mirroring the "instanced mesh draw extraction" precedent
  (`M-instanced-mesh-draw-extraction.md`): caller-owned resource, no internal allocation.
- **`previewFrameLoop.test.ts`** mocks `GraphFrameExecutor` entirely as `class { execute =
  executeMock; }` with no `dispose` method — adding a `dispose()` call in `previewFrameLoop.ts`'s
  `destroy()` breaks this test unless the mock also gains `dispose = vi.fn()`.

## Fix

1. **Give `PassGraph` an explicit, additive `readbackTargets` field, and make `inferTextureUsage`
   and `buildIndependentPassGraph` agree on it — the blocking fix.** `display` stays what it always
   meant (the one canonical/final target, used by `computeLifetimes` for within-frame retention);
   "needs `COPY_SRC` because a host will read it back" is a genuinely separate concept, so it gets
   its own field rather than overloading `display` into a set:
   ```ts
   // frameGraph/types.ts — additive, optional; no existing F2.1–F2.3 test constructs it, none break
   export interface PassGraph {
     targets: ResourceTarget[];
     passes: Pass[];
     display: string;
     readbackTargets?: readonly string[];
   }
   ```
   ```ts
   // frameGraph/realize.ts — inferTextureUsage, updated
   if (graph.display === targetId || graph.readbackTargets?.includes(targetId)) {
     usage |= GPUTextureUsage.COPY_SRC;
   }
   ```
   ```ts
   // graphFramePlan.ts — buildIndependentPassGraph, updated
   return {
     targets,
     passes: passes.map((pass) => ({ consumerId: pass.targetId, writeTarget: pass.targetId, reads: [] })),
     display: passes[0]?.targetId ?? '',
     readbackTargets: passes.map((pass) => pass.targetId) // every independently-planned target is read back
   };
   ```
   This touches two files nominally owned by F2.2 (`types.ts`) and F2.3 (`realize.ts`,
   `graphFramePlan.ts`'s mechanical update), but strictly additively — the new field is optional,
   every existing hand-constructed `PassGraph` literal in `order.test.ts`/`realize.test.ts` simply
   omits it and keeps its prior behavior exactly. This is the fix F2.4 needs to exist at all: without
   it, this milestone's own executor would fail WebGPU validation the first time it ran against any
   multi-target independent graph.
2. **`executeFullscreenFragment`/`PipelineGraphExecutor.execute` accept a caller-owned render
   target — no internal creation or destruction:**
   ```ts
   // fullscreenFragment.ts
   export interface FullscreenFragmentInput {
     device: GPUDevice;
     graph: GraphDocument;
     output: PortRef;
     resolver?: WgslModuleResolver;
     width: number;
     height: number;
     host: ShaderToyHostInputs;
     target: GPUTexture; // caller-owned; not destroyed here
   }
   ```
   Remove the `device.createTexture(...)`/`texture.destroy()` calls from
   `executeFullscreenFragment`'s body; use `input.target` everywhere `texture` was used (the color
   attachment view, the `copyTextureToBuffer` source). The readback-to-`Uint8Array` behavior is
   otherwise unchanged — callers still receive the same `FullscreenFragmentResult` shape.
   `PipelineGraphInput` gains the same `target: GPUTexture` field and passes it straight through.
3. **`GraphFrameExecutor` owns a lazily-created, device-keyed `ResourceRealizer`** — following this
   file's existing `private` field convention (not `realize.ts`'s `#`-private convention; match
   each file's own style):
   ```ts
   import { buildPassOrder } from './frameGraph/order.js';
   import type { PassGraph } from './frameGraph/types.js'; // NOT order.js — order.ts never re-exports it
   import { ResourceRealizer } from './frameGraph/realize.js';
   import { buildIndependentPassGraph, planIndependentGraphFramePasses } from './graphFramePlan.js';

   export class GraphFrameExecutor {
     private readonly pipeline = new PipelineGraphExecutor();
     private realizer: ResourceRealizer | undefined;
     private realizerDevice: GPUDevice | undefined;

     private realizerFor(device: GPUDevice): ResourceRealizer {
       if (this.realizerDevice !== device) {
         this.realizer?.dispose();
         this.realizer = new ResourceRealizer(device);
         this.realizerDevice = device;
       }
       return this.realizer;
     }

     dispose(): void {
       this.realizer?.dispose();
       this.realizer = undefined;
       this.realizerDevice = undefined;
     }

     async execute(input: GraphFrameExecuteInput): Promise<GraphFrameExecuteResult> {
       // see step 3
     }
   }
   ```
   Keyed-by-device-identity rather than created once at construction: `GraphFrameExecutor`'s public
   constructor takes no arguments today (confirmed the only caller does `new GraphFrameExecutor()`
   with `device` supplied per `execute()` call) — this preserves that API exactly, while still
   correctly disposing and recreating if a device is ever replaced (device loss/recreation),
   instead of silently mixing GPU resources from two different devices.
4. **`execute` builds a `PassGraph` from the same document-derivation as today (unchanged), runs it
   through `buildPassOrder`, realizes it, and executes in the order the algorithm produces —
   resolving `consumerId → Pass.writeTarget → GraphFramePass` explicitly, throwing on a mismatch
   rather than silently skipping:**
   ```ts
   async execute(input: GraphFrameExecuteInput): Promise<GraphFrameExecuteResult> {
     const passes = planIndependentGraphFramePasses(input.graph);
     const passGraph = buildIndependentPassGraph(passes);
     const order = buildPassOrder(passGraph);
     const realizer = this.realizerFor(input.device);
     realizer.realizeAll(passGraph, { width: input.width, height: input.height });

     const planPassByConsumerId = new Map(passGraph.passes.map((pass) => [pass.consumerId, pass]));
     const framePassByTargetId = new Map(passes.map((pass) => [pass.targetId, pass]));
     const targets: Record<string, Uint8Array> = {};
     for (const consumerId of order.order) {
       const planPass = planPassByConsumerId.get(consumerId);
       if (!planPass) {
         throw new Error(`buildPassOrder returned unknown consumer "${consumerId}"`);
       }
       const framePass = framePassByTargetId.get(planPass.writeTarget);
       if (!framePass) {
         throw new Error(
           `Pass "${consumerId}" writes target "${planPass.writeTarget}" with no matching GraphFramePass`
         );
       }
       const host: ShaderToyHostInputs = {
         iTime: input.host.iTime,
         iFrame: input.host.iFrame,
         iMouse: input.host.pointers[framePass.targetId] ?? [0, 0, 0, 0]
       };
       const result = await this.pipeline.execute({
         device: input.device,
         graph: input.graph,
         output: framePass.fieldOutput,
         resolver: input.resolver,
         width: input.width,
         height: input.height,
         host,
         target: realizer.resolve(framePass.targetId).write as GPUTexture
       });
       targets[framePass.targetId] = result.pixels;
     }
     realizer.advanceFrame();
     return { width: input.width, height: input.height, targets };
   }
   ```
   `order.order` contains `Pass.consumerId` values, not `GraphFramePass.targetId` values — they are
   numerically identical for every graph `buildIndependentPassGraph` produces today
   (`consumerId: pass.targetId, writeTarget: pass.targetId`), but the types keep them conceptually
   distinct, and a lookup keyed by the wrong one plus a silent `continue` would hide a real
   correctness break the moment that equality ever stopped holding (e.g. once real dependency
   graphs exist). Resolving through `writeTarget` explicitly, and throwing on either lookup miss,
   makes the current 1:1 correspondence a verified invariant instead of an assumed one — for any
   graph constructible today, neither `throw` is reachable, so this changes robustness, not
   observable behavior. `realizer.advanceFrame()` runs once per `execute()` call regardless —
   harmless no-op today (no `history`-lifetime targets exist yet in an independently-planned
   graph), correct steady state once they do.
5. **`previewFrameLoop.ts`'s `destroy()` disposes the executor** — the concrete fix for the
   resource-leak gap a stateful realizer introduces:
   ```ts
   destroy() {
     destroyed = true;
     cancelAnimationFrame(rafId);
     listeners.clear();
     executor.dispose();
     device?.destroy();
     device = null;
     executor = new GraphFrameExecutor();
   }
   ```
6. **Update the two test files whose fakes/mocks the above changes make incompatible** (mechanical,
   not behavioral):
   - `graphFrameExecutor.test.ts`'s fake-device test: give `device` mocked `createBuffer`/
     `createTexture` (returning distinct fake objects with a `destroy: vi.fn()`, matching
     `realize.test.ts`'s `mockDevice()` shape) so `ResourceRealizer.realizeAll` doesn't throw
     against a bare `{}`.
   - `fullscreenFragment.test.ts`'s two real-device tests: create a `device.createTexture(...)`
     before calling `executeFullscreenFragment`, pass it as `target`, and `.destroy()` it after
     (alongside the existing `device.destroy()`).
   - `previewFrameLoop.test.ts`'s mocked `GraphFrameExecutor` class: add `dispose = vi.fn()`.

## Gate

1. A test proves `buildPassOrder` and `ResourceRealizer.realizeAll` are genuinely invoked once per
   `execute()` call (spies on the real functions/methods) — the concrete proof this milestone
   routes execution through the generic machinery, not just a claim in prose.
2. A test proves texture identity is **reused** across two consecutive `execute()` calls with an
   unchanged graph/viewport: `createTexture` is called once per unique target across both calls,
   not once per call — the direct, observable fix for today's per-frame reallocation
   (`fullscreenFragment.ts`'s old create-then-destroy-every-call pattern).
3. **Regression, revision 2 finding 1 (blocking), unit-level:** a test proves every target in a
   multi-target independent `PassGraph` (built via `buildIndependentPassGraph` from two or more
   pipeline passes) resolves to a `usage` including `GPUTextureUsage.COPY_SRC` — not only the first
   / `display` one — proving `readbackTargets` actually reaches `inferTextureUsage`.
4. **Regression, revision 2 finding 1 (blocking), real-device:** `it.skipIf(!hasWebGPU)` — a new
   test builds a document with **two or more** independent pipeline display targets (following
   `graphFramePlan.test.ts`'s local `dualDisplayGraph()` pattern), runs it through
   `GraphFrameExecutor.execute` against a real device, and confirms every target's readback
   succeeds without a WebGPU validation error — the concrete end-to-end proof the existing
   single-target real-device test could not catch.
5. **Regression, revision 2 finding 2:** a test proves the executor throws a clear error rather than
   silently skipping a pass when `buildPassOrder`'s returned `consumerId` cannot be resolved
   through `Pass.writeTarget` to a known `GraphFramePass` — constructed via a deliberately
   inconsistent plan (e.g. a `PassGraph`/`GraphFramePass[]` pair built by hand so the two do not
   correspond), proving the resolution is a real, load-bearing check and not dead code.
6. A test proves `advanceFrame()` is called exactly once per `execute()` call (spy on
   `ResourceRealizer.prototype.advanceFrame`).
7. A test proves `GraphFrameExecutor.dispose()` disposes the internal realizer (spy on
   `ResourceRealizer.prototype.dispose`) when one exists, and is a safe no-op when called before
   any `execute()` call has ever happened.
8. A test proves a device identity change between two `execute()` calls disposes the stale realizer
   and allocates fresh resources against the new device, rather than mixing resources from two
   devices or throwing.
9. `graphFrameExecutor.test.ts`'s existing "passes the same iTime and iFrame to every pass in a
   frame" test still passes after its fake device gains `createBuffer`/`createTexture` mocks —
   proving the iTime/iFrame-sharing behavior this test exists for is unaffected by the realizer
   wiring.
10. `graphFrameExecutor.test.ts`'s existing real-device test ("renders all independent targets on a
    device") still passes unmodified.
11. `fullscreenFragment.test.ts`'s two real-device tests still pass after updating to create and own
    their target texture externally, proving `executeFullscreenFragment` renders correctly into a
    caller-provided texture rather than one it allocates itself.
12. `previewFrameLoop.test.ts` still passes after its mocked `GraphFrameExecutor` gains `dispose =
    vi.fn()`; a new assertion confirms it's called once after `loop.destroy()`.
13. `check` **and** `test` green for `runtime-webgpu`, `graph-editor`, full workspace (clear every
    `packages/*/dist` first, per `AGENTS.md`'s documented stale-dist issue).

## Out of scope

A channel-read primitive or any WGSL emission for sampling another pass's texture as a fragment
input (no primitive/codegen path exists yet; see Context — flagged as a real gap for whichever
milestone briefs the Foundation 2 proof next). Document-driven non-trivial `PassGraph.reads`
(`planIndependentGraphFramePasses` is unchanged; `buildIndependentPassGraph` only gains the
additive `readbackTargets` field from Fix step 1, still producing zero-read passes). Compute
dispatch or any buffer-feedback execution path (Foundation 3 — no compute kernels exist). Bind-group
layout derivation beyond what `pipeline.getBindGroupLayout(0)` (WebGPU's `layout: 'auto'`) already
provides today — unchanged, not this milestone's concern. Resource pooling/aliasing across
different target ids. Any change to `PipelineGraphResourceCache` (geometry realization) —
unrelated, untouched. Any change to Foundation 1's milestones beyond Fix step 1's narrow, additive
`readbackTargets` field (no existing F2.1–F2.3 gate item is invalidated), or this document's own
Foundation 2 sequencing beyond noting the proof-step gap
above.

## Handoff

→ `GraphFrameExecutor` now allocates render targets exactly once and reuses them across frames via
a persistent, device-keyed `ResourceRealizer`, instead of creating and destroying a texture on
every single rendered frame; every independently-planned target correctly carries `COPY_SRC` via
the new `readbackTargets` field, not just the first/`display` one; execution proceeds in the order
`buildPassOrder` produces via a verified `consumerId → writeTarget → GraphFramePass` resolution
(currently a no-op reordering for today's all-independent graphs, but the correct, robust hook once
real dependencies exist); `dispose()` closes the resource-leak gap the new persistent state
introduces, wired into `previewFrameLoop.ts`'s teardown. **Before briefing the Foundation 2 proof
step:** decide whether
its own contract should absorb the minimal channel-read primitive (texture case) — a small,
ShaderToy-scoped addition using the already-reserved `HostBinding{context:'read-resource'}`, not
Foundation-3-grade generic kernel bindings — and whether the buffer-feedback sample needs to wait
for at least a minimal compute-dispatch capability, or whether Foundation 2's proof should be
resequenced/split to fit what's actually buildable before Foundation 3 lands. This milestone
deliberately surfaces that decision rather than making it unilaterally.
