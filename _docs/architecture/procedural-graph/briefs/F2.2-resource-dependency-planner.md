## Revision note

**Revision 2** fixed two confirmed correctness defects in revision 1, caught before routing:

1. **`feedbackTargets` conflated persistence with history.** Revision 1 mapped every
   non-`transient` target (both `persistent` and `history`) into `feedbackTargets`, and this
   brief's own Handoff/Out-of-scope text told F2.3 to physically double-buffer everything that list
   contains. But `persistent` means exactly one physical allocation and `history` means two,
   ping-ponged — including `persistent` targets would tell F2.3 to double-buffer resources that
   must not be. Fixed below: `collectFeedbackTargets` now checks `lifetime.kind === 'history'`
   only. This also exposed that the old unconditional `if (graph.display) feedback.add(...)` line
   was conflating two different concerns — within-frame display retention (already fully handled by
   `computeLifetimes`'s existing end-of-frame `lastRead` extension for `graph.display`) and
   across-frame ping-pong need (what `feedbackTargets` is actually for). The display-add line is
   removed; a target's retention already follows from its own `lifetime`, not from being displayed.
2. **`ResourceTarget` let shape and size disagree.** Revision 1's flat `{ shape, size }` interface
   permitted a buffer paired with a `screen-relative`/`fixed` (pixel) size, or a texture paired with
   `element-count` — nonsensical combinations with no runtime meaning. Fixed below: `ResourceTarget`
   is now a discriminated union with a buffer-shaped variant (paired only with `element-count` size)
   and a texture-shaped variant (paired only with `screen-relative`/`fixed` size), so a mismatched
   pairing fails to type-check rather than being merely undocumented.

Both are correctness issues the stronger F2.1 lifetime/shape model exposed, not implementation
details — revision 1's types compiled and its own gate tests would have passed, but F2.3 building
against either defect would have propagated a real bug (wrong double-buffering decisions; targets
with self-contradictory size semantics).

## Context

Second of Foundation 2's sequence (F2.1 ✅ → **F2.2** → F2.3 → F2.4 → proof). This milestone
generalizes `frameGraph/types.ts`'s texture-only shapes to F2.1's resource union, and does the two
things F2.1's own brief explicitly deferred here:

1. "Materializing `ResourceInstance` from a real `GraphDocument` (F2.2)."
2. "Attaching `ResourceBinding[]` to `Pass` (F2.2 — planner-level, independent of any
   kernel/command graph node shape; Foundation 3 later translates kernel/command nodes into this
   shape, it does not define it, so F2.2 never waits on Foundation 3)."

**No algorithmic changes to the ordering/validation/lifetime functions themselves** —
`validatePassGraph`, `buildAdjacency`, `topologicalOrder`, `computeLifetimes`, `buildPassOrder`
already reason only about opaque string IDs; this was verified directly when the Foundation 2 plan
was written. This milestone changes what fields those opaque IDs and booleans are read from, plus
adds one new validation rule the resource-history model makes possible for the first time (item 4
below) — it does not change the topological-sort or lifetime-computation logic itself.

## Verified current state

- `frameGraph/types.ts` (32 lines) is the entire texture-only surface: `RenderTarget { id, format:
  GPUTextureFormat, size: TargetSize, persistent?: boolean }`, `ChannelRead { channel, target,
  previousFrame?: boolean, sampler? }`, `Pass { consumerId, writeTarget, reads, iterations?,
  pure? }`, `PassGraph { targets, passes, display }`.
- `order.ts` (282 lines) reads exactly these fields: `target.id` (`targetIds`), `target.persistent`
  (`collectFeedbackTargets`), `read.previousFrame` (`sameFrameReads`, `collectFeedbackTargets`,
  `validatePassGraph`'s read-write-same-pass check), `target.format`/`target.size`
  (`resolveTargetSizes` only — the rest of the file never touches them). Confirmed by direct read:
  no other field is consulted anywhere in the ordering/validation/lifetime code.
- `graphFramePlan.ts`'s `buildIndependentPassGraph` constructs `PassGraph` literals directly
  (`format: 'rgba8unorm'`, `size: { kind: 'screen-relative', scale: 1 }`, no `reads`) — this is the
  only other in-repo constructor of a `PassGraph`, confirmed via
  `grep -rl 'RenderTarget|ChannelRead|PassGraph'` across the whole monorepo (only
  `runtime-webgpu/src/{frameGraph/,graphFramePlan.ts,index.ts}` and their tests match — no
  `apps/*` consumer exists; the `ProceduralRenderTarget`/`WaterRenderTarget` types under
  `apps/scene-editor` are unrelated homonyms for the planet renderer's draw-order batching, not
  this frame graph).
- `runtime-webgpu`'s `package.json` already depends on `@world-lab/graph` — no new workspace
  dependency needed to import `ResourceInstance` et al.
- `packages/graph/src/executionRoots.ts` is the precedent for document-materialization: `doc.nodes
  .filter((node) => getPrimitive(node.primitive)?.implementation.kind === 'sink')`. This milestone's
  `collectResourceInstances` follows the identical shape, swapping the primitive kind and building
  an instance instead of just filtering.
- No `kind: 'resource'` primitive is registered anywhere in the codebase yet (F2.1 only defined the
  contract shape; real resource primitives like a hypothetical `buffer.persist` come later, once a
  consumer needs one). This milestone's document-materialization gate test registers a test-only
  primitive to prove the mechanism, exactly as F1.3's `test.compatibilitySink` proved the sink
  mechanism ahead of real consumers existing.

## Fix

1. **Generalize the target shape as a discriminated union, keeping identity flat.** Rename
   `RenderTarget` → `ResourceTarget` (the old name actively misleads once buffers flow through the
   same list). Only buffer- and texture-shaped resources are frame-graph targets — a sampler is
   read-only configuration, never something a pass writes to, so exclude it at the type level rather
   than merely documenting the gap. **Shape and size must be structurally paired, not a free
   product** — a buffer's size is an element count, a texture's size is pixel dimensions, and
   nothing else makes sense for either:
   ```ts
   import type { ResourceLifetime, ResourceShape } from '@world-lab/graph';

   export interface BufferResourceTarget {
     id: string;
     shape: Extract<ResourceShape, { kind: 'buffer' }>;
     lifetime: ResourceLifetime;
     size: { kind: 'element-count'; count: number };
   }

   export interface TextureResourceTarget {
     id: string;
     shape: Extract<ResourceShape, { kind: 'texture' }>;
     lifetime: ResourceLifetime;
     size:
       | { kind: 'screen-relative'; scale: number }
       | { kind: 'fixed'; width: number; height: number };
   }

   export type ResourceTarget = BufferResourceTarget | TextureResourceTarget;
   ```
   Both variants discriminate on `shape.kind` (`'buffer'` vs. `'texture'`), so `target.shape.kind
   === 'buffer'` narrows `target` to `BufferResourceTarget` — including its `size` — throughout
   `order.ts` without a separate discriminant field. `ResourceTarget.persistent?: boolean` is gone —
   replaced by each variant's `lifetime: ResourceLifetime`. `target.persistent` (old) becomes
   `target.lifetime.kind !== 'transient'` (new) where that generalization is still needed (it is
   **not** the right check for `collectFeedbackTargets` — see step 4).
2. **Generalize reads: `previousFrame?: boolean` → `version?: 'previous'`.** Reading "the previous
   frame's contents" only makes physical sense for a ping-ponged (`history`) resource — a
   `persistent`-but-not-`history` resource has exactly one physical allocation, so there is no
   second copy to read without aliasing the write. Making this explicit is new expressiveness F2.1's
   `ResourceLifetime.history` case makes possible for the first time (F2.1's own brief flagged wiring
   this in as F2.2's job):
   ```ts
   export interface ResourceRead {
     channel: number;
     target: string; // ResourceTarget.id
     version?: 'previous'; // omitted = current frame; 'previous' = prior frame's committed contents
     sampler?: { filter: 'nearest' | 'linear'; wrap: 'clamp' | 'repeat' }; // texture-shaped only
   }
   ```
3. **Reserve (do not yet consume) planner-level bindings on `Pass`.** This is the field F2.1's brief
   assigned here to avoid the F2.2/Foundation-3 circularity — Foundation 3 will attach real
   `ResourceBinding` values once kernel/command nodes exist to derive them from. This milestone only
   gives the field a home and proves its presence doesn't perturb ordering (Gate item 9):
   ```ts
   import type { ResourceBinding } from '@world-lab/graph';

   export interface Pass {
     consumerId: string;
     writeTarget: string;
     reads: ResourceRead[];
     bindings?: ResourceBinding[]; // planner-level; unconsumed until F2.3/Foundation 3 need it
     iterations?: number;
     pure?: boolean;
   }

   export interface PassGraph {
     targets: ResourceTarget[];
     passes: Pass[];
     display: string;
   }
   ```
   Do not cross-validate `bindings` against `reads`/`writeTarget` for consistency in this milestone
   — the real relationship between them isn't decided until F2.3/Foundation 3 has a concrete
   consumer. Validating a contract against itself with no consumer risks encoding the wrong
   relationship and having to unwind it later.
4. **Update `order.ts`'s field accesses to match (mechanical, not algorithmic), add one new
   validation rule, and correct `collectFeedbackTargets` to stop conflating persistence with
   history:**
   - `sameFrameReads`: `reads.filter((read) => read.version !== 'previous')` (was `!read.previousFrame`).
   - `validatePassGraph`'s read-write-same-pass check: `read.target === pass.writeTarget &&
     read.version !== 'previous'` (was `!read.previousFrame`).
   - **New rule:** a read with `version: 'previous'` against a target whose `lifetime.kind !==
     'history'` is invalid — there is no second physical slot to read from without aliasing the
     write. Add the issue kind and check:
     ```ts
     export type FrameGraphIssue =
       | { kind: 'intra-frame-cycle'; cycle: string[] }
       | { kind: 'dangling-target'; pass: string; target: string }
       | { kind: 'read-write-same-pass'; pass: string; target: string }
       | { kind: 'invalid-history-read'; pass: string; target: string };
     ```
     Check it in `validatePassGraph`'s existing per-read loop (where dangling-target is already
     checked), and add the corresponding case to `formatIssue`'s switch.
   - **`collectFeedbackTargets` now means exactly "needs physical double-buffering," and only
     `history` means that:**
     ```ts
     function collectFeedbackTargets(graph: PassGraph): string[] {
       const feedback = new Set<string>();
       for (const target of graph.targets) {
         if (target.lifetime.kind === 'history') {
           feedback.add(target.id);
         }
       }
       return [...feedback];
     }
     ```
     This drops two things from the old body: the `persistent`-target loop (a `persistent` target
     is exactly one allocation — including it here would tell F2.3 to double-buffer something that
     must not be) and the unconditional `if (graph.display) feedback.add(graph.display)` line
     (display retention **within the current frame** is already fully handled by
     `computeLifetimes`'s existing special case — see below — which is a different concern from
     "needs a second physical buffer across frames"; a `transient`-lifetime display target needs no
     special cross-frame handling at all). The separate loop over `pass.reads` checking
     `read.version === 'previous'` is also dropped as redundant: `buildPassOrder` always calls
     `validatePassGraph` first and throws on any issue, and the new `invalid-history-read` rule
     already guarantees every `version: 'previous'` read's target has `history` lifetime by the time
     `collectFeedbackTargets` runs — so the lifetime check alone is exhaustive for validated input.
     `computeLifetimes`'s own display-retention special case (extending `graph.display`'s `lastRead`
     to the end of the pass order) is **unchanged** — retention-until-end-of-frame and
     needs-ping-pong-across-frames are different concerns, and only the latter was ever
     `collectFeedbackTargets`'s job.
5. **Add a resource-kind-aware sizing function, parallel to the texture one — not an overload of
   it** (per the Foundation 2 plan's explicit instruction). The discriminated union from step 1 lets
   both functions narrow on `shape.kind` directly, so a texture target's `size` is statically known
   to be pixel-shaped and a buffer target's `size` is statically known to be an element count —
   neither function can access the wrong `size` shape for a given `target`:
   ```ts
   export function resolveTargetSizes(
     graph: PassGraph,
     viewport: { width: number; height: number },
   ): Record<string, { width: number; height: number }>; // unchanged behavior, now iterating only
     // targets narrowed to shape.kind === 'texture'

   export function resolveBufferSizes(
     graph: PassGraph,
   ): Record<string, { elementCount: number }>; // new — iterates targets narrowed to
     // shape.kind === 'buffer'; a pure passthrough of size.count, no viewport dependency
   ```
   Byte-size/alignment computation (std140/std430 layout rules for `device.createBuffer`) is **not**
   this function's job — that requires WGSL struct-layout rules and belongs with F2.3's actual GPU
   allocation, which needs precise layout anyway. `resolveBufferSizes` only resolves the
   already-declared `element-count`, mirroring how `resolveTargetSizes` only resolves already-declared
   pixel dimensions — neither function invents information the target's own `size` value doesn't
   already carry.
6. **Materialize `ResourceInstance`s from a real `GraphDocument`** — the piece F2.1's brief deferred
   here. Lives in `@world-lab/graph` (document/primitive-walking is a graph-package concern, not a
   runtime-webgpu one), next to F2.1's `resources.ts`, following `executionRoots.ts`'s exact pattern:
   ```ts
   // packages/graph/src/resources.ts, extended
   export function collectResourceInstances(doc: GraphDocument): ResourceInstance[] {
     const instances: ResourceInstance[] = [];
     for (const node of doc.nodes) {
       const implementation = getPrimitive(node.primitive)?.implementation;
       if (implementation?.kind === 'resource') {
         instances.push({ id: node.id, ...implementation.template });
       }
     }
     return instances;
   }
   ```
   Returns the full `ResourceInstance` union (buffer/texture/sampler) — unlike `ResourceTarget`,
   nothing here excludes samplers, since a sampler can legitimately be a named `ResourceInstance`
   that a `ResourceBinding` refers to even though it's never a pass write-target.
   **Pairing a materialized instance with a concrete `ResourceSize` to form a full `ResourceTarget`
   is out of scope** — that requires resource-primitive-specific size derivation (e.g. reading a
   node's `params.elementCount`) that doesn't exist as a contract yet, since no real `kind:'resource'`
   primitive is registered anywhere today. Gate tests exercise this with a synthetic test primitive,
   matching F1.3's `test.compatibilitySink` precedent.
7. **Update `graphFramePlan.ts`'s `buildIndependentPassGraph`** to construct the renamed/generalized
   `ResourceTarget` shape (`{ id, shape: { kind: 'texture', ... }, lifetime: { kind: 'transient' },
   size: { kind: 'screen-relative', scale: 1 } }` in place of the old `{ id, format, size }`
   literal). This is mechanical fallout of step 1's rename, not a behavior change — the function
   still produces the same zero-cross-pass-read, screen-relative rgba8unorm targets it does today.
   Wiring this (or anything else) into real GPU execution is F2.4's job, not this milestone's.

## Gate

1. A `PassGraph.targets` array containing both a buffer-shaped `ResourceTarget` (`lifetime: {kind:
   'history', slots: 2}`, `size: {kind: 'element-count', count: 256}`) and a texture-shaped one in
   the same list type-checks and round-trips through `buildPassOrder` — the direct proof both kinds
   coexist in one generic model, not two parallel systems.
2. A test proves a sampler-shaped `TypeRef` fails to type-check as either `ResourceTarget` variant's
   `shape` (an `@ts-expect-error` test, matching F2.1's Gate item 1 convention exactly) — the
   concrete proof samplers are excluded from frame-graph targets at the type level, not just by
   convention.
3. A test proves a buffer shape paired with a `screen-relative`/`fixed` size, and a texture shape
   paired with an `element-count` size, both fail to type-check as `ResourceTarget` (two
   `@ts-expect-error` cases) — the direct regression proof for the shape/size correlation defect:
   the union structurally forbids the mismatch rather than merely documenting it.
4. `order.test.ts`'s existing three cases are updated for the field renames, and the chain-ordering
   case's feedback assertion is **corrected, not merely renamed**: with `B` (the display target)
   given no explicit `history` lifetime, `result.feedbackTargets` must **not** contain `'B'` —
   proving the old unconditional display-inclusion is genuinely gone, not just relabeled. The other
   two cases (previousFrame/`version` self-read as feedback, read-write-same-pass rejection) keep
   identical observable behavior, with the self-read fixture's target given an explicit `history`
   lifetime (now required — `lifetime` is mandatory, not optional, on `ResourceTarget`).
5. A new test proves a buffer-shaped, `history`-lifetime target with a `version: 'previous'` read
   correctly appears in `collectFeedbackTargets`'s output — the concrete generalization proof that
   feedback detection isn't texture-only, using a buffer instead of a texture for the first time.
6. A new test proves a `persistent`-lifetime target (texture or buffer, single allocation, no
   ping-pong) does **not** appear in `feedbackTargets` even though it survives across frames — the
   direct regression test for the persistence/history conflation defect, and the concrete check
   that F2.3 will only be told to double-buffer what genuinely needs two physical slots.
7. A test proves `computeLifetimes` still extends `graph.display`'s `lastRead` to the end of the
   pass order regardless of its lifetime — confirming within-frame display retention is preserved
   exactly as before, unaffected by `collectFeedbackTargets`'s correction (the two concerns are
   independent, per the Revision note).
8. A new test proves `validatePassGraph` rejects a `version: 'previous'` read against a
   `transient`-lifetime target and against a `persistent`-lifetime target (two cases, both invalid)
   with `{ kind: 'invalid-history-read' }`, and accepts the identical read shape against a
   `history`-lifetime target (one more case, valid) — proving the new rule fires exactly when it
   should and not otherwise.
9. A test proves `Pass.bindings` is optional and inert: two otherwise-identical `PassGraph`s, one
   with `bindings` populated on every pass and one without, produce identical `buildPassOrder`
   output (`order`, `feedbackTargets`, `lifetimes` all equal) — the concrete proof the reservation
   doesn't perturb the ordering algorithm, now that the field actually exists.
10. A test proves `resolveTargetSizes` resolves only texture-shaped targets from a mixed-shape
    `PassGraph`, and the new `resolveBufferSizes` resolves only buffer-shaped targets from the same
    mixed graph — each ignoring the other's targets, run against one shared fixture graph so the
    split is proven directly rather than in isolation.
11. A test registers a test-only `kind: 'resource'` primitive (e.g. `test.resourceInstance`,
    following `test.compatibilitySink`'s precedent) with a fixed `ResourceTemplate`, builds a
    `GraphDocument` with two nodes using that primitive, and proves `collectResourceInstances`
    returns two `ResourceInstance`s with different `id`s (matching each `node.id`) and identical
    `shape`/`lifetime` — the end-to-end proof of F2.1's ownership split working through a real
    document, not just at the isolated-type level F2.1's own gate proved.
12. `graphFramePlan.test.ts` passes unchanged in behavior after `buildIndependentPassGraph`'s literal
    is updated to the new `ResourceTarget` shape — proving the mechanical update didn't change what
    the zero-dependency preview path produces.
13. `check` **and** `test` green for `graph`, `runtime-webgpu`, full workspace (clear every
    `packages/*/dist` first, per `AGENTS.md`'s documented stale-dist issue).

## Out of scope

GPU resource allocation, bind-group layout derivation, byte-size/alignment computation for
`device.createBuffer` (all F2.3). Wiring `GraphFrameExecutor`/`PipelineGraphExecutor` to actually
call `buildPassOrder`/`resolveTargetSizes`/`resolveBufferSizes` (F2.4 — today's independent-only
bypass is untouched by this milestone). Deriving a target's `size` from a real resource primitive's
node params (no such primitive is registered yet; this milestone only defines the shape and a
document-walking function over whatever gets registered later). Cross-validating `Pass.bindings`
against `reads`/`writeTarget` for consistency (deferred to F2.3/Foundation 3, once there's a
concrete consumer to define the real relationship). Multi-target passes (`Pass.writeTarget` stays
singular — not called for by the plan). N-deep (>2 slot) history. Any change to Foundation 1's
milestones, F2.1's landed contract, or this document's own Foundation 2 sequencing beyond this one
milestone.

## Handoff

→ `frameGraph/types.ts` is fully generalized to F2.1's resource union as a discriminated union
(buffer and texture targets coexist in one `PassGraph.targets` list, each with a size shape that
structurally matches its own resource shape; sampler-shaped targets are excluded at the type level);
the previous-frame-read mechanism is generalized to `version` and newly validated against the
resource actually having ping-pong capacity (`history` lifetime) rather than silently accepting a
read that would alias an in-flight write; `collectFeedbackTargets` means exactly "needs physical
double-buffering" and only `history`-lifetime targets satisfy that, with within-frame display
retention left entirely to `computeLifetimes`, where it already lived; `Pass.bindings` has a
reserved home without a premature consumer; `collectResourceInstances` proves F2.1's
template/instance split materializes correctly from a real document. Once this lands: F2.3 (runtime
resource realization — actually allocating WebGPU buffers and textures from these descriptors,
deriving bind-group layouts and usage flags, physically double-buffering exactly what
`collectFeedbackTargets` flags and single-allocating everything else `lifetime.kind !== 'transient'`
marks persistent) is next — request its own contract at that point.
