# Brief — F3.6.5: vertex-kernel primitive with real varying output

**Type:** new real primitive, real varying dataflow, fifth milestone of the F3.6 pipeline-kernels
sequence — the largest F3.6 milestone so far; if the implementer finds it too large to land as one
stage commit, split along Fix steps 1-4 (varying plumbing) vs. 5-6 (the primitive + paired proof)
and say so rather than cutting scope silently · **Package:** `@world-lab/graph` (new primitive,
`procedural.uv` override) and `@world-lab/runtime-webgpu` (`emitGraphEval` extensions,
`vertexKernelPosition.ts` extension, `kernelFragment.ts` extension) · **Depends on:** F3.6.1
(`29a3208`), F3.6.2 (`9183107`), F3.6.3 (`a26d8fb`), F3.6.4 (`3fcc253`), all landed · **Design
authority:** [f3.6-pipeline-kernels-design.md, (B) and (K)](../f3.6-pipeline-kernels-design.md),
recommendation B1 · [f3.6.4-vertex-kernel-invocation-design.md, section
(K)](../f3.6.4-vertex-kernel-invocation-design.md) (explicitly deferred this milestone's own
scoping here) · **Contract author:** Opus · **Recommended executor:** Cursor or Codex · **Status:**
ready to route

## Context

Fifth of F3.6's tentative sequence, and the one every prior milestone has been building toward:
`stage.vertex`/`stage.fragment` connect through a single, *structural*, opaque `varyings`-typed port
— nothing today produces or consumes a real, graph-computed varying value end to end.
`assertVaryingsMatch` (F3.2, landed) and `varyingsStructWgsl` (F3.2, landed) have never had a real
caller. F3.6.4 proved a vertex kernel's *position* output can be graph-authored and compiles; this
milestone proves the same for a *varying* — B1's recommendation (one `emitGraphEval` call per
output) applied for real, for the first time.

**Three real gaps surfaced by grounding this contract, none of them resolved by any prior
milestone — flagged here, not discovered mid-implementation:**

1. **No `vec2f` variant of `emitGraphEval` exists.** `emitGraphScalarEval`/`emitGraphVec3Eval`/
   `emitGraphVec4Eval` are the only three exported wrappers (`emitGraphEval.ts:200-222`) — a
   `vec2f`-typed varying (the natural, minimal first case — a UV) has no way to be emitted at all
   today. Fix step 1.
2. **`procedural.uv`'s `emitHostInput` branch has no override, unlike every other invocation-context
   host input.** Confirmed by direct read (`emitGraphEval.ts:90-92`): `if (binding.context ===
   'invocation' && binding.key === 'uv') return \`let ${variable} = vec2<f32>(u, v);\`;` — no
   `opts?.uvExpr` fallback, unlike `metricPosition`'s sibling branch two lines below it. Reusing
   `procedural.uv` for "the current varying UV, wherever it's sourced from" (mirroring
   `metricPosition`'s own already-reviewed reuse for vertex-grid position — same shape of idea: a
   host-input primitive that already means "the current surface coordinate," now given a second,
   assembly-supplied source) needs this override added first. Fix step 2.
3. **`stage.fragmentKernel`'s own assembly function (`assembleKernelFragmentModuleAsync`, F3.6.3)
   has never been called with `varyings` at all** — `assembleStageEntry`'s fragment branch changes
   *shape* when varyings are present (`fn fs_main(input: VSOut)` instead of `fn
   fs_main(@builtin(position) position: vec4f)`, confirmed `stageEntry.ts:206-214`) — this doesn't
   break `buildKernelGraphEvalFn`'s existing wrapper (its own parameter names are caller-chosen,
   independent of the entry's own parameter shape — `callArgs` just needs to become
   `['input.position', 'input.uv']` instead of `['position']`), but nothing in `kernelFragment.ts`
   exercises this today. Fix step 5.

**This milestone does not wire `PipelineGraphExecutor`, `GraphFrameExecutor`, or the editor** —
matching F3.6.3's and F3.6.4's own precedent exactly (both proved their real primitive/mechanism via
a standalone, direct real-device test, not through the full executor). `stage.vertex`/
`stage.fragment` stay completely untouched; a real vertex+fragment kernel *pair* is proven to
compile, link, and render correctly via a direct call to this milestone's own assembly functions,
not through any document-driven pipeline path. Wiring `PipelineGraphExecutor`'s vertex-side branch
(there is currently none — `resolveVertexAssembly` is called unconditionally by every execution
path, regardless of what `stage.vertex`-role primitive a document actually uses) is F3.6.6's job,
which needs that wiring anyway to make a bundled sample reachable from the editor. No manual/visual
gate here for the same reason F3.6.3/F3.6.4 had none.

**Deliberately narrow scope for "real varying output": exactly one, named `uv: vec2f`.** Not a
general N-varyings mechanism. `assembleStageEntry`'s `opts.varyings: VaryingDecl[]` is already
caller-suppliable as an arbitrary array — nothing here prevents a future milestone from generalizing
to more varyings or other types — but proving *one* real, named, typed varying flows correctly end
to end (vertex-computed → interpolated → fragment-consumed, with `assertVaryingsMatch` actually
validating the pairing) is the actual, narrowly-scoped goal, matching every prior F3.6 milestone's
"prove one real thing, don't generalize speculatively" discipline.

## Verified current state

- **`emitGraphEval.ts`'s three exported wrappers** (`emitGraphScalarEval`/`emitGraphVec3Eval`/
  `emitGraphVec4Eval`, lines 200-222), all thin calls into the shared, module-private `emitGraphEval`
  function with a hardcoded `expectedOutputType`. A fourth, `emitGraphVec2Eval`, is a mechanical
  addition of the identical shape.
- **`procedural.uv`'s exact current registration** (`packages/graph/src/primitives/uv.ts`):
  `binding: {context:'invocation', key:'uv'}`, `outputs:[{name:'uv', dataType:'vec2f'}]`,
  `evalCPU` reads `ctx.procedural?.uv`. Widely used elsewhere in this codebase for field/terrain
  evaluation (unaudited exhaustively — same open-risk caveat F3.6.4's design document already
  raised for `metricPosition`'s reuse applies here identically; the override is per-call, not
  global, so existing callers supplying no `uvExpr` are unaffected).
- **`assembleStageEntry`'s fragment branch, both shapes** (`packages/compiler/src/stageEntry.ts:206-214`):
  with `varyings.length === 0` (F3.6.2/F3.6.3's current usage), `fn fs_main(@builtin(position)
  position: vec4f)`; with `varyings.length > 0`, `fn fs_main(input: ${structName})` where
  `${structName}` (default `'VSOut'`) always includes `@builtin(position) position: vec4f` as its
  own first field (`varyingsStructWgsl`, `stageEntry.ts:133-141`) — `input.position` replaces the
  bare `position` identifier, `input.uv` (or whatever varying name) becomes newly available.
  `callArgs` is fully caller-controlled text passed verbatim to the generated call — `buildKernelGraphEvalFn`'s
  own wrapper-function parameter names don't need to match `input.<field>`'s own names, only the
  `callArgs` strings supplied at the `assembleStageEntry` call site need to reference them correctly.
- **F3.6.4's exact landed `assembleVertexKernelPositionModuleAsync`**
  (`packages/runtime-webgpu/src/vertexKernelPosition.ts`) — position-only today (`varyings` omitted
  from its own `assembleStageEntry` call, defaulting to `[]`), plus the `graphModuleSources` helper
  a post-landing review flagged as missing `generateWgsl`'s cycle-detection guard (`visiting:
  Set<string>`) — this milestone's Fix step 4 resolves that exact, previously-flagged gap by
  replacing it with a direct `compileGraph` call, not by patching in the missing guard.
- **F3.6.3's exact landed `assembleKernelFragmentModuleAsync`/`KernelFragmentAssembly`/
  `executeKernelFragment`** (`packages/runtime-webgpu/src/consumers/kernelFragment.ts`) — no
  `varyings` parameter anywhere; `assembleStageEntry` called with `callArgs: ['position']` only.
- **`stage.vertex`/`stage.fragmentKernel`'s exact current port shapes**
  (`packages/graph/src/primitives/pipeline/index.ts`): `stage.vertex` — `inputs:[{mesh:geometry}]`,
  `outputs:[{varyings:varyings}]`; `stage.fragmentKernel` — `inputs:[{varyings:varyings},
  {color:vec4f}], outputs:[{texture:texture}]`. Both ports named exactly `varyings`/`mesh`/`color`/
  `texture` per F3.6.1's port-name-contract — `stage.vertexKernel` must match `stage.vertex`'s
  `mesh`/`varyings` port names exactly (plus its own new inputs) for `planPipelineGraph`'s
  `requireEdge` calls to accept it structurally; extra ports beyond the two required ones are fine
  (`requireEdge` only checks specific edges exist, not that no other ports do).
- **`bindingDeclKindForTemplate`/binding machinery is irrelevant here** — varyings are not
  `KernelBindingTemplate`-shaped resources; they flow through `assembleStageEntry`'s dedicated
  `varyings`/`callArgs` mechanism entirely, confirmed by direct read of `VaryingDecl`'s shape
  (`name`, `wgslType` only — no `resourceKind`/`access`/`binding` fields at all).

## Fix

### 1. `emitGraphVec2Eval`

`packages/runtime-webgpu/src/emitGraphEval.ts`, alongside the three existing wrappers:

```ts
export function emitGraphVec2Eval(
	doc: GraphDocument,
	output: PortRef,
	opts?: EmitGraphEvalOptions
): EmittedGraphEval {
	return emitGraphEval(doc, output, 'vec2f', 'emitGraphVec2Eval requires vec2f output', opts);
}
```

### 2. `procedural.uv` override

`EmitGraphEvalOptions` gains `uvExpr?: string`. `emitHostInput`'s `uv` branch
(`emitGraphEval.ts:90-92`):

```ts
if (binding.context === 'invocation' && binding.key === 'uv') {
	return `let ${variable} = ${opts?.uvExpr ?? 'vec2<f32>(u, v)'};`;
}
```

No change to `procedural.uv`'s own registration, `evalCPU`, or any other `emitHostInput` branch.

### 3. Extend `vertexKernelPosition.ts` for a real varying

`assembleVertexKernelPositionModuleAsync`'s input gains an optional second graph output:

```ts
export interface VertexKernelPositionInput {
	graph: GraphDocument;
	output: PortRef;
	/** Optional vec2f-producing graph output for the 'uv' varying. */
	uvOutput?: PortRef;
	geo?: PipelineGeometryParams;
	resolver: WgslModuleResolver;
}

export interface VertexKernelPositionAssembly {
	code: string;
	vertexCount: number;
	/** Empty when uvOutput was not supplied. */
	varyings: VaryingDecl[];
}
```

**Correction, caught by a pre-routing review of an earlier draft: `input.uvOutput`'s emission needs
the same `positionExpr` override the position output already gets, not a bare
`emitGraphVec2Eval(input.graph, input.uvOutput)` call.** `procedural.metricPosition`'s
`emitHostInput` branch (Fix step 3's own sibling, F3.6.4-landed) only substitutes `opts?.positionExpr`
when it's supplied — omitted, it falls back to `vec3<f32>(u, v, 0.0)`, and `u`/`v` are
field-evaluation-context variables that do not exist as identifiers anywhere in this milestone's
vertex wrapper functions (only `vid`/`iid` are ever in scope, per `buildVertexPositionEvalFn`'s own
signature). A `uvOutput` graph rooted in `procedural.metricPosition` without this override would
emit WGSL referencing undefined `u`/`v` — this milestone's own Gate item 5 (the varying's value
must *visibly, measurably* affect rendered output, not just compile) cannot be satisfied by a
uniform/broken fallback anyway, so the fix and the requirement point the same direction: supply the
identical `positionExpr: planeGridPositionExpr(geo)` to `emitGraphVec2Eval` that `output`'s own
`emitGraphVec3Eval` call already receives.

```ts
const emitted = emitGraphVec3Eval(input.graph, input.output, { positionExpr });
const uvEmitted = input.uvOutput
	? emitGraphVec2Eval(input.graph, input.uvOutput, { positionExpr })
	: undefined;
```

**This milestone's own `uvOutput` test graph must be rooted in `procedural.metricPosition`** (e.g.
extracting the grid position's `xy` via an existing vector-component extractor into the `uv`
output), sharing the upstream node with `output` — matching Gate item 3's own "sharing upstream
nodes" requirement, which was already the intent, just not correctly wired through here.
`procedural.uv` on the vertex side is explicitly out of scope for this milestone (see Out of
scope) — it has no equivalent override mechanism, and `procedural.metricPosition` already supplies
a real, working, per-vertex source, so nothing requires solving that separately here.

Build a second wrapper function analogous to `buildVertexPositionEvalFn` but for `uv`:

```ts
function buildVertexVaryingEvalFn(name: string, body: string[], resultExpr: string): string {
	return `fn graph_eval_${name}(vid: u32, iid: u32) -> vec2f {
${body.map((line) => `\t${line}`).join('\n')}
\treturn ${resultExpr};
}`;
}
```

pass `varyings: [{name:'uv', wgslType:'vec2f'}]` and extend `outputFns`/`output` handling in the
`assembleStageEntry` call to include `uv: 'graph_eval_uv'` (`assembleStageEntry`'s vertex branch
already assigns `out.<varying> = <varyingFn>(<callArgs>)` per varying — `stageEntry.ts:222-230` —
this is calling existing, tested machinery, not extending it). Return the resolved `varyings` array
alongside `code`/`vertexCount` — F3.6.5's own fragment-side extension (Fix step 5) needs to know
which varyings exist to build a matching `VaryingDecl[]` for `assertVaryingsMatch`.

### 4. Replace `graphModuleSources` with `compileGraph`

Resolves the gap F3.6.4's post-landing review flagged, not a new concern. Remove
`graphModuleSources` entirely. Reuse the exact pattern `assembleKernelFragmentModuleAsync`/
`assembleFullscreenFragmentModuleAsync` already establish: call `compileGraph(graph, resolver,
{consumers: [{type:'image', id:'vertex-kernel-position', stage:'vertex', outputs: [...]}]})` to get
`consumerShader.code` (the graph's own callable-function library, module-resolved and
cycle-detection-guarded by `generateWgsl`'s already-tested logic), then prepend
`planeModule.source.trim()` and the wrapper function(s) on top, exactly mirroring how
`kernelFragment.ts` combines `consumerShader.code` with `buildKernelGraphEvalFn`'s own output today.
`outputs` in the consumer descriptor needs both `position` and, when present, `uv` — `compileGraph`
compiles whatever the *declared* `doc.outputs` reference, so the graph passed to
`assembleVertexKernelPositionModuleAsync` needs both outputs declared (matching how a real pipeline
document would declare them) — verify this against `compileGraph`'s actual behavior for multiple
declared outputs sharing overlapping upstream nodes (not exhaustively traced here; a real,
non-mocked test proving both `position` and `uv` compile together, sharing common upstream nodes
without duplication or error, is Gate item 2 below, not assumed to just work).

### 5. Extend `kernelFragment.ts` to accept varyings

**Blocking correction, caught by a pre-routing review of an earlier draft — this step as first
written would produce a fragment module with two conflicting `struct VSOut` definitions.**
`assembleKernelFragmentModuleAsync` (`kernelFragment.ts:174` in the current landed file) always
calls `resolveVertexAssembly(graph, resolver)` and includes its returned `vertexWgsl` — which itself
contains a hand-written, **position-only** `struct VSOut { @builtin(position) position: vec4f, };` —
in the final assembled `code`. That's correct and necessary for every existing (non-varying) caller,
which is fully self-contained (no external vertex module to pair with). But `assembleStageEntry`'s
fragment branch, when `varyings.length > 0`, does **not** itself emit `struct VSOut` — it only
references it in `fn fs_main(input: VSOut)`'s signature (confirmed by direct read,
`stageEntry.ts:206-214` — no call to `varyingsStructWgsl` on the fragment side at all). The struct is
expected to come from **whatever else the caller concatenates in** — confirmed by
`kernelVaryingsDeviceCompile.test.ts`'s own established, working pattern: it builds vertex and
fragment `ConsumerShader`s **separately**, then joins `` `${vertex.code}\n\n${fragment.code}` `` into
**one** combined string and compiles **one** `device.createShaderModule` from it (not two separate
modules) — the struct, emitted once by the vertex side, is shared by both entry points within that
single module.

So: when `varyings` is supplied, `assembleKernelFragmentModuleAsync` must **not** call
`resolveVertexAssembly` at all — its position-only struct would collide with the paired
vertex-kernel module's own `{position, uv}`-shaped struct (from `varyingsStructWgsl`, called inside
`assembleVertexKernelPositionModuleAsync`'s own `assembleStageEntry` call) once the two are
concatenated, a duplicate-declaration compile error. The returned assembly's `code` becomes
fragment-stage-only (the graph's own function library plus `stage.code`, no vertex WGSL, no struct
of its own) — pairing code (this milestone's own Gate item 4/5 tests, and eventually F3.6.6) is
responsible for concatenating it with a separately-assembled vertex-kernel module's `code`, exactly
mirroring `kernelVaryingsDeviceCompile.test.ts`'s existing approach. `vertexCount` is meaningless
for a fragment-only assembly in this mode — take it from the paired vertex-kernel assembly's own
`vertexCount` instead when driving a real draw call.

```ts
const paramsStruct = hasGraphParams ? buildParamsStructWgsl(emitted.params) : '';
const codeParts = input.varyings
	? [paramsStruct, stage.code]
	: [paramsStruct, (await resolveVertexAssembly(graph, resolver)).vertexWgsl, stage.code];
const code = codeParts.filter(Boolean).join('\n\n');
```

The rest of this step, unchanged from the earlier draft:

- `assembleStageEntry`'s call gains `varyings: input.varyings`, and `callArgs` becomes
  `['input.position', ...input.varyings.map((v) => \`input.${v.name}\`)]` instead of `['position']`.
- `buildKernelGraphEvalFn`'s wrapper signature gains matching parameters:
  `fn graph_eval_${outputName}(position: vec4f, uv: vec2f) -> vec4f` (built dynamically from
  `input.varyings`, not hardcoded to `uv` specifically — the function should accept the resolved
  `VaryingDecl[]` and build parameter names/types from it, even though this milestone's own real
  usage only ever supplies one).
- The emitted graph body (`emitGraphVec4Eval` call) needs `uvExpr` supplied when the graph
  references `procedural.uv` — `{uvExpr: 'uv'}` (the wrapper's own local parameter name, matching
  Fix step 2's override mechanism, analogous to how `host.fragCoord`'s `fragCoordExpr` already
  defaults to `'position.xy'`, the wrapper's own local parameter).

`executeKernelFragment`/`KernelFragmentInput` do **not** need changes — varyings are purely an
assembly-time (WGSL text) concern; nothing about binding resolution, bind-group construction, or
resource execution changes.

### 6. Register `stage.vertexKernel`

`packages/graph/src/primitives/pipeline/index.ts`, alongside `stage.vertex`/`stage.fragment`/
`stage.fragmentKernel`:

```ts
{
	id: 'stage.vertexKernel',
	category: 'stage',
	inputs: [
		{ name: 'mesh', dataType: 'geometry', metadata: { semantic: 'geometry-resource' } },
		{ name: 'position', dataType: 'vec3f', metadata: { semantic: 'vertex-position' } },
		{ name: 'uv', dataType: 'vec2f', metadata: { semantic: 'vertex-uv-varying' } }
	],
	outputs: [{ name: 'varyings', dataType: 'varyings', metadata: { semantic: 'fragment-varyings' } }],
	params: noParams,
	implementation: { kind: 'kernel', stage: 'vertex', bindings: [] },
	metadata: {
		description: 'Graph-driven vertex kernel stage with a graph-authored position and uv varying.',
		pure: true,
		deterministic: true,
		role: 'pipelineStage',
		pipelineStageKind: 'vertex'
	}
}
```

`bindings: []` for the same reason `stage.fragmentKernel` declares it — nothing about this
primitive's own registration is document-derived; its real capability comes from the graph wired
into its `position`/`uv` inputs, resolved by Fix steps 3-4 at assembly time, not by anything
`resolveKernelBindings`-shaped. `mesh` stays structurally required (matching `stage.vertex`'s own
port, per F3.6.1's port-name-contract) even though — like `stage.vertex` — this primitive's own
assembly logic doesn't read the wired mesh *value*, only requires the edge to exist.

## Gate

1. `emitGraphVec2Eval` requires `vec2f` output, mirrors the other three wrappers' exact error-message
   pattern and behavior — a direct unit test analogous to existing coverage for the other three.
2. `procedural.uv`'s new `uvExpr` override: a direct `emitGraphEval.test.ts`-style test proves a
   graph wired to `procedural.uv` emits the overridden expression when `uvExpr` is supplied, and the
   unchanged `vec2<f32>(u, v)` fallback when it isn't — regression proof this doesn't silently change
   existing callers.
3. `assembleVertexKernelPositionModuleAsync` with `uvOutput` supplied, sharing upstream nodes with
   `output` (e.g. both derived from the same `procedural.metricPosition`/graph-authored expression,
   not fully independent subgraphs) compiles cleanly via `compileGraph` with no duplicate-declaration
   or module-resolution errors — the specific case Fix step 4's note flagged as unverified, not
   assumed.
4. **Real-device paired compile/link proof, the core deliverable:** a `stage.vertexKernel` graph
   with `position` and `uv` both rooted in the same `procedural.metricPosition` node (per Fix step
   3's correction — `uv` extracts the grid position's `xy`, both outputs sharing the
   `positionExpr`-supplied `plane_grid_position(...)` expression, giving a real, per-vertex,
   non-uniform value, not the broken/uniform fallback) paired with a `stage.fragmentKernel` graph
   consuming that `uv` via `procedural.uv` (with `uvExpr: 'uv'` supplied, per Fix step 5).
   `assertVaryingsMatch(vertexVaryings, fragmentVaryings)` (F3.2, its first real caller) asserted to
   pass with no mismatch. Assemble both sides, **concatenate their `code` into one combined string**
   (`` `${vertexAssembly.code}\n\n${fragmentAssembly.code}` ``, mirroring
   `kernelVaryingsDeviceCompile.test.ts`'s own established pattern exactly — not two separate
   modules; per Fix step 5's correction, the fragment assembly's own `code` has no `struct VSOut` of
   its own to conflict with the vertex side's), then **one** `device.createShaderModule`,
   `getCompilationInfo()` errors asserted empty, then a real `createRenderPipelineAsync` with
   `vertex: {module, entryPoint:'vs_main'}` and `fragment: {module, entryPoint:'fs_main'}` both
   referencing the **same** compiled module — `expect(pipeline).toBeDefined()`.
5. A second real-device test renders the paired pipeline into a real target texture and reads back
   pixels, confirming the `uv` varying's value visibly, measurably affects the rendered output (e.g.
   the fragment kernel colors based on `uv`, producing a non-uniform image across the target) — not
   merely that it compiles, mirroring the "prove it's live, not just present in WGSL text" standard
   every prior F3.6 milestone's real-device gate has held to.
6. `assertVaryingsMatch` genuinely rejects a mismatched pair — a test pairing this milestone's real
   vertex kernel (declaring `uv: vec2f`) against a hand-built fragment `VaryingDecl[]` naming a
   *different* varying or type, asserting `assertVaryingsMatch` throws — proof the check is
   load-bearing here, not merely called and ignored.
7. `getPrimitive('stage.vertexKernel')` registration correctness (port names/types, `bindings: []`,
   `pipelineStageKind: 'vertex'`, `role: 'pipelineStage'`) — `stage.vertex`/`stage.fragment`/
   `stage.fragmentKernel` confirmed byte-identical to before this milestone, extending the same
   registration test F3.6.3 already established.
8. Regression: every existing test in `fullscreenFragment.test.ts`, `kernelFragment.test.ts`,
   `vertexKernelPosition.test.ts`, `emitGraphEval.test.ts`, `pipeline.test.ts`, `pipelineGraph.test.ts`
   passes unmodified except where Fix steps 1-5 add genuinely new, additive parameters with safe
   defaults (`varyings`/`uvOutput`/`uvExpr` all optional, all default to today's exact behavior when
   omitted) — direct proof this milestone doesn't change `stage.vertex`/`stage.fragment`/the
   fragment-kernel-without-varyings path's own behavior.
9. `check` **and** `test` green for `graph`, `runtime-webgpu`, full workspace (clear every
   `packages/*/dist` first). Real-device tests independently confirmed to execute, not silently
   skip, when re-verified (`--reporter=verbose` scoped to the package).

## Out of scope

Wiring `PipelineGraphExecutor`'s vertex-side branch, `GraphFrameExecutor`, the editor preview UI, or
a bundled sample — F3.6.6's job entirely, which needs this wiring anyway to make anything reachable
from the editor; this milestone's proof is standalone, matching F3.6.3's and F3.6.4's own precedent.
Generalizing to more than one varying, or to varying types beyond `vec2f` — proven narrowly with
`uv: vec2f` only; the mechanism (`assembleStageEntry`'s already-array-shaped `varyings` option)
supports more without further design work, but this milestone doesn't build or test more than one.
The raw `u32`-index consumer gap F3.6.4 flagged (no coercion path from `host.vertexIndex`'s output
into existing f32-based graph math) — still not resolved here, and not needed here either; Fix
step 3's `procedural.metricPosition`-rooted `uv` graph gives Gate item 4/5's proof a real,
non-uniform per-vertex source without touching `host.vertexIndex`/`host.instanceIndex` at all.
`procedural.uv` on the vertex side — no vertex-context override mechanism exists for its
`emitHostInput` branch (unlike `metricPosition`'s `positionExpr`), and nothing in this milestone
needs one; `procedural.metricPosition` already provides a real, working per-vertex source for the
one varying this milestone proves. Any change to `stage.vertex`/`stage.fragment`'s own behavior,
registration, or assembly path. Any change to F3.6.1-4's landed contracts beyond the explicitly-scoped
extension points (Fix steps 3-5, all purely additive with safe defaults).

## Handoff

→ A real, additive vertex-kernel primitive (`stage.vertexKernel`) exists, pairs with F3.6.3's real
fragment-kernel primitive through a genuinely graph-computed, typed varying — not the structural,
opaque pass-through every prior milestone left untouched — validated by `assertVaryingsMatch`'s
first real call, and proven live via a real-device render where the varying's value visibly affects
output pixels, not merely compiles. `emitGraphVec2Eval` and `procedural.uv`'s `uvExpr` override are
now available for any future milestone needing a second real varying. F3.6.6 (the bundled,
editor-visible proof sample) is next: it needs to wire `PipelineGraphExecutor`'s vertex-side branch
(mirroring F3.6.2's fragment-side branch, currently entirely absent — `resolveVertexAssembly` runs
unconditionally today regardless of which vertex-stage primitive a document actually uses) before a
real `stage.vertexKernel`+`stage.fragmentKernel` pipeline document can render through the normal
execution path at all. Request F3.6.6's own contract when picking it up; it is not included here.
