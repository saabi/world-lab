# Brief — F3.1: kernel & binding type algebra

**Type:** type-system extension, first milestone of the Foundation 3 generic-kernels sequence ·
**Package:** `@world-lab/graph`, `@world-lab/compiler` · **Depends on:** Foundation 2, complete
(F2.1–F2.5 landed) — specifically F2.1's `ResourceAccess`/`ResourceBinding`
(`packages/graph/src/implementation.ts`) and F2.2's `Pass.bindings`
(`packages/runtime-webgpu/src/frameGraph/types.ts`) — and F1.3's `PrimitiveImplementation`
(`{ kind: 'kernel'; stage: ShaderStage }` already reserved as a placeholder) · **Design authority:**
[foundation-3-generic-kernels-plan.md](../foundation-3-generic-kernels-plan.md) ·
[elemental-webgpu-architecture-review.md, Foundation 3](../elemental-webgpu-architecture-review.md#foundation-3-generic-kernels)
· **Contract author:** Opus, incorporating two pre-routing review rounds (see Context) ·
**Recommended executor:** Cursor or Codex · **Status:** ready to route

## Context

First of Foundation 3's sequence (F3.1 → F3.2 → F3.3 → F3.4 → F3.5). **This milestone is pure
types and pure resolution functions — no GPU allocation, no runtime dispatch, no `GPUBindGroup`
construction.** It gives `{ kind: 'kernel'; stage: ShaderStage }` (currently a bare placeholder,
`packages/graph/src/implementation.ts:71`) its real declared shape.

This contract was shaped by two pre-routing review rounds, both incorporated directly below rather
than left as later revision rounds:

**Round 1:**

1. **Graph-level binding templates stay independent of the compiler's `BindingDecl`** — two
   separate types in two separate packages, joined later by a pure, one-directional translation
   function, never merged into one shape. `BindingDecl` (`packages/compiler/src/stageEntry.ts`) is
   a WGSL-emission concern (it carries a literal `wgslType` string derived from a resolved
   resource's concrete shape); the graph-level template only knows abstract resource kind, not WGSL
   syntax. This also respects the actual dependency direction — see Verified current state.
2. **A pure function specifies exactly how a kernel's static, id-less binding templates map to
   per-pass `ResourceBinding.resourceId`** — this is the milestone's real deliverable, not just a
   type declaration. It mirrors F2.1's own id-less-template/id-bearing-instance split: F2.1 defined
   `ResourceTemplate`/`ResourceInstance` and deferred document-walking to F2.2; this milestone
   defines `KernelBindingTemplate` and a resolution function that takes a caller-supplied
   name→resourceId table, deferring *deriving that table from a real `GraphDocument`* to whichever
   later milestone actually walks one (not this one — see Out of scope).
3. **Each binding template carries five fields**: shader name, binding index, resource type,
   access, and stage visibility — enumerated exactly in Fix step 1 below.

**Round 2** (found two blocking issues in round 1's draft, plus one non-blocking gap):

4. **Blocking — resolution must not discard binding identity.** Round 1's `resolveKernelBindings`
   returned plain `ResourceBinding[]` (`{ resourceId, access }`), losing `name`/`binding`/
   `resourceKind`/`stages` on the way out. Any later milestone consuming the result (F3.4, wiring a
   resolved kernel into real frame execution) would need to zip the output back against the input
   templates by array index to recover which resolved entry is which — a fragile parallel-array
   reconstruction. Fixed by `ResolvedKernelBinding` (Fix step 1a): extends `ResourceBinding`, so it
   remains a valid `ResourceBinding` anywhere one is expected (e.g. `Pass.bindings`), while carrying
   the template metadata forward.
5. **Blocking — stage visibility was never checked against the owning kernel's own stage.** Round
   1's `KernelBindingTemplate.stages` was validated for internal consistency (non-empty, etc.) but
   nothing checked it against the `{ kind: 'kernel'; stage }` primitive that actually owns the
   binding — so a `{ stage: 'compute' }` kernel could declare a binding with `stages: ['fragment']`
   only, a binding its own compute entry point could never legally reference, and this would pass
   registration silently. Fixed by threading the owning kernel's `stage` into both
   `validateKernelBindingTemplates` and `resolveKernelBindings` (Fix steps 2 and 4) and requiring
   `template.stages.includes(ownerStage)`.
6. **Shader-name validation was missing.** `KernelBindingTemplate.name` had no format check — an
   empty string, a string that isn't identifier-shaped, or a reserved WGSL keyword would pass
   validation and only fail much later, opaquely, wherever it's finally interpolated into emitted
   WGSL. Added to `validateKernelBindingTemplates` (Fix step 2).

A separate, non-blocking correction to the Foundation 3 plan document itself (not this brief):
the plan's F3 sequence description claimed `assembleStageEntry`'s dead compute template "already"
emits a correct `@compute @workgroup_size(...)` entry. Direct read of
`packages/compiler/src/stageEntry.ts:63` shows it actually emits `@workgroupSize` (no underscore),
which is not valid WGSL — and `stageEntry.test.ts:45` asserts that same wrong string, so the bug is
enshrined in its own test, not merely a typo nobody noticed. This is corrected in the plan document
directly (not fixed in code here — it's dead code with zero runtime consumers today, and fixing it
is F3.3's job, since F3.3 is the milestone that gives the compute template its first real caller;
see that plan bullet's amended wording).

## Verified current state

- **`{ kind: 'kernel'; stage: ShaderStage }` has zero consumers anywhere**
  (`packages/graph/src/implementation.ts:71`), confirmed by grep — the only match for `'kernel'` in
  non-test code is the type declaration itself. `ShaderStage = 'vertex' | 'fragment' | 'compute'`
  (`implementation.ts:9`) is already the right shape and is not touched by this milestone.
- **`Pass.bindings?: ResourceBinding[]`** (`packages/runtime-webgpu/src/frameGraph/types.ts`,
  reserved by F2.2) **has exactly one consumer today**: `resolveBufferUsage`
  (`packages/runtime-webgpu/src/frameGraph/realize.ts:59`), for usage-flag inference only. Nothing
  today turns `Pass.bindings` into an actual `GPUBindGroup`. `ResourceBinding` itself
  (`implementation.ts:57-60`) is `{ resourceId: string; access: ResourceAccess }` — flat, with no
  name/slot field — confirming there is no existing mechanism linking a *named* binding to a
  resolved resource id; this milestone adds exactly that, one layer up.
- **`ResourceShape`** (`implementation.ts:41-44`) is already `Extract<TypeRef, { kind: 'buffer' |
  'texture' | 'sampler' }>` — this milestone's `resourceKind` field reuses `ResourceShape['kind']`
  directly rather than inventing a parallel three-value union.
- **`HostBinding.stages?: ShaderStage[]`** (`implementation.ts:21`) is an existing, working
  precedent for stage-visibility — used today by `host.fragCoord`
  (`stages: ['fragment']`, `packages/graph/src/primitives/host/fragCoord.ts:14`) — **but it is
  declared, not enforced**: confirmed by grep, `emitGraphEval.ts` never reads `binding.stages`
  anywhere. This milestone's `KernelBindingTemplate.stages` reuses the same field shape (a
  `ShaderStage[]`), but **required, not optional** (every resource binding must declare its
  visibility — unlike a host-input value, which may be legitimately stage-agnostic), and is
  actually validated at both registration time and resolution time (Fix steps 2 and 4). Retrofitting
  enforcement onto `HostBinding.stages`'s own pre-existing, separate gap is explicitly out of scope
  (see below) — this milestone closes the gap only for the new binding templates it introduces.
- **`BindingDecl.kind`** (`packages/compiler/src/stageEntry.ts:8`) is
  `'uniform' | 'storage-read' | 'texture' | 'sampler'` — **no read-write storage variant**. This is
  not hypothetical: F2.5 hit it directly. Its buffer-feedback sample's WGSL declares
  `var<storage, read_write> next: array<f32>`, and F2.5's own brief states explicitly it bypasses
  `assembleStageEntry` entirely "so there is exactly one place these bindings are declared"
  (`F2.5-foundation-2-proof.md`, Fix C, item 11) precisely because the generic path couldn't
  express it. `bindingVar` (`stageEntry.ts:36-40`) is the one place the WGSL address-space string is
  derived from `BindingDecl.kind` — confirmed by direct read, a three-way ternary with no
  read-write branch.
- **Package dependency direction is one-way and already enforced by the workspace**, confirmed via
  each package's `package.json`: `@world-lab/graph` depends only on `@world-lab/schema`;
  `@world-lab/compiler` depends on `@world-lab/graph` (never the reverse); `@world-lab/runtime-webgpu`
  depends on both. `KernelBindingTemplate` must live in `@world-lab/graph`; the pure translation to
  `BindingDecl` must live in `@world-lab/compiler`, which can import from `graph`.
- **`registerPrimitive`** (`packages/graph/src/registry.ts:98`) already performs structural
  validation at registration time (port type/dataType mismatch, implementation/wgsl mismatch,
  duplicate id — confirmed by direct read of the existing throw sites). This is the established,
  idiomatic place to add the same discipline for a `kind: 'kernel'` primitive's declared bindings,
  rather than only validating the first time someone calls a resolution function against a real
  document.
- **No `stage.compute` primitive exists**, and **`assembleStageEntry`'s compute template is dead
  code** — zero `GPUComputePipeline` creation or `dispatchWorkgroups` calls exist anywhere in
  `runtime-webgpu` (confirmed by grep). This milestone does not change that; it only gives kernels a
  real binding-declaration shape so F3.3 has something concrete to bind against later.

## Fix

### 1. `KernelBindingTemplate` — the graph-level, compiler-independent binding declaration

In `packages/graph/src/implementation.ts`, alongside the existing `ShaderStage`/`HostBinding`
declarations:

```ts
export type KernelResourceKind = ResourceShape['kind']; // 'buffer' | 'texture' | 'sampler'

export interface KernelBindingTemplate {
	/** Shader name — the WGSL variable this binding declares. */
	name: string;
	/** Author-declared binding index, local to this kernel's own resource-binding block.
	 *  Final absolute `@binding(n)` numbering — after any preceding host-uniform/param
	 *  bindings in an assembled module — is an assembly-time concern for whichever later
	 *  milestone assembles a complete WGSL module (not this one; see Out of scope). */
	binding: number;
	/** Resource type — reuses `ResourceShape`'s own kind union, not a parallel one. */
	resourceKind: KernelResourceKind;
	/** Read/write access. `sampler` bindings must be `'read'`; `texture` bindings must be
	 *  `'read'` until storage textures are supported (F2.3's own deferral) — enforced by
	 *  `validateKernelBindingTemplates` (Fix step 2), not just documented here. */
	access: ResourceAccess;
	/** Which stages this binding is visible in. Reuses `HostBinding.stages`'s field shape,
	 *  but **required**, not optional — every resource binding must declare its visibility. */
	stages: readonly ShaderStage[];
}
```

Extend `PrimitiveImplementation`'s kernel variant to carry its declared bindings:

```ts
export type PrimitiveImplementation =
	| ... // unchanged
	| { kind: 'kernel'; stage: ShaderStage; bindings: readonly KernelBindingTemplate[] };
```

### 1a. `ResolvedKernelBinding` — resolution output that keeps binding identity

Also in `implementation.ts`, alongside `ResourceBinding`:

```ts
export interface ResolvedKernelBinding extends ResourceBinding {
	name: string;
	binding: number;
	resourceKind: KernelResourceKind;
	stages: readonly ShaderStage[];
}
```

`resolveKernelBindings` (Fix step 4) returns `ResolvedKernelBinding[]`, not plain
`ResourceBinding[]` — every entry remains usable anywhere a `ResourceBinding` is expected (e.g.
`Pass.bindings`, structurally), while carrying `name`/`binding`/`resourceKind`/`stages` forward so a
later consumer (F3.4) never has to reconstruct which resolved entry came from which template by
array-index correlation.

### 2. `validateKernelBindingTemplates` — the single, shared validator

New file `packages/graph/src/kernelBinding.ts` (mirrors `resources.ts`'s existing split: types in
`implementation.ts`, functions alongside it in a sibling file):

```ts
const WGSL_IDENTIFIER = /^[A-Za-z_][A-Za-z0-9_]*$/;
const WGSL_RESERVED_WORDS = new Set([
	'alias',
	'break',
	'case',
	'const',
	'const_assert',
	'continue',
	'continuing',
	'default',
	'diagnostic',
	'discard',
	'else',
	'enable',
	'false',
	'fn',
	'for',
	'if',
	'let',
	'loop',
	'override',
	'requires',
	'return',
	'struct',
	'switch',
	'true',
	'var',
	'while'
]);

export function validateKernelBindingTemplates(
	templates: readonly KernelBindingTemplate[],
	ownerStage: ShaderStage
): void {
	const names = new Set<string>();
	const bindings = new Set<number>();
	for (const template of templates) {
		if (!WGSL_IDENTIFIER.test(template.name)) {
			throw new Error(`Kernel binding has an invalid shader name: "${template.name}"`);
		}
		if (WGSL_RESERVED_WORDS.has(template.name)) {
			throw new Error(`Kernel binding uses a reserved WGSL keyword as its shader name: "${template.name}"`);
		}
		if (names.has(template.name)) {
			throw new Error(`Duplicate kernel binding name: "${template.name}"`);
		}
		names.add(template.name);
		if (!Number.isInteger(template.binding) || template.binding < 0) {
			throw new Error(
				`Kernel binding "${template.name}" has an invalid binding index: ${template.binding}`
			);
		}
		if (bindings.has(template.binding)) {
			throw new Error(
				`Duplicate kernel binding index ${template.binding} ("${template.name}")`
			);
		}
		bindings.add(template.binding);
		if (template.stages.length === 0) {
			throw new Error(`Kernel binding "${template.name}" declares no visible stages`);
		}
		if (!template.stages.includes(ownerStage)) {
			throw new Error(
				`Kernel binding "${template.name}" is not visible in its owning kernel's stage ` +
					`"${ownerStage}" (declares stages: ${template.stages.join(', ')})`
			);
		}
		if (template.resourceKind === 'sampler' && template.access !== 'read') {
			throw new Error(
				`Kernel binding "${template.name}" is a sampler and must declare access:'read'`
			);
		}
		if (template.resourceKind === 'texture' && template.access !== 'read') {
			throw new Error(
				`Kernel binding "${template.name}" is a texture binding and must declare ` +
					`access:'read' (storage textures are deferred — see F2.3)`
			);
		}
	}
}
```

`ownerStage` is the enclosing `{ kind: 'kernel'; stage }` primitive's own `stage` — every binding a
kernel declares must be visible in the stage that kernel itself executes as (Round 2, finding 5). A
binding may still additionally declare visibility in other stages (e.g. a resource shared across a
vertex/fragment kernel pair once F3.2 lands typed varyings) — the check is `includes`, not
equality.

### 3. `isBindingVisibleInStage` — the stage-visibility predicate

```ts
export function isBindingVisibleInStage(
	binding: Pick<KernelBindingTemplate, 'stages'>,
	stage: ShaderStage
): boolean {
	return binding.stages.includes(stage);
}
```

Deliberately a separate, independently-testable pure predicate rather than folded into resolution
(step 4) — `Pass.bindings` is a single flat list covering a whole pass, not stage-partitioned;
per-stage filtering is a distinct concern a later milestone's per-stage WGSL assembly will need.

### 4. `resolveKernelBindings` — static binding slots → per-pass `ResourceBinding.resourceId`

The milestone's actual deliverable — same file:

```ts
export function resolveKernelBindings(
	templates: readonly KernelBindingTemplate[],
	ownerStage: ShaderStage,
	resourceIds: ReadonlyMap<string, string> // KernelBindingTemplate.name -> ResourceInstance.id
): ResolvedKernelBinding[] {
	validateKernelBindingTemplates(templates, ownerStage);
	return templates.map((template) => {
		const resourceId = resourceIds.get(template.name);
		if (resourceId === undefined) {
			throw new Error(`Kernel binding "${template.name}" has no resolved resource id`);
		}
		return {
			resourceId,
			access: template.access,
			name: template.name,
			binding: template.binding,
			resourceKind: template.resourceKind,
			stages: template.stages
		};
	});
}
```

`resourceIds` is caller-supplied — deriving that name→id table from a real `GraphDocument`'s wiring
is explicitly not this milestone's job (mirroring F2.1's own deferral of document-walking to F2.2).
`ownerStage` is the same value the caller would pass because it already has the enclosing
`{ kind: 'kernel'; stage }` primitive in hand — this function does not infer it.

### 5. Registration-time enforcement

`registerPrimitive`'s actual structural checks live inside `normalizeImplementation`
(`packages/graph/src/registry.ts`, called from `normalizePrimitiveInput` — confirmed by direct
read; e.g. the existing `wgsl`/implementation mismatch throw at line ~74). Add the same discipline
there, alongside the existing checks, using the same `implementation`/`input.id` variables already
in scope:

```ts
if (implementation.kind === 'kernel') {
	validateKernelBindingTemplates(implementation.bindings, implementation.stage);

```

A badly-declared kernel primitive (duplicate binding index, a sampler with `access:'write'`, etc.)
now fails at registration, not silently until some later document happens to call
`resolveKernelBindings` against it.

### 6. `BindingDecl.kind` gains `storage-read-write`

In `packages/compiler/src/stageEntry.ts`:

```ts
export interface BindingDecl {
	group: number;
	binding: number;
	name: string;
	kind: 'uniform' | 'storage-read' | 'storage-read-write' | 'texture' | 'sampler';
	wgslType: string;
}
```

Update `bindingVar`'s address-space table to match:

```ts
function bindingVar(b: BindingDecl): string {
	const addr =
		b.kind === 'uniform'
			? '<uniform> '
			: b.kind === 'storage-read'
				? '<storage, read> '
				: b.kind === 'storage-read-write'
					? '<storage, read_write> '
					: ' ';
	return `@group(${b.group}) @binding(${b.binding}) var${addr}${b.name}: ${b.wgslType};`;
}
```

### 7. `bindingDeclKindForTemplate` — the pure, one-directional translation

Same file (`stageEntry.ts`), importing the type only (compiler depends on graph, never the
reverse):

```ts
import type { KernelBindingTemplate } from '@world-lab/graph';

export function bindingDeclKindForTemplate(
	template: Pick<KernelBindingTemplate, 'resourceKind' | 'access'>
): BindingDecl['kind'] {
	if (template.resourceKind === 'sampler') {
		if (template.access !== 'read') {
			throw new Error(`sampler kernel bindings must declare access:'read'`);
		}
		return 'sampler';
	}
	if (template.resourceKind === 'texture') {
		if (template.access !== 'read') {
			throw new Error(
				`texture kernel bindings must declare access:'read' (storage textures are deferred — see F2.3)`
			);
		}
		return 'texture';
	}
	return template.access === 'read' ? 'storage-read' : 'storage-read-write';
}
```

This function independently re-checks the same two rules `validateKernelBindingTemplates` already
enforces — deliberately, not a code-sharing gap: the two functions live in different packages with
a one-way dependency, and duplicating a two-line check here is preferable to inventing a
cross-package validation-sharing scheme for something this small. It assumes nothing about *how*
`group`/`binding`/`wgslType` get filled in for a real `BindingDecl` — turning a resolved
`KernelBindingTemplate` into a fully-formed `BindingDecl` (with a real WGSL type derived from the
bound resource's concrete shape, and a final, module-wide binding index) is explicitly deferred —
see Out of scope.

Do **not** implement any GPU allocation, bind-group construction, real per-document WGSL assembly,
or dispatch here — this milestone is types and pure resolution functions only.

## Gate

1. A test proves `KernelBindingTemplate.resourceKind` only accepts `'buffer' | 'texture' |
   'sampler'` — a `@ts-expect-error` test assigning an invalid literal must fail to compile,
   matching this project's existing `Extract`-based exhaustiveness-test convention.
2. A test proves `validateKernelBindingTemplates` rejects two templates sharing the same `name`.
3. A test proves `validateKernelBindingTemplates` rejects two templates sharing the same `binding`
   index.
4. A test proves `validateKernelBindingTemplates` rejects a negative or non-integer `binding` index.
5. A test proves `validateKernelBindingTemplates` rejects a template with an empty `stages` array.
6. A test proves `validateKernelBindingTemplates` rejects a `sampler` template with `access !==
   'read'`, and separately rejects a `texture` template with `access !== 'read'`.
7. A test proves `validateKernelBindingTemplates` rejects an empty-string `name`, a `name` that
   is not identifier-shaped (e.g. starting with a digit, containing a space or `-`), and a reserved
   WGSL keyword such as `fn` — the direct proof of Round 2 finding 6.
8. A test proves `validateKernelBindingTemplates` rejects a template whose `stages` does not
   include the `ownerStage` argument (e.g. `stages: ['fragment']` validated against
   `ownerStage: 'compute'`), and separately accepts one where `stages` includes `ownerStage` plus at
   least one other stage — the direct proof of Round 2 finding 5.
9. A test proves `resolveKernelBindings` maps each template's `name` to the caller-supplied
   `resourceId`, producing a `ResolvedKernelBinding[]` in the same order as the input templates,
   preserving each template's declared `access`, and carrying `name`/`binding`/`resourceKind`/
   `stages` forward on each resolved entry — the direct proof of finding 2 (Context) and Round 2
   finding 4 (`ResolvedKernelBinding` retains identity, not just `{ resourceId, access }`).
10. A test proves `resolveKernelBindings` throws a clear error when `resourceIds` is missing an
    entry for a declared template name.
11. A test proves `resolveKernelBindings` throws `validateKernelBindingTemplates`'s error (not a
    missing-resourceId error) when both a validation defect and a missing `resourceIds` entry are
    present simultaneously — proving validation runs first, not opportunistically.
12. A test proves `resolveKernelBindings` throws the owner-stage-visibility error (not a
    missing-resourceId error) when a binding fails the `ownerStage` check even though `resourceIds`
    has an entry for it — proving the `ownerStage` check also runs before resolution, not after.
13. A test proves `isBindingVisibleInStage` returns `true`/`false` correctly for both single- and
    multi-stage `stages` arrays.
14. A test proves `registerPrimitive` throws when registering a `kind: 'kernel'` primitive whose
    `bindings` violate `validateKernelBindingTemplates` (e.g. duplicate binding index), **and**
    separately when a binding's `stages` excludes the primitive's own `stage` — the registration-time
    enforcement gate (Fix step 5), extending `registry.test.ts`'s existing structural-validation
    coverage.
15. A test proves `bindingDeclKindForTemplate` maps `{resourceKind:'buffer', access:'read'}` →
    `'storage-read'`, `{resourceKind:'buffer', access:'read-write'}` and
    `{resourceKind:'buffer', access:'write'}` → `'storage-read-write'`,
    `{resourceKind:'texture', access:'read'}` → `'texture'`, `{resourceKind:'sampler',
    access:'read'}` → `'sampler'`, and throws for `{resourceKind:'texture', access:'write'}` and
    `{resourceKind:'sampler', access:'write'}`.
16. A test proves a `BindingDecl` with `kind:'storage-read-write'` round-trips through
    `assembleStageEntry` producing `var<storage, read_write>` in the emitted code — extending
    `stageEntry.test.ts`'s existing `bindings` fixture (currently covering `uniform`/`storage-read`
    at lines 24-27) with a third entry, the direct regression proof for Fix step 6.
17. `check` **and** `test` green for `graph`, `compiler`, full workspace (clear every
    `packages/*/dist` first, per `AGENTS.md`'s documented stale-dist issue).

## Out of scope

Deriving a real `GraphDocument`'s wiring into the `resourceIds` name→id table `resolveKernelBindings`
consumes (a later milestone's job, mirroring F2.2's role relative to F2.1). Turning a resolved
`KernelBindingTemplate` into a fully-formed `BindingDecl` with a concrete `group`/final `binding`
index/`wgslType` derived from an actual bound resource (assembly-time — F3.4's concern once real
per-document kernel WGSL modules are built). Any GPU allocation, `GPUBindGroupLayout`/`GPUBindGroup`
construction, or dispatch (F3.3/F3.4). Typed varyings between vertex and fragment kernels (F3.2).
Built-in stage-input identifiers beyond what `HostBinding` already declares (`vertex_index`,
`instance_index`, `global_invocation_id`, etc. stay exactly as `HostBinding`'s existing mechanism
already handles them — this milestone does not add new built-in-input primitives). Retrofitting
stage-visibility *enforcement* onto `HostBinding.stages`'s own pre-existing, separate,
currently-unenforced gap (`emitGraphEval.ts` never checks it) — a real gap, but not one this
milestone's new bindings introduced, and not touched here. Storage textures (`access: 'write'` or
`'read-write'` on a `texture`-kind binding) — stays rejected, per F2.3's own deferral, until a later
milestone explicitly un-defers it. Multiple bind groups (every binding in this milestone is
implicitly group 0, matching every existing consumer's own convention). `{ kind: 'command';
command: GpuCommandKind }` — stays untouched; that placeholder is Foundation 4's seam, not this
one's. Any change to Foundation 1's or Foundation 2's landed contracts.

## Handoff

→ `{ kind: 'kernel'; stage }` has real, validated, resolvable binding declarations
(`KernelBindingTemplate`, validated at both registration time and resolution time via one shared
function that also checks every binding's declared stages against its owning kernel's own stage),
independent of the compiler's `BindingDecl` but joined to it by a pure, precondition-safe
translation (`bindingDeclKindForTemplate`) — and `BindingDecl` itself can finally express a
read-write storage binding, closing the exact gap F2.5's buffer-feedback sample had to route around.
`resolveKernelBindings` gives `Pass.bindings`/`ResourceBinding[]` its first consumer beyond usage
inference, returning `ResolvedKernelBinding[]` so identity (name/binding/resourceKind/stages) survives
resolution rather than being discarded. Nothing here allocates GPU resources, assembles a real
per-document WGSL module, or dispatches anything — that's F3.2 (typed varyings, the direct Mode A
vertex-displacement fix) next; request its own contract at that point.
