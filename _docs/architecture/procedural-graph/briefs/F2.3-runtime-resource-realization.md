# Brief — F2.3: runtime resource realization

**Type:** new GPU resource allocator, first real consumer of `Pass.bindings` · **Package:**
`@world-lab/runtime-webgpu` · **Depends on:** F2.1 (`04f5319`), F2.2 (`397af7f`), both landed —
`ResourceTarget`/`BufferResourceTarget`/`TextureResourceTarget`, `resolveTargetSizes`/
`resolveBufferSizes`, `resolveBufferUsage`, `Pass.bindings` · **Design authority:**
[foundation-2-generic-resources-plan.md](../foundation-2-generic-resources-plan.md) · **Contract
author:** Opus · **Recommended executor:** Cursor or Codex · **Status:** ready to route

## Context

Third of Foundation 2's sequence (F2.1 ✅ → F2.2 ✅ → **F2.3** → F2.4 → proof). The Foundation 2
plan's own one-line description of this milestone ("allocate real WebGPU resources... derive
bind-group layouts and usage flags...") is narrowed here in one respect, deliberately, before
routing — the same kind of pre-implementation correction F2.1 and F2.2 each needed after the fact:

**Bind-group layout derivation is out of scope for this milestone.** A `GPUBindGroupLayoutEntry`
needs a binding index and shader-stage visibility, and today nothing produces either — those come
from a kernel's actual shader interface, which is Foundation 3's job (`packages/runtime-webgpu`'s
own `elemental-webgpu-architecture-review.md` lists "typed vertex/fragment/compute interfaces" and
"arbitrary resource bindings" as Foundation 3 scope, not Foundation 2's). Attempting layout
derivation now would mean inventing a binding-index/stage convention with no real consumer to
validate it against — exactly the F2.2/Foundation-3 circularity F2.1's own revision 3 already
caught and fixed for `ResourceBinding` attachment. This milestone allocates physical resources and
derives **usage flags** only; Foundation 3 derives bind-group layouts once kernels exist.

This milestone is also the **first real consumer of `Pass.bindings`** — F2.2 reserved it
specifically so a buffer's declared usage floor could be combined with its actual per-pass access
pattern via F2.1's `resolveBufferUsage`. Texture usage has no equivalent per-binding-declared floor
(`TypeRef`'s texture variant never grew a `usages` field — F2.1 didn't need one, and adding one now
would reopen a closed milestone), so texture usage is inferred structurally from `PassGraph` shape
instead: written targets need `RENDER_ATTACHMENT`, read targets need `TEXTURE_BINDING`, the display
target additionally needs `COPY_SRC` for CPU readback — exactly what `fullscreenFragment.ts`
already hand-derives for its single, hardcoded target today, generalized to a graph of them.

## Verified current state

- **Nothing today does generic, descriptor-driven GPU allocation.** Every `device.createBuffer`/
  `device.createTexture` call in `packages/runtime-webgpu/src/consumers/*.ts` is hand-authored per
  consumer with hardcoded usage flags (confirmed by direct grep across the package) —
  `fullscreenFragment.ts:260` creates its one render target with
  `RENDER_ATTACHMENT | COPY_SRC`, matching exactly the texture-usage inference this milestone
  generalizes.
- **`PipelineGraphExecutor`/`GraphFrameExecutor` don't call any of F2.2's resolve functions** —
  confirmed in the Foundation 2 plan's own investigation; wiring this milestone's realizer into
  that execution path is F2.4's job, not this one's.
- **`packages/runtime-webgpu/src/buffers.ts`** already has `alignTo(size, alignment)` (rounds up to
  a multiple, used for the 4-byte buffer-size alignment WebGPU requires) and
  `createStorageBuffer(device, { label?, size, usage? })` — reuse `alignTo` directly; this
  milestone's own buffer allocation needs derived (not caller-supplied) usage flags and byte size,
  so it calls `device.createBuffer` directly rather than through `createStorageBuffer`, but shares
  the alignment helper.
- **No scalar-to-byte-size table exists anywhere in the codebase** (confirmed by grep for
  `byteSizeOf`/`scalarSize`/similar — no hits). This milestone adds a minimal one, scoped to what
  `BufferResourceTarget.shape.element` can actually be validated against today.
- **`packages/runtime-webgpu/src/graphFrameExecutor.test.ts`** establishes the existing headless
  GPU-testing convention this milestone follows: a `hasWebGPU` capability check
  (`navigator.gpu !== undefined`) gates `it.skipIf(!hasWebGPU)` tests that need a real device via
  `requestGpuDevice()` (`./device.ts`); everything else uses a fake `device: {} as GPUDevice` with
  `vi.fn()`-mocked `createBuffer`/`createTexture` to test logic (caching, parity, usage-flag
  derivation) without a real GPU. This milestone's gate follows the same split.
- **`PipelineGraphResourceCache`** (`pipelineGraph.ts`) is the existing precedent for a small,
  stateful, per-execution-session cache object owned across frames (there: geometry realization
  keyed by a content fingerprint). This milestone's realizer follows the same shape — a class
  instance the caller keeps alive across frames, not a pure function called fresh each time.

## Fix

New file `packages/runtime-webgpu/src/frameGraph/realize.ts`.

1. **A stateful realizer, lazily allocating and caching one physical resource per target id** (two,
   for `history` — ping-ponged), reallocating only when a target's resolved size or shape actually
   changes:
   ```ts
   export interface RealizedTarget {
     write: GPUBuffer | GPUTexture; // this frame's write destination / any same-frame reader's source
     readPrevious?: GPUBuffer | GPUTexture; // only present for history-lifetime targets
   }

   export class ResourceRealizer {
     constructor(private readonly device: GPUDevice) {}

     /** Idempotent: (re)allocates only targets that are missing or whose resolved size/shape changed. */
     realizeAll(graph: PassGraph, viewport: { width: number; height: number }): void;

     /** Throws if `targetId` hasn't been realized (call `realizeAll` first). */
     resolve(targetId: string): RealizedTarget;

     /** Flips which physical slot is "current" for every history-lifetime target. Call once per frame. */
     advanceFrame(): void;

     /** Destroys every allocated GPUBuffer/GPUTexture and clears the cache. */
     dispose(): void;
   }
   ```
   `resolve`'s `write`/`readPrevious` split matches `ResourceRead.version`'s two cases directly:
   `version` omitted (current-frame read) and same-frame writes both resolve to `write`;
   `version: 'previous'` resolves to `readPrevious` — which F2.2's own validation already guarantees
   only exists on `history`-lifetime targets, so `resolve` never needs to guess.
2. **`realizeAll` reallocates on any of: missing entry, slot-count mismatch (lifetime changed to/from
   `history`), shape-kind mismatch, or resolved-size mismatch** — destroying the old resource(s)
   before replacing them:
   ```ts
   realizeAll(graph, viewport) {
     const textureSizes = resolveTargetSizes(graph, viewport);
     const bufferSizes = resolveBufferSizes(graph);
     for (const target of graph.targets) {
       const requiredSlots = target.lifetime.kind === 'history' ? 2 : 1;
       const resolvedSize = target.shape.kind === 'texture'
         ? textureSizes[target.id] : bufferSizes[target.id];
       const existing = this.#entries.get(target.id);
       if (existing && existing.kind === target.shape.kind
           && existing.slots.length === requiredSlots
           && sizeMatches(existing.resolvedSize, resolvedSize)) {
         continue;
       }
       existing?.slots.forEach((resource) => resource.destroy());
       const usage = target.shape.kind === 'buffer'
         ? toGPUBufferUsage(resolveBufferUsage(target.shape.usages, bindingsFor(graph, target.id)))
         : inferTextureUsage(graph, target.id);
       const allocate = () => target.shape.kind === 'buffer'
         ? this.device.createBuffer({
             size: resolveBufferByteSize(target.shape, resolvedSize.elementCount),
             usage
           })
         : allocateTexture(this.device, target, resolvedSize, usage);
       this.#entries.set(target.id, {
         kind: target.shape.kind,
         slots: Array.from({ length: requiredSlots }, allocate),
         resolvedSize
       });
     }
   }
   ```
   `sizeMatches` is a plain deep-equality check (`JSON.stringify` comparison is sufficient — sizes
   are small plain objects, matching the pattern F2.1's own lifetime round-trip test already used).
   A resize necessarily drops a `history` target's prior content — both slots become fresh,
   identically-sized allocations — this is inherent to reallocating, not a gap this milestone needs
   to solve (there is nothing meaningful to preserve across a size change).
3. **`resolve`/`advanceFrame` are pure bookkeeping over the cache — no GPU calls:**
   ```ts
   resolve(targetId) {
     const entry = this.#entries.get(targetId);
     if (!entry) throw new Error(`Resource "${targetId}" has not been realized`);
     if (entry.slots.length === 1) return { write: entry.slots[0] };
     const current = this.#frameParity;
     return { write: entry.slots[current], readPrevious: entry.slots[1 - current] };
   }
   advanceFrame() {
     this.#frameParity = this.#frameParity === 0 ? 1 : 0;
   }
   ```
4. **Buffer usage: the first real consumer of `Pass.bindings`.** Collect every binding across every
   pass that references a given resource id, and combine with the template's declared floor via
   F2.1's own `resolveBufferUsage` — no new usage-combination logic, just wiring the function F2.2
   reserved the field for:
   ```ts
   function bindingsFor(graph: PassGraph, resourceId: string): ResourceBinding[] {
     return graph.passes.flatMap((pass) =>
       (pass.bindings ?? []).filter((binding) => binding.resourceId === resourceId)
     );
   }

   const BUFFER_USAGE_BITS: Record<BufferUsageFlag, GPUBufferUsageFlags> = {
     vertex: GPUBufferUsage.VERTEX, index: GPUBufferUsage.INDEX,
     uniform: GPUBufferUsage.UNIFORM, storage: GPUBufferUsage.STORAGE,
     'copy-src': GPUBufferUsage.COPY_SRC, 'copy-dst': GPUBufferUsage.COPY_DST,
     indirect: GPUBufferUsage.INDIRECT, 'query-resolve': GPUBufferUsage.QUERY_RESOLVE
   };
   function toGPUBufferUsage(flags: readonly BufferUsageFlag[]): GPUBufferUsageFlags {
     return flags.reduce((acc, flag) => acc | BUFFER_USAGE_BITS[flag], 0);
   }
   ```
5. **Texture usage: inferred structurally from `PassGraph`, since `TypeRef`'s texture variant has no
   declared-usage floor to combine with (unlike buffers):**
   ```ts
   export function inferTextureUsage(graph: PassGraph, targetId: string): GPUTextureUsageFlags {
     let usage: GPUTextureUsageFlags = 0;
     if (graph.passes.some((pass) => pass.writeTarget === targetId)) {
       usage |= GPUTextureUsage.RENDER_ATTACHMENT;
     }
     if (graph.passes.some((pass) => pass.reads.some((read) => read.target === targetId))) {
       usage |= GPUTextureUsage.TEXTURE_BINDING;
     }
     if (graph.display === targetId) {
       usage |= GPUTextureUsage.COPY_SRC;
     }
     return usage;
   }
   ```
   This is exactly `fullscreenFragment.ts`'s existing hardcoded `RENDER_ATTACHMENT | COPY_SRC`,
   generalized: that consumer's one target is always written and always the display, so it always
   gets both flags today; this function produces the same result for that case and the correct,
   narrower result for a graph with more than one target.
6. **Byte-size derivation, scoped to scalar elements only — explicitly, not silently:**
   ```ts
   const SCALAR_BYTE_SIZE: Partial<Record<ScalarType, number>> = { f32: 4, i32: 4, u32: 4, f16: 2 };

   export function resolveBufferByteSize(
     shape: Extract<ResourceShape, { kind: 'buffer' }>,
     elementCount: number
   ): number {
     if (shape.element.kind !== 'scalar') {
       throw new Error(
         `Buffer element kind "${shape.element.kind}" is not yet supported by resource realization ` +
         `(scalar only — vector/matrix/array/struct byte-sizing needs std430 alignment rules and is deferred)`
       );
     }
     const size = SCALAR_BYTE_SIZE[shape.element.scalar];
     if (!size) {
       throw new Error(`Scalar type "${shape.element.scalar}" cannot be stored in a host-shareable buffer`);
     }
     return alignTo(size * elementCount, 4); // reuse buffers.ts's existing alignment helper
   }
   ```
   `bool` is deliberately excluded (WGSL forbids `bool` in host-shareable address spaces — this is
   correct rejection, not a gap). Vector/matrix/array/struct elements need real WGSL std430/std140
   alignment rules to size correctly (e.g. a tightly-packed `array<vec3<f32>>` still strides at 16
   bytes, not 12) — genuinely separate, non-trivial work with no current consumer; the Foundation 2
   proof's own buffer-feedback sample only needs a flat scalar array, so this scope cut doesn't
   block it.
7. **Texture allocation, scoped to `dimension: '2d'` only — explicitly, not silently:**
   ```ts
   function allocateTexture(
     device: GPUDevice, target: TextureResourceTarget,
     size: { width: number; height: number }, usage: GPUTextureUsageFlags
   ): GPUTexture {
     if (target.shape.dimension !== '2d') {
       throw new Error(
         `Texture dimension "${target.shape.dimension}" is not yet supported by resource realization ` +
         `(2d only — 1d/3d sizing and cube's array-layer + view-dimension mapping are deferred)`
       );
     }
     return device.createTexture({
       size: [size.width, size.height],
       format: (target.shape.format ?? 'rgba8unorm') as GPUTextureFormat,
       usage
     });
   }
   ```
   Every texture target either resolve function can produce today is 2D (`resolveTargetSizes` only
   ever returns `{ width, height }`) — this check is a defensive guard against a future
   dimension value reaching here before this function is extended, not a currently-reachable case.

## Gate

1. A test proves a `transient`/`persistent` target is allocated once: two consecutive `realizeAll`
   calls with an unchanged graph/viewport, followed by `resolve`, return the **same** `GPUBuffer`/
   `GPUTexture` identity both times (mocked `createBuffer`/`createTexture` call count stays at 1).
2. A test proves a `history`-lifetime target allocates two distinct resources, and `resolve(id)`'s
   `write`/`readPrevious` swap identities after `advanceFrame()` — the core ping-pong proof.
3. A test proves a `transient`/`persistent` target's `resolve(id).readPrevious` is `undefined` —
   consistent with F2.2's validation that only `history` targets can ever be read as `'previous'`.
4. A test proves resizing a `screen-relative` texture target (a second `realizeAll` call with a
   different viewport) destroys the old `GPUTexture` (mocked `.destroy()` called) and allocates a
   new one sized to the new viewport.
5. A test proves buffer usage flags are the real combination: a target with declared
   `usages: ['uniform']` and a `write`-access `ResourceBinding` elsewhere in the graph produces a
   `createBuffer` call whose `usage` includes both `GPUBufferUsage.UNIFORM` and
   `GPUBufferUsage.STORAGE` — proving `Pass.bindings` is genuinely consumed, not just present.
6. A test proves `inferTextureUsage`: a target that is both a `writeTarget` and read by another pass
   gets `RENDER_ATTACHMENT | TEXTURE_BINDING`; the `graph.display` target additionally gets
   `COPY_SRC`; a target that is only ever read (never written, hypothetically) gets
   `TEXTURE_BINDING` alone.
7. A test proves `resolveBufferByteSize`: scalar element types produce the correct byte count
   (`f32`/`i32`/`u32` → 4 bytes/element, `f16` → 2) times element count, rounded up to a 4-byte
   multiple; a `vector`/`matrix`/`array`/`struct` element throws a clear error rather than producing
   a silently-wrong size.
8. A test proves texture allocation rejects a non-`'2d'` `dimension` with a clear error rather than
   silently mis-sizing or ignoring it.
9. A test proves `dispose()` calls `.destroy()` on every allocated resource (both slots, for
   `history` targets) exactly once, and a subsequent `resolve()` call throws.
10. `it.skipIf(!hasWebGPU)`: one real-device test allocates a small mixed graph (one buffer, one
    texture, one history buffer) via `requestGpuDevice()`, confirms no exception, and disposes
    cleanly — the concrete proof the derived descriptors are valid `GPUBufferDescriptor`/
    `GPUTextureDescriptor` values, not just internally-consistent mock call arguments.
11. `check` **and** `test` green for `runtime-webgpu`, full workspace (clear every `packages/*/dist`
    first, per `AGENTS.md`'s documented stale-dist issue).

## Out of scope

Bind-group layout derivation (Foundation 3 — needs kernel binding-index/shader-stage information
that doesn't exist yet; see Context). Wiring `ResourceRealizer` into `GraphFrameExecutor`/
`PipelineGraphExecutor` (F2.4). Vector/matrix/array/struct buffer-element byte-sizing (needs WGSL
std430/std140 alignment rules; no current consumer). Texture dimensions other than `2d` (1d/3d
sizing, cube's array-layer + view-dimension mapping). Any WGSL codegen change needed to actually
sample another pass's texture output in a shader, or to declare a compute kernel's storage-buffer
binding (Foundation 3/4 — kernels need typed bindings before generated shader code can reference
these resources at all; this milestone only makes the physical resources exist and know their own
usage flags). Resource pooling/aliasing across *different* target ids for memory reuse (explicitly
out of scope for the whole frame-graph effort, per the original brief and this plan). Any change to
Foundation 1's milestones, F2.1/F2.2's landed contracts, or this document's own Foundation 2
sequencing beyond this one milestone.

## Handoff

→ Physical `GPUBuffer`/`GPUTexture` allocation exists for every `ResourceTarget` shape, cached and
reallocated only when size/shape genuinely changes; `history`-lifetime targets are physically
double-buffered with a simple frame-parity flip; buffer usage flags are the real combination of a
template's declared floor and its actual per-pass `ResourceBinding` access (the first real use of
the field F2.2 reserved); texture usage flags are inferred from the same structural information
`fullscreenFragment.ts` already hand-derives for its one hardcoded target. Bind-group layout
derivation deliberately waits for Foundation 3's kernel interfaces, so this milestone doesn't invent
a binding-index convention nothing consumes yet. Once this lands: F2.4 (generic frame executor —
replacing `GraphFrameExecutor`'s independent-only bypass with real execution over
`ResourceRealizer`, incrementally, keeping today's zero-dependency case working throughout) is
next — request its own contract at that point.
