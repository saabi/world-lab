sequence ¬∑ **Package:** `@world-lab/runtime-webgpu` (derivation logic, kernel-fragment path

## Context

Third of F3.6's tentative sequence. Sub-problem (A), landing on top of F3.6.2's real branch point:
F3.1's binding model is **static per-primitive** (`KernelBindingTemplate[]` lives on
`PrimitiveImplementation`, declared once at registration ‚Äî confirmed, `Node` itself has no field
that could carry a per-instance binding set, see (A)'s full analysis). `stage.vertex`/
`stage.fragment` are reused by every pipeline graph, and each document wires a *different* set of
host uniforms, texture channels, and params into them ‚Äî a fixed, per-primitive binding list can't
express that. F3.6.2 proved the *execution path* branches correctly on a kernel-based fragment
stage, but only for a hand-declared, fixed, test-only binding shape (one `tint` buffer) ‚Äî explicitly
scaffolding, not a real answer to "what does a graph-authored fragment kernel's binding set actually
look like."

**This milestone's recommendation, per (A): derive the binding set from graph structure, don't
author it (Option A2).** Not a new idea ‚Äî it's exactly what `assembleFullscreenFragmentModuleAsync`
already does **by hand, today**: it inspects the emitted graph slice
(`usesShaderToyHost`/`emitted.usedChannels`/`hasGraphParams`, all computed dynamically per call) and
builds the right `BindingDecl[]` for whatever's actually wired in. F3.6.3 generalizes that existing,
proven, one-off inline logic into small, reusable, tested functions, and applies it to a **new**
primitive (`stage.fragmentKernel` ‚Äî additive, per (D)'s D2 recommendation; `stage.fragment` stays
exactly as it is, forever, until a separate later decision retires it, not made here).

**Explicitly not unifying `HostBinding` and `KernelBindingTemplate`** (per (A)'s own recommendation)
‚Äî host-uniform/channel/param derivation stays the *existing*, simpler value-injection mechanism;
this milestone reuses it exactly as-is, it does not touch `emitGraphEval.ts` or any `host.*`/
`input.channel` primitive. The new primitive registers with `bindings: []` ‚Äî **zero** statically
kernel-declared bindings ‚Äî because everything real about it comes from what's wired into its `color`
input, discovered per document, not authored on the primitive. F3.6.2's own kernel-declared-binding
mechanism (`KernelBindingTemplate[]` resolved via F3.1's `resolveKernelBindings` chain) is not
removed ‚Äî a primitive *can* still declare static bindings and combine them with derived ones (this
milestone's Fix steps preserve and extend that path, not replace it) ‚Äî but the new primitive this
milestone registers doesn't need any.

**Generalizes all three of `assembleFullscreenFragmentModuleAsync`'s existing derivation
concerns ‚Äî uniforms, params, channels ‚Äî not just channels.** F3.6.2 explicitly rejected combining
kernel bindings with ShaderToy uniform host inputs (`host.iTime`/`host.iResolution`/`host.iMouse`)
or channel textures (`input.channel`), deferring both to this exact milestone. Graph params were
already supported by F3.6.2 unchanged. This milestone lifts both remaining rejections ‚Äî a real
fragment kernel can now use `host.iTime` and/or `input.channel` exactly as freely as `stage.fragment`
already can, because the binding set is now genuinely derived from what the graph actually
references, the same way the ShaderToy path has always worked.

**No new "kernel body" combination contract needed ‚Äî confirmed, not assumed.** F3.6.2's Out-of-scope
worried about designing "how a real primitive's body relates to its own declared bindings." That
question doesn't arise here: `input.channel`/`host.iTime`/etc. are read **inside the graph-emitted
body itself** via `emitGraphEval`'s existing `emitHostInput` handling (e.g. `input.channel` already
emits `textureSample(channel0, channel0Sampler, ...)` directly into `emitted.body` ‚Äî confirmed,
`emitGraphEval.ts:106-114`) ‚Äî this milestone only has to make sure the **binding declarations** those
already-emitted references need actually exist, not change what the body itself contains. Because
the new primitive declares zero static bindings, there is no "how does a kernel-declared binding get
referenced from graph-authored WGSL" question to design either ‚Äî sidestepped entirely, not solved.

**No manual/visual gate, no bundled sample.** `stage.fragment` and every existing sample are
untouched. `stage.fragmentKernel` is real (not test-only), but nothing wires it into
`GraphFrameExecutor`'s per-pass loop, the editor's preview UI, or a bundled sample graph yet ‚Äî that's
explicitly F3.6.6's job in the original tentative sequencing, once F3.6.3 (this) and F3.6.4/F3.6.5
(vertex-side) both exist. This milestone's proof is a direct, real-device call to
`executeKernelFragment`, exactly matching F3.6.2's own proof shape.

## Fix

### 1. Extract and export channel-binding derivation from `fullscreenFragment.ts`

Replace the inline loop (`fullscreenFragment.ts:168-192`) with a call to a new, exported function ‚Äî
pure refactor, the produced `BindingDecl[]`/`channelBindings` must be byte-identical to today's
inline version:

```ts
export interface ChannelBindingAssembly {
	bindings: BindingDecl[];
	channelBindings: ReadonlyMap<number, { textureBinding: number; samplerBinding: number }>;
	/** Lowest binding index not used by any channel binding ‚Äî chain further derived bindings here. */
	nextBindingIndex: number;
}

export function deriveChannelBindingDecls(
	usedChannels: readonly number[],
	startBinding: number
): ChannelBindingAssembly {
	const bindings: BindingDecl[] = [];
	const channelBindings = new Map<number, { textureBinding: number; samplerBinding: number }>();
	let nextBinding = startBinding;
	for (const channel of usedChannels) {
		const textureBinding = nextBinding++;
		const samplerBinding = nextBinding++;
		channelBindings.set(channel, { textureBinding, samplerBinding });
		bindings.push(
			{ group: 0, binding: textureBinding, name: `channel${channel}`, kind: 'texture', wgslType: 'texture_2d<f32>' },
			{ group: 0, binding: samplerBinding, name: `channel${channel}Sampler`, kind: 'sampler', wgslType: 'sampler' }
		);
	}
	return { bindings, channelBindings, nextBindingIndex: nextBinding };
}

export function buildChannelBindGroupEntries(
	device: GPUDevice,
	channelBindings: ReadonlyMap<number, { textureBinding: number; samplerBinding: number }>,
	channelTargets: ReadonlyMap<number, GPUTexture>
): GPUBindGroupEntry[] {
	const entries: GPUBindGroupEntry[] = [];
	for (const [channel, decl] of channelBindings) {
		const target = channelTargets.get(channel);
		if (!target) {
			throw new Error(`Missing channel target for channel ${channel}`);
		}
		entries.push(
			{ binding: decl.textureBinding, resource: target.createView() },
			{
				binding: decl.samplerBinding,
				resource: device.createSampler({
					magFilter: 'linear',
					minFilter: 'linear',
					addressModeU: 'clamp-to-edge',
					addressModeV: 'clamp-to-edge'
				})
			}
		);
	}
	return entries;
}
```

`assembleFullscreenFragmentModuleAsync` calls `deriveChannelBindingDecls(emitted.usedChannels,
nextBinding)` in place of its own loop, appends the result's `bindings` to its own `bindings` array,
and advances its own `nextBinding` local to the result's `nextBindingIndex` (unused after, since
channels are already last in that function's own order ‚Äî kept for symmetry/clarity, not because
anything after it needs the value). `executeFullscreenFragment` calls
`buildChannelBindGroupEntries(device, channelBindings, input.channelTargets ?? new Map())` in place
of its own `for (const [channel, bindings] of channelBindings)` loop ‚Äî but keep the **existing**
`for (const channel of channelBindings.keys()) { if (!input.channelTargets?.has(channel)) throw ... }`
pre-check exactly as it is (fail-fast before pipeline creation; `buildChannelBindGroupEntries`'s own
internal check is a second, cheaper-to-hit safety net, not a replacement for the early one).

**Gate for this step: every existing `fullscreenFragment.test.ts` test passes completely
unmodified** ‚Äî the direct, mechanical proof this refactor changed nothing observable.

### 2. Generalize `assembleKernelFragmentModuleAsync`

`consumers/kernelFragment.ts`. **First, and easy to miss ‚Äî a pre-routing review of an earlier draft
of this brief caught that it was missing entirely:** `buildKernelGraphEvalFn`
(`kernelFragment.ts:82-87`) still hardcodes `return ${resultExpr} * tint[0];`. Removing only
`assertSupportedKernelFragmentShape` (below) is not enough on its own ‚Äî `stage.fragmentKernel`
declares `bindings: []`, so this line would compile WGSL referencing an **undeclared** `tint`
identifier the moment any real, non-scaffold graph runs through this path. Fix `buildKernelGraphEvalFn`
itself:

```ts
function buildKernelGraphEvalFn(outputName: string, body: string[], resultExpr: string): string {
	return `fn graph_eval_${outputName}(position: vec4f) -> vec4f {
${body.map((line) => `\t${line}`).join('\n')}
\treturn ${resultExpr};
}`;
}
```

No parameter, no conditional ‚Äî the tint-multiply was always scaffold-only, and (per Context) there is
still no general mechanism for graph-authored WGSL to reference an arbitrary kernel-declared binding
by name; inventing a parameterized version of this function would be designing that mechanism, which
is explicitly out of scope. **Consequence, called out explicitly so it isn't mistaken for an
unrelated regression:** F3.6.2's own real-device test, `'uses a live tint buffer to affect rendered
pixels'`, depended on this exact multiply ‚Äî with it removed, `tint`'s value no longer affects
anything the WGSL body computes, so that specific assertion (`halfTint` ‚âà half of `fullTint`) will
now fail if left as-is. Gate item 2 below folds its replacement into Gate item 7's binding-order
proof rather than deleting the coverage outright ‚Äî a kernel-declared `tint` binding, still declared
and still resolved into a real bind group entry, just no longer read by the WGSL body it doesn't
need to be for this milestone's purposes.

Second, remove `assertSupportedKernelFragmentShape` (the function and its one call site) entirely ‚Äî
`buildKernelBindingDecls`'s own existing call to `bindingDeclKindForTemplate` already throws a clear
error for a structurally malformed template (bad `resourceKind`/`access` combination); no additional
scaffold-shape gate is needed once real binding-set support exists. Remove both rejection `throw`
blocks (ShaderToy uniform, channel textures). Replace the single-kernel-bindings-only assembly with
the full ordering, matching Verified current state's documented order (kernel-declared, then
uniform, then params, then channels):

```ts
const kernelBindingDecls = buildKernelBindingDecls(bindings, wgslTypes);
const usesShaderToyHost = wgslReferencesShaderToyUniform(libraryWithEval);
const hasGraphParams = emitted.params.length > 0;

let nextBinding = nextFreeBindingIndex(bindings);
const derivedBindings: BindingDecl[] = [];
if (usesShaderToyHost) {
	derivedBindings.push({ group: 0, binding: nextBinding, name: 'u', kind: 'uniform', wgslType: 'ShaderToyUniforms' });
	nextBinding += 1;
}
const paramsBindingIndex = nextBinding;
if (hasGraphParams) {
	derivedBindings.push({ group: 0, binding: paramsBindingIndex, name: 'params', kind: 'uniform', wgslType: 'GraphParams' });
	nextBinding += 1;
}
const channelAssembly = deriveChannelBindingDecls(emitted.usedChannels, nextBinding);
derivedBindings.push(...channelAssembly.bindings);

const allBindings: BindingDecl[] = [...kernelBindingDecls, ...derivedBindings];
```

`paramsBindingIndex` stays an unconditionally-computed `number` (reserved whether or not
`hasGraphParams` is true) ‚Äî matching F3.6.2's existing field exactly, not a breaking type change.
`code` assembly gains the same `SHADERTOY_UNIFORM_STRUCT_WGSL` prepend
`assembleFullscreenFragmentModuleAsync` already does when `usesShaderToyHost`:

```ts
const paramsStruct = hasGraphParams ? buildParamsStructWgsl(emitted.params) : '';
const { vertexWgsl, vertexCount } = await resolveVertexAssembly(graph, resolver);
const codeParts = usesShaderToyHost
	? [SHADERTOY_UNIFORM_STRUCT_WGSL, paramsStruct, vertexWgsl, stage.code]
	: [paramsStruct, vertexWgsl, stage.code];
const code = codeParts.filter(Boolean).join('\n\n');
```

`KernelFragmentAssembly` gains two fields: `usesShaderToyHost: boolean` and `channelBindings:
ReadonlyMap<number, {textureBinding: number; samplerBinding: number}>` (from
`channelAssembly.channelBindings`), both returned alongside the existing fields. This step's new
imports: `SHADERTOY_UNIFORM_STRUCT_WGSL` from `./shadertoyUniforms.js`, `deriveChannelBindingDecls`
from `./fullscreenFragment.js` (both used in `assembleKernelFragmentModuleAsync` itself).
Fix step 3 below needs two more from `./shadertoyUniforms.js` ‚Äî `packShaderToyUniforms` and
`SHADERTOY_UNIFORM_BYTE_LENGTH` ‚Äî and `buildChannelBindGroupEntries` from `./fullscreenFragment.js`,
used in `executeKernelFragment`, not here.

### 3. Extend `executeKernelFragment`

`KernelFragmentInput` gains `host: ShaderToyHostInputs` (required, matching
`FullscreenFragmentInput`'s own existing convention exactly ‚Äî even a call that doesn't end up using
ShaderToy uniforms still passes one) and `channelTargets?: ReadonlyMap<number, GPUTexture>`
(optional, matching the existing ShaderToy-path convention). Import `ShaderToyHostInputs` as a type
from `./fullscreenFragment.js` (already exported there).

```ts
export async function executeKernelFragment(input: KernelFragmentInput): Promise<FullscreenFragmentResult> {
	const { device, graph, output, bindings, width, height, host, kernelBindings } = input;
	if (width <= 0 || height <= 0) {
		throw new RangeError('width and height must be positive');
	}

	const assembly = await assembleKernelFragmentModuleAsync({
		graph, output, bindings, wgslTypes: kernelBindings.wgslTypes, resolver: input.resolver
	});
	for (const channel of assembly.channelBindings.keys()) {
		if (!input.channelTargets?.has(channel)) {
			throw new Error(`Missing channel target for channel ${channel}`);
		}
	}

	const pipeline = await createKernelFragmentPipeline(device, assembly.code);
	const resolved = resolveKernelBindings(bindings, 'fragment', kernelBindings.resourceIds);
	const bindGroupEntries = buildComputeBindGroupEntries(resolved, kernelBindings.resources);

	let uniformBuffer: GPUBuffer | null = null;
	if (assembly.usesShaderToyHost) {
		const uBinding = assembly.bindings.find((b) => b.name === 'u');
		if (!uBinding) throw new Error('assembly reports usesShaderToyHost but declared no u binding');
		const uniformData = packShaderToyUniforms({
			width, height, iTime: host.iTime, iMouse: host.iMouse, iFrame: host.iFrame
		});
		uniformBuffer = device.createBuffer({
			label: 'kernel-fragment-shadertoy-uniforms',
			size: alignTo(SHADERTOY_UNIFORM_BYTE_LENGTH, 16),
			usage: GPUBufferUsage.UNIFORM | GPUBufferUsage.COPY_DST
		});
		device.queue.writeBuffer(uniformBuffer, 0, uniformData);
		bindGroupEntries.push({ binding: uBinding.binding, resource: { buffer: uniformBuffer } });
	}

	let graphParamsBuffer: GPUBuffer | null = null;
	if (assembly.params.length > 0) {
		// ...unchanged from F3.6.2's existing block, still keyed on assembly.paramsBindingIndex...
	}

	if (assembly.channelBindings.size > 0) {
		bindGroupEntries.push(
			...buildChannelBindGroupEntries(device, assembly.channelBindings, input.channelTargets ?? new Map())
		);
	}

	// ...pipeline/bind group/render pass/readback unchanged from F3.6.2's existing body, plus
	// uniformBuffer?.destroy() alongside the existing graphParamsBuffer?.destroy()...
}
```

## Gate

## Out of scope

Combining kernel-*declared* bindings (F3.1's `KernelBindingTemplate[]`, resolved via
`resolveKernelBindings`) with a graph-authored body that references them **by name** ‚Äî there is
still no mechanism for graph-authored WGSL to reference an arbitrary kernel-declared binding the way
`input.channel`/`host.iTime` reference their own; `stage.fragmentKernel` sidesteps this by declaring
zero kernel bindings, and this milestone does not design that mechanism (not proven to be needed
yet ‚Äî no primitive requires it). **Whether a kernel-declared binding the body never reads can even
survive a real `layout:'auto'` pipeline creation** ‚Äî a pre-routing review flagged that WebGPU derives
the bind group layout from bindings the entry point actually references, so an unused declared
binding may not appear in the derived layout at all. Gate item 7 proves the assembly math only, not
a real render of that combination; if a future milestone needs a kernel-declared binding that
survives a real render, it will need either a body that references it (the "by name" mechanism just
above) or an explicit, non-`auto` pipeline layout ‚Äî neither designed here. Unifying
`HostBinding`/`KernelBindingTemplate` into one model ‚Äî (A)'s
own explicit non-recommendation, unchanged. Any vertex-side kernel path, invocation model, or new
vertex primitive (F3.6.4, F3.6.5). Wiring `GraphFrameExecutor`'s own per-pass loop, the editor preview
UI, or a bundled sample graph to make `stage.fragmentKernel` reachable from the running editor ‚Äî
F3.6.6's job once F3.6.4/F3.6.5 also land. Retiring or deprecating `stage.vertex`/`stage.fragment` ‚Äî
not decided, not this milestone's call, per (D)'s D2 recommendation. Any change to
`emitGraphEval.ts`, `host.*`/`input.channel` primitive registrations, or how the graph body itself
references host inputs ‚Äî all reused completely unchanged. Any change to F3.1-F3.6.1's landed
contracts.

## Handoff

‚Üí A real, additive fragment-kernel primitive (`stage.fragmentKernel`) exists whose binding set ‚Äî
ShaderToy uniforms, graph params, and texture channels, all genuinely *derived* from what a document
wires into it, not hand-declared ‚Äî generalizes logic `assembleFullscreenFragmentModuleAsync` already
proved out for the ShaderToy path years ago. A separate, *static* binding-declaration path (F3.1's
`KernelBindingTemplate[]`, resolved via `resolveKernelBindings`) still exists unchanged for a
primitive that wants one, though nothing currently reads such a binding from a graph-authored body ‚Äî
`stage.fragmentKernel` itself needs none, declaring `bindings: []`. Proven end to end by a
real-device test: a live GPU channel texture with a known color measurably appears in the rendered
output through the kernel-fragment path. `stage.fragment` and every existing sample are provably
unaffected (Gate items 1-2). Nothing wires this new primitive into the running editor yet ‚Äî that's
F3.6.6's job, once F3.6.4 (vertex kernel invocation model) and F3.6.5 (the real vertex-kernel
primitive) also exist, since a genuinely useful bundled sample plausibly wants both a real fragment
*and* a real vertex kernel to be worth showing. Request F3.6.4's own contract when picking it up; it
is not included here.
