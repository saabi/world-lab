## Context

Second of Foundation 3's sequence (F3.1 → **F3.2** → F3.3 → F3.4 → F3.5). This is the direct fix
for the parent review's "Mode A vertex-displacement gap": today there is no way to author vertex
logic in the graph at all, only to pick a grid resolution (see Verified current state). F3.2 gives
`assembleStageEntry`'s vertex/fragment templates (`packages/compiler/src/stageEntry.ts`) the
ability to declare and share a **typed varyings struct** beyond `position` — the missing piece
before a vertex kernel could ever hand a fragment kernel anything more than its clip-space
position.

**This milestone is compiler-internals only.** It does not wire a real vertex kernel into the
render pipeline, does not touch `packages/graph`, and does not add or change any
`emitGraphEval`-family function in `runtime-webgpu`. The existing fixed plane/grid path
(`pipelineVertex.ts`, `fullscreenFragment.ts`) is untouched and remains the only thing that
actually draws pixels — exactly the plan's own "the existing fixed grid path must keep working
throughout" constraint. Wiring a real graph-authored vertex kernel through these new templates,
replacing `stage.vertex`'s hardcode, is F3.4's job, not this one's.

Because nothing rendered changes and there is no new graph-authored behavior a person could open
the app and see, this milestone has **no manual/visual gate** and needs **no bundled sample** —
the same situation as F3.1 (pure types/codegen, no runtime consequence yet). The project's
standing "every brief with a manual/visual gate needs a bundled sample" instruction does not apply
here; the real visual proof (an actual displaced-vertex sample) is F3.5's job once F3.4 wires a
real kernel through this machinery.

## Verified current state

- **`assembleStageEntry`'s vertex branch is dead code today, exactly like its compute branch was
  before F3.1** (`packages/compiler/src/stageEntry.ts:91-93`, current landed state after F3.1).
  Confirmed by grep across `packages/*/src` (excluding tests): **zero** production call sites pass
  `shader.stage === 'vertex'` into `assembleStageEntry`. The one place that generically *could*
  reach it — `compileStagedConsumer`'s fallback branch in
  `packages/graph-editor/src/compiledWgsl.ts:207-212` (`callArgs: stage === 'vertex' ? ['vid',
  'iid'] : ...`) — is itself unreachable in practice: grep for `stage: 'vertex'` across
  `packages/*/src` (excluding tests) returns zero matches, so no `ProceduralConsumer` is ever
  constructed with a vertex stage. `stageEntry.test.ts:62-67`'s "supports a vertex stage" test is
  the *only* thing currently exercising this branch.
- **The real "vertex stage" today is a completely separate, hand-written code path that never
  goes through `assembleStageEntry` at all**: `assemblePipelineVertexWgsl`
  (`packages/runtime-webgpu/src/pipelineVertex.ts:50-72`) emits its own `struct VSOut { @builtin(position)
  position: vec4f }` and its own `@vertex fn vs_main(...)` directly as a template string, driven by
  `resolvePipelineGeometryParams`'s grid-shape params (`resU`/`resV`/`width`/`height`/rotation) —
  never a graph-computed position. `fullscreenFragment.ts`'s `resolveVertexAssembly`
  (lines 82-98) calls this, not `assembleStageEntry`, and concatenates its output ahead of
  `assembleStageEntry`'s **fragment**-only output (line 194-201: `assembleStageEntry` is called
  there with `consumerShader` whose `stage` is always `'fragment'`). This confirms two separate,
  independent "vertex stage" concepts exist in the codebase today — the dead generic template in
  `stageEntry.ts`, and the real hand-rolled one in `pipelineVertex.ts` — and this milestone touches
  only the former.
- **The `emitGraphVec4Eval`/`emitGraphVec3Eval`/`emitGraphScalarEval` family lives in
  `runtime-webgpu`, not `compiler`** (`packages/runtime-webgpu/src/emitGraphEval.ts:194-216`),
  confirmed by direct read. Each one walks a single output port's upstream slice and returns one
  `body`/`resultExpr` pair (`EmittedGraphEval`, lines 34-43) — there is no existing mechanism for
  emitting *several* named, typed outputs (e.g. `position` plus two independent varyings) from one
  shared graph slice without calling the family multiple times and accepting the redundant
  upstream-node re-walk between calls. The plan's phrase "reusing the same graph-slicing/emission
  machinery already proven for fragment kernels... not a new codegen path" describes a **forward
  compatibility** property this milestone must preserve (whatever eventually produces a varying's
  WGSL body must be able to plug into `outputFns` the same way `emitGraphVec4Eval`'s output already
  does), not an in-milestone deliverable — no `runtime-webgpu` production code changes here. See
  Out of scope.
- **`StageEntryOptions` already has the right shape to extend for this**
  (`packages/compiler/src/stageEntry.ts:14-27`): `outputFns: Record<string, string>` is already a
  name→fn-name map (currently used for exactly one name, whichever `output` drives the entry);
  `callArgs` is already a single shared argument list applied to whatever fn is called. Adding
  varyings only requires *more entries* in `outputFns` (one per varying, keyed by the varying's own
  `name`) called with the same `callArgs` — no new argument-plumbing concept needed.
- **F2.5's "reject the mismatch clearly" precedent** (`F2.5-foundation-2-proof.md`, Fix step 3:
  `gridWidth === input.width && gridHeight === input.height`, reject a mismatch with a clear error
  rather than silently producing an inconsistent result) is the model for this milestone's
  `assertVaryingsMatch` — a single, clear thrown error, not a silently-broken shared struct
  referencing an undeclared field.
- **This milestone has no code dependency on F3.1's landed types.** `KernelBindingTemplate`/
  `ResolvedKernelBinding` (F3.1, `packages/graph/src/implementation.ts`) describe *resource*
  bindings (buffers/textures/samplers); varyings are an unrelated concept (typed data passed
  between a vertex and fragment invocation within one draw call, never a `ResourceBinding`). F3.2
  does not import anything from `@world-lab/graph` beyond what `stageEntry.ts` already imports
  (`KernelBindingTemplate`, used only by F3.1's unrelated `bindingDeclKindForTemplate`). The plan
  sequences F3.2 after F3.1 for narrative/roadmap reasons (both are prerequisites for F3.4), not
  because either milestone's code depends on the other's.

## Fix

### 1. `VaryingDecl` — one typed varying field

In `packages/compiler/src/stageEntry.ts`, alongside `BindingDecl`:

```ts
/** A named, typed value passed from a vertex kernel to its paired fragment kernel. */
export interface VaryingDecl {
	/** WGSL struct field name — also the `@location(n)` field identifier. */
	name: string;
	/** WGSL scalar/vector type, e.g. 'f32', 'vec2f', 'vec3f', 'vec4f'. */
	wgslType: string;
}
```

### 2. `varyingsStructWgsl` — the shared struct, declared once

Same file:

```ts
export function varyingsStructWgsl(structName: string, varyings: readonly VaryingDecl[]): string {
	const seen = new Set<string>();
	for (const varying of varyings) {
		if (seen.has(varying.name)) {
			throw new Error(`Duplicate varying name: "${varying.name}"`);
		}
		seen.add(varying.name);
	}
	const fields = varyings
		.map((varying, index) => `\t@location(${index}) ${varying.name}: ${varying.wgslType},\n`)
		.join('');
	return `struct ${structName} {\n\t@builtin(position) position: vec4f,\n${fields}};`;
}
```

Locations are assigned sequentially from the array's declaration order — the caller controls
layout by controlling array order. `varyingsStructWgsl(structName, [])` must produce
byte-identical text to today's hardcoded position-only struct (Gate item 1) — this is what keeps
the zero-varyings case an exact no-op.

This struct is declared **once** by the vertex side (Fix step 3) and referenced by name (not
redeclared) by the fragment side (Fix step 4) — both entry points end up in the same concatenated
WGSL module, exactly as `position`-only vertex/fragment output already is today
(`fullscreenFragment.ts` concatenates `vertexWgsl` ahead of `stage.code`, one module, two entry
points). `assembleStageEntry` remains text-assembly-only (per its own existing doc comment,
line 69-70) — it does not verify the struct it references is actually declared elsewhere in the
final module; that stays the caller's responsibility, same as it already is for `bindings` today.

### 3. Vertex template gains `varyings`

Replace the vertex branch (`stageEntry.ts:91-93` in the current landed file) with:

```ts
} else if (shader.stage === 'vertex') {
	const varyings = opts.varyings ?? [];
	const structName = opts.varyingsStructName ?? 'VSOut';
	const struct = varyingsStructWgsl(structName, varyings);
	const assignments = varyings.map((varying) => {
		const varyingFn = opts.outputFns[varying.name];
		if (varyingFn === undefined) {
			throw new Error(
				`No output fn provided for varying '${varying.name}' (consumer ${shader.consumerId})`
			);
		}
		return `\tout.${varying.name} = ${varyingFn}(${args});`;
	});
	const body = [
		`\tvar out: ${structName};`,
		`\tout.position = ${fn}(${args});`,
		...assignments,
		`\treturn out;`
	];
	entry = `${struct}\n@vertex\nfn vs_main(@builtin(vertex_index) vid: u32, @builtin(instance_index) iid: u32) -> ${structName} {\n${body.join('\n')}\n}`;
}
```

With `opts.varyings` omitted or empty, this produces byte-identical output to today's vertex
branch (Gate item 4) — `struct` collapses to the existing hardcoded text (Fix step 2) and `body`
collapses to today's exact three lines.

### 4. Fragment template accepts a varyings-struct parameter

Replace the fragment branch (`stageEntry.ts:86-87`) with:

```ts
} else if (shader.stage === 'fragment') {
	const varyings = opts.varyings ?? [];
	if (varyings.length > 0) {
		const structName = opts.varyingsStructName ?? 'VSOut';
		entry = `@fragment\nfn fs_main(in: ${structName}) -> @location(0) vec4f {\n\treturn ${fn}(${args});\n}`;
	} else {
		entry = `@fragment\nfn fs_main(@builtin(position) position: vec4f) -> @location(0) vec4f {\n\treturn ${fn}(${args});\n}`;
	}
}
```

With `opts.varyings` empty (the default), this is byte-identical to today's fragment branch (Gate
item 7). With varyings present, the parameter becomes the shared struct (`in: VSOut` by default);
callers reference `in.position`/`in.<varyingName>` inside their `callArgs`/graph-eval body — this
is new API surface, not a change to the existing bare-`position`-param convention.

### 5. `StageEntryOptions` gains the two new fields

```ts
export interface StageEntryOptions {
	// ...unchanged...
	/** Named, typed values shared between a vertex/fragment kernel pair (vertex-only: produced;
	 *  fragment-only: consumed via the struct parameter). Empty/omitted is a no-op. */
	varyings?: VaryingDecl[];
	/** Struct type name shared by both stages. Defaults to `'VSOut'`. */
	varyingsStructName?: string;
}
```

### 6. `assertVaryingsMatch` — the "reject the mismatch clearly" check

Same file:

```ts
export function assertVaryingsMatch(
	vertexVaryings: readonly VaryingDecl[],
	fragmentVaryings: readonly VaryingDecl[]
): void {
	const vertexByName = new Map(vertexVaryings.map((v) => [v.name, v.wgslType]));
	const fragmentByName = new Map(fragmentVaryings.map((v) => [v.name, v.wgslType]));
	const problems: string[] = [];

	for (const [name, wgslType] of fragmentByName) {
		const vertexType = vertexByName.get(name);
		if (vertexType === undefined) {
			problems.push(`fragment expects varying "${name}" but the vertex kernel does not produce it`);
		} else if (vertexType !== wgslType) {
			problems.push(
				`varying "${name}" type mismatch: vertex produces ${vertexType}, fragment expects ${wgslType}`
			);
		}
	}
	for (const name of vertexByName.keys()) {
		if (!fragmentByName.has(name)) {
			problems.push(`vertex produces varying "${name}" that the fragment kernel does not declare`);
		}
	}

	if (problems.length > 0) {
		throw new Error(`Vertex/fragment varying mismatch:\n${problems.join('\n')}`);
	}
}
```

Deliberately an **exact** set match in both directions (not "fragment may consume a subset of what
vertex produces") — the simpler, more conservative rule, matching F2.5's own strict-equality
dimension check rather than inventing a subsetting policy nothing yet needs. This is not called
from inside `assembleStageEntry` (which only ever sees one stage's varyings at a time) — it is a
standalone check a caller (F3.4) runs against both sides' declared `varyings` arrays before
assembling either stage.

Do **not** wire this into `fullscreenFragment.ts`, `pipelineVertex.ts`, or any real render
pipeline — this milestone is template/codegen capability plus tests only.

### 7. One confirmatory device-validation test in `runtime-webgpu`

New file `packages/runtime-webgpu/src/kernelVaryingsDeviceCompile.test.ts`, mirroring
`typeRefDeviceCompile.test.ts`'s exact pattern (same `hasWebGPU` check, same `it.skipIf`, same
`requestGpuDevice`/`createShaderModule`/`getCompilationInfo` shape):

```ts
import { requestGpuDevice } from './device.js';
import {
	assembleStageEntry,
	varyingsStructWgsl,
	type ConsumerShader
} from '@world-lab/compiler';
import { describe, it, expect } from 'vitest';

const hasWebGPU =
	typeof globalThis.navigator !== 'undefined' &&
	'gpu' in globalThis.navigator &&
	globalThis.navigator.gpu !== undefined;

describe('kernel varyings device compile', () => {
	it.skipIf(!hasWebGPU)('compiles a shared varyings struct across a vertex/fragment pair', async () => {
		const vertexShader: ConsumerShader = {
			consumerId: 'v',
			stage: 'vertex',
			outputs: ['position'],
			code: 'fn vertex_pos() -> vec4f { return vec4f(0.0, 0.0, 0.0, 1.0); }\nfn vertex_uv() -> vec2f { return vec2f(0.0, 0.0); }',
			moduleIds: []
		};
		const fragmentShader: ConsumerShader = {
			consumerId: 'f',
			stage: 'fragment',
			outputs: ['color'],
			code: 'fn frag_color(in_uv: vec2f) -> vec4f { return vec4f(in_uv, 0.0, 1.0); }',
			moduleIds: []
		};
		const varyings = [{ name: 'uv', wgslType: 'vec2f' }];

		const vertex = assembleStageEntry(vertexShader, {
			output: 'position',
			outputFns: { position: 'vertex_pos', uv: 'vertex_uv' },
			varyings
		});
		const fragment = assembleStageEntry(fragmentShader, {
			output: 'color',
			outputFns: { color: 'frag_color' },
			callArgs: ['in.uv'],
			varyings
		});
		const code = `${vertex.code}\n\n${fragment.code}`;

		const { device } = await requestGpuDevice();
		try {
			const module = device.createShaderModule({ code });
			const info = await module.getCompilationInfo();
			expect(info.messages.filter((m) => m.type === 'error')).toEqual([]);
		} finally {
			device.destroy();
		}
	});
});
```

This proves the shared-struct design compiles as real WGSL, not just that the generated strings
match expected text.

## Gate

1. `varyingsStructWgsl(structName, [])` produces byte-identical text to today's hardcoded
   position-only `VSOut` struct — the direct backward-compatibility proof for Fix step 2.
2. `varyingsStructWgsl` emits one `@location(n)` field per varying in declaration order for a
   2-varying case (e.g. `vec2f` then `f32`), locations `0` and `1` respectively.
3. `varyingsStructWgsl` throws a clear error on a duplicate varying `name`.
4. `stageEntry.test.ts:62-67`'s existing "supports a vertex stage" test passes **unmodified** — no
   regression when `opts.varyings` is omitted.
5. `assembleStageEntry`'s vertex template, given 2 `varyings`, emits `out.position = ...` followed
   by one `out.<name> = <fn>(<args>)` assignment per varying in declaration order, each fn looked
   up from `outputFns` by the varying's own `name`, and returns the struct type name (default
   `'VSOut'`).
6. `assembleStageEntry` throws a clear error when a declared `varying.name` has no matching
   `outputFns` entry.
7. `stageEntry.test.ts:10-21`'s existing fragment test passes **unmodified** — the bare
   `@builtin(position) position: vec4f` param stays the default when `opts.varyings` is omitted.
8. `assembleStageEntry`'s fragment template, given non-empty `varyings`, emits `fn fs_main(in:
   VSOut) -> @location(0) vec4f` (struct-typed param) and does **not** re-declare the struct.
9. `assembleStageEntry`'s fragment template, given a custom `varyingsStructName`, types `fs_main`'s
   param with that name instead of the default `'VSOut'`.
10. `assertVaryingsMatch` accepts two equal varying lists declared in different orders without
    throwing.
11. `assertVaryingsMatch` throws a clear error listing every varying the fragment expects that the
    vertex does not produce.
12. `assertVaryingsMatch` throws a clear error listing every varying whose `wgslType` differs
    between vertex and fragment.
13. `assertVaryingsMatch` throws a clear error listing every varying the vertex produces that the
    fragment does not declare — proving the check is symmetric, not one-directional.
14. The new `runtime-webgpu` device-validation test (Fix step 7) compiles cleanly on a real device
    where available (`it.skipIf(!hasWebGPU)`, same convention as `typeRefDeviceCompile.test.ts`).
15. `check` **and** `test` green for `compiler`, `runtime-webgpu`, full workspace (clear every
    `packages/*/dist` first, per `AGENTS.md`'s documented stale-dist issue).

## Out of scope

Wiring a real vertex/fragment kernel pair into `fullscreenFragment.ts`, `pipelineVertex.ts`, or any
other real render pipeline — the fixed plane/grid path stays exactly as-is and is not touched by
this milestone (F3.4's job). Any change to `packages/graph` — no new `HostBinding` context or
graph-level primitive for reading a varying inside a fragment kernel's graph body (that requires
real per-document kernel wiring, deferred alongside F3.4). Any new function or change in
`runtime-webgpu/src/emitGraphEval.ts` — `emitGraphVec3Eval`/`emitGraphVec4Eval`/
`emitGraphScalarEval` are untouched; a future milestone calls the existing family once per
varying-producing output and feeds the resulting fn names into `outputFns`, which this milestone's
extension already accepts. Fragment-side varying subsetting (a fragment consuming fewer varyings
than the vertex produces) — `assertVaryingsMatch` requires an exact set match in both directions;
relaxing this is a straightforward future extension, not built here. Any bundled/pickable visual
sample — nothing rendered changes in this milestone (see Context). Compute kernels or dispatch
domains (F3.3). Resource bindings / `KernelBindingTemplate` (F3.1, already landed and untouched
here). `{ kind: 'command'; command: GpuCommandKind }` (Foundation 4's seam). Any change to
Foundation 1's, Foundation 2's, or F3.1's landed contracts.

## Handoff

→ `assembleStageEntry`'s vertex/fragment templates can express a shared, typed varyings struct
beyond `position` (`VaryingDecl`, `varyingsStructName`), with a single clear rejection point
(`assertVaryingsMatch`) for when a vertex/fragment pair disagrees on what's carried between them —
proven to compile as real WGSL on a real device, not just string-matched. The existing fixed
plane/grid rendering path is completely untouched and remains the only thing driving real pixels.
F3.3 (compute kernels and dispatch domains) is next in the plan's sequence — it has no code
dependency on this milestone either, and its own contract should be requested when appropriate.
Wiring a real graph-authored vertex kernel through these new templates, replacing `stage.vertex`'s
hardcode, is F3.4's job.
