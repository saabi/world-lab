## Context

Fourth of F3.6's tentative sequence, and the last purely-preparatory one — F3.6.5 (the real
vertex-kernel primitive) cannot be scoped correctly until this milestone answers three questions the
parent design document's own pre-drafting review flagged as genuinely open: what can a graph-authored
vertex kernel read (`vertex_index`/`instance_index`), how does it sample "this vertex's grid
position," and how does a graph-computed `vec3f` position become the clip-space `vec4f` the vertex
entry actually needs. The linked design discussion (three review rounds, no findings outstanding)
answers all three with a recurring theme: **reuse existing mechanisms, don't invent new ones** —
`host.fragCoord`'s exact registration shape for the two new host inputs, `procedural.metricPosition`'s
already-existing `positionExpr` override for geometry sampling, and the same caller-side wrapper
pattern `buildGraphEvalFn`/`buildKernelGraphEvalFn` already use for the `vec4f` lift.

**This milestone does not register a vertex-kernel stage primitive, and does not touch
`GraphFrameExecutor`, the editor, or any sample.** It builds and proves, in isolation, the three
pieces F3.6.5 will assemble into a real primitive: two new host-input primitives, two new
`emitGraphEval` branches, and one new, standalone function
(`assembleVertexKernelPositionModuleAsync`) that proves a graph wired to `procedural.metricPosition`
can be assembled into **real, compiling** vertex-stage WGSL that samples the pipeline's existing
plane-grid geometry — a "thin vertical slice," per the design document's own framing, not the full
vertex-kernel primitive. No manual/visual gate, no bundled sample — nothing here is reachable from
the editor; F3.6.6 is still the milestone that makes anything visible, once F3.6.5 also exists.

**A genuine new finding, surfaced while grounding this contract, not caught by the design
document's own three review rounds — flagged here rather than silently designed around.**
`resolveCoercion` (`packages/graph/src/coercion.ts:65-78`), the mechanism that lets an edge connect
two structurally-different-but-compatible port types, handles exactly two cases: `identity` (same
type) and `vec2f→vec3f` padding. **Nothing coerces a `u32` value into anything** — confirmed by
direct read, and confirmed no primitive anywhere in `packages/graph/src/primitives/math/` accepts or
produces a `u32` value (grepped, zero matches). This means `host.vertexIndex`/`host.instanceIndex`'s
own `u32` output, as scoped by the design document and reviewed correct on its own terms (`type:
{kind:'scalar', scalar:'u32'}`, not `dataType`), **has no real, non-test-only consumer anywhere in
this codebase today** — no graph author can wire it into existing f32-based math (a `remap`, a
`multiply`, anything) without a type-mismatch error, since no cast/coercion path exists. This does
**not** block this milestone's own gate items (see Gate item 2's test-only probe primitive, mirroring
an already-established pattern in this codebase for testing host-input dispatch in isolation), and it
does **not** affect section (I)'s geometry-position mechanism at all (`positionExpr` is raw WGSL-text
substitution, entirely outside the port/edge type system) — but it does mean this milestone delivers
a proven *mechanism*, not yet a *usable* graph-authoring capability for raw vertex/instance indices.
Whether that gap needs closing (a `math.u32ToF32`-style cast primitive, or extending
`resolveCoercion`) is a real, separate question this milestone does not resolve — see Out of scope.

## Verified current state

- **`host.fragCoord`'s exact registration shape** (`packages/graph/src/primitives/host/fragCoord.ts`),
  the direct template for both new primitives: `id`, `category`, `inputs: []`, one output port,
  `params: Type.Object({})`, `implementation: {kind:'host-input', binding: HostBinding}`, `metadata`,
  and an `evalCPU`. All four currently-registered `host.*`/`procedural.*` host-input primitives
  (`fragCoord`, `iResolution`, `iTime`, `channel`) declare `evalCPU`; none is a useful precedent for
  "no meaningful CPU value exists" since all four *do* have one.
- **`emitHostInput`'s exact current dispatch structure**
  (`packages/runtime-webgpu/src/emitGraphEval.ts:79-119`) — six `if` branches keyed on
  `binding.context`/`binding.key`, falling through to a final `throw` for anything unrecognized. Two
  new branches, matching this exact shape, are the entire code change needed for the vertex-invocation
  builtins:
  ```ts
  if (binding.context === 'stage-builtin' && binding.key === 'vertexIndex') {
  	return `let ${variable}: u32 = vid;`;
  }
  if (binding.context === 'stage-builtin' && binding.key === 'instanceIndex') {
  	return `let ${variable}: u32 = iid;`;
  }
  ```
  `vid`/`iid` are free identifiers here — valid only because whatever wraps the emitted body into a
  real WGSL function declares them as its own parameters (matching `assembleStageEntry`'s vertex
  branch signature, `packages/compiler/src/stageEntry.ts:237`, and this milestone's own
  `graph_eval_position(vid: u32, iid: u32)` wrapper below).
- **Port typing, confirmed exactly as the design document's own reviewed correction states:**
  `ValueDataType` (`packages/graph/src/types.ts:5`) has no `u32` member; `typeRefToDataType`
  (`packages/graph/src/dataType.ts:48-52`) returns `undefined` for a `u32` scalar `TypeRef`; `Port.type?:
  TypeRef` (`types.ts:69-70`) is the canonical field, `Port.dataType?: DataType` is `@deprecated...
  absent when no DataType equivalent exists` — exactly this case. `typeRefToWgsl`
  (`dataType.ts:102-114`) returns `'u32'` for `{kind:'scalar', scalar:'u32'}` directly — WGSL codegen
  itself has no issue with `u32`, only the legacy display-alias system does.
- **`resolveVertexAssembly`'s exact geometry-module-resolution pattern**
  (`packages/runtime-webgpu/src/consumers/fullscreenFragment.ts:92-107`), the pattern this
  milestone's new function must mirror, not reinvent: `const planeModule = await
  resolver.resolve('geometry.plane'); ... assemblePipelineVertexWgsl(geo, planeModule.source)` — a
  resolver call keyed on the literal moduleId `'geometry.plane'`, whose result is prepended verbatim
  ahead of the hand-written `vs_main`. This milestone's new assembly function performs the identical
  `resolver.resolve('geometry.plane')` call itself — it does not import or call
  `resolveVertexAssembly`/`assemblePipelineVertexWgsl` (both produce a *complete*, hand-written,
  non-graph-driven `vs_main` — the wrong shape to build on top of here).
- **`resolvePipelineGeometryParams`, exported, unchanged, directly reusable**
  (`packages/runtime-webgpu/src/pipelineVertex.ts:19-34`) — resolves `resU`/`resV`/`width`/`height`/
  rotation from a document's geometry source node. This milestone's tests construct a
  `PipelineGeometryParams` value directly (a plain object, no document/plan needed) rather than
  routing through this function, since there is no real pipeline document to resolve one from yet —
  F3.6.5 is the milestone that wires a real document's geometry params through.
- **`emitGraphVec3Eval`, landed F3.1-era, confirmed still unused by any real caller**
  (`packages/runtime-webgpu/src/emitGraphEval.ts:210-216`, a thin wrapper requiring `expectedOutputType:
  'vec3f'`) — this milestone gives it its first real caller, continuing this session's own repeated
  pattern (F3.6.2 did this for `buildComputeBindGroupEntries`, F3.6.3 for `deriveChannelBindingDecls`'s
  generalization).
- **`kernelVaryingsDeviceCompile.test.ts`'s exact real-device compile-proof pattern**
  (`packages/runtime-webgpu/src/kernelVaryingsDeviceCompile.test.ts`), the direct template for this
  milestone's own device-compile gate: `assembleStageEntry` to build WGSL text, `requestGpuDevice()`
  (from `./device.js`), `device.createShaderModule({code})`,
  `(await module.getCompilationInfo()).messages.filter(m => m.type === 'error')` asserted empty, then
  `device.createRenderPipelineAsync(...)` (paired with a trivial throwaway fragment shader, since a
  vertex module alone cannot form a complete pipeline) to prove it's link-usable, not just
  syntactically valid text.
- **`emitGraphEval.test.ts`'s existing `'dispatches host inputs by binding metadata rather than
  primitive id'` test** (`emitGraphEval.test.ts:62-92`) — the direct, already-established precedent
  for testing a new `emitHostInput` branch via a throwaway, test-only-registered primitive declaring
  the relevant `HostBinding`, independent of whether any *real* primitive with a compatible output
  type could wire into it today. This milestone's own emission-proof test follows this exact,
  existing pattern (see Context's coercion-gap finding for why a throwaway probe, not a real graph,
  is the correct and only currently-possible way to prove this).

### 3. New file: `packages/runtime-webgpu/src/vertexKernelPosition.ts`

```ts
import { assembleStageEntry, type WgslModuleResolver } from '@world-lab/compiler';
import type { GraphDocument, PortRef } from '@world-lab/graph';

import { emitGraphVec3Eval } from './emitGraphEval.js';
import {
	DEFAULT_PIPELINE_GEOMETRY_PARAMS,
	planeGridVertexCount,
	type PipelineGeometryParams
} from './pipelineVertex.js';

export interface VertexKernelPositionInput {
	graph: GraphDocument;
	/** The vec3f-producing graph output feeding the vertex kernel's clip-space position. */
	output: PortRef;
	geo?: PipelineGeometryParams;
	resolver: WgslModuleResolver;
}

export interface VertexKernelPositionAssembly {
	code: string;
	vertexCount: number;
}

// Duplicated from pipelineVertex.ts's own module-private helper of the same name/body — trivial
// (one ternary), not exported there, not worth a cross-module export for something this small and
// this unlikely to drift. Not an oversight; see F3.6.3's own precedent for the opposite call
// (resolveVertexAssembly/packGraphParams were exported because their logic is real and non-trivial).
function formatWgslFloat(value: number): string {
	return Number.isInteger(value) ? `${value}.0` : String(value);
}

/** WGSL call to the pipeline's existing plane-grid geometry function — the positionExpr override
 *  procedural.metricPosition's emitHostInput branch substitutes verbatim. Mirrors
 *  assemblePipelineVertexWgsl's own param-interpolation exactly (pipelineVertex.ts:50-72) — this is
 *  not a new geometry model, only a new place the same params get plugged in. */
function planeGridPositionExpr(geo: PipelineGeometryParams): string {
	return (
		`plane_grid_position(vid, ${geo.resU}u, ${geo.resV}u, ${formatWgslFloat(geo.width)}, ` +
		`${formatWgslFloat(geo.height)}, ${formatWgslFloat(geo.rotationX)}, ` +
		`${formatWgslFloat(geo.rotationY)}, ${formatWgslFloat(geo.rotationZ)})`
	);
}

function buildVertexPositionEvalFn(body: string[], resultExpr: string): string {
	return `fn graph_eval_position(vid: u32, iid: u32) -> vec4f {
${body.map((line) => `\t${line}`).join('\n')}
\treturn vec4f(${resultExpr}, 1.0);
}`;
}

export async function assembleVertexKernelPositionModuleAsync(
	input: VertexKernelPositionInput
): Promise<VertexKernelPositionAssembly> {
	const geo = input.geo ?? DEFAULT_PIPELINE_GEOMETRY_PARAMS;
	const planeModule = await input.resolver.resolve('geometry.plane');

	const emitted = emitGraphVec3Eval(input.graph, input.output, {
		positionExpr: planeGridPositionExpr(geo)
	});
	const evalFn = buildVertexPositionEvalFn(emitted.body, emitted.resultExpr);
	const libraryWithEval = `${planeModule.source.trim()}\n\n${evalFn}`;

	const stage = assembleStageEntry(
		{
			consumerId: 'vertex-kernel-position',
			stage: 'vertex',
			outputs: ['position'],
			code: libraryWithEval,
			moduleIds: []
		},
		{ output: 'position', outputFns: { position: 'graph_eval_position' }, callArgs: ['vid', 'iid'] }
	);

	return { code: stage.code, vertexCount: planeGridVertexCount(geo.resU, geo.resV) };
}
```

`assembleStageEntry`'s vertex branch with `opts.varyings` omitted (defaults to `[]`,
`stageEntry.ts:219`) produces a `VSOut` struct with only `@builtin(position)` — a position-only
vertex kernel, matching this milestone's explicit scope (varyings are F3.6.5's problem, per section
(K)). `ConsumerShader`'s exact shape (`consumerId`/`stage`/`outputs`/`code`/`moduleIds`) mirrors
`kernelFragment.ts`'s own `{...consumerShader, code: libraryWithEval}` construction pattern — no new
shape invented.

## Out of scope

**A `u32`→`f32` cast/coercion primitive, or extending `resolveCoercion` to handle `u32`** — the real
consequence of Context's coercion-gap finding: `host.vertexIndex`/`host.instanceIndex` have no
real (non-test-only) consumer in this codebase today, and this milestone does not change that. Not
proven necessary by any concrete use case yet — if F3.6.5 or a real sample later needs raw
per-vertex/per-instance index math combined with existing f32-based graph primitives, that's its own,
separate, evidence-backed decision. Registering any real vertex-kernel stage primitive, or wiring
`assembleVertexKernelPositionModuleAsync` into `PipelineGraphExecutor`/`GraphFrameExecutor`/the
editor — F3.6.5's job entirely; this milestone's new function is a standalone, tested building block,
not consumed by anything yet. Varying outputs, `assertVaryingsMatch`, or anything about the fragment
side of a vertex/fragment kernel pair — section (K), explicitly F3.6.5's problem. Enforcing
`HostBinding.stages` (making `validateGraph` reject `host.vertexIndex` wired into a fragment-stage
graph) — section (L), a real, separate, already-known gap, not created or fixed here. Any change to
`emitGraphVec3Eval`/`emitGraphScalarEval`/`emitGraphVec4Eval`'s own type-checking, `resolveCoercion`,
or any existing `host.*`/`procedural.*` primitive's registration. Any change to F3.6.1-3's landed
contracts.

## Post-landing verification (commit `3fcc253`)

Diff read in full against every Fix step — matches closely, with two things worth recording.

**A gap in this contract itself, not in the implementation.** Fix step 3's sketch of
`assembleVertexKernelPositionModuleAsync` only resolved `geometry.plane`'s WGSL module, on the
(incomplete) assumption that no other module dependency would ever be needed for a vertex-position
expression. The landed code adds `graphModuleSources`, a function this brief never asked for:
it walks `upstreamNodeIds`, collects each upstream primitive's `callableWgslSource` (if any), and
recursively resolves each module's own `dependencies` — closely mirroring
`packages/compiler/src/codegen.ts`'s `generateWgsl`, which does the same walk for the fragment-side
`compileGraph` path. The landed test suite proves this is real, exercised logic, not dead code: the
second `vertexKernelPosition.test.ts` test wires `transform.translate`/`vector.vec3f` — both
`wgsl-function`-backed primitives with their own callable modules — into the position expression,
and the assembled code correctly includes `fn makeVec3f(...)`/`fn translate(...)` alongside
`plane_grid_position`.

**One real, moderate-severity, non-blocking gap in that new function, relative to the code it
mirrors:** `generateWgsl`'s own `visit()` has a third set, `visiting: Set<string>`, specifically to
break circular module dependencies (`if (visiting.has(moduleId)) return;`, checked before resolving
a module's own dependencies). `graphModuleSources`' `visit()` has no equivalent — a circular module
dependency would recurse indefinitely here where the original function it mirrors doesn't. Neither
landed test exercises a circular case (none exists among `geometry.plane`/`transform.translate`/
`vector.vec3f`'s actual dependencies today), so this is not a currently-reachable bug, and both
tests pass. It is, however, a real correctness gap in new code F3.6.5 is expected to build directly
on top of. Two reasonable fixes, neither attempted here since this reviewer does not edit an
implementer's landed files: (a) replace `graphModuleSources` with a direct `compileGraph` call,
matching `kernelFragment.ts`'s/`fullscreenFragment.ts`'s own established pattern for exactly this
concern — the more consistent fix, and it would delete this duplicated logic entirely rather than
patch it; or (b) add the missing `visiting` guard if there's a reason to keep the standalone walker.
Flagged here, and in `_TASK_BOARD.md`'s closeout, for F3.6.5's contract to pick up explicitly — not
silently inherited.

Independently re-verified, not diff-only: `check`/`test`/`build` re-run clean after clearing every
`packages/*/dist` (`runtime-webgpu` 182 passed/8 skipped, up from F3.6.3's 178/8 baseline; `graph`
clean; full workspace `check` clean across all 12 packages and both apps). Both new real-device
compile/link tests independently re-run scoped to their own package with `--reporter=verbose`,
confirmed to execute (231ms and 15ms respectively, real adapter/device warnings logged) rather than
silently skip. The workspace-wide `dataType.test.ts` guard fix (`if (port.dataType === undefined)
continue;`) is correct and appropriately narrow — it only skips the canonical-form check for ports
