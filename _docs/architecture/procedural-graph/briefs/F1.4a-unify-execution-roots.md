## Context

Last of five milestones (F1.1 → F1.2 → F1.5 → F1.3 → **F1.4a**). **This is the final milestone —
once it lands, Foundation 1 is complete** and a follow-up document sequences Foundation 2
separately (not written yet, deliberately).

`GraphDocument.outputs` + `GraphDocument.consumers` + role-metadata node detection (`isPipelineTarget`/
`isMeshTarget`) are three overlapping mechanisms for "what should execute," reconciled today by
hand-written glue (`effectiveOutputs`/`effectiveConsumers`/`derivePipelineConsumers`/
`deriveMeshTargets`). This milestone replaces all three with one rule — sink nodes (F1.3's
`implementation.kind === 'sink'`) are the only execution roots — and ships a real `GraphDocument`
schema migration (v1 → v2) so no existing saved graph silently loses its execution roots when
`consumers` is retired.

**Standing instruction for this brief specifically:** bundle hardcoded, checked-in sample graphs
demonstrating the migration paths a human can load and visually verify — not just headless test
fixtures. See Fix step 8 and Gate item 3.

## Fix

Work through these roughly in order — later steps depend on earlier ones existing.

1. Move `mintNodeId`/`mintEdgeId`/`collectNodeIds`/`collectEdgeIds`/`dedupeGraphIds` (all five,
   `packages/graph-editor/src/graphIds.ts`) into `packages/graph`. `graph-editor` re-exports them
   for existing call sites. Do the same for a canonical default-geometry constant: define it (or a
   graph-owned constructor producing it) in `packages/graph`; `runtime-webgpu`'s
   `DEFAULT_PIPELINE_GEOMETRY_PARAMS` references that canonical value instead of migration needing
   to import from `runtime-webgpu`.
2. Extract a standalone `compileConsumers` operation in `packages/compiler`:
   ```ts
   export async function compileConsumers(
     doc: GraphDocument,
     descriptors: ProceduralConsumer[],
     resolver: WgslModuleResolver
   ): Promise<GraphCompileResult>; // today's per-descriptor sliceGraph + generateWgsl loop,
                                    // extracted verbatim from compileGraph
   ```
   `compileGraph` becomes a thin wrapper: `compileConsumers(doc, opts.consumers ?? doc.consumers,
   resolver)`. Implement the `legacy.consumerSink` `SinkCompilerAdapter` (F1.3's interface, in
   `packages/compiler`) to translate a resolved `SinkInvocation`'s `{type, stage, outputs}` payload
   into a `ProceduralConsumer` descriptor. **Collect every legacy-sink invocation across the whole
   document first, then call `compileConsumers(doc, allDescriptors, resolver)` once** — calling it
   per-sink resets `moduleUseCount`/`sharedModuleIds` bookkeeping on each call, so a module shared
   across two legacy sinks would never be detected as shared. This is the concrete regression the
   test in Gate item 4 exists to catch.
3. Type the v1/v2 boundary explicitly:
   ```ts
   interface GraphDocumentV1 extends /* today's shape */ { version: '1'; consumers: ProceduralConsumer[] }
   interface GraphDocumentV2 extends /* new shape */ { version: '2' } // no consumers field
   type GraphDocument = GraphDocumentV2; // canonical, forward-looking alias

   function migrateGraphDocument(doc: GraphDocumentV1 | GraphDocumentV2): GraphDocumentV2;
   ```
   `deserializeGraph` (`packages/graph/src/serialize.ts`) parses as `unknown`/a union, branches on
   `version`, and returns `migrateGraphDocument`'s result — composing with the existing
   `normalizeGraphSemantics` call already there, not replacing it.
4. Build the v1→v2 migration function around a **registry**, keyed by consumer `type`:
   ```ts
   interface GraphMigrationPatch { nodes: Node[]; edges: Edge[]; outputs?: GraphOutput[] }
   type LegacyConsumerMigration = (consumer: ProceduralConsumer, doc: GraphDocument) => GraphMigrationPatch;
   ```
   - `'preview'` → synthesize a `preview.fieldSink` node (new primitive, `kind: 'sink'`, per F1.3 —
     its `SinkDefinition` and `params` schema follow the exact shape F1.3's `test.compatibilitySink`
     already proved: a real TypeBox `params` schema holding the referenced output name,
     `deriveInvocation` resolving that name against `doc.outputs`, `dependencies` populated from the
     resolved port). Any type is valid for this generic field-preview sink.
   - `'image'` → **only if the wired output's type is verified `vec4f`** (check per-document, not
     assumed): synthesize the full chain — a default `geometry.plane` (using the canonical
     graph-owned default geometry from step 1) → `buffer.persist` → `stage.vertex` →
     `stage.fragment` → `target.display`, wired from `consumer.outputs`. This makes today's
     `resolveVertexAssembly` implicit try/catch fallback (`packages/runtime-webgpu/src/consumers/fullscreenFragment.ts`)
     an explicit, permanent part of the migrated document. If the wired type is not `vec4f` (a
     malformed/synthetic edge case), fall through to `legacy.consumerSink` instead of forcing it.
   - Everything else — `'vertex-pass'`, `'fragment-pass'` when not `vec4f`, `'veg-compute'`,
     `'terrain-mesh'`, and any type not in the registry — migrates to a new **`legacy.consumerSink`**
     primitive (`kind: 'sink'`, params schema + `deriveInvocation` following the same
     `test.compatibilitySink`-proven shape, `payload` carrying `{type, stage, outputs}` verbatim).
     Its runtime handler is the batched `compileConsumers`-calling bridge from step 2 — it does not
     invent new compilation logic. **Never silently discard an unrecognized type** — this is the
     primary safety net (this exact question has been under-researched twice already in this
     project's own history), not a backstop for hypotheticals.
   - **Process every `consumers` entry independently.** A document can have a real `target.display`
     sink satisfying one entry and a separate, unrelated entry with no matching node at all —
     checking "does the document have any real sink node" once and dropping the whole array on a
     match would silently discard the second entry. Check each entry individually; only entries
     with an already-existing matching sink are skipped.
5. Extend `pruneOutputsAndConsumers` (`packages/graph-editor/src/irAdapter.ts:107`) — or its
   post-migration v2 equivalent — so deleting an output referenced by a `preview.fieldSink`/
   `legacy.consumerSink` node's `params` updates or removes that node, mirroring today's behavior
   for `doc.consumers`. This is a routine editor mutation going forward, not a one-time migration
   concern.
6. Retire `GraphDocument.consumers`/`ProceduralConsumer` as a live authoring field once migration
   is proven. Update the 6 files reading `.consumers` (listed in Verified current state) to derive
   the same information from reachable sink nodes instead, one file at a time, with a parity test
   per file.
7. Add `discoverExecutionRoots(doc: GraphDocument): Node[]` — find every `kind: 'sink'` node, full
   stop, independent of pipeline shape. Teach `PipelineGraphPlan`
   (`packages/runtime-webgpu/src/pipelineGraph.ts`) to use it for sink-finding, replacing the
   hardcoded `findNode(doc, 'target.display')`/`findNode(doc, 'buffer.persist')` calls — but do
   **not** generalize `PipelineGraphPlan`'s planning logic beyond today's one fixed pipeline shape;
   that's Foundation 4's job.
8. **Bundle hardcoded, pickable sample graphs demonstrating both migration paths, per the standing
   instruction for this brief:**
   - Add a new sample to `graphBuilders.ts`/`samples.ts` that is a hand-written **v1**
     `GraphDocument`, deliberately sink-node-free, matching `fullscreenFragment.test.ts:65-86`'s
     exact shape (`host.fragCoord`/`host.iResolution`/`host.iTime` → `effect.cosinePalette`,
     `consumers: [{type: 'image', stage: 'fragment', outputs: [...]}]`, no `stage.vertex`/
     `stage.fragment`/`target.display` nodes at all). Label it something like "Migration — legacy
     fullscreen fragment (pre-sink)" so its purpose is obvious to whoever opens it. Loading this
     sample exercises the full `geometry.plane → buffer.persist → stage.vertex → stage.fragment →
     target.display` synthesis path end to end, visually — this is the migration path with no
     existing discoverable example.
   - Add the existing `defaultPreviewGraph()` to the pickable `GRAPH_SAMPLES` list too (it is
     currently load-on-start only, not in the picker) with a label noting it demonstrates the
     `'preview'`-type sink-free migration, so both migration paths are independently
     re-loadable/re-verifiable by a human at any time, not just implicitly exercised.

## Gate

1. Every already-shipped feature depending on `consumers`/`outputs`/role-detection — image-pipeline
   preview, mesh-gen preview, compiled-WGSL view, markup export, graph validation, undo/redo
   compile-signature caching — matches its appropriate parity category: byte-identical WGSL for
   compiled shader text; pixel-equivalent for rendered previews; canonical v2 serialization (not
   byte-identical v1 text) for markup export; a **browser visual check on both new bundled samples
   from Fix step 8** — load each in the editor, confirm it renders correctly, not headless-green
   alone.
2. The full fixture corpus (every bundled sample, `'preview'`, verified-`vec4f` `'image'`,
   `'vertex-pass'`, non-`vec4f` `'fragment-pass'`, `'veg-compute'`, `'terrain-mesh'`, and a
   hand-written unknown-type document) migrates v1→v2 with no silent loss of execution roots.
3. **The two new bundled samples from Fix step 8 are loaded and visually confirmed correct in the
   browser** — the legacy fullscreen-fragment sample renders the cosine-palette effect after full
   chain synthesis; `defaultPreviewGraph` (now picker-visible) renders identically to its current
   load-on-start appearance. Paste a screenshot of each per this project's visual-gate convention.
4. A test with at least two `legacy.consumerSink`-routed entries sharing a common upstream module
   proves that module is correctly detected as shared (`sharedModuleIds`) after migration — the
   concrete regression case for the batched-`compileConsumers` requirement; calling it per-sink
   would fail this test.
5. A test proves `compileGraph` (pre-migration path) and the `legacy.consumerSink` bridge
   (post-migration path, called once with the complete descriptor list) both call
   `compileConsumers` and produce identical `GraphCompileResult`s for the same descriptor set.
6. A test proves a document with a real `target.display` sink *and* a separate, unrelated
   `legacy.consumerSink`-routed entry migrates both correctly — neither dropped because the other's
   real node already existed.
7. A test proves `deserializeGraph` branches on `version` and returns a normalized `GraphDocumentV2`
   for both v1 and v2 input, and that `GraphDocumentV1`/`GraphDocumentV2` are distinct enough at the
   type level that v2-shape-assuming code doesn't type-check against a raw v1 value without going
   through `migrateGraphDocument` first.
8. A test proves deleting an output referenced by a `preview.fieldSink`/`legacy.consumerSink`
   node's `params` updates or removes that node, exercised as a routine editor mutation.
9. `effectiveOutputs`/`effectiveConsumers`/`derivePipelineConsumers`/`deriveMeshTargets` are
   rehomed (their logic already lives in F1.3's `SinkDefinition.deriveInvocation` implementations),
   not deleted outright until the standalone reconciliation functions have zero remaining callers.
10. A test wires a novel sink kind and proves `discoverExecutionRoots` finds it via
    `implementation.kind` alone — not `PipelineGraphPlan`, which stays scoped to the one known
    pipeline shape by design.
11. Migration determinism and idempotence: migrating the same v1 document twice independently
    produces identical node/edge IDs and serialized output; migrating an already-v2 document is a
    no-op; synthesized IDs are collision-free (using the relocated `packages/graph` ID facility,
    including `dedupeGraphIds`, from step 1).
12. Loading an old v1 saved document still opens and behaves identically after migration.
13. `check` **and** `test` green for every package, full workspace (clear every `packages/*/dist`
    first, per `AGENTS.md`'s documented stale-dist issue), with special attention to `graph-editor`.
