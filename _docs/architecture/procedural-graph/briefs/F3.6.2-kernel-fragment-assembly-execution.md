**Type:** new execution path + branch point, second milestone of the F3.6 pipeline-kernels

## Context

Second of F3.6's tentative sequence. F3.6.1 gave `target.display`'s presentation logic and
`planPipelineGraph` a way to *recognize* a non-standard vertex/fragment stage node
(`pipelineStageKind`) — but recognizing one and *doing something different once found* are two
separate gaps. Sub-problem (E), confirmed by direct read: `PipelineGraphExecutor.execute`
(`pipelineGraph.ts`) calls `planPipelineGraph` (which now correctly *discovers*
`plan.fragmentStageNode`/`plan.vertexStageNode` for a non-standard stage primitive, per F3.6.1),
then **discards both node ids** and calls `executeFullscreenFragment` unconditionally — which
always assembles the ShaderToy-style ad-hoc uniform/channel-binding module
(`assembleFullscreenFragmentModuleAsync`), regardless of what primitive
`plan.fragmentStageNode` actually is. A graph-authored kernel-based fragment stage discovered
correctly by F3.6.1 would still be **assembled and executed exactly like `stage.fragment`** —
wrong WGSL, wrong bindings, wrong resources — because nothing downstream of discovery branches on
what was discovered.

**This milestone builds the branch and a real second path, proven by a test-only fixture — not a
new production stage primitive.** Per this document's own sequencing note (and a prior review of
an earlier F3.6.x draft that found a "no-op/pass-through branch" claim unbuildable/untestable — see
[F3.6.1's brief](./F3.6.1-pipeline-stage-discovery.md), an analogous lesson from that milestone's
own pre-routing review), a branch with nothing real on the other side proves nothing. The real,
document-derived fragment kernel (F3.6.3) doesn't need to exist yet — a test-only fixture with a
**fixed** (not derived) binding set is enough to prove the wiring, since F3.1's binding model
already works for fixed binding sets (F3.6.3's job is making the binding set *derived from the
graph*, not making binding resolution work at all).

**Fragment side only.** Sub-problem (F) (vertex kernel invocation model — what a vertex kernel can
even read, `vertex_index`/`instance_index`/geometry attributes) is unresolved and is its own
milestone (F3.6.4) before its own primitive (F3.6.5). A fragment kernel has no equivalent gap:
`emitGraphEval`'s existing host-input vocabulary (`host.fragCoord`, `host.iTime`, `host.iResolution`,
`input.channel`) already works for fragment bodies — it's what the *existing* ShaderToy path uses
today. This milestone still threads `plan.vertexStageNode`'s identity through
`PipelineGraphExecutor`/`kernelFragment.ts` (cheap, and needed for a future vertex branch to hang
off), but does not add a second vertex-assembly path — `stage.vertex`'s fixed-grid vertex WGSL is
reused unchanged for both the ShaderToy path and the new kernel path in this milestone.

**Reuses, not reinvents, F3.1/F3.3's binding machinery.** This milestone resolves the design
document's own open "resource-routing" question: a kernel-based fragment stage's bindings route
through F3.1's `resolveKernelBindings` → `buildKernelBindingDecls` → `assembleStageEntry` →
`buildComputeBindGroupEntries` chain — the exact same functions F3.3's `executeComputeKernel`
already uses, given their first real **render**-pipeline consumer (`buildComputeBindGroupEntries`'s
name is compute-specific; its logic, confirmed by direct read, is not — it maps `ResolvedKernelBinding[]`
+ a resource map to `GPUBindGroupEntry[]`, nothing compute-stage-specific about it). Not renamed
here — reused exactly as landed, matching this session's standing rule of not touching F3.1-F3.5's
landed contracts.

**No manual/visual gate, no bundled sample.** No existing sample's rendering changes — `stage.fragment`
still exists, unmodified, still the only fragment-stage primitive any real sample uses. The new path
is proven by a real-device test rendering the test-only fixture directly, not by anything reachable
from the editor UI. A genuinely visible bundled sample is F3.6.6's job, once F3.6.3's real primitive
exists.

## Verified current state

- **`PipelineGraphExecutor.execute`, in full** (`pipelineGraph.ts:258-274`):
  ```ts
  async execute(input: PipelineGraphInput): Promise<FullscreenFragmentResult> {
  	const plan = planPipelineGraph(
  		input.graph,
  		input.output ? { output: input.output } : {}
  	);
  	this.cache.realizeGeometry(geometryCacheFingerprint(input.graph, plan));
  	return executeFullscreenFragment({
  		device: input.device,
  		graph: input.graph,
  		output: plan.fieldOutput,
  		resolver: input.resolver,
  		width: input.width,
  		height: input.height,
  		host: input.host,
  		target: input.target,
  		channelTargets: input.channelTargets
  	});
  }
  ```
  `plan.fragmentStageNode`/`plan.vertexStageNode` are computed by `planPipelineGraph` and then never
  read again — confirmed, `plan` is destructured for exactly `fieldOutput` (plus
  `geometryCacheFingerprint`'s own separate read of `plan.persistNode`/`plan.geometryNode`/
  `plan.geometryPrimitive`).
- **`executeFullscreenFragment`/`assembleFullscreenFragmentModuleAsync`, in full**
  (`consumers/fullscreenFragment.ts`) — the exact shape any new kernel path needs to run alongside,
  not replace:
  - `assembleFullscreenFragmentModuleAsync(graph, output, resolver)` compiles the graph slice
    feeding `output` (**not** the fragment-stage node itself — confirmed: `stage.fragment`/
    `n_fragment` is never passed to `compileGraph`; only whatever's wired into its `color` input is,
    via `output = plan.fieldOutput`). Emits via `emitGraphVec4Eval(graph, output, {shaderToy:true})`,
    wraps the result in `buildGraphEvalFn` (a hand-templated `fn graph_eval_${outputName}(position:
    vec4f) -> vec4f {...}` wrapper), derives `BindingDecl[]` ad hoc (ShaderToy uniform at binding 0
    if referenced, graph-params uniform next, then one texture+sampler pair per `input.channel`
    index used), and calls `assembleStageEntry` with `outputFns: {[outputName]:
    'graph_eval_${outputName}'}, callArgs: ['position']`.
  - `resolveVertexAssembly(graph, resolver)` (module-**private**, `fullscreenFragment.ts:82-98`)
    builds the fixed-grid vertex WGSL via `planPipelineGraph`+`resolvePipelineGeometryParams`+
    `assemblePipelineVertexWgsl` — needs exporting so the new kernel path can reuse it unchanged
    (Fix step 1); duplicating this logic would create two independently-maintained copies of the
    vertex-assembly path, a real drift risk.
  - `executeFullscreenFragment` creates the render pipeline (`createRenderPipeline`, tiny,
    module-private, `fullscreenFragment.ts:220-233` — trivial enough to duplicate locally in the new
    file rather than exporting a third thing), builds bind-group entries from whichever of
    uniform/params/channel bindings are present, draws, and reads back RGBA8 pixels
    (`fullscreenFragment.ts:333-361` — the exact readback pattern the new kernel path's own
    `FullscreenFragmentResult` needs to match, since it returns the same result type).
- **`EmitGraphEvalOptions.shaderToy` is confirmed dead — do not rely on it to gate anything.**
  Grepped `emitGraphEval.ts` for every read of `opts.shaderToy`/`opts?.shaderToy`: zero matches.
  `host.iTime`/`host.fragCoord`/`host.iResolution` emit `u.iTime`/`position.xy`/`u.iResolution`
  **unconditionally** (`emitGraphEval.ts:97-104`), regardless of the `shaderToy` flag — the only
  real overrides are the separate `iTimeExpr`/`fragCoordExpr`/`iResolutionExpr` options, which
  neither this milestone nor the existing ShaderToy path supplies. Passing `{shaderToy: false}` in
  Fix step 2 below is therefore a no-op, not a real switch to "procedural" host-input semantics —
  it's included only for parity with the existing call site's shape. The **real** mechanism that
  prevents a ShaderToy-uniform-referencing graph from silently producing broken WGSL is the
  post-hoc `wgslReferencesShaderToyUniform(wgsl)` text check (already exported,
  `fullscreenFragment.ts:111-113`) — the same mechanism the *existing* ShaderToy path already uses
  to derive `usesShaderToyHost` (`assembleFullscreenFragmentModuleAsync` doesn't branch on
  `shaderToy` either; it derives `usesShaderToyHost` from the emitted text after the fact, same as
  this milestone's rejection check does).
- **F3.1's binding chain, confirmed already stage-agnostic, zero changes needed:**
  `resolveKernelBindings(templates, ownerStage: ShaderStage, resourceIds)`
  (`packages/graph/src/kernelBinding.ts:98-118`) takes any `ShaderStage` including `'fragment'` —
  F3.3's own compute-only usage (`executeComputeKernel`) is a caller choice, not a signature
  restriction. `bindingDeclKindForTemplate`/`assembleStageEntry`'s fragment-stage branch
  (`packages/compiler/src/stageEntry.ts:206-214`, the `varyings.length === 0` case) accept
  `bindings: BindingDecl[]` unconditionally — no stage gating gates `storage-read`/`storage-read-write`
  bindings out of the fragment branch, confirmed by direct read of `bindingVar`
  (`stageEntry.ts:49-59`) and the fragment-branch template itself. `buildKernelBindingDecls`/
  `buildComputeBindGroupEntries` (`computeKernel.ts:40-85`) are equally generic — nothing in either
  function's body references `'compute'` specifically.
- **`PrimitiveImplementation`'s `{kind:'kernel'}` variant** (`packages/graph/src/implementation.ts:88`):
  `{ kind: 'kernel'; stage: ShaderStage; bindings: readonly KernelBindingTemplate[] }` — already
  supports `stage: 'fragment'`, confirmed by type, never yet exercised by any registered primitive
  (`target.computeBuffer`, F3.4's only kernel primitive, declares `stage` implicitly via its own
  compute-only executor, not via this field's `'fragment'` variant).
- **Not reachable without a graph-level field slice.** A kernel-based fragment stage still needs
  something wired into its `color` input (per F3.6.1's own port-name-contract note) — `compileGraph`
  compiles *that*, exactly as it does for `stage.fragment` today. This milestone's new path reuses
  `compileGraph`+`emitGraphVec4Eval` unchanged for that part; only the *binding declarations* and
  *bind group construction* differ from the ShaderToy path.

### 2. New file: `packages/runtime-webgpu/src/consumers/kernelFragment.ts`

```ts
import {
	assembleStageEntry,
	compileGraph,
	type BindingDecl,
	type WgslModuleResolver
} from '@world-lab/compiler';
import {
	resolveKernelBindings,
	type GraphDocument,
	type KernelBindingTemplate,
	type PortRef
} from '@world-lab/graph';

import { alignTo, rgba8BufferByteLength } from '../buffers.js';
import {
	buildComputeBindGroupEntries,
	buildKernelBindingDecls,
	type ComputeKernelResource
} from '../computeKernel.js';
import { buildParamsStructWgsl, emitGraphVec4Eval } from '../emitGraphEval.js';
import { createStandardLibraryResolver } from '../moduleResolver.js';
import {
	packGraphParams,
	resolveVertexAssembly,
	wgslReferencesShaderToyUniform,
	type FullscreenFragmentResult
} from './fullscreenFragment.js';

export interface KernelFragmentBindingInput {
	/** Binding name -> concrete WGSL type string (same shape as ComputeKernelInput). */
	wgslTypes: ReadonlyMap<string, string>;
	/** Binding name -> resourceId. */
	resourceIds: ReadonlyMap<string, string>;
	/** resourceId -> the real, already-allocated GPU resource. */
	resources: ReadonlyMap<string, ComputeKernelResource>;
}

export interface KernelFragmentInput {
	device: GPUDevice;
	graph: GraphDocument;
	/** The field-color port wired into the kernel fragment stage's `color` input. */
	output: PortRef;
	bindings: readonly KernelBindingTemplate[];
	resolver?: WgslModuleResolver;
	width: number;
	height: number;
	target: GPUTexture;
	kernelBindings: KernelFragmentBindingInput;
}

function findOutputName(doc: GraphDocument, output: PortRef): string {
	const match = doc.outputs.find(
		(candidate) => candidate.from.node === output.node && candidate.from.port === output.port
	);
	if (!match) {
		throw new Error(`Output port is not declared in graph.outputs: ${output.node}.${output.port}`);
	}
	return match.name;
}

/**
 * Combines the graph-emitted color expression with the kernel's own bindings. This one
 * multiply-by-tint shape is test-fixture-specific, not a generalized "kernel fragment body"
 * contract — designing that contract (how a real F3.6.3 primitive's body relates to its own
 * declared bindings) is explicitly out of scope here; see the brief's Out of scope.
 */
function buildKernelGraphEvalFn(outputName: string, body: string[], resultExpr: string): string {
	return `fn graph_eval_${outputName}(position: vec4f) -> vec4f {
${body.map((line) => `\t${line}`).join('\n')}
	return ${resultExpr} * tint[0];
}`;
}

/**
 * This milestone hardwires exactly one binding shape (a single read-only 'tint' buffer,
 * combined into the body by `buildKernelGraphEvalFn`'s hardcoded multiply) — it does not
 * implement a general kernel-fragment binding/body contract (that's F3.6.3's job). Any
 * `{kind:'kernel', stage:'fragment'}` primitive not matching this exact shape must fail loudly,
 * here, before any WGSL assembly — not produce a shader referencing an undefined `tint` binding
 * that only fails at device-compile time with no useful message. Binding *index* is not
 * constrained (see Fix step's binding-index note below) — only name/resourceKind/access/stage.
 */
function assertSupportedKernelFragmentShape(bindings: readonly KernelBindingTemplate[]): void {
	const tint = bindings.length === 1 ? bindings[0] : undefined;
	const matches =
		tint !== undefined &&
		tint.name === 'tint' &&
		tint.resourceKind === 'buffer' &&
		tint.access === 'read' &&
		tint.stages.includes('fragment');
	if (!matches) {
		throw new Error(
			'executeKernelFragment only supports the F3.6.2 scaffold binding shape (exactly one ' +
				'read-only buffer binding named "tint") — a general kernel-fragment binding/body ' +
				`contract is F3.6.3's job, not built here. Got: ${JSON.stringify(bindings)}`
		);
	}
}

/** Lowest binding index not already used by `bindings` — NOT `bindings.length`, since
 *  KernelBindingTemplate.binding is caller-declared and may be sparse/nonzero (F3.1 allows
 *  arbitrary nonnegative indices, not contiguous-from-0 ones). Using `.length` here would
 *  silently collide graph-params' own binding with a kernel binding at index 1; a higher sparse
 *  index would still put params at the wrong unused slot, just without the direct collision. */
function nextFreeBindingIndex(bindings: readonly KernelBindingTemplate[]): number {
	return bindings.reduce((max, binding) => Math.max(max, binding.binding), -1) + 1;
}

async function createKernelFragmentPipeline(
	device: GPUDevice,
	shaderCode: string
): Promise<GPURenderPipeline> {
	const module = device.createShaderModule({ label: 'kernel-fragment', code: shaderCode });
	return device.createRenderPipeline({
		label: 'kernel-fragment',
		layout: 'auto',
		vertex: { module, entryPoint: 'vs_main' },
		fragment: { module, entryPoint: 'fs_main', targets: [{ format: 'rgba8unorm' }] },
		primitive: { topology: 'triangle-list' }
	});
}

export async function executeKernelFragment(
	input: KernelFragmentInput
): Promise<FullscreenFragmentResult> {
	const { device, graph, output, bindings, width, height, kernelBindings } = input;
	if (width <= 0 || height <= 0) {
		throw new RangeError('width and height must be positive');
	}
	assertSupportedKernelFragmentShape(bindings);

	const resolver = input.resolver ?? createStandardLibraryResolver();
	const outputName = findOutputName(graph, output);
	const compiled = await compileGraph(graph, resolver, {
		consumers: [{ type: 'image', id: 'kernel-fragment', stage: 'fragment', outputs: [outputName] }]
	});
	const consumerShader = compiled.shaders[0];
	if (!consumerShader) {
		throw new Error('compileGraph produced no shaders');
	}

	const emitted = emitGraphVec4Eval(graph, output, { shaderToy: false });
	// Deferred to F3.6.3 — see Out of scope: combining kernel bindings with ShaderToy uniform
	// host inputs (host.iTime/host.iResolution/host.iMouse) or channel textures in the same
	// fragment kernel. host.fragCoord is NOT one of these — it emits `position.xy`, a plain
	// function parameter already passed to graph_eval_${outputName} regardless of path (see
	// Verified current state), so it needs no binding and works unmodified here. Confirmed:
	// wgslReferencesShaderToyUniform's regex only matches `u.iResolution|iTime|iFrame|iMouse` —
	// `position.xy` never matches it, so host.fragCoord correctly falls through this check.
	const evalFn = buildKernelGraphEvalFn(outputName, emitted.body, emitted.resultExpr);
	const libraryWithEval = `${consumerShader.code}\n\n${evalFn}`;
	if (wgslReferencesShaderToyUniform(libraryWithEval)) {
		throw new Error(
			'Kernel-based fragment stages do not yet support ShaderToy host uniform inputs ' +
				'(host.iResolution/host.iTime/host.iMouse) — deferred, see F3.6.2 Out of scope. ' +
				'(host.fragCoord is fine — no uniform binding needed.)'
		);
	}
	if (emitted.usedChannels.length > 0) {
		throw new Error(
			'Kernel-based fragment stages do not yet support channel textures (input.channel) — ' +
				'deferred, see F3.6.2 Out of scope'
		);
	}

	const kernelBindingDecls = buildKernelBindingDecls(bindings, kernelBindings.wgslTypes);
	const hasGraphParams = emitted.params.length > 0;
	const paramsBindingIndex = nextFreeBindingIndex(bindings);
	const allBindings: BindingDecl[] = [...kernelBindingDecls];
	if (hasGraphParams) {
		allBindings.push({
			group: 0,
			binding: paramsBindingIndex,
			name: 'params',
			kind: 'uniform',
			wgslType: 'GraphParams'
		});
	}

	const stage = assembleStageEntry(
		{ ...consumerShader, code: libraryWithEval },
		{ bindings: allBindings, outputFns: { [outputName]: `graph_eval_${outputName}` }, callArgs: ['position'] }
	);

	const paramsStruct = hasGraphParams ? buildParamsStructWgsl(emitted.params) : '';
	const { vertexWgsl, vertexCount } = await resolveVertexAssembly(graph, resolver);
	const code = [paramsStruct, vertexWgsl, stage.code].filter(Boolean).join('\n\n');

	const pipeline = await createKernelFragmentPipeline(device, code);
	const resolved = resolveKernelBindings(bindings, 'fragment', kernelBindings.resourceIds);
	const bindGroupEntries = buildComputeBindGroupEntries(resolved, kernelBindings.resources);

	let graphParamsBuffer: GPUBuffer | null = null;
	if (hasGraphParams) {
		const graphParamsData = packGraphParams(width, height, emitted.params, graph);
		graphParamsBuffer = device.createBuffer({
			label: 'kernel-fragment-graph-params',
			size: alignTo(graphParamsData.byteLength, 16),
			usage: GPUBufferUsage.UNIFORM | GPUBufferUsage.COPY_DST
		});
		device.queue.writeBuffer(
			graphParamsBuffer,
			0,
			graphParamsData.buffer,
			graphParamsData.byteOffset,
			graphParamsData.byteLength
		);
		bindGroupEntries.push({
			binding: paramsBindingIndex,
			resource: { buffer: graphParamsBuffer }
		});
	}

	const bindGroup = device.createBindGroup({
		layout: pipeline.getBindGroupLayout(0),
		entries: bindGroupEntries
	});

	const encoder = device.createCommandEncoder({ label: 'kernel-fragment' });
	const pass = encoder.beginRenderPass({
		label: 'kernel-fragment',
		colorAttachments: [
			{ view: input.target.createView(), loadOp: 'clear', storeOp: 'store', clearValue: { r: 0, g: 0, b: 0, a: 1 } }
		]
	});
	pass.setPipeline(pipeline);
	pass.setBindGroup(0, bindGroup);
	pass.draw(vertexCount);
	pass.end();

	// Readback: identical pattern to executeFullscreenFragment (fullscreenFragment.ts:333-361) —
	// same padded-row/copyTextureToBuffer/mapAsync sequence, not reproduced here verbatim; match it
	// exactly so FullscreenFragmentResult's contract (tight-packed RGBA8, no row padding) holds.
	const pixelBytes = rgba8BufferByteLength(width, height);
	const bytesPerRow = alignTo(width * 4, 256);
	const readbackBuffer = device.createBuffer({
		label: 'kernel-fragment-readback',
		size: bytesPerRow * height,
		usage: GPUBufferUsage.COPY_DST | GPUBufferUsage.MAP_READ
	});
	encoder.copyTextureToBuffer(
		{ texture: input.target },
		{ buffer: readbackBuffer, bytesPerRow, rowsPerImage: height },
		{ width, height }
	);
	device.queue.submit([encoder.finish()]);
	await readbackBuffer.mapAsync(GPUMapMode.READ);
	const mapped = new Uint8Array(readbackBuffer.getMappedRange());
	const pixels = new Uint8Array(pixelBytes);
	const rowBytes = width * 4;
	for (let y = 0; y < height; y++) {
		pixels.set(mapped.subarray(y * bytesPerRow, y * bytesPerRow + rowBytes), y * rowBytes);
	}
	readbackBuffer.unmap();
	graphParamsBuffer?.destroy();
	readbackBuffer.destroy();

	return { width, height, pixels };
}
```

(The encoder is created before the readback buffer above only for exposition ordering in this
snippet — implementer should create `encoder` once, before the render pass, matching
`executeFullscreenFragment`'s own ordering exactly; fix the snippet's ordering during
implementation, this is not a load-bearing detail.)

		input.graph,
