## Context

Third of five milestones (F1.1 → F1.2 → **F1.5** → F1.3 → F1.4a). **This milestone is now frozen
and approved for implementation** — do not revisit F1.1/F1.2 (done) or F1.3/F1.4a's design here;
each gets its own contract when it becomes next. F1.5 runs *before* F1.3 deliberately: F1.3's
`resource`/`kernel`/`command` primitive-kind placeholders want `TypeRef` to already exist, and
F1.1's deferred runtime-buffer element-typing needs it too — doing this later would mean designing
those placeholders twice.

`DataType` is a flat, closed union of exactly 20 string values (verified:
`packages/graph/src/types.ts:5-19` — 5 value types, 3 external-resource types, 8 pipeline-resource
types, 4 tuple types since F1.1's rename). No integer or matrix types exist at all. This milestone
introduces a structural type algebra (`TypeRef`) alongside `DataType` — not replacing it — capable
of expressing integers, matrices, and (as type-level stubs only, not fully wired — see Fix step 1)
structs, buffers, and textures with real parametrization, once Foundation 2/3 build the resource
and kernel model those need to actually function.

**The exact `TypeRef` shape, from the parent architecture review
(`elemental-webgpu-architecture-review.md:111-122`), since "as the parent review specifies" is a
pointer, not something to implement cold:**
```ts
type ScalarType = 'bool' | 'i32' | 'u32' | 'f32' | 'f16';

type TypeRef =
  | { kind: 'scalar'; scalar: ScalarType }
  | { kind: 'vector'; element: ScalarType; width: 2 | 3 | 4 }
  | { kind: 'matrix'; element: 'f32' | 'f16'; columns: 2 | 3 | 4; rows: 2 | 3 | 4 }
  | { kind: 'array'; element: TypeRef; length?: number }
  | { kind: 'struct'; id: string; fields: StructField[] }
  | { kind: 'buffer'; element: TypeRef; access: BufferAccess; usages: BufferUsage[] }
  | { kind: 'texture'; dimension: TextureDimension; sample: SampleType;
      format?: string; access?: StorageTextureAccess }
  | { kind: 'sampler'; filtering: boolean; comparison: boolean }
  | { kind: 'mesh'; vertex: TypeRef; index?: ScalarType }
  | { kind: 'command'; command: string }
  | { kind: 'legacy'; alias: DataType }; // added by this migration, not in the parent review —
                                          // see Fix step 2

interface StructField { name: string; type: TypeRef }
type BufferAccess = 'read' | 'read-write';
type BufferUsage = string; // placeholder — real usage-flag modeling is Foundation 2's job
type TextureDimension = '1d' | '2d' | '3d' | 'cube';
type SampleType = 'float' | 'depth' | 'sint' | 'uint';
type StorageTextureAccess = 'read' | 'write' | 'read-write';
```
**Only `scalar`, `vector`, `matrix`, `struct` (bare, for one test — see Gate), and `legacy` need to
actually work** for this milestone's own gate (compile, round-trip, or feed a real test primitive).
`array`/`buffer`/`texture`/`sampler`/`mesh`/`command` are introduced as **type-level stubs only** —
they must exist and type-check as part of the union (so the shape is complete and Foundation 2/3
don't need to redesign it), but nothing in this milestone exercises them with real codegen or
runtime behavior. Do not attempt to wire real buffer/texture semantics here — that's explicitly
Foundation 2's job (see Out of scope).

## Verified current state (beyond what the parent plan's own research named)

- **The three promotion functions are at these exact locations** (unchanged by F1.1/F1.2's
  renames, re-verified): `packages/compiler/src/groupCodegen.ts:32`, `packages/runtime-webgpu/src/emitGraphEval.ts:108`
  (both `function promoteExpr(expr: string, fromType: DataType, toType: DataType): string`), and
  `packages/runtime-cpu/src/evalGraph.ts:32` (`function coerceInputValue(value: CpuValue,
  fromType: DataType, toType: DataType): CpuValue`). All three currently only handle one case:
  `vec2f → vec3f`.
- **The 5 exhaustive-switch files, confirmed unchanged:** `packages/graph-editor/src/graphValidation.ts`,
  `previewBackend.ts`, `nodePaletteModel.ts`, `irAdapter.ts`, `previewBuffers.ts` — each has a
  `default: { const _exhaustive: never = dataType; ... }` pattern that must be updated last, once
  `TypeRef` is proven, per the plan's own sequencing.
- **The integer-schema rejection in `groupCodegen.ts` (lines ~44-160, `kind === 'integer'` checks,
  throwing `'Group param mappings do not support integer schemas (i32 graph ports are out of
  scope)'`) is a *different, unrelated* restriction — do not touch it.** That code rejects integer
  **group params** (a TypeBox `Type.Integer()` schema on a `GroupParamMapping`), not integer
  **ports**. This milestone's new integer-port test primitive (Fix step 6) should declare
  `params: Type.Object({})` (no params) to sidestep this entirely — proving an integer *port*
  works end-to-end doesn't require touching group-param support at all.
- **A within-package headless GPU test pattern already exists** for the device-compile gate item:
  `packages/runtime-webgpu/src/graphFrameExecutor.test.ts:12,43-45` — `const hasWebGPU = ...
  navigator.gpu !== undefined`, then `it.skipIf(!hasWebGPU)('...', async () => { const {
  requestGpuDevice } = await import('./device.js'); const { device } = await requestGpuDevice();
  ... })`. Use this exact pattern for the new integer/matrix test primitives' device-compile test —
  it belongs in `packages/runtime-webgpu`, not the scene-editor app's separate
  `apps/scene-editor/src/lib/planet/gpu/wgslCompile.test.ts` (a different, higher-level pattern for
  a different package — don't conflate the two).
- **`packages/schema`'s `TSchema`/`Type` (TypeBox) surface**, for the narrow value-schema
  conversion (Fix step 7): `packages/schema/src/schema.ts` exports `check`, `create`, `fields`,
  `fieldKind`, plus TypeBox's own `Type.Object`/`Type.Number`/etc. re-exported. The conversion only
  needs to handle scalar/vector/struct **value** shapes (matching `TypeRef`'s `scalar`/`vector`/
  `struct` kinds) — do not attempt buffer/texture/sampler conversion, they have no TypeBox
  equivalent at all.

## Fix

1. Add `TypeRef` (the full shape above, including the type-level-only stub variants) to
   `packages/graph/src/types.ts`, alongside `DataType` — do not replace it.
2. Add the `DataType ↔ TypeRef` mapping in `packages/graph/src/dataType.ts` (co-locate with
   `canonicalDataType`/`dataTypeToWgsl`):
   - `dataTypeToTypeRef(dataType: DataType): TypeRef` — **total, exhaustive, every one of the 20
     current `DataType` values named explicitly** (an `_exhaustive: never` check on the input
     switch, so this fails to compile if a future `DataType` value is added without a
     corresponding case). Value types (`f32`→`{kind:'scalar',scalar:'f32'}`,
     `vec2f`/`vec3f`/`vec4f`→`{kind:'vector',...}`, `bool`→`{kind:'scalar',scalar:'bool'}`) get real
     structural shapes. Everything else — `image`/`mesh`/`audio`/`geometry`/`varyings`/`texture`/
     `vertexBuffer`/`indexBuffer`/`renderTarget`/`bindGroup`/`storageBuffer` and all 4
     `tuple<T>` variants — maps to `{ kind: 'legacy', alias: dataType }` (the tuple case can unwrap
     its inner type recursively into `{kind:'legacy', alias: 'tuple<...>'}`, keeping the whole
     `DataType` string as the alias rather than trying to model it structurally — that's this
     milestone's explicit scope boundary, not a gap to fill).
   - `typeRefToDataType(type: TypeRef): DataType | undefined` — **partial**, returns `undefined`
     (not throwing) for any `TypeRef` with no legacy equivalent (struct, array, buffer, texture,
     sampler, mesh, command, or a scalar/vector/matrix combination that happens not to match an
     existing `DataType`, e.g. `{kind:'scalar', scalar:'i32'}`). Returns the alias directly for
     `{kind:'legacy', alias}`, and the matching `DataType` for scalar/vector shapes that do have one
     (f32/bool/vec2f/vec3f/vec4f).
3. Add `type: TypeRef` to `PortSpec` (`packages/graph/src/primitive.ts`) and `Port`
   (`packages/graph/src/types.ts`) as the canonical field — `dataType: DataType` becomes optional,
   a deprecated authoring alias, not a second source of truth:
   ```ts
   type PortSpecInput = Omit<PortSpec, 'type' | 'dataType'> &
     ({ type: TypeRef; dataType?: DataType } | { type?: never; dataType: DataType });

   interface PortSpec {
     type: TypeRef;
     dataType?: DataType;
   }
   ```
   Wrap the primitive-registration boundary the same way, so bare `dataType`-only registrations
   keep type-checking without every one of the 138 call sites needing a source change:
   ```ts
   interface NodePrimitiveInput extends Omit<NodePrimitive, 'inputs' | 'outputs'> {
     inputs: PortSpecInput[];
     outputs: PortSpecInput[];
   }
   function normalizePrimitiveInput(input: NodePrimitiveInput): NodePrimitive; // shared, derives
                                                                              // type from dataType
                                                                              // via step 2's mapping
   ```
   Update **both** `registerPrimitive` and `replacePrimitive`
   (`packages/graph/src/registry.ts` — note F1.2 already added a `normalizePrimitiveSemantics`
   wrapper here for `semantics`; this milestone's `normalizePrimitiveInput` should compose with or
   subsume that existing normalization, not duplicate it as a third pass) to accept
   `NodePrimitiveInput` and normalize through the same shared function — not two independently
   maintained normalization paths.
4. Split the promotion logic into a shared decision plus two backend-specific appliers — the three
   existing functions have genuinely different signatures (WGSL-text vs. actual-value) and cannot
   literally merge into one:
   ```ts
   interface CoercionPlan { /* whatever shape resolveCoercion needs to describe e.g. "pad Z with 0" */ }
   function resolveCoercion(from: TypeRef, to: TypeRef): CoercionPlan | null; // packages/graph,
                                                                              // shared by both backends
   function emitCoercion(plan: CoercionPlan, expr: string): string; // packages/compiler or
                                                                     // packages/runtime-webgpu —
                                                                     // replaces both promoteExpr()
                                                                     // copies
   function applyCoercion(plan: CoercionPlan, value: CpuValue): CpuValue; // packages/runtime-cpu —
                                                                          // replaces coerceInputValue()
   ```
   `resolveCoercion` must reproduce today's one existing rule (vec2f→vec3f, pad with 0) exactly —
   this is a behavior-preserving refactor of the promotion logic, not a redesign of what promotions
   are legal.
5. Migrate the 5 exhaustive-switch files (`graphValidation.ts`, `previewBackend.ts`,
   `nodePaletteModel.ts`, `irAdapter.ts`, `previewBuffers.ts`) **last**, once steps 1-4 are proven
   via the new test primitives (step 6) — these still switch on `DataType`, not `TypeRef`; no
   change to what they switch on, just confirm they still compile against the updated `PortSpec`/
   `Port` shape (which now also carries `type`, unused by these files for now).
6. Register two new test-only primitives (not part of the standard library, scoped to test files)
   proving the closed-type-system problem is actually fixed, not just "the union got bigger":
   - One primitive with an integer port (`{kind: 'scalar', scalar: 'i32'}` or `'u32'`), `params:
     Type.Object({})` (no group params — sidesteps the unrelated integer-param restriction in
     `groupCodegen.ts`, per the verified-state note above).
   - One primitive with a matrix port (`{kind: 'matrix', element: 'f32', columns: 3, rows: 3}` or
     similar).
   Both need: an `evalCPU` round-trip, correct WGSL generation, and an actual device-level compile
   (using the `it.skipIf(!hasWebGPU)` / `requestGpuDevice` pattern from `graphFrameExecutor.test.ts`,
   living in `packages/runtime-webgpu`).
7. Add a narrow, explicitly-scoped TypeBox-compatible conversion helper — **only** for the
   scalar/vector/struct value-schema subset that overlaps cleanly with `packages/schema`'s
   `TSchema`. Not a general bridge; buffer/texture/sampler/etc. have no TypeBox equivalent and
   should not be forced through this helper.

## Gate

1. All 138 existing primitive registrations validate and compile identically via the
   `DataType → TypeRef` mapping and the derived `type` field — zero behavior change. Run the full
   `graph`/`compiler`/`runtime-webgpu`/`runtime-cpu`/`graph-editor` test suites before and after,
   diff for any changed outcome.
2. `resolveCoercion` + `emitCoercion` produce identical WGSL text to both current `promoteExpr()`
   copies for the existing vec2f→vec3f case; `resolveCoercion` + `applyCoercion` produce identical
   values to `coerceInputValue()` for the same case.
3. The new integer-port and matrix-port test primitives (Fix step 6) round-trip through `evalCPU`,
   compile to correct WGSL, **and** pass an actual device-level compile via
   `it.skipIf(!hasWebGPU)`/`requestGpuDevice` in `packages/runtime-webgpu` — WGSL text generation
   and a green CPU eval alone are not sufficient; invalid generated WGSL passes both and fails only
   on GPU.
4. A test proves `typeRefToDataType` returns `undefined` for a bare `{kind: 'struct', id: 'test',
   fields: []}` (or similar, no legacy equivalent) — not throwing, not guessing.
5. A test proves `storageBuffer`/`texture`/`image`/`mesh`/`audio`/every `tuple<T>` variant all map
   to `{ kind: 'legacy', alias: <the original DataType string> }`, and that this round-trips back
   through `typeRefToDataType` correctly (returns the original alias).
6. A test proves `replacePrimitive` normalizes a `dataType`-only `NodePrimitiveInput` identically
   to `registerPrimitive` — both call the same `normalizePrimitiveInput` — and that this composes
   correctly with F1.2's existing `semantics`-normalization rather than duplicating or bypassing it.
7. An exhaustiveness test (`_exhaustive: never`-style, on `dataTypeToTypeRef`'s own switch) fails to
   compile if a `DataType` value is missing from the mapping.
8. `check` **and** `test` green for `graph`, `compiler`, `runtime-webgpu`, `runtime-cpu`,
   `graph-editor`, and the full workspace (clear every `packages/*/dist` first, per `AGENTS.md`'s
   documented stale-dist issue).

## Out of scope

The full WebGPU capability/validation model (Foundation 2). Storage/texture usage-flag inference
from edges (Foundation 2). Wiring real codegen/runtime behavior for the `array`/`buffer`/`texture`/
`sampler`/`mesh`/`command` `TypeRef` variants beyond their type-level shape existing (Foundation
2/3). Removing `DataType` — it stays as a convenience alias layer indefinitely. A general
TypeBox↔TypeRef bridge beyond the narrow scalar/vector/struct value-schema subset. Touching
`groupCodegen.ts`'s integer-*param* rejection (a separate, unrelated restriction — see verified
state). Any change to F1.1, F1.2, F1.3, or F1.4a's design.

## Handoff

→ `TypeRef` exists as a real structural type algebra alongside `DataType`, with a total forward
mapping and a partial reverse one, proven by two genuinely new capabilities (an integer port and a
matrix port, both compiling and running on a real device) rather than just a bigger union.
`PortSpec`/`Port` carry canonical `type`, registration normalizes both old and new authoring shapes
through one shared function. Once this lands: F1.3 (discriminated primitive-implementation union +
group registry) is next — request its own contract at that point.
