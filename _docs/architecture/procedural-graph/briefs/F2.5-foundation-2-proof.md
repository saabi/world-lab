# Brief — F2.5: Foundation 2 proof (two-pass texture feedback + buffer feedback)

**Type:** two bundled, hardcoded sample graphs proving the generic resource-history model — the
first real, document-driven consumer of cross-pass reads · **Package:** `@world-lab/graph`,
`@world-lab/compiler`, `@world-lab/runtime-webgpu`, `@world-lab/graph-editor` · **Depends on:**
F2.1–F2.4, all landed (`04f5319`, `397af7f`, `f355221`, `37f496a`) · **Design authority:**
[foundation-2-generic-resources-plan.md](../foundation-2-generic-resources-plan.md) · **Contract
author:** Opus · **Recommended executor:** Cursor or Codex · **Status:** ready to route

## Context

Fifth and final item of Foundation 2's sequence (F2.1 ✅ → F2.2 ✅ → F2.3 ✅ → F2.4 ✅ → **proof**).
Per the plan: two bundled, hardcoded samples — a two-pass render-target sample (one pass feeds
another, both textures) and a small buffer-feedback sample (storage-buffer ping-pong) — proving the
resource-history model generalizes, not just typechecks. F2.4's own brief flagged that neither
sample could exist as a real document without more work; this brief is that work, scoped as
narrowly as the two samples actually require — **not** Foundation 3's generic kernel/binding model.

**Why this is narrower than it sounds.** Two pieces of infrastructure for exactly this were already
reserved and unused:

1. **`HostBinding{context:'read-resource'}`** (`packages/graph/src/implementation.ts`) — reserved
   since F1.3, zero consumers anywhere (confirmed by grep). This milestone is its first use.
2. **`BindingDecl.kind: 'texture' | 'sampler' | 'storage-read'`**
   (`packages/compiler/src/stageEntry.ts`) — the bind-group-declaration machinery for exactly these
   binding kinds already exists and already emits correct WGSL (`bindingVar`'s `<storage, read>`
   branch, and the unqualified `var name: texture_2d<f32>`/`var name: sampler` forms) — it has just
   never been asked to emit anything but `'uniform'`. **`layout: 'auto'`** (WebGPU's automatic
   bind-group-layout derivation, already used by `createRenderPipeline` in
   `fullscreenFragment.ts`) means no bind-group-*layout* derivation code is needed at all — WebGPU
   derives it from whatever bindings the WGSL declares, exactly as it already does for the
   `ShaderToyUniforms`/`GraphParams` bindings today.

**What's genuinely new, and why it's still small:**

- One new `host-input` primitive (`input.channel`) and one new `emitHostInput` branch
  (`packages/runtime-webgpu/src/emitGraphEval.ts:76-106`) — the exact same shape as the five
  branches already there for `iTime`/`iResolution`/`fragCoord`/`uv`/`metricPosition`. This emits a
  `textureSample(...)` expression, nothing more.
- One new `BindingDecl.kind: 'storage-read-write'` (buffer sample only) — one more line in
  `bindingVar`'s address-space switch, alongside the existing `uniform`/`storage-read` cases.
- A small, purpose-built function that detects channel-read wiring in a document and produces real
  `ResourceRead`s — **not** a change to `buildIndependentPassGraph`/`planIndependentGraphFramePasses`
  (both stay exactly as they are, for the — still far more common — zero-read case).
- The buffer-feedback sample's fragment entry is **hand-written via the already-existing
  `entryOverride` escape hatch** (`StageEntryOptions.entryOverride`, `stageEntry.ts:24`), not a new
  generic "storage write as a graph primitive" mechanism. This is deliberate and matches the plan's
  own framing — a *hardcoded* proof sample, not a new authoring capability. A graph-authorable
  storage-write primitive, if ever wanted, is Foundation 3/4 territory (compute kernels needing a
  real place to write from).

This keeps the milestone inside "wire two fixed, illustrative samples through the resource model,"
not "build the general multi-pass authoring system" — that generalization work (arbitrary
same-frame reads authored freely in the editor, compute-dispatch primitives) stays Foundation 3/4's
job, unchanged from F2.4's own Out-of-scope framing.

## Verified current state

- **`derivePipelinePresentations`/`presentationForDisplay`** (`packages/graph/src/pipeline.ts`)
  only walk a `target.display` node's own `color ← stage.fragment ← <field>` chain — nothing in
  this path looks for a channel-read node inside that field subgraph referencing another display's
  output. Confirmed by direct read: no cross-display awareness anywhere in this file.
- **`emitGraphEval`** (`packages/runtime-webgpu/src/emitGraphEval.ts:240-`) calls `emitHostInput(node,
  primitive.implementation.binding, opts)` for every `kind:'host-input'` node during its single
  topological pass — the exact, already-existing extension point for a new binding case. `opts` is
  already an open bag (`EmitGraphEvalOptions`) the caller controls, matching how `iResolutionExpr`/
  `iTimeExpr`/`fragCoordExpr` are already threaded through from `fullscreenFragment.ts`.
- **`EmittedGraphEval`** (`emitGraphEval.ts:33-40`) returns `{body, params, resultExpr}` only — no
  record of which host-input bindings were actually used. `fullscreenFragment.ts` currently detects
  the ShaderToy-uniform case by **regexing the generated WGSL text**
  (`wgslReferencesShaderToyUniform`) rather than being told directly — workable for one fixed
  binding, but not a pattern to repeat for channel bindings with a dynamic channel index.
- **`assembleFullscreenFragmentModuleAsync`/`executeFullscreenFragment`**
  (`consumers/fullscreenFragment.ts`) construct `bindGroupEntries` by hand per binding kind present
  (`usesShaderToyHost`, `hasGraphParams`) — the precedent this milestone's texture/sampler/storage
  entries extend, not replace.
- **No `evalCPU` requirement is universal** — `host.iTime`/etc. all define one, but nothing in the
  primitive contract requires it; a GPU-only primitive with an `evalCPU` that throws a clear "GPU
  pipeline only" error is an accepted, existing pattern (confirmed: `NodePrimitiveInput.evalCPU` is
  optional in the type; several host/structural primitives already omit or narrow it).
- **Sample registration** (`packages/graph-editor/src/samples.ts`) is a flat `{id, label,
  build(): GraphDocument}[]` list (`GRAPH_SAMPLES`), with builder functions living in
  `graphBuilders.ts` — the exact pattern F1.4a's two migration samples already used, reused here.

## Fix

### A. Shared: the channel-read primitive and its WGSL emission

1. **New primitive `input.channel`** (`packages/graph/src/primitives/host/channel.ts`, new file),
   following `host.iTime`'s exact shape:
   ```ts
   const channel: NodePrimitiveInput = {
     id: 'input.channel',
     category: 'ShaderToy',
     inputs: [],
     outputs: [{ name: 'color', dataType: 'vec4f' }],
     params: Type.Object({ channel: Type.Number({ default: 0 }) }),
     implementation: { kind: 'host-input', binding: { context: 'read-resource', key: 'channel' } },
     evalCPU() {
       throw new Error('input.channel is GPU-pipeline only; no CPU evaluation path');
     }
   };
   ```
   `params.channel` (not the port list) selects which physical channel binding (`channel0`,
   `channel1`, …) — the node itself carries no wiring to "the other pass," because there is no
   graph-level output port representing an entire rendered target (a `target.display` sink is a
   sink, not a value producer). The connection from `input.channel`'s `params.channel` index to a
   *specific other display node* is made by the sample graph's own document-level convention (see
   Fix D) — this milestone does not build a generalized "which pass feeds channel N" authoring UI.
2. **New `emitHostInput` branch** (`emitGraphEval.ts`), same shape as the five existing cases:
   ```ts
   if (binding.context === 'read-resource' && binding.key === 'channel') {
     const channelIndex = typeof node.params?.channel === 'number' ? node.params.channel : 0;
     return `let ${variable} = textureSample(channel${channelIndex}, channel${channelIndex}Sampler, uv);`;
   }
   ```
   `uv` is already an available body-local (emitted by the existing `invocation`/`uv` host-input
   case whenever a graph uses it) — the sample graphs wire `input.channel` alongside `input.uv` so
   `uv` is in scope; do not invent a second position source.
3. **`EmittedGraphEval` gains `usedChannels: number[]`** — the emitter tracks which channel indices
   it actually emitted a `textureSample` for, instead of requiring callers to regex the output text:
   ```ts
   export interface EmittedGraphEval {
     body: string[];
     params: GraphParamField[];
     resultExpr: string;
     usedChannels: number[]; // NEW — distinct channel indices referenced via input.channel
   }
   ```
4. **`assembleFullscreenFragmentModuleAsync` pushes one `texture`+one `sampler` `BindingDecl` per
   entry in `usedChannels`** (binding indices allocated after the existing uniform bindings, same
   pattern as today's `paramsBinding = usesShaderToyHost ? 1 : 0` allocation), and
   **`executeFullscreenFragment` accepts the resolved channel textures as an additional input**:
   ```ts
   export interface FullscreenFragmentInput {
     /* ...existing fields... */
     channelTargets?: readonly GPUTexture[]; // index = channel number; caller-resolved, caller-owned
   }
   ```
   Bind-group entries: `{binding: N, resource: channelTargets[i].createView()}` (texture) and
   `{binding: N+1, resource: device.createSampler({...})}` (sampler — a fresh default sampler per
   call is fine; no caching needed for two hardcoded samples). `layout: 'auto'` derives the rest.

### B. Texture-feedback sample: real cross-pass `PassGraph` derivation

5. **A small, purpose-built function detects channel wiring and augments the plan — it does not
   touch `buildIndependentPassGraph`/`planIndependentGraphFramePasses`:**
   ```ts
   // graphFramePlan.ts, new export, additive
   export interface ChannelDependency {
     consumerId: string;   // the pass whose field subgraph reads the channel
     channel: number;
     sourceTargetId: string; // resolved target id of the display node it reads
   }

   export function resolveChannelDependencies(
     doc: GraphDocument,
     passes: readonly GraphFramePass[]
   ): ChannelDependency[] {
     // Walk each pass's field subgraph for `input.channel` nodes; resolve `params.channel` and
     // (sample-graph convention, see Fix D) an accompanying node.params.sourceDisplayId to the
     // matching pass's targetId. Throws a clear error if sourceDisplayId doesn't match any pass —
     // fail loud, this is authoring-time wiring, not a runtime-recoverable state.
   }

   export function buildPassGraphWithChannelReads(
     passes: readonly GraphFramePass[],
     dependencies: readonly ChannelDependency[]
   ): PassGraph {
     // Same target/lifetime construction as buildIndependentPassGraph, but each pass's `reads`
     // includes a ResourceRead per matching ChannelDependency (`{channel, target: sourceTargetId}`,
     // version omitted — same-frame read of the other pass's current output).
   }
   ```
   `GraphFrameExecutor.execute` calls `resolveChannelDependencies` once per frame; if it returns
   `[]` (the common case today), behavior is byte-for-byte identical to F2.4's landed path
   (`buildIndependentPassGraph` unchanged). If non-empty, it calls
   `buildPassGraphWithChannelReads` instead — the *only* branch point this milestone adds to the
   executor, keeping F2.4's own gate items intact.
6. **`GraphFrameExecutor` resolves each pass's channel textures before calling `pipeline.execute`**,
   using `ResourceRealizer.resolve(dependency.sourceTargetId).write` (same-frame read — the other
   pass's just-produced texture; F2.2's validation already guarantees a same-frame `ResourceRead`
   never needs `.readPrevious`), passed through as `channelTargets` (Fix step 4).
7. **The sample graph itself** (`graphBuilders.ts`, new builder, e.g. `channelFeedbackPipelineGraph`):
   two `target.display` sinks (`n_display_a` writes a base pattern; `n_display_b`'s field subgraph
   wires `input.channel{channel:0}` with `node.params.sourceDisplayId: 'n_display_a'` and combines
   it with its own procedural field). Bundled and pickable (Fix E).

### C. Buffer-feedback sample: storage binding + hand-written entry

8. **New `BindingDecl.kind: 'storage-read-write'`** (`stageEntry.ts`):
   ```ts
   function bindingVar(b: BindingDecl): string {
     const addr =
       b.kind === 'uniform' ? '<uniform> '
       : b.kind === 'storage-read' ? '<storage, read> '
       : b.kind === 'storage-read-write' ? '<storage, read_write> '
       : ' ';
     return `@group(${b.group}) @binding(${b.binding}) var${addr}${b.name}: ${b.wgslType};`;
   }
   ```
9. **A dedicated, hand-written consumer function** (`consumers/bufferFeedback.ts`, new file) — not
   routed through `emitGraphEval`/`assembleStageEntry`'s default templates, using `entryOverride`
   directly:
   ```ts
   const entry = `
     @group(0) @binding(0) var<storage, read> previous: array<f32>;
     @group(0) @binding(1) var<storage, read_write> next: array<f32>;
     @fragment
     fn fs_main(@builtin(position) position: vec4f) -> @location(0) vec4f {
       let index = u32(position.x) + u32(position.y) * ${gridWidth}u;
       let alive = previous[index];
       // ...next-state rule (e.g. a fixed cellular-automaton step)...
       next[index] = nextState;
       return vec4f(nextState, nextState, nextState, 1.0);
     }
   `;
   ```
   This bypasses graph-node-driven codegen entirely for the feedback mechanics — deliberately, per
   Context. The *visualization* (color from state) can still be a trivial expression; the
   next-state *rule* is a fixed, hardcoded WGSL body for this one sample, not a graph-authorable
   primitive. Bind group: `previous` = `ResourceRealizer.resolve(id).readPrevious`, `next` =
   `.write` (F2.2/F2.3 already guarantee these are the correct, distinct physical buffers for a
   `history`-lifetime target — this milestone only has to bind them).
10. **The buffer-feedback `PassGraph`** is hand-constructed (mirroring how `order.test.ts`/
    `realize.test.ts` already build `PassGraph` literals directly) — a single buffer-shaped
    `history`-lifetime target, one pass reading it with `version:'previous'` and writing it. No
    document-driven derivation is needed for this sample specifically (there is no
    graph-authorable node representing "the previous state of this storage buffer" yet — building
    one is exactly the generalization Context says stays out of scope), so this sample's frame-graph
    plumbing is exercised directly, not through `GraphDocument`/`emitGraphEval` at all.
11. **A small, dedicated executor path** (e.g. `BufferFeedbackExecutor` or a function
    `executeBufferFeedback(device, realizer, passGraph, gridWidth)` in the new consumer file) drives
    this one hand-built `PassGraph` through `buildPassOrder` + `ResourceRealizer`, exactly as
    `GraphFrameExecutor` does for pipeline graphs — proving the *same* generic ordering/realization
    machinery drives both a document-derived graph (Fix B) and a hand-built one (this), which is
    itself part of the proof.

### D. Sample-graph wiring convention (texture case)

12. **`input.channel`'s `sourceDisplayId` param is document convention, not IR-enforced.** Add it as
    a second param on the primitive (`Type.String({ default: '' })`) purely as *data*
    `resolveChannelDependencies` reads — the graph/primitive system does not validate this string
    against anything at authoring time (no new port, no new edge kind). This is the minimal,
    unambitious wiring mechanism the two hardcoded samples need; a real multi-pass authoring UX
    (drag a display node onto a channel slot, live-validated) is explicitly editor/Foundation-3/4
    work, not this milestone's.

### E. Bundling

13. **Register both samples** in `GRAPH_SAMPLES` (`graph-editor/src/samples.ts`), builders in
    `graphBuilders.ts`, following the exact `migration-default-preview`/`migration-fullscreen-fragment`
    precedent (`{id, label, build}`). Per standing instruction, both must be pickable in the editor,
    not headless-only fixtures.

## Gate

1. A test proves `emitGraphEval` on a graph using two `input.channel` nodes (channels 0 and 1)
   returns `usedChannels: [0, 1]` and emits two distinct `textureSample` expressions referencing
   `channel0`/`channel1`.
2. A test proves `assembleFullscreenFragmentModuleAsync` pushes one texture + one sampler
   `BindingDecl` per used channel, at binding indices after the existing uniform bindings, and that
   the resulting WGSL round-trips through `WgslValidator`.
3. A test proves `resolveChannelDependencies` returns `[]` for every existing sample/fixture graph
   with no `input.channel` node — the direct regression proof that F2.4's landed, all-independent
   execution path is untouched for every graph constructible before this milestone.
4. A test proves `resolveChannelDependencies` correctly resolves a two-display sample graph's
   `input.channel`/`sourceDisplayId` wiring into a `ChannelDependency`, and throws a clear error if
   `sourceDisplayId` doesn't match any pass's target id.
5. `it.skipIf(!hasWebGPU)`: the texture-feedback sample renders on a real device; pass B's output
   visibly incorporates pass A's texture (assert on specific pixel values reflecting the known
   combination, not just "no exception") — the concrete two-pass visual proof.
6. A test proves the buffer-feedback `PassGraph`'s two slots are correctly bound as `previous`/
   `next` across two consecutive frame executions (`advanceFrame` between them) — mirroring
   `realize.test.ts`'s existing ping-pong proof, but through the new `executeBufferFeedback` path.
7. `it.skipIf(!hasWebGPU)`: the buffer-feedback sample runs N frames on a real device and produces
   the expected deterministic next-state sequence for a small, known initial pattern — the concrete
   storage-buffer visual/behavioral proof, and the direct test that the resource-history model
   didn't quietly stay texture-only (per the plan's own explicit requirement for this sample).
8. Both samples are selectable from the editor's sample picker and render without error — a
   screenshot or manual confirmation is required per this project's visual-gate convention (cannot
   be verified in a headless review environment).
9. `check` **and** `test` green for `graph`, `compiler`, `runtime-webgpu`, `graph-editor`, full
   workspace (clear every `packages/*/dist` first, per `AGENTS.md`'s documented stale-dist issue).

## Out of scope

A general, editor-authorable way to wire "read another pass's output" (drag-and-drop channel
binding, live validation, arbitrary channel counts) — the two samples use a fixed
`params.sourceDisplayId` string convention, not a new port/edge kind. A graph-authorable
storage-buffer-write primitive or compute dispatch (Foundation 3/4 — `entryOverride` is used
directly for the one buffer sample instead). Bind-group *layout* derivation beyond
`layout: 'auto'` (unnecessary — WebGPU already derives it from the WGSL bindings this milestone
adds). Any change to `buildIndependentPassGraph`/`planIndependentGraphFramePasses` (both stay
exactly as landed; `resolveChannelDependencies` is a fully separate, additive function). CPU
evaluation of `input.channel` (GPU-pipeline only, throws clearly). N-channel generality beyond what
the two samples need (2 channels is enough to prove the mechanism; a third channel is no different
in kind). Any change to Foundation 1's milestones or F2.1–F2.4's landed contracts.

## Handoff

→ Foundation 2's resource-history model is proven end-to-end with two real, pickable, rendering
samples — one texture-to-texture cross-pass read, one storage-buffer ping-pong — using the same
`buildPassOrder`/`ResourceRealizer` machinery for both, confirming it was never texture-only. The
channel-read primitive and its WGSL emission are narrowly scoped (one primitive, one emission
branch, a document convention rather than a new port/edge kind) and deliberately do not anticipate
Foundation 3's actual kernel/binding model. Foundation 2 is complete. Foundation 3 (generic
vertex/fragment/compute kernels, arbitrary resource bindings, typed varyings) is next, whenever
you want its own sequencing plan — it is a larger, from-scratch design effort, not a follow-on
contract in the same shape as F2.1–F2.5.
