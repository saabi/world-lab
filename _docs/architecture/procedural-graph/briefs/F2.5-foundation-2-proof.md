## Revision note

**Revision 2** fixed eight confirmed defects in revision 1, caught before routing:

1. **Blocking — `uv` is not safely available to `input.channel`.** The generated fragment
   function's only parameter is `position: vec4f` (`buildGraphEvalFn`); there is no `uv` in scope,
   and even `input.uv`'s own emission produces a mangled `v_<node>_<port>` variable, never a literal
   `uv`. Sibling host-input nodes also have no ordering guarantee relative to each other. Fixed:
   the channel-read expression derives its own UV directly from `position.xy` and
   `textureDimensions(...)`, depending on nothing but its own binding and the function's actual
   parameter.
2. **Blocking — `sourceDisplayId` cannot map to `GraphFramePass.targetId`.** `GraphFramePass`
   retains only `targetId` (a *derived* name, e.g. `pipeline_image_n_display_a` via
   `outputNameForField`) and `fieldOutput` — not the raw display node id the sample naturally
   stores. Fixed: resolution goes through `derivePipelinePresentations(doc)`'s existing
   `displayNodeId ↔ outputName` pairing, not a direct string-equality assumption.
3. **Blocking — the buffer sample has no editor execution path.** `GRAPH_SAMPLES` only stores a
   `GraphDocument` builder; nothing dispatches a buffer-feedback document to the hand-written
   executor. Fixed: a new, minimal `target.bufferFeedback` sink primitive marks the document; a
   small derivation function finds it; `GraphFrameExecutor` recognizes it and merges the
   hand-written path's readback pixels into the same `targets` record the normal path produces —
   reusing the existing image-preview UI, no new panel.
4. **Blocking — the buffer sample's render/init resources were incomplete.** A fragment pipeline
   needs a color attachment, but the hand-built `PassGraph` had only a buffer target, and `Pass`
   has one `writeTarget`. Fixed: the presentation texture is an explicit, caller-owned,
   incidental output — allocated directly, outside `PassGraph`/`ResourceRealizer` entirely — with
   `RENDER_ATTACHMENT | COPY_SRC` usage; the buffer's declared usage gains `'copy-dst'` so its
   initial pattern can be seeded via `queue.writeBuffer` before frame 1; grid dimensions are stated
   to match the buffer's element count exactly; determinism is asserted via texture readback.
5. **High — channel-binding metadata didn't reach execution; a dense array is unsafe for sparse
   channel indices.** Fixed: `assembleFullscreenFragmentModuleAsync` returns the actual binding
   plan (`channelBindings: ReadonlyMap<number, {textureBinding, samplerBinding}>`);
   `executeFullscreenFragment` takes `channelTargets: ReadonlyMap<number, GPUTexture>` and throws
   clearly if a declared channel has no matching entry.
6. **High — `entryOverride` and the proposed `BindingDecl` kind duplicated the same declarations.**
   Fixed: the buffer sample does not call `assembleStageEntry` at all — it constructs its own,
   fully self-contained WGSL module string (bindings + entry together), so there is exactly one
   place the storage bindings are declared. The `BindingDecl.kind: 'storage-read-write'` addition
   is dropped as unnecessary. (Separately noted: `assembleStageEntry` validates `output`/`outputFns`
   *before* checking `entryOverride` — a real, pre-existing gap in `stageEntry.ts` — but this
   milestone doesn't call that function for the buffer path, so it isn't touched here.)
7. **Medium — channel indices were unvalidated.** `Type.Number` permits negative/fractional values,
   which would generate invalid WGSL identifiers (`channel-1`). Fixed: `Type.Integer({minimum: 0,
   maximum: 3})` at the schema level, plus a defensive re-check at emission time.
8. **Terminology — the texture sample is a same-frame cross-pass read, not history/feedback.**
   Renamed throughout: "cross-pass texture read sample." Only the buffer sample is `history`
   -lifetime/previous-frame feedback; the texture sample's two targets are both `transient` and
   ordered by `buildPassOrder` for the first time against a real dependency — it needs no
   ping-pong at all. This also simplifies the texture case: it exercises ordering and cross-pass
   binding, not `ResourceRealizer`'s double-buffering (which the buffer sample alone proves).

## Context

Fifth and final item of Foundation 2's sequence (F2.1 ✅ → F2.2 ✅ → F2.3 ✅ → F2.4 ✅ → **proof**).
Per the plan: two bundled, hardcoded samples — a cross-pass texture read (one pass feeds another,
both textures, same-frame) and a small buffer-feedback sample (storage-buffer ping-pong, genuinely
previous-frame history) — proving the resource-history model generalizes, not just typechecks.
F2.4's own brief flagged that neither sample could exist as a real document without more work; this
brief is that work, scoped as narrowly as the two samples actually require — **not** Foundation 3's
generic kernel/binding model.

**Why this is narrower than it sounds.** Two pieces of infrastructure for exactly this were already
reserved and unused:

1. **`HostBinding{context:'read-resource'}`** (`packages/graph/src/implementation.ts`) — reserved
   since F1.3, zero consumers anywhere (confirmed by grep). This milestone is its first use.
2. **`BindingDecl.kind: 'texture' | 'sampler' | 'storage-read'`**
   (`packages/compiler/src/stageEntry.ts`) — the bind-group-declaration machinery for exactly these
   binding kinds already exists and already emits correct WGSL (`bindingVar`'s `<storage, read>`
   branch, and the unqualified `var name: texture_2d<f32>`/`var name: sampler` forms) — it has just
   never been asked to emit anything but `'uniform'`. **`layout: 'auto'`** (WebGPU's automatic
   bind-group-layout derivation, already used by `createRenderPipeline` in
   `fullscreenFragment.ts`) means no bind-group-*layout* derivation code is needed at all — WebGPU
   derives it from whatever bindings the WGSL declares, exactly as it already does for the
   `ShaderToyUniforms`/`GraphParams` bindings today.

**What's genuinely new, and why it's still small:**

- One new `host-input` primitive (`input.channel`) and one new `emitHostInput` branch
  (`packages/runtime-webgpu/src/emitGraphEval.ts:76-106`) — the exact same shape as the five
  branches already there for `iTime`/`iResolution`/`fragCoord`/`uv`/`metricPosition`. This emits a
  self-contained `textureSample(...)` expression, nothing more.
- A small, purpose-built function that detects channel-read wiring in a document and produces real
  `ResourceRead`s — **not** a change to `buildIndependentPassGraph`/`planIndependentGraphFramePasses`
  (both stay exactly as they are, for the — still far more common — zero-read case).
- The buffer sample's fragment entry is a **fully self-contained, hand-written WGSL module** (not
  routed through `assembleStageEntry`'s templates at all), not a new generic "storage write as a
  graph primitive" mechanism. This is deliberate and matches the plan's own framing — a
  *hardcoded* proof sample, not a new authoring capability. A graph-authorable storage-write
  primitive, if ever wanted, is Foundation 3/4 territory (compute kernels needing a real place to
  write from).
- One new, minimal sink primitive (`target.bufferFeedback`) so the buffer sample is a real,
  dispatchable `GraphDocument`, not a standalone function nothing ever calls.

This keeps the milestone inside "wire two fixed, illustrative samples through the resource model,"
not "build the general multi-pass authoring system" — that generalization work (arbitrary
same-frame reads authored freely in the editor, compute-dispatch primitives) stays Foundation 3/4's
job, unchanged from F2.4's own Out-of-scope framing.

## Verified current state

- **`buildGraphEvalFn`** (`emitGraphEval.ts:109-116`) emits `fn graph_eval_<name>(position: vec4f)
  -> vec4f`. `position` is the **only** parameter — confirmed by direct read. There is no `u`/`v`
  or `uv` variable in scope unless a node explicitly emits one under that exact name, which nothing
  does (`input.uv`'s own output variable is `portVar(node.id, outPort.id)`, a mangled name).
- **`derivePipelinePresentations`/`presentationForDisplay`** (`packages/graph/src/pipeline.ts`)
  only walk a `target.display` node's own `color ← stage.fragment ← <field>` chain — nothing in
  this path looks for a channel-read node inside that field subgraph referencing another display's
  output. Each `PipelinePresentation` carries both `displayNodeId` (the raw node id) and
  `outputName` (the derived name that becomes `GraphFramePass.targetId` via
  `planIndependentGraphFramePasses`) — this pairing is exactly what's needed to resolve a
  human-authored `sourceDisplayId` to the correct runtime target id.
- **`emitGraphEval`** (`emitGraphEval.ts:240-`) calls `emitHostInput(node,
  primitive.implementation.binding, opts)` for every `kind:'host-input'` node during its single
  topological pass, and separately runs `topologicalSort(doc, output.node)` to collect every node
  upstream of a given output — the exact, already-existing reachability computation this
  milestone's channel-detection reuses rather than reimplementing.
- **`assembleStageEntry`** (`packages/compiler/src/stageEntry.ts:44-53`) validates `output`/
  `outputFns` **before** checking whether `opts.entryOverride` is set — so even an override-only
  caller must satisfy that validation today. This milestone's buffer sample does not call this
  function at all (see Fix C), so the gap doesn't block it, but it's a real, separate latent issue
  in `stageEntry.ts` worth flagging for whoever next uses `entryOverride`.
- **`assembleFullscreenFragmentModuleAsync`/`executeFullscreenFragment`**
  (`consumers/fullscreenFragment.ts`) construct `bindGroupEntries` by hand per binding kind present
  (`usesShaderToyHost`, `hasGraphParams`) — the precedent this milestone's texture/sampler entries
  extend, not replace.
- **No `evalCPU` requirement is universal** — `host.iTime`/etc. all define one, but nothing in the
  primitive contract requires it; a GPU-only primitive with an `evalCPU` that throws a clear "GPU
  pipeline only" error is an accepted, existing pattern (confirmed: `NodePrimitiveInput.evalCPU` is
  optional in the type; several host/structural primitives already omit or narrow it).
- **Sample registration** (`packages/graph-editor/src/samples.ts`) is a flat `{id, label,
  build(): GraphDocument}[]` list (`GRAPH_SAMPLES`), with builder functions living in
  `graphBuilders.ts` — the exact pattern F1.4a's two migration samples already used, reused here.
  This is document-only: nothing about *executing* a selected sample lives in this registry, which
  is exactly why the buffer sample needs its own dispatch marker (Fix D), not just a registry entry.

## Fix

### A. Shared: the channel-read primitive and its WGSL emission

1. **New primitive `input.channel`** (`packages/graph/src/primitives/host/channel.ts`, new file),
   following `host.iTime`'s exact shape, with a validated integer channel index:
   ```ts
   const channel: NodePrimitiveInput = {
     id: 'input.channel',
     category: 'ShaderToy',
     inputs: [],
     outputs: [{ name: 'color', dataType: 'vec4f' }],
     params: Type.Object({
       channel: Type.Integer({ default: 0, minimum: 0, maximum: 3 }),
       sourceDisplayId: Type.String({ default: '' })
     }),
     implementation: { kind: 'host-input', binding: { context: 'read-resource', key: 'channel' } },
     evalCPU() {
       throw new Error('input.channel is GPU-pipeline only; no CPU evaluation path');
     }
   };
   ```
   `params.sourceDisplayId` (document convention, not IR-enforced — see Fix D) names the *other*
   display node this channel reads. There is no graph-level output port representing an entire
   rendered target (a `target.display` sink is a sink, not a value producer), so this connection
   cannot be an ordinary edge — it is data `resolveChannelDependencies` reads, nothing more.
2. **New `emitHostInput` branch** (`emitGraphEval.ts`) — self-contained, depends only on `position`
   (the function's actual parameter) and the channel's own texture binding, never a sibling node's
   output:
   ```ts
   if (binding.context === 'read-resource' && binding.key === 'channel') {
     const channelIndex = node.params?.channel;
     if (typeof channelIndex !== 'number' || !Number.isInteger(channelIndex) || channelIndex < 0) {
       throw new Error(`input.channel node ${node.id} has an invalid channel index`);
     }
     return (
       `let ${variable} = textureSample(channel${channelIndex}, channel${channelIndex}Sampler, ` +
       `position.xy / vec2<f32>(textureDimensions(channel${channelIndex})));`
     );
   }
   ```
   `textureDimensions` reads the actual bound texture's size at runtime, so this is correct
   regardless of viewport/render-target size — no externally-computed UV or extra `opts` field
   needed.
3. **`EmittedGraphEval` gains `usedChannels: number[]`** — the emitter tracks which distinct
   channel indices it actually emitted a `textureSample` for, instead of requiring callers to regex
   the output text:
   ```ts
   export interface EmittedGraphEval {
     body: string[];
     params: GraphParamField[];
     resultExpr: string;
     usedChannels: number[]; // NEW — sorted, distinct channel indices referenced via input.channel
   }
   ```
4. **`assembleFullscreenFragmentModuleAsync` returns the actual channel-binding plan, not just the
   raw indices** — the single source of truth both the WGSL declarations and the executor's
   bind-group entries read from, closing the gap where binding-index allocation could silently
   diverge between the two:
   ```ts
   // FullscreenFragmentAssembly, extended
   channelBindings: ReadonlyMap<number, { textureBinding: number; samplerBinding: number }>;
   ```
   Allocated after the existing uniform bindings (same pattern as today's `paramsBinding =
   usesShaderToyHost ? 1 : 0`), one texture + one sampler binding per entry in `usedChannels`.
5. **`executeFullscreenFragment` accepts resolved channel textures keyed by channel number, not
   position — and rejects a missing entry clearly:**
   ```ts
   export interface FullscreenFragmentInput {
     /* ...existing fields... */
     channelTargets?: ReadonlyMap<number, GPUTexture>; // caller-resolved, caller-owned
   }
   ```
   For each `[channel, {textureBinding, samplerBinding}]` in `channelBindings`: look up
   `input.channelTargets?.get(channel)`; if missing, throw
   `` `Missing channel target for channel ${channel}` `` before building the bind group — never
   silently bind a wrong or empty resource. `layout: 'auto'` derives the rest.

### B. Cross-pass texture read sample: real `PassGraph` derivation

6. **A small, purpose-built function detects channel wiring and augments the plan — it does not
   touch `buildIndependentPassGraph`/`planIndependentGraphFramePasses`.** Resolution goes through
   `derivePipelinePresentations`'s `displayNodeId ↔ outputName` pairing, correcting revision 1's
   assumption that a `sourceDisplayId` string equals a `targetId` directly:
   ```ts
   // graphFramePlan.ts, new export, additive
   export interface ChannelDependency {
     consumerId: string;     // the pass whose field subgraph reads the channel (== its targetId)
     channel: number;
     sourceTargetId: string; // resolved via presentation.displayNodeId -> presentation.outputName
   }

   export function resolveChannelDependencies(
     doc: GraphDocument,
     passes: readonly GraphFramePass[]
   ): ChannelDependency[] {
     const targetIdByDisplayNodeId = new Map(
       derivePipelinePresentations(doc).map((p) => [p.displayNodeId, p.outputName])
     );
     const dependencies: ChannelDependency[] = [];
     for (const pass of passes) {
       // Reuse emitGraphEval's own upstream-reachability walk from pass.fieldOutput.node to find
       // every input.channel node feeding this pass (do not reimplement topological reachability).
       for (const node of channelNodesUpstreamOf(doc, pass.fieldOutput.node)) {
         const sourceTargetId = targetIdByDisplayNodeId.get(String(node.params?.sourceDisplayId));
         if (!sourceTargetId) {
           throw new Error(
             `input.channel node ${node.id} references unknown display "${node.params?.sourceDisplayId}"`
           );
         }
         dependencies.push({ consumerId: pass.targetId, channel: Number(node.params?.channel), sourceTargetId });
       }
     }
     return dependencies;
   }

   export function buildPassGraphWithChannelReads(
     passes: readonly GraphFramePass[],
     dependencies: readonly ChannelDependency[]
   ): PassGraph {
     // Same target/lifetime construction as buildIndependentPassGraph — every target stays
     // `transient` (this sample needs no history/ping-pong at all, only ordering) — but each
     // pass's `reads` includes a ResourceRead per matching ChannelDependency
     // (`{channel, target: sourceTargetId}`, version omitted: a same-frame read of the other
     // pass's current output).
   }
   ```
   `GraphFrameExecutor.execute` calls `resolveChannelDependencies` once per frame; if it returns
   `[]` (the common case today), behavior is byte-for-byte identical to F2.4's landed path
   (`buildIndependentPassGraph` unchanged). If non-empty, it calls
   `buildPassGraphWithChannelReads` instead — the *only* branch point this milestone adds to the
   executor, keeping F2.4's own gate items intact.
7. **`GraphFrameExecutor` resolves each pass's channel textures before calling `pipeline.execute`**,
   building a `Map<number, GPUTexture>` from each pass's `ChannelDependency` list via
   `ResourceRealizer.resolve(dependency.sourceTargetId).write` (same-frame read — the other pass's
   just-produced texture; F2.2's validation already guarantees a same-frame `ResourceRead` never
   needs `.readPrevious`), passed through as `channelTargets` (Fix step 5). This requires
   `buildPassOrder`'s `order.order` to be respected for the first time with real consequence: pass A
   must execute (and its texture resolve to final content) before pass B reads it.
8. **The sample graph itself** (`graphBuilders.ts`, new builder, e.g. `crossPassTextureReadGraph`):
   two `target.display` sinks (`n_display_a` writes a base pattern; `n_display_b`'s field subgraph
   wires `input.channel{channel:0, sourceDisplayId:'n_display_a'}` and combines it with its own
   procedural field). Bundled and pickable (Fix F).

### C. Buffer-feedback sample: self-contained WGSL, incidental presentation texture

9. **A dedicated, hand-written consumer function** (`consumers/bufferFeedback.ts`, new file)
   constructs its **own, fully self-contained WGSL module string** — bindings and entry together —
   and does **not** call `assembleStageEntry` at all, so there is exactly one place these bindings
   are declared (no `BindingDecl` extension, avoiding the duplicate-declaration contradiction):
   ```ts
   const gridWidth = /* ... */;
   const code = `
     @group(0) @binding(0) var<storage, read> previous: array<f32>;
     @group(0) @binding(1) var<storage, read_write> next: array<f32>;
     @fragment
     fn fs_main(@builtin(position) position: vec4f) -> @location(0) vec4f {
       let index = u32(position.x) + u32(position.y) * ${gridWidth}u;
       let alive = previous[index];
       // ...fixed next-state rule (e.g. one cellular-automaton step)...
       next[index] = nextState;
       return vec4f(nextState, nextState, nextState, 1.0);
     }
   `;
   ```
   Bind group: `previous` = `ResourceRealizer.resolve(id).readPrevious`, `next` = `.write` — F2.2/
   F2.3 already guarantee these are the correct, distinct physical buffers for a `history`-lifetime
   target; this milestone only has to bind them.
10. **The buffer `PassGraph` models only the buffer** — the actual resource-history proof target.
    It is hand-constructed (mirroring how `order.test.ts`/`realize.test.ts` already build
    `PassGraph` literals directly): one buffer-shaped, `history`-lifetime target; one `Pass` whose
    `writeTarget` is that buffer, reading it with `version:'previous'`. **The presentation texture
    is deliberately not part of this `PassGraph` or `ResourceRealizer` at all** — `Pass.writeTarget`
    stays singular and refers only to the buffer being proven; the texture is incidental
    visualization output, allocated and owned directly by `bufferFeedback.ts`:
    ```ts
    const presentationTexture = device.createTexture({
      size: [gridWidth, gridHeight],
      format: 'rgba8unorm',
      usage: GPUTextureUsage.RENDER_ATTACHMENT | GPUTextureUsage.COPY_SRC
    });
    ```
    The buffer's declared `TypeRef.buffer.usages` must include `'copy-dst'` alongside `'storage'`
    (not just `'storage'`) so `resolveBufferUsage`/`ResourceRealizer` realize it with `COPY_DST`,
    which the initial-seed write (next step) requires. **`gridWidth * gridHeight` must equal the
    buffer's declared `element-count`** — state this explicitly as the sample's own invariant; the
    fragment shader's `index` computation and the render-pass viewport both depend on it.
11. **`executeBufferFeedback(device, realizer, passGraph, gridWidth, gridHeight)`**: calls
    `realizer.realizeAll` once, then **seeds the initial pattern into `readPrevious` via
    `device.queue.writeBuffer`** before the first frame (a real, necessary detail revision 1 missed
    — without it, frame 1 reads uninitialized/undefined memory, not a known pattern). Each frame:
    encode a render pass into `presentationTexture` (color attachment) with the buffer bind group
    from step 9, submit, `realizer.advanceFrame()`. Determinism is asserted via **texture readback**
    (`copyTextureToBuffer` + `mapAsync` on `presentationTexture`, the same readback code path
    already used elsewhere) — not a separate buffer-readback mechanism — since the presentation
    texture already encodes state as grayscale by construction.

### D. Document marker and dispatch (buffer sample's editor execution path)

12. **New, minimal sink primitive `target.bufferFeedback`** (`kind:'sink'`), carrying `gridWidth`/
    `gridHeight` as params — just enough to be a valid execution root for graph-validity purposes
    (`discoverExecutionRoots` must see *some* sink so the document isn't flagged as dead/unreachable);
    its `SinkDefinition.deriveInvocation` can be minimal, since the actual execution bypasses
    `compileGraph` entirely (Fix C).
13. **A tiny derivation function**, mirroring `presentationForDisplay`'s shape:
    ```ts
    export function deriveBufferFeedbackTarget(
      doc: GraphDocument
    ): { sinkNodeId: string; gridWidth: number; gridHeight: number } | null;
    ```
14. **`GraphFrameExecutor.execute` checks for a buffer-feedback target** (in addition to its normal
    independent/channel-read pass loop): if `deriveBufferFeedbackTarget(input.graph)` returns
    non-null, it also runs `executeBufferFeedback` and merges the resulting readback pixels into
    the **same** `targets: Record<string, Uint8Array>` result under the sink node's id — the buffer
    sample lands in the exact same preview/pixel-display UI every other target already uses, with
    no new preview panel or family required.

### E. Sample-graph wiring convention (texture case)

15. `input.channel`'s `sourceDisplayId` (Fix step 1) is document convention, not IR-enforced — the
    graph/primitive system does not validate this string against anything at authoring time (no new
    port, no new edge kind). This is the minimal, unambitious wiring mechanism the two hardcoded
    samples need; a real multi-pass authoring UX (drag a display node onto a channel slot,
    live-validated) is explicitly editor/Foundation-3/4 work, not this milestone's.

### F. Bundling

16. **Register both samples** in `GRAPH_SAMPLES` (`graph-editor/src/samples.ts`), builders in
    `graphBuilders.ts`, following the exact `migration-default-preview`/`migration-fullscreen-fragment`
    precedent (`{id, label, build}`). Per standing instruction, both must be pickable in the editor,
    not headless-only fixtures.

## Out of scope

A general, editor-authorable way to wire "read another pass's output" (drag-and-drop channel
binding, live validation, arbitrary channel counts) — the two samples use a fixed
`params.sourceDisplayId` string convention, not a new port/edge kind. A graph-authorable
storage-buffer-write primitive or compute dispatch (Foundation 3/4 — the buffer sample's WGSL is
fully hand-written instead). Bind-group *layout* derivation beyond `layout: 'auto'` (unnecessary —
WebGPU already derives it from the WGSL bindings this milestone adds). Any change to
`buildIndependentPassGraph`/`planIndependentGraphFramePasses` (both stay exactly as landed;
`resolveChannelDependencies` is a fully separate, additive function). Fixing `stageEntry.ts`'s
