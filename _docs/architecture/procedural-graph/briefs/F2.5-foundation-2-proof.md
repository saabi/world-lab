## Revision note

**Revision 3** fixed three more confirmed blocking defects and two smaller ones in revision 2,
caught before routing:

1. **Blocking — the buffer path cannot share `GraphFrameExecutor`'s `ResourceRealizer`.**
   `realizeAll()` prunes any resource absent from its *current* `PassGraph` — the normal pipeline
   graph's call would destroy the buffer's history state every frame, and the buffer graph's call
   would then destroy the pipeline's texture targets. Both paths calling `advanceFrame()` on the
   same instance would also double-flip parity. Fixed: a fully separate, persistent, device-keyed
   **`BufferFeedbackExecutor`** class owns its own `ResourceRealizer`, presentation texture,
   one-time seed state, and a grid/device fingerprint — structurally mirroring
   `GraphFrameExecutor` itself. `GraphFrameExecutor` owns one instance of it (alongside
   `PipelineGraphExecutor`) and its `dispose()` disposes both.
2. **Blocking — the buffer sink still would not appear or execute in the editor.**
   `previewFrameLoop.ts`'s `tick()` returns early with "No pipeline preview targets" whenever
   `planIndependentGraphFramePasses` finds zero passes — **before** `GraphFrameExecutor.execute` is
   ever called, so a buffer-only document never reaches it regardless of what
   `GraphFrameExecutor` itself does. `enumeratePreviewBuffers` also has no case for
   `target.bufferFeedback`, so no selectable row exists even if it did render. Fixed: the early-exit
   guard also checks `deriveBufferFeedbackTarget`; `enumeratePreviewBuffers` gains a
   `deriveMeshTargets`-style loop producing one `family: 'image'` buffer whose `id` equals the
   executor result key. `previewFrameLoop.ts`/`previewBuffers.ts` and their tests are added to this
   milestone's ownership.
3. **Blocking — per-target dimensions conflict with `GraphFrameExecuteResult`'s single global
   size.** The result type has one `width`/`height` for every target's pixels, but the buffer
   sink declares its own `gridWidth`/`gridHeight`. Restructuring the result to per-target
   `{width, height, pixels}` is too broad for this milestone. Fixed: require
   `gridWidth === input.width && gridHeight === input.height`, reject a mismatch with a clear
   error, and author the sample's sink params to match the preview loop's actual viewport.
4. **Medium — "caller-owned" presentation-texture ownership was ambiguous.** Resolved by finding
   1's fix: `BufferFeedbackExecutor` allocates, resizes (on fingerprint change), and destroys the
   presentation texture; the per-frame render step only uses it.
5. **Medium — defensive channel validation was incomplete.** The emission check omitted the upper
   bound (`> 3`) the schema declares, and dependency resolution used `Number(...)`, which silently
   coerces non-numeric values instead of rejecting them; nothing rejected two `input.channel` nodes
   in one pass assigning the same channel to two different sources. Fixed: one shared, strict
   `parseChannelIndex` helper used by both emission and dependency resolution; a same-channel/
   different-source conflict now throws.

## Context

Fifth and final item of Foundation 2's sequence (F2.1 ✅ → F2.2 ✅ → F2.3 ✅ → F2.4 ✅ → **proof**).
Per the plan: two bundled, hardcoded samples — a cross-pass texture read (one pass feeds another,
both textures, same-frame) and a small buffer-feedback sample (storage-buffer ping-pong, genuinely
previous-frame history) — proving the resource-history model generalizes, not just typechecks.
F2.4's own brief flagged that neither sample could exist as a real document without more work; this
brief is that work, scoped as narrowly as the two samples actually require — **not** Foundation 3's
generic kernel/binding model.

**Why this is narrower than it sounds.** Two pieces of infrastructure for exactly this were already
reserved and unused:

1. **`HostBinding{context:'read-resource'}`** (`packages/graph/src/implementation.ts`) — reserved
   since F1.3, zero consumers anywhere (confirmed by grep). This milestone is its first use.
2. **`BindingDecl.kind: 'texture' | 'sampler' | 'storage-read'`**
   (`packages/compiler/src/stageEntry.ts`) — the bind-group-declaration machinery for exactly these
   binding kinds already exists and already emits correct WGSL. **`layout: 'auto'`** (WebGPU's
   automatic bind-group-layout derivation, already used by `createRenderPipeline` in
   `fullscreenFragment.ts`) means no bind-group-*layout* derivation code is needed at all — WebGPU
   derives it from whatever bindings the WGSL declares. (The buffer sample ends up not using
   `BindingDecl`/`assembleStageEntry` at all — see Fix C — but the texture case's `texture`/
   `sampler` kinds are exactly this precedent.)

**What's genuinely new, and why it's still small:**

- One new `host-input` primitive (`input.channel`) and one new `emitHostInput` branch, the exact
  same shape as the five branches already there for `iTime`/`iResolution`/`fragCoord`/`uv`/
  `metricPosition`. This emits a self-contained `textureSample(...)` expression, nothing more.
- A small, purpose-built function that detects channel-read wiring in a document and produces real
  `ResourceRead`s — **not** a change to `buildIndependentPassGraph`/`planIndependentGraphFramePasses`
  (both stay exactly as they are, for the — still far more common — zero-read case).
- The buffer sample's fragment entry is a **fully self-contained, hand-written WGSL module**, not a
  new generic "storage write as a graph primitive" mechanism — matching the plan's own framing, a
  *hardcoded* proof sample, not a new authoring capability.
- One new, minimal sink primitive (`target.bufferFeedback`) and a small, separate
  `BufferFeedbackExecutor` so the buffer sample is a real, dispatchable, editor-visible
  `GraphDocument` — not a standalone function nothing ever calls, and not sharing state with
  `GraphFrameExecutor`'s own `ResourceRealizer`.

This keeps the milestone inside "wire two fixed, illustrative samples through the resource model,"
not "build the general multi-pass authoring system" — that generalization work stays Foundation
3/4's job, unchanged from F2.4's own Out-of-scope framing.

## Verified current state

- **`buildGraphEvalFn`** emits `fn graph_eval_<name>(position: vec4f) -> vec4f`. `position` is the
  **only** parameter. There is no `u`/`v`/`uv` variable in scope unless a node explicitly emits one
  under that exact name, which nothing does today.
- **`derivePipelinePresentations`/`presentationForDisplay`** only walk a `target.display` node's
  own `color ← stage.fragment ← <field>` chain. Each `PipelinePresentation` carries both
  `displayNodeId` (raw node id) and `outputName` (the derived name that becomes
  `GraphFramePass.targetId`) — exactly what's needed to resolve a human-authored `sourceDisplayId`.
- **`emitGraphEval`** runs `topologicalSort(doc, output.node)` to collect every node upstream of a
  given output — the reachability computation this milestone's channel-detection reuses.
- **`assembleStageEntry`** validates `output`/`outputFns` before checking `opts.entryOverride` — a
  real, separate latent gap in `stageEntry.ts`. This milestone's buffer path doesn't call this
  function at all (Fix C), so the gap doesn't block it and isn't touched.
- **`previewFrameLoop.ts`'s `tick()`** (`packages/graph-editor/src/previewFrameLoop.ts:65-76`)
  computes `planIndependentGraphFramePasses(options.graph)` and returns immediately with
  `error: 'No pipeline preview targets'` if it's empty — **before** `device` is requested or
  `executor.execute` is ever called. Confirmed by direct read: this is the actual, only call site
  that invokes `GraphFrameExecutor` from the editor; nothing else routes a loaded sample to it.
- **`enumeratePreviewBuffers`** (`packages/graph-editor/src/previewBuffers.ts:167-232`) already has
  exactly the precedent needed: a dedicated loop over `deriveMeshTargets(doc)` producing one
  `PreviewBuffer` per mesh sink via `bufferFromMeshSink`, appended after the pipeline-sink loop.
  `PreviewBuffer.id` is the key consumers use to look up rendered pixels — for `target.display`'s
  own texture-only case (`bufferFromPipelineSink`'s no-`fieldOutput` branch), `id` is the raw sink
  node id and `family` is `'image'`, with `source: { sinkNode }` — no `fieldOutput` to evaluate, the
  panel just indexes the frame snapshot's `targets` record by `id` directly. This is the exact
  shape `target.bufferFeedback`'s own `PreviewBuffer` needs, reusing the same panel code path.
- **`GraphFrameExecuteResult`** (`graphFrameExecutor.ts`) has one `width`/`height` for the whole
  frame, shared by every entry in `targets: Record<string, Uint8Array>`. Confirmed: no per-target
  size exists anywhere in this type today.
- **Sample registration** (`packages/graph-editor/src/samples.ts`) is a flat `{id, label,
  build(): GraphDocument}[]` list (`GRAPH_SAMPLES`) — document-only. Nothing about *executing* a
  selected sample lives in this registry, which is why the buffer sample's real dispatch path runs
  through `previewFrameLoop.ts`/`GraphFrameExecutor`, not the registry itself.

## Fix

### A. Shared: the channel-read primitive, its WGSL emission, and one strict index parser

1. **New primitive `input.channel`** (`packages/graph/src/primitives/host/channel.ts`, new file),
   following `host.iTime`'s exact shape:
   ```ts
   const channel: NodePrimitiveInput = {
     id: 'input.channel',
     category: 'ShaderToy',
     inputs: [],
     outputs: [{ name: 'color', dataType: 'vec4f' }],
     params: Type.Object({
       channel: Type.Integer({ default: 0, minimum: 0, maximum: 3 }),
       sourceDisplayId: Type.String({ default: '' })
     }),
     implementation: { kind: 'host-input', binding: { context: 'read-resource', key: 'channel' } },
     evalCPU() {
       throw new Error('input.channel is GPU-pipeline only; no CPU evaluation path');
     }
   };
   ```
   `params.sourceDisplayId` (document convention, not IR-enforced — see Fix E) names the *other*
   display node this channel reads; `resolveChannelDependencies` reads it as data, nothing more.
2. **One shared, strict channel-index parser** — used by both emission and dependency resolution,
   so validation can't drift between the two call sites:
   ```ts
   // graphFramePlan.ts (or a small shared util both graph/runtime-webgpu import)
   export function parseChannelIndex(value: unknown, context: string): number {
     if (typeof value !== 'number' || !Number.isInteger(value) || value < 0 || value > 3) {
       throw new Error(`${context}: invalid channel index ${JSON.stringify(value)} (expected an integer 0-3)`);
     }
     return value;
   }
   ```
3. **New `emitHostInput` branch** (`emitGraphEval.ts`) — self-contained, depends only on `position`
   (the function's actual parameter) and the channel's own texture binding, never a sibling node's
   output, and rejects out-of-range indices via the shared parser:
   ```ts
   if (binding.context === 'read-resource' && binding.key === 'channel') {
     const channelIndex = parseChannelIndex(node.params?.channel, `input.channel node ${node.id}`);
     return (
       `let ${variable} = textureSample(channel${channelIndex}, channel${channelIndex}Sampler, ` +
       `position.xy / vec2<f32>(textureDimensions(channel${channelIndex})));`
     );
   }
   ```
   `textureDimensions` reads the actual bound texture's size at runtime — correct regardless of
   viewport/render-target size, no externally-computed UV or extra `opts` field needed.
4. **`EmittedGraphEval` gains `usedChannels: number[]`** (sorted, distinct channel indices actually
   emitted), avoiding a regex-the-output-text approach for a dynamic channel count.
5. **`assembleFullscreenFragmentModuleAsync` returns the actual channel-binding plan**, the single
   source of truth both the WGSL declarations and the executor's bind-group entries read from:
   ```ts
   channelBindings: ReadonlyMap<number, { textureBinding: number; samplerBinding: number }>;
   ```
   Allocated after the existing uniform bindings, one texture + one sampler binding per entry in
   `usedChannels`.
6. **`executeFullscreenFragment` accepts resolved channel textures keyed by channel number, and
   rejects a missing entry clearly:**
   ```ts
   export interface FullscreenFragmentInput {
     /* ...existing fields... */
     channelTargets?: ReadonlyMap<number, GPUTexture>; // caller-resolved, caller-owned
   }
   ```
   For each `[channel, {textureBinding, samplerBinding}]` in `channelBindings`: look up
   `input.channelTargets?.get(channel)`; if missing, throw
   `` `Missing channel target for channel ${channel}` `` before building the bind group. `layout:
   'auto'` derives the rest.

### B. Cross-pass texture read sample: real `PassGraph` derivation

7. **A small, purpose-built function detects channel wiring and augments the plan — it does not
   touch `buildIndependentPassGraph`/`planIndependentGraphFramePasses`.** Resolution goes through
   `derivePipelinePresentations`'s `displayNodeId ↔ outputName` pairing; a same-channel conflict
   across two `input.channel` nodes in one pass is rejected explicitly:
   ```ts
   // graphFramePlan.ts, new export, additive
   export interface ChannelDependency {
     consumerId: string;     // the pass whose field subgraph reads the channel (== its targetId)
     channel: number;
     sourceTargetId: string; // resolved via presentation.displayNodeId -> presentation.outputName
   }

   export function resolveChannelDependencies(
     doc: GraphDocument,
     passes: readonly GraphFramePass[]
   ): ChannelDependency[] {
     const targetIdByDisplayNodeId = new Map(
       derivePipelinePresentations(doc).map((p) => [p.displayNodeId, p.outputName])
     );
     const dependencies: ChannelDependency[] = [];
     for (const pass of passes) {
       const byChannel = new Map<number, string>();
       // Reuse emitGraphEval's own upstream-reachability walk from pass.fieldOutput.node to find
       // every input.channel node feeding this pass (do not reimplement topological reachability).
       for (const node of channelNodesUpstreamOf(doc, pass.fieldOutput.node)) {
         const channel = parseChannelIndex(node.params?.channel, `input.channel node ${node.id}`);
         const sourceDisplayId = String(node.params?.sourceDisplayId ?? '');
         const sourceTargetId = targetIdByDisplayNodeId.get(sourceDisplayId);
         if (!sourceTargetId) {
           throw new Error(`input.channel node ${node.id} references unknown display "${sourceDisplayId}"`);
         }
         const existing = byChannel.get(channel);
         if (existing && existing !== sourceTargetId) {
           throw new Error(
             `Pass "${pass.targetId}" has conflicting sources for channel ${channel}: "${existing}" vs "${sourceTargetId}"`
           );
         }
         byChannel.set(channel, sourceTargetId);
         dependencies.push({ consumerId: pass.targetId, channel, sourceTargetId });
       }
     }
     return dependencies;
   }

   export function buildPassGraphWithChannelReads(
     passes: readonly GraphFramePass[],
     dependencies: readonly ChannelDependency[]
   ): PassGraph {
     // Same target/lifetime construction as buildIndependentPassGraph — every target stays
     // `transient` (this sample needs no history/ping-pong, only ordering) — but each pass's
     // `reads` includes a ResourceRead per matching ChannelDependency (`{channel, target:
     // sourceTargetId}`, version omitted: a same-frame read of the other pass's current output).
   }
   ```
   `GraphFrameExecutor.execute` calls `resolveChannelDependencies` once per frame; if it returns
   `[]` (the common case today), behavior is byte-for-byte identical to F2.4's landed path. If
   non-empty, it calls `buildPassGraphWithChannelReads` instead — the *only* branch point this
   milestone adds to the executor's existing pipeline-target handling, keeping F2.4's own gate
   items intact.
8. **`GraphFrameExecutor` resolves each pass's channel textures before calling `pipeline.execute`**,
   using `ResourceRealizer.resolve(dependency.sourceTargetId).write` (same-frame read; F2.2's
   validation already guarantees a same-frame `ResourceRead` never needs `.readPrevious`), passed
   through as `channelTargets`. This requires `buildPassOrder`'s `order.order` to be respected for
   the first time with real consequence: pass A must execute before pass B reads it.
9. **The sample graph itself** (`graphBuilders.ts`, new builder, e.g. `crossPassTextureReadGraph`):
   two `target.display` sinks (`n_display_a` writes a base pattern; `n_display_b`'s field subgraph
   wires `input.channel{channel:0, sourceDisplayId:'n_display_a'}` and combines it with its own
   procedural field). Bundled and pickable (Fix G).

### C. Buffer-feedback sample: a fully separate, persistent `BufferFeedbackExecutor`

10. **`BufferFeedbackExecutor`** (`packages/runtime-webgpu/src/consumers/bufferFeedback.ts`, new
    file) — a standalone class, structurally mirroring `GraphFrameExecutor` itself, owning
    everything the buffer sample needs so it never touches `GraphFrameExecutor`'s own
    `ResourceRealizer`/`PipelineGraphExecutor`:
    ```ts
    export class BufferFeedbackExecutor {
      private realizer: ResourceRealizer | undefined;
      private presentationTexture: GPUTexture | undefined;
      private device: GPUDevice | undefined;
      private gridWidth: number | undefined;
      private gridHeight: number | undefined;
      private seeded = false;

      private ensureAllocated(device: GPUDevice, gridWidth: number, gridHeight: number): void {
        const unchanged =
          this.device === device && this.gridWidth === gridWidth && this.gridHeight === gridHeight;
        if (unchanged && this.realizer && this.presentationTexture) return;
        this.dispose();
        this.realizer = new ResourceRealizer(device);
        this.presentationTexture = device.createTexture({
          size: [gridWidth, gridHeight],
          format: 'rgba8unorm',
          usage: GPUTextureUsage.RENDER_ATTACHMENT | GPUTextureUsage.COPY_SRC
        });
        this.device = device;
        this.gridWidth = gridWidth;
        this.gridHeight = gridHeight;
        this.seeded = false;
      }

      async execute(device: GPUDevice, gridWidth: number, gridHeight: number): Promise<{ pixels: Uint8Array }> {
        this.ensureAllocated(device, gridWidth, gridHeight);
        const passGraph = buildBufferFeedbackPassGraph(gridWidth * gridHeight); // pure, cheap to rebuild
        this.realizer!.realizeAll(passGraph, { width: gridWidth, height: gridHeight });
        if (!this.seeded) {
          // device.queue.writeBuffer(this.realizer!.resolve(BUFFER_ID).readPrevious as GPUBuffer, ...)
          this.seeded = true;
        }
        // encode a render pass into this.presentationTexture using the bound buffer slots from
        // this.realizer!.resolve(BUFFER_ID) (previous/next), submit, this.realizer!.advanceFrame(),
        // then read back this.presentationTexture via copyTextureToBuffer + mapAsync.
      }

      dispose(): void {
        this.realizer?.dispose();
        this.presentationTexture?.destroy();
        this.realizer = undefined;
        this.presentationTexture = undefined;
        this.device = undefined;
        this.gridWidth = undefined;
        this.gridHeight = undefined;
        this.seeded = false;
      }
    }
    ```
    `ensureAllocated`'s fingerprint (device identity + grid dimensions) is what F2.3's
    `ResourceRealizer.realizeAll` itself already does for individual resources — this class does
    the same at the *executor* level, since a grid-dimension change means a different buffer
    element count and a different-sized presentation texture, both requiring a fresh instance and
    a fresh seed.
11. **The hand-written WGSL module is fully self-contained** — bindings and entry together, **not**
    routed through `assembleStageEntry` at all, so there is exactly one place these bindings are
    declared:
    ```ts
    const code = `
      @group(0) @binding(0) var<storage, read> previous: array<f32>;
      @group(0) @binding(1) var<storage, read_write> next: array<f32>;
      @fragment
      fn fs_main(@builtin(position) position: vec4f) -> @location(0) vec4f {
        let index = u32(position.x) + u32(position.y) * ${gridWidth}u;
        let alive = previous[index];
        // ...fixed next-state rule (e.g. one cellular-automaton step)...
        next[index] = nextState;
        return vec4f(nextState, nextState, nextState, 1.0);
      }
    `;
    ```
12. **The hand-built `PassGraph` models only the buffer** — the actual resource-history proof
    target — mirroring how `order.test.ts`/`realize.test.ts` already build `PassGraph` literals
    directly: one buffer-shaped, `history`-lifetime target; one `Pass` whose `writeTarget` is that
    buffer, reading it with `version:'previous'`. The presentation texture is **not** part of this
    `PassGraph`/`ResourceRealizer` — `Pass.writeTarget` stays singular, referring only to the
    buffer. The buffer's declared `TypeRef.buffer.usages` includes `'copy-dst'` alongside
    `'storage'` (so the realized buffer supports the initial-seed write). **`gridWidth * gridHeight`
    must equal the buffer's declared element count** — this milestone's own fixture invariant.

### D. Document marker, dispatch, and dimension contract (buffer sample's editor execution path)

13. **New, minimal sink primitive `target.bufferFeedback`** (`kind:'sink'`), carrying `gridWidth`/
    `gridHeight` as params — just enough to be a valid execution root
    (`discoverExecutionRoots`); its `SinkDefinition.deriveInvocation` is minimal, since the actual
    execution bypasses `compileGraph` entirely.
14. **A tiny derivation function**, mirroring `presentationForDisplay`'s shape:
    ```ts
    export function deriveBufferFeedbackTarget(
      doc: GraphDocument
    ): { sinkNodeId: string; gridWidth: number; gridHeight: number } | null;
    ```
15. **`GraphFrameExecutor` owns one persistent `BufferFeedbackExecutor` instance** (alongside its
    existing `PipelineGraphExecutor`), and `execute` checks for a buffer-feedback target **in
    addition to** its normal pipeline-target handling:
    ```ts
    const bufferTarget = deriveBufferFeedbackTarget(input.graph);
    if (bufferTarget) {
      if (bufferTarget.gridWidth !== input.width || bufferTarget.gridHeight !== input.height) {
        throw new Error(
          `target.bufferFeedback grid (${bufferTarget.gridWidth}x${bufferTarget.gridHeight}) ` +
          `must match the frame viewport (${input.width}x${input.height})`
        );
      }
      const { pixels } = await this.bufferFeedbackExecutor.execute(
        input.device, bufferTarget.gridWidth, bufferTarget.gridHeight
      );
      targets[bufferTarget.sinkNodeId] = pixels;
    }
    ```
    This is the deliberate, narrow resolution to the size-contract gap: `GraphFrameExecuteResult`
    keeps its single global `width`/`height` (restructuring it per-target is out of scope for this
    milestone), and a mismatch is rejected clearly rather than silently producing an
    inconsistently-sized pixel array under the same snapshot. The bundled sample's sink params are
    authored to equal the preview loop's actual configured viewport (Fix G).
16. **`GraphFrameExecutor.dispose()` disposes both executors**:
    ```ts
    dispose(): void {
      this.realizer?.dispose();
      this.bufferFeedbackExecutor.dispose();
      this.realizer = undefined;
      this.realizerDevice = undefined;
    }
    ```
17. **`previewFrameLoop.ts`'s early-exit guard also checks for a buffer-feedback target**, so a
    buffer-only document reaches `GraphFrameExecutor.execute` at all:
    ```ts
    const passes = planIndependentGraphFramePasses(options.graph);
    const bufferFeedbackTarget = deriveBufferFeedbackTarget(options.graph);
    if (passes.length === 0 && !bufferFeedbackTarget) {
      emit({ ..., error: 'No pipeline preview targets' });
      return;
    }
    ```
18. **`enumeratePreviewBuffers` gains a `deriveBufferFeedbackTarget`-driven loop**, mirroring the
    existing `deriveMeshTargets` loop exactly, producing a `family: 'image'` buffer whose `id`
    equals the sink node id — the same id `GraphFrameExecutor` uses as the `targets` key (Fix
    step 15), so the frame snapshot's pixels are found by the panel exactly the way `target.display`'s
    own texture-only case already works (`source: { sinkNode }`, no `fieldOutput` to evaluate):
    ```ts
    const bufferFeedbackTarget = deriveBufferFeedbackTarget(doc);
    if (bufferFeedbackTarget) {
      buffers.push({
        id: bufferFeedbackTarget.sinkNodeId,
        label: labelForNode(doc, bufferFeedbackTarget.sinkNodeId),
        source: { sinkNode: bufferFeedbackTarget.sinkNodeId },
        dataType: 'storageBuffer',
        family: 'image',
        inferred: true
      });
    }
    ```
    `resolvePreviewBufferPort`/`resolveMeshPreviewRequest` need no changes — this buffer's pixels
    never go through port-based evaluation, only the frame snapshot's `targets` record.

### E. Sample-graph wiring convention (texture case)

19. `input.channel`'s `sourceDisplayId` (Fix step 1) is document convention, not IR-enforced — the
    graph/primitive system does not validate this string against anything at authoring time (no new
    port, no new edge kind). This is the minimal, unambitious wiring mechanism the two hardcoded
    samples need; a real multi-pass authoring UX is explicitly editor/Foundation-3/4 work.

## Out of scope

A general, editor-authorable way to wire "read another pass's output" (drag-and-drop channel
binding, live validation, arbitrary channel counts) — the two samples use a fixed
`params.sourceDisplayId` string convention, not a new port/edge kind. A graph-authorable
storage-buffer-write primitive or compute dispatch (Foundation 3/4 — the buffer sample's WGSL is
fully hand-written instead). Bind-group *layout* derivation beyond `layout: 'auto'`. Any change to
`buildIndependentPassGraph`/`planIndependentGraphFramePasses`. Fixing `stageEntry.ts`'s
validate-before-`entryOverride`-check ordering (a real, separate gap, not touched — this
milestone's buffer path avoids that function entirely). Restructuring `GraphFrameExecuteResult` to
per-target `{width, height, pixels}` (out of scope per finding 3's own resolution — a strict
dimension-match requirement is used instead). CPU evaluation of `input.channel`. N-channel
generality beyond what the two samples need. Any change to Foundation 1's milestones or F2.1–F2.4's
landed contracts.

## Handoff

→ Foundation 2's resource-history model is proven end-to-end with two real, pickable, rendering
samples — one same-frame cross-pass texture read (ordering, no ping-pong needed), one
previous-frame storage-buffer feedback (the genuine history/ping-pong proof, driven by its own
fully independent `BufferFeedbackExecutor` so it can never collide with `GraphFrameExecutor`'s own
resource lifecycle) — both reachable from the actual editor preview loop and both selectable in the
buffer-list UI. The channel-read primitive and its WGSL emission are narrowly scoped and
deliberately do not anticipate Foundation 3's actual kernel/binding model. Foundation 2 is complete
once this lands. Foundation 3 (generic vertex/fragment/compute kernels, arbitrary resource
bindings, typed varyings) is next, whenever you want its own sequencing plan — it is a larger,
from-scratch design effort, not a follow-on contract in the same shape as F2.1–F2.5.
