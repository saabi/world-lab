## Objective

Finish the `M-node-model-foundation` decomposition follow-up correctly before
[`M-pipeline-nodes-s0`](./M-pipeline-nodes-s0.md) starts.

The latest committed docs say `math.remap` / `sdf.opSubtract` decomposition landed, but
the current working tree contains a correctness fallback that makes them atomic again.
That fallback prevents invalid WGSL, but it does not satisfy the intended node-group
architecture.

Implement the intended group-backed decomposition while preserving public primitive
contracts and dependency correctness.

## Pinned contract

The executor must implement this contract; do not choose a different group-param or
built-in-group representation.

### Group parameters

Extend the graph group model with an optional param interface:

- `GroupInterface.params?: GroupParamMapping[]`
- each mapping has only `name` and `target: PortRef`
- `GroupDefinition.params?: TSchema` is a TypeBox object schema and is the sole source of
  param type/default/widget/range metadata
- mapping names must exactly match the object schema properties; do not add a parallel
  `ParamSpec`-like metadata shape

`groupToFunction()` emits function arguments in this order: interface inputs, then
interface params. It maps both kinds to internal input ports, but emits them under
separate `inputs:` and `params:` frontmatter sections. It must serialize the
loader-supported TypeBox metadata needed to reconstruct an equivalent primitive param
schema. Groups with no params retain their current output.

For `math.remap`, the canonical order is
`x, inMin, inMax, outMin, outMax`; `x` is the only input and the four bounds are params.

### Canonical built-in groups

Canonical `GroupDefinition`s for `math.remap` and `sdf.opSubtract` live under
`packages/procedural-wgsl/src/groups/`. Their exported standard-library module sources
must be generated by `groupToFunction()`, then passed through `loadWgslPrimitive()` to
derive the declared `dependencies`/imports. Do not hand-write composed WGSL plus a manual
dependency list.

`@virtual-planet/procedural-wgsl` may add direct workspace dependencies on
`@virtual-planet/graph` and `@virtual-planet/schema`; it already depends on the compiler.
The graph package continues to register matching `NodePrimitive`s manually because graph
must not depend on compiler. Add parity tests between those registrations and the loaded
generated contracts. Parity means the mechanical contract (id, category, inputs, params,
outputs, WGSL entry/argument classification) plus role/help/usage. Preserve graph-only
keywords/pure/deterministic metadata; do not expand `GroupDefinition` metadata in this
task.

### Scalar min/max entries

Preserve primitive/module ids `math.min` and `math.max`, but rename their WGSL entry
functions and `NodePrimitive.wgsl.entry` values to non-builtin names (`mathMin` and
`mathMax`). The module body may call WGSL builtins `min`/`max`; it must not declare
`fn min` or `fn max` and recursively shadow the builtin.

## Current dirty-tree context

At handoff, the working tree contains an uncommitted correctness fallback:

- `packages/procedural-wgsl/src/modules/math/remap.ts` was restored to atomic WGSL.
- `packages/procedural-wgsl/src/modules/sdf/ops.ts` was restored to atomic `opSubtract`.
- `packages/procedural-wgsl/src/modules/use-deps.test.ts` adds a guard against inert raw
  `// @use` comments.
- `packages/compiler/src/groupCodegen.test.ts` adds explicit port directions.

Preserve the correctness insight: raw procedural-wgsl module source comments like
`// @use math.subtract` are inert unless represented as declared dependencies. The fix is
not to put undeclared `// @use` comments in raw module constants.

## Requirements

1. Keep or improve the `use-deps.test.ts` regression guard.
2. Ensure all elemental ops needed by groups exist in both graph and procedural-wgsl:
   - `math.add`
   - `math.subtract`
   - `math.multiply`
   - `math.divide`
   - `math.min`
   - `math.max`
   - add `math.negate` for `sdf.opSubtract`
3. Restore `math.remap` as a generated group/function over:
   - `subtract(x, inMin)`
   - `subtract(inMax, inMin)`
   - `divide(...)`
   - `subtract(outMax, outMin)`
   - `multiply(...)`
   - `add(outMin, ...)`
4. Preserve `math.remap`'s public primitive contract:
   - graph input: `x`
   - params: `inMin`, `inMax`, `outMin`, `outMax`
   - output: `value`
   - `paramInputPorts` / `promotableParams` tests still pass
5. Restore `sdf.opSubtract` as a group over max + negate, preserving:
   - inputs: `a`, `b`
   - output: `distance`
6. Do not reintroduce `sdf.opUnion` / `sdf.opIntersect` as preferred aliases. They remain
   deprecated in favor of `math.min` / `math.max` + help text.
7. Add tests proving group-generated WGSL declares and loads dependencies through the M3
   loader path, not through inert raw source comments.
8. Add or adjust tests so linked WGSL for `remap` and `opSubtract` includes dependency
   modules and does not call undefined functions.
9. Rename scalar min/max WGSL entries exactly as pinned above and test that their module
   sources do not declare `fn min(` or `fn max(`.
