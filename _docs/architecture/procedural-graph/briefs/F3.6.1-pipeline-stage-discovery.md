## Context

First of F3.6's tentative sequence (F3.6.1 → F3.6.2 → F3.6.3 → F3.6.4 → F3.6.5 → F3.6.6, per the
design document — each gets its own contract; this is the first). Sub-problem (C) from that
document, in full, already reviewed through three rounds there before this contract was drafted:
`target.display`'s only mechanism for identifying "the fragment stage" (and, transitively, "the
vertex stage") is a hardcoded primitive-id string comparison — `fragmentNode.primitive !==
'stage.fragment'` — at four separate call sites, with no fallback and no structural/role check. A
graph-authored kernel-based fragment or vertex stage under any other primitive id makes every one
of these checks fail silently (return `null`/throw), even once such a primitive exists and is
correctly wired.

**This milestone registers no new stage primitive and changes no rendered pixel.** It only swaps
the *discovery mechanism* four hardcoded checks use, from primitive-id string comparison to a new,
purpose-built metadata field — the same "walk the document structurally, don't hardcode an id"
idiom this codebase already uses for `isPipelineTarget`/`isBufferFeedbackTarget`/
`isComputeBufferTarget`. `stage.vertex`/`stage.fragment` keep behaving exactly as they do today;
every existing sample renders identically. This is the lowest-risk possible first F3.6 slice,
and a clean prerequisite for F3.6.2 (which needs *something* to recognize a stage node by, before
it can decide what to do once it does).

**No manual/visual gate, no bundled sample.** Nothing rendered changes — this is purely a discovery-
mechanism swap underneath primitives that already exist and already work. The real visual proof is
F3.6.6's job, once a real graph-authored stage primitive exists to render with.

**Pre-routing review found Gate item 7 (site 4's coverage) unbuildable as first drafted.** The
original wording asked for a test proving `resolvePipelineTarget`'s legacy fallback recognizes a
target-role node under a different id "returning a plan" — but `resolvePipelineTarget` is
module-private (not exported), and a genuinely *successful*, full `planPipelineGraph()` call can
never actually be reached through that fallback branch: `planPipelineGraph` independently
re-validates the fragment node's `pipelineStageKind` immediately after `resolvePipelineTarget`
returns (site 3, Fix step 5), and the *only* way `derivePipelinePresentations` misses a
presentation in the first place (forcing the fallback) is when the upstream node isn't a recognized
fragment stage — which then fails that same site-3 check moments later regardless of which branch
found the display node. Chasing a "full success via the fallback" scenario would have been a real
trap for whoever implemented this. The gate below is rewritten around a scenario that *is*
reachable and *is* meaningful: a deliberately minimal, otherwise-disconnected document, where the
fallback is reached and the **specific error message thrown** distinguishes old (id-based, fails
immediately, never finds the node at all) from new (role-based, finds the node, fails one check
later on the next missing piece) behavior — see Gate item 7 and its Fix-step note.

## Verified current state

- **Four call sites hardcode a primitive-id string check, not two or three — confirmed by direct
  read, not assumed:**
  1. `packages/graph/src/pipeline.ts:34`, inside `fieldOutputForDisplay`:
     `fragmentNode.primitive !== 'stage.fragment'`.
  2. `packages/runtime-webgpu/src/pipelineGraph.ts:121`, inside `resolveVertexForFragment`:
     `vertex.primitive !== 'stage.vertex'`.
  3. `packages/runtime-webgpu/src/pipelineGraph.ts:230`, inside `planPipelineGraph` itself:
     `fragment.primitive !== 'stage.fragment'`.
  4. `packages/runtime-webgpu/src/pipelineGraph.ts:188-190`, inside `resolvePipelineTarget`'s
     **legacy fallback branch** (reached only when `derivePipelinePresentations(doc)[0]` is
     `undefined` — i.e. when the primary, already-role-based discovery path finds no complete
     presentation):
     ```ts
     const display = discoverExecutionRoots(doc).find(
     	(node) => node.primitive === 'target.display'
     );
     ```
  Sites 1-3 identify a *vertex- or fragment-stage* node. Site 4 identifies a *target/sink* node —
  a different kind of thing — and needs a different fix (see Fix step 4). This fallback branch is
  currently **untested** by `pipelineGraph.test.ts` (confirmed by grep — zero references to
  `discoverExecutionRoots`/"execution root" in that file); this milestone's Gate item 7 is the
  first test ever to exercise it.
- **`isPipelineTarget` is the exact, already-proven precedent for the discovery-mechanism fix**
  (`packages/graph/src/pipeline.ts:189-192`, exported):
  ```ts
  export function isPipelineTarget(node: Node): boolean {
  	const primitive = getPrimitive(node.primitive);
  	return primitive?.metadata?.role === PIPELINE_TARGET_ROLE; // 'pipelineTarget'
  }
  ```
  `target.display` already declares `role: 'pipelineTarget'`
  (`packages/graph/src/primitives/pipeline/index.ts:70`) and `derivePipelinePresentations`'s
  *primary* path already uses `isPipelineTarget` (`pipeline.ts:104-105`) — only site 4's fallback,
  reached when that primary path finds nothing, still uses the old hardcoded id check.
- **`PrimitiveMetadata.role` cannot be reused for stage-kind discovery — confirmed, not assumed.**
  `role` is a single, optional string (`packages/graph/src/primitive.ts:19`, `role?: string`, not
  an array), and `stage.vertex`/`stage.fragment` **already both declare the same role**,
  `role: 'pipelineStage'` (`packages/graph/src/primitives/pipeline/index.ts:39,56`) — identical for
  both primitives, distinguishing neither from the other. That role is also the exact key
  `swapFamily()` (`packages/graph/src/contract.ts:31`) groups primitives by for the editor's
  node-palette/accent-color UI (`packages/graph-editor/src/nodeAccentColor.ts`,
  `nodePaletteModel.ts`). Repurposing or replacing it would change that UI grouping as a side effect
  of a discovery-mechanism fix — out of scope here, not investigated, and not this milestone's
  problem to solve. A **new**, separate `PrimitiveMetadata` field is required.
- **`PrimitiveMetadata`'s exact current shape** (`packages/graph/src/primitive.ts:7-24`):
  `description?/pure?/deterministic?/color?/icon?/keywords?/role?/help?/usage?` — all optional,
  plenty of room for one more additive optional field.
- **`resolveVertexForFragment`/`planPipelineGraph`'s fragment check/`fieldOutputForDisplay` are the
  *only* places in the codebase that inspect a stage node's primitive for identity purposes** —
  confirmed by grep for `'stage.vertex'`/`'stage.fragment'` as string literals across
  `packages/*/src` (excluding tests and the primitive registration itself): no other production
  code compares against these ids directly.
- **A second, separate constraint this milestone does not touch: port names.** Independent of the
  four primitive-id checks, `planPipelineGraph`'s `requireEdge` calls
  (`pipelineGraph.ts:234-241`) and `pipeline.ts`'s `fieldOutputForDisplay`/`resolveVertexForFragment`
  also hardcode the *port names* wired between stages (`mesh`/`varyings` on the vertex side;
  `varyings`/`color`/`texture` on the fragment side). This milestone's fix doesn't change or relax
  that — any future graph-authored stage primitive (F3.6.3+) still has to use these exact port ids
  regardless of what this milestone does. Noted here so the test-only fixture primitive this
  milestone registers (Fix step 6) uses matching port names, and so nobody mistakes this milestone
  for having generalized port shape too.

## Fix

### 1. `PrimitiveMetadata` gains `pipelineStageKind`

In `packages/graph/src/primitive.ts`, alongside the existing fields:

```ts
export type PipelineStageKind = 'vertex' | 'fragment';

export interface PrimitiveMetadata {
	description?: string;
	pure?: boolean;
	deterministic?: boolean;
	color?: string;
	icon?: string;
	keywords?: string[];
	role?: string;
	help?: string;
	usage?: string;
	/**
	 * Which pipeline stage this primitive fills, if any — a graph-authored kernel-based
	 * stage primitive declares the matching value to be recognized by `isPipelineStage`
	 * wherever `target.display`'s presentation logic currently hardcodes
	 * `'stage.vertex'`/`'stage.fragment'`. Independent of `role` (which groups `swapFamily()`
	 * swap candidates, a separate UI concern) — see F3.6.1's brief, Verified current state.
	 */
	pipelineStageKind?: PipelineStageKind;
}
```

### 2. `stage.vertex`/`stage.fragment` declare it

In `packages/graph/src/primitives/pipeline/index.ts`, add one field to each primitive's existing
`metadata` object — `role: 'pipelineStage'` stays exactly as-is on both:

```ts
// stage.vertex's metadata:
metadata: {
	description: 'Vertex stage node for pipeline graph execution.',
	pure: true,
	deterministic: true,
	role: 'pipelineStage',
	pipelineStageKind: 'vertex'
}
// stage.fragment's metadata:
metadata: {
	description: 'Fragment stage node that writes a field color into a texture resource.',
	pure: false,
	deterministic: false,
	role: 'pipelineStage',
	pipelineStageKind: 'fragment'
}
```

### 3. `isPipelineStage` — the new discovery predicate

In `packages/graph/src/pipeline.ts`, alongside `isPipelineTarget` (same file, same idiom):

```ts
export function isPipelineStage(node: Node, kind: PipelineStageKind): boolean {
	const primitive = getPrimitive(node.primitive);
	return primitive?.metadata?.pipelineStageKind === kind;
}
```

Import `PipelineStageKind` as a type from `./primitive.js` alongside this file's existing imports.
JS function-declaration hoisting already lets `isPipelineTarget` (declared at line ~189) be called
by code earlier in the same file (`derivePipelinePresentations`, line ~104) — `isPipelineStage`
follows the identical pattern; place it near `isPipelineTarget` for locality, not necessarily above
its own call site.

### 4. Fix site 1 — `fieldOutputForDisplay`

`packages/graph/src/pipeline.ts:29-39`:

```ts
function fieldOutputForDisplay(doc: GraphDocument, displayNode: Node): PortRef | null {
	const toDisplay = incomingEdge(doc, displayNode.id, 'color');
	if (!toDisplay) return null;

	const fragmentNode = doc.nodes.find((node) => node.id === toDisplay.from.node);
	if (!fragmentNode || !isPipelineStage(fragmentNode, 'fragment')) return null;
	if (toDisplay.from.port !== 'texture') return null;

	const fieldEdge = incomingEdge(doc, fragmentNode.id, 'color');
	return fieldEdge?.from ?? null;
}
```

### 5. Fix sites 2 and 3 — `resolveVertexForFragment` and `planPipelineGraph`

`packages/runtime-webgpu/src/pipelineGraph.ts`. Import `isPipelineStage` and `isPipelineTarget` as
types/values from `@world-lab/graph` alongside the file's existing import block (lines 2-7).

```ts
function resolveVertexForFragment(doc: GraphDocument, fragmentStageNode: string): Node {
	const varyingsEdge = incoming(doc, fragmentStageNode, 'varyings');
	if (!varyingsEdge) {
		throw new Error(`Pipeline fragment ${fragmentStageNode} is missing its varyings input`);
	}
	const vertex = doc.nodes.find((candidate) => candidate.id === varyingsEdge.from.node);
	if (!vertex || !isPipelineStage(vertex, 'vertex')) {
		throw new Error(`Pipeline graph is missing a vertex stage for fragment ${fragmentStageNode}`);
	}
	return vertex;
}
```

```ts
// inside planPipelineGraph, replacing the existing fragment check:
const fragment = doc.nodes.find((candidate) => candidate.id === target.fragmentStageNode);
if (!fragment || !isPipelineStage(fragment, 'fragment')) {
	throw new Error(`Pipeline graph is missing a fragment stage for display ${target.displayTargetNode}`);
}
```

Error message text changes deliberately (`"missing stage.vertex"` → `"missing a vertex stage"`,
same for fragment): once this fix lands, a *different* primitive declaring the matching
`pipelineStageKind` can also satisfy the check, so an error message naming `stage.vertex`
specifically would be misleading. `buffer.persist`'s own hardcoded check
(`pipelineGraph.ts:225`, `persist.primitive !== 'buffer.persist'`) and the already-role-based
geometry-source check (`findPipelineGeometrySource`, using `PIPELINE_GEOMETRY_SOURCE_ROLE`) are
**not** touched by this milestone — out of scope, see below.

### 6. Fix site 4 — `resolvePipelineTarget`'s legacy fallback

`packages/runtime-webgpu/src/pipelineGraph.ts:188-190`:

```ts
const display = discoverExecutionRoots(doc).find(isPipelineTarget);
```

`isPipelineTarget` needs adding to this file's `@world-lab/graph` import block — it is not
currently imported there. This site identifies a *target* node, not a stage node — it is the one
place `pipelineStageKind` does not apply (see Verified current state); `isPipelineTarget` is
already correct and already used by this same function's primary path two branches earlier, just
not reused at this one remaining fallback.

**How to reach and test this branch (see Gate item 7).** `resolvePipelineTarget` is called with
`options = {}` from `planPipelineGraph`'s default (no-`output`) path, so the fallback is reached
whenever `derivePipelinePresentations(doc)[0]` is `undefined` for the whole document — the simplest
reliable way to guarantee that is a document whose target-role node has **no incoming `color` edge
at all** (not merely an unrecognized fragment stage — a completely absent one). With nothing wired,
`derivePipelinePresentations` trivially returns `[]`, and the fallback's own first check
(`incoming(doc, display.id, 'color')`) also fails — but *which* error is thrown first depends
entirely on whether `display` was found:
- **Before this fix:** a target-role node under any id other than the literal string
  `'target.display'` is never found by `discoverExecutionRoots(doc).find((node) => node.primitive
  === 'target.display')`, so `resolvePipelineTarget` throws `'Pipeline graph is missing
  target.display execution root'` — the node's existence was never even checked structurally.
- **After this fix:** `discoverExecutionRoots(doc).find(isPipelineTarget)` finds the same node (it
  declares `role: 'pipelineTarget'` regardless of its id), so `resolvePipelineTarget` proceeds one
  step further and throws the *next* check's error instead — `'Pipeline display is missing its
  fragment texture input'`.

The two distinct error messages are the proof — a minimal, deliberately-disconnected document is
enough; no attempt is needed (or possible, see Context) to make the overall `planPipelineGraph()`
call succeed through this branch.

### 7. Test-only fixture primitives — registration discipline

Gate items 4-8 collectively need up to three test-only primitives (a fragment-stage fixture, a
vertex-stage fixture, an alt-id target fixture). `registerPrimitive` has no matching `unregister`
(confirmed by direct read of `packages/graph/src/registry.ts` — `primitives`/`insertionOrder` are
module-level state with no removal API), so:

- Every test-only fixture id must be **unique across the whole workspace test run**, not just
  within one file — prefix with the owning test file's own namespace (e.g.
  `test.f361FragmentStageFixture`, not a generic `test.fragmentStage` some other suite might also
  reach for).
- Register each fixture **once, idempotently**, mirroring the existing `testPrimitive` helper
  pattern already used in `packages/runtime-webgpu/src/typeRefDeviceCompile.test.ts` (`const
  existing = getPrimitive(id); if (existing) return existing; registerPrimitive(input); return
  getPrimitive(id)!;`) — not a bare `registerPrimitive(...)` call inside a `beforeEach`/at the top
  of every `it`, which throws `"Primitive already registered"` on a second run within the same
  process (vitest can re-run files without a fresh module registry in watch mode, and multiple
  `it` blocks in the same file share one module-level `primitives` map).
- Do not register a test-only fixture whose id collides with any real, already-registered
  primitive — confirmed there is no existing `test.*`-prefixed primitive in
  `packages/graph/src/primitives/` today (grep before picking exact ids, to be safe against
  drift by the time this lands).

### 8. Export

`packages/graph/src/index.ts` already re-exports everything from `./pipeline.js` and

surface through those without any new export line needed (verify against the current file; add one
only if the existing wildcard exports don't already cover the new names).

## Gate

1. `PrimitiveMetadata` accepts `pipelineStageKind: 'vertex' | 'fragment'` (type-level; a
   `@ts-expect-error` test assigning an invalid literal fails to compile, matching this project's
   existing exhaustiveness-test convention).
2. `getPrimitive('stage.vertex').metadata?.pipelineStageKind` is `'vertex'`;
   `getPrimitive('stage.fragment').metadata?.pipelineStageKind` is `'fragment'`; both primitives'
   `metadata.role` is still exactly `'pipelineStage'`, unchanged.
3. `isPipelineStage` returns `true` only for a node whose primitive declares the matching
   `pipelineStageKind` — `false` for the other kind, `false` for a primitive with no
   `pipelineStageKind` at all (e.g. `buffer.persist`), `false` for an unregistered primitive id.
4. A test registers a **test-only fragment-stage primitive** under a different id (not
   `stage.fragment`), declaring `pipelineStageKind: 'fragment'` and the same port shape
   `stage.fragment` uses (`varyings`/`color` in, `texture` out — per the port-name constraint noted
   in Verified current state), wires it into an otherwise-normal pipeline document in place of
   `stage.fragment`, and proves `fieldOutputForDisplay`/`presentationForDisplay`/
   `derivePipelinePresentations` recognize it — the direct proof that discovery is genuinely
   role/kind-based now, not still secretly id-based under a different name.
5. The same test-only primitive (or a matching test-only vertex-stage primitive, declaring
   `pipelineStageKind: 'vertex'`) proves `resolveVertexForFragment` recognizes it in place of
   `stage.vertex`.
6. The same substitution proves `planPipelineGraph` succeeds end-to-end with the test-only
   fragment-stage primitive in place of `stage.fragment` (all other pipeline pieces —
   `stage.vertex`, `buffer.persist`, `target.display` — unchanged), returning a
   `PipelineGraphPlan` with `fragmentStageNode` pointing at the substituted node.
7. **New coverage for a previously-untested path** (see Fix step 6's exact reachability note — do
   **not** attempt a document where `planPipelineGraph()` succeeds through this branch; that's
   structurally impossible, not just hard, per Context): a minimal document containing a
   `target.display`-role node under a **different primitive id** (declaring `role: 'pipelineTarget'`,
   `implementation.kind: 'sink'`) with **zero edges anywhere** in the document. Calling
   `planPipelineGraph(doc)` on it throws `'Pipeline display is missing its fragment texture input'`
   (proving `discoverExecutionRoots(doc).find(isPipelineTarget)` found the alt-id node and proceeded
   past the existence check) — **not** `'Pipeline graph is missing target.display execution root'`
   (which is what the old `node.primitive === 'target.display'` check would throw instead, since it
   would never find the node at all). The specific error message is the proof.
8. Every existing pipeline sample's tests (`pipeline.test.ts`, `primitives/pipeline/pipeline.test.ts`,
   `pipelineGraph.test.ts`, and any `graphFrameExecutor.test.ts`/`fullscreenFragment.test.ts`
   coverage exercising `stage.vertex`/`stage.fragment`) pass **unmodified** — zero regression, direct
   proof nothing about `stage.vertex`/`stage.fragment`'s own behavior changed.
9. `check` **and** `test` green for `graph`, `runtime-webgpu`, full workspace (clear every
   `packages/*/dist` first, per `AGENTS.md`'s documented stale-dist issue).

## Out of scope

Registering any real, non-test-only graph-authored kernel-based stage primitive — this milestone's
only new primitive is the test-only fixture used to prove discovery generalizes (Gate items 4-6);
the real one is F3.6.3's job. Generalizing or touching port names (`mesh`/`varyings`/`color`/
`texture`) — noted as a real, separate constraint in Verified current state, not addressed or
relaxed here. `buffer.persist`'s own hardcoded `persist.primitive !== 'buffer.persist'` check
(`pipelineGraph.ts:225`) — a resource/buffer concern, not a stage-discovery one; not part of this
milestone's four sites. The already-role-based geometry-source check
(`findPipelineGeometrySource`/`PIPELINE_GEOMETRY_SOURCE_ROLE`) — already structural, nothing to fix.
Repurposing or changing `metadata.role`/`swapFamily()`'s grouping behavior in any way. Threading
`plan.fragmentStageNode`/`plan.vertexStageNode` through `PipelineGraphExecutor.execute` into real
assembly/execution branching, or the binding-set resource-routing question — that's F3.6.2's job
(sub-problem (E) in the design document), not touched here; this milestone only fixes *discovery*,
not what happens once a stage is discovered. Vertex kernel invocation semantics (F3.6.4, sub-problem
(F)). Any change to `stage.vertex`/`stage.fragment`'s own rendering behavior, `pipelineVertex.ts`,
`fullscreenFragment.ts`, or any existing sample graph. Any change to Foundation 1's, Foundation 2's,
or F3.1-F3.5's landed contracts.

## Handoff

→ `target.display`'s presentation-derivation logic and `planPipelineGraph` both recognize a
vertex/fragment stage by a new, purpose-built `pipelineStageKind` metadata field instead of a
hardcoded primitive-id string — proven not just by type but by a real test-only primitive under a
different id passing every check `stage.vertex`/`stage.fragment` themselves pass.

`target.display`-id-based) and covered by a test for the first time. Every existing sample renders

included here.
