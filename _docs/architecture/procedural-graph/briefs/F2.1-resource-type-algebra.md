# Brief — F2.1: generic resource type algebra

**Type:** type-system extension, first milestone of the Foundation 2 generic-resources sequence ·
**Package:** `@world-lab/graph`, `@world-lab/runtime-webgpu` · **Depends on:** Foundation 1,
complete — specifically F1.5's `TypeRef` (`buffer`/`texture`/`sampler` variants already exist as
type-level stubs) and F1.3's `PrimitiveImplementation` (`{kind: 'resource'; descriptor:
ResourceDescriptor}` already reserved as a placeholder) · **Design authority:**
[foundation-2-generic-resources-plan.md](../foundation-2-generic-resources-plan.md) ·
[elemental-webgpu-architecture-review.md, Foundation 2](../elemental-webgpu-architecture-review.md#roadmap-realignment)
· **Contract author:** Opus (revision 3 — see Revision note) · **Recommended executor:** Cursor or
Codex · **Status:** ready to route once the Foundation 2 plan is approved

## Revision note

**Revision 3** fixed two more confirmed defects in revision 2, again caught before routing:

1. **Gate item 6 (the declared/inferred additive-usage requirement) had no function to satisfy
   it.** `inferBufferUsage(bindings)` alone can't produce a declared-plus-inferred union — it
   never receives the template's declared `usages` as input. Fixed by adding
   `resolveBufferUsage(declared, bindings)` as the actual entry point that combines them;
   `inferBufferUsage` is now explicitly the internal aggregation step it calls, not the whole
   answer.
2. **F2.2 and Foundation 3 formed a circular dependency.** Revision 2 said attaching
   `ResourceBinding` to graph structure was "F2.2's concern once there's a concrete kernel/command
   shape to attach it to" — but Foundation 3 (which defines kernel/command shapes) doesn't start
   until Foundation 2 finishes. Fixed by making `ResourceBinding[]` attach to `Pass`
   (`frameGraph/types.ts`'s existing, already-kernel-independent planner concept) instead of to
   raw kernel/command graph nodes — F2.2 never waits on Foundation 3; Foundation 3 later
   translates kernel/command nodes into this already-defined planner-level shape.

**Revision 2** fixed two confirmed defects in revision 1, caught before it was ever routed:

1. **Resource identity was at the wrong ownership level.** Revision 1 put `id: string` directly on
   `ResourceDescriptor` and attached that descriptor to `PrimitiveImplementation`. But
   `PrimitiveImplementation` lives on the *registered primitive* (`NodePrimitive.implementation`,
   `packages/graph/src/primitive.ts:81`) — one value per primitive **type**, shared by every graph
   node that uses it. An `id` baked in there would be the same literal string for every node
   instance of that primitive, which is wrong: two different resource nodes in the same graph need
   two different resource identities. This is exactly the same class of mistake F1.3's own review
   caught for `SinkDefinition` ("static, per-primitive metadata — it cannot hold per-node data,"
   `F1.3-primitive-implementation-union.md`) — a lesson this brief should have generalized and
   didn't. Fixed below: the primitive-level value is an **ID-less template**; instance identity
   (`node.id`) is attached later, by whichever milestone actually walks a real `GraphDocument`.
2. **Usage inference had no authoritative signal for access direction.** Revision 1's
   `inferBufferUsage(doc, resourceNodeId)` promised to derive read/write access from "how the
   resource is wired," but graph edges alone don't carry that signal at the port/binding level — a
   buffer can be read by one command and written by another, and `TypeRef.buffer.access` is a
   single aggregate value, not a per-binding one. Fixed below: access becomes an explicit,
   first-class per-binding value (`ResourceAccess`), and the inference function becomes a pure
   aggregator over explicit bindings, not a function that invents access direction from topology.

## Context

First of Foundation 2's sequence (F2.1 → F2.2 → F2.3 → F2.4 → proof). **This milestone is pure
types and inference — no GPU allocation, no runtime execution.** It gives `ResourceDescriptor`
(currently `{ placeholder?: never }`, `packages/graph/src/implementation.ts:35-37`) its real
shape, and extends F1.5's already-introduced `TypeRef` buffer/texture/sampler variants rather than
inventing a parallel type system.

**Do not reinvent what F1.5 already built.** `TypeRef` already has:
```ts
// packages/graph/src/types.ts, current, unchanged since F1.5
export type BufferAccess = 'read' | 'read-write';
export type BufferUsage = string; // placeholder — this milestone's job
export type TextureDimension = '1d' | '2d' | '3d' | 'cube';
export type SampleType = 'float' | 'depth' | 'sint' | 'uint';
export type StorageTextureAccess = 'read' | 'write' | 'read-write';

type TypeRef =
  | ... // scalar/vector/matrix/array/struct/legacy, all real since F1.5
  | { kind: 'buffer'; element: TypeRef; access: BufferAccess; usages: BufferUsage[] }
  | { kind: 'texture'; dimension: TextureDimension; sample: SampleType;
      format?: string; access?: StorageTextureAccess }
  | { kind: 'sampler'; filtering: boolean; comparison: boolean };
```
This milestone's job is: (1) replace the `BufferUsage = string` placeholder with a real WebGPU
usage-flag union, (2) define resource **templates** (id-less) as a lifetime-qualified wrapper
around these existing `TypeRef` buffer/texture/sampler shapes, (3) define per-binding
`ResourceAccess` as an explicit, first-class value, and (4) write a usage-inference function that
*aggregates* explicit per-binding accesses into allocation-level usage flags — it does not invent
access direction from graph topology, because there is no signal in raw edges to invent it from.

**Do not confuse this with the pre-existing `ResourceDependency`/`ResourceDataType`**
(`packages/graph/src/types.ts:109-118`) — that's the M8-era external image/mesh/audio resource
declaration mechanism, a document-level input concept, unrelated to GPU buffer/texture allocation.

## Verified current state

- `ResourceDescriptor { placeholder?: never }` and `GpuCommandKind = string`
  (`packages/graph/src/implementation.ts:35-39`) are F1.3's deliberate stubs, unchanged.
- `NodePrimitive.implementation: PrimitiveImplementation` (`packages/graph/src/primitive.ts:81`)
  is registered once per primitive **id**, via `registerPrimitive`/`replacePrimitive` — confirmed
  by re-reading the registration path — never once per graph node. Any per-node data (an id, a
  wired value, anything instance-specific) cannot live here; it has to live in `node.params` or be
  computed by something that walks the actual `GraphDocument`, exactly as F1.3 already established
  for `SinkDefinition`'s per-node payload data.
- `frameGraph/types.ts`'s `RenderTarget { id, format: GPUTextureFormat, size: TargetSize,
  persistent?: boolean }` (`packages/runtime-webgpu/src/frameGraph/types.ts`) is the texture-only
  precedent this milestone's lifetime model generalizes — note `RenderTarget.id` is already an
  **instance**-level identifier (one per pass-graph target, not one per primitive type), which is
  the correct level for identity to live at; this brief's template/instance split keeps that
  precedent, it doesn't change it.
- `collectFeedbackTargets` (`packages/runtime-webgpu/src/frameGraph/order.ts:139-161`) already
  treats `target.persistent` and `read.previousFrame` as the two signals for "this needs
  ping-pong" — confirmed generic (string IDs and booleans only). F2.1's lifetime model must supply
  values these two signals can continue to key off, generalized to buffers too.
- No usage-inference function exists anywhere today — every current resource-adjacent primitive
  (`buffer.persist`, the pipeline `geometry`/`texture`/`storageBuffer` `DataType` values) declares
  its role by primitive identity, not by inferring usage from graph structure. There is also no
  existing per-binding access concept anywhere in the codebase to build on — this is genuinely new.

## Fix

1. Replace the `BufferUsage = string` placeholder with a real, closed union matching WebGPU's
   actual buffer usage flags:
   ```ts
   export type BufferUsageFlag =
     | 'vertex' | 'index' | 'uniform' | 'storage'
     | 'copy-src' | 'copy-dst' | 'indirect' | 'query-resolve';
   export type BufferUsage = BufferUsageFlag; // keep the exported name stable for existing TypeRef callers
   ```
2. Define resource lifetime as an explicit, shared concept (not per-resource-kind duplicated):
   ```ts
   export type ResourceLifetime =
     | { kind: 'transient' } // scoped to one frame, freed/reused after
     | { kind: 'persistent' } // survives across frames, single physical allocation
     | { kind: 'history'; slots: 2 }; // survives across frames, ping-ponged (current/previous);
                                      // `slots` fixed at 2 for now — N-deep history is out of scope
   ```
   This generalizes `RenderTarget.persistent?: boolean` (`true` → `persistent`, `undefined`/`false`
   → `transient`) and adds the `history` case explicitly. F2.2 wires `ResourceLifetime.history` and
   `collectFeedbackTargets`'s existing detection together — this milestone only defines the type.
3. **Define an id-less resource template, not an id-bearing descriptor.** This is what a
   primitive's `PrimitiveImplementation` actually holds — it describes the *shape* every node of
   this primitive will have, not any one node's identity:
   ```ts
   export interface ResourceTemplate {
     shape: Extract<TypeRef, { kind: 'buffer' | 'texture' | 'sampler' }>;
     lifetime: ResourceLifetime;
   }
   ```
   Update `PrimitiveImplementation`'s `{ kind: 'resource'; descriptor: ResourceDescriptor }`
   variant (`packages/graph/src/implementation.ts`) to `{ kind: 'resource'; template:
   ResourceTemplate }` — renaming the field from `descriptor` to `template` is deliberate, so the
   id-less nature is visible at the call site, not just in a type comment.
   **Materializing a per-node instance is explicitly F2.2's job, not this milestone's** — F2.2
   (which walks a real `GraphDocument`) constructs `{ id: node.id, ...template }` for each node
   using a `kind: 'resource'` primitive. This brief only defines the shape that materialization
   produces, so F2.2 has a concrete type to target:
   ```ts
   export interface ResourceInstance extends ResourceTemplate {
     id: string; // node.id, assigned by F2.2 when walking a real document — never by this milestone
   }
   ```
4. **Define per-binding access as an explicit, first-class value — not inferred from topology:**
   ```ts
   export type ResourceAccess = 'read' | 'write' | 'read-write';

   export interface ResourceBinding {
     resourceId: string; // the ResourceInstance.id this binding refers to
     access: ResourceAccess;
   }
   ```
   `ResourceBinding` is deliberately minimal and doesn't yet specify *how* a binding gets attached
   to anything — this milestone only fixes the contract shape so usage resolution (step 6) has
   something real to consume. **Attachment is planner-level, not kernel-level, and is entirely
   F2.2's job:** `ResourceBinding[]` attaches to `Pass` (`frameGraph/types.ts`'s existing,
   already-generic concept — a `Pass` already exists independent of any kernel/command graph node
   shape), not to raw graph kernel/command nodes. This deliberately avoids a circular dependency:
   F2.2 must not wait on Foundation 3's kernel shapes to exist, since Foundation 3 itself doesn't
   start until Foundation 2 is done. Foundation 3, once it lands, translates real kernel/command
   graph nodes into `Pass`-level `ResourceBinding[]` entries — it produces values in this shape,
   it does not define the shape.
5. **Usage inference is a pure aggregator over explicit bindings, not a topology-inference
   function:**
   ```ts
   export function inferBufferUsage(bindings: readonly ResourceBinding[]): BufferUsageFlag[];
   ```
   Note the signature change from revision 1's `(doc, resourceNodeId)` — this function no longer
   tries to derive access from a `GraphDocument` at all. Given a list of bindings that already know
   their own access direction, it aggregates: any `'write'`/`'read-write'` binding contributes
   `'storage'` (write-capable); any `'read'` binding also contributes `'storage'` (read-capable,
   same flag — WebGPU's `STORAGE` usage flag doesn't split read/write at the allocation level, only
   at the WGSL binding-declaration level, which is a `StorageTextureAccess`/binding-layout concern,
   not an allocation-usage one). Do not attempt full usage-flag inference for every WebGPU binding
   scenario (e.g. vertex/index buffer usage inference from a kernel's actual binding layout is
   Foundation 3's concern, once kernels exist to bind against) — keep this minimal and correct for
   the buffer-feedback and storage-read/write cases Foundation 2's own proof needs.
6. **Clarify `TypeRef.buffer.usages`/`.access` vs. inferred usage with an actual function, not just
   a description — `inferBufferUsage` alone cannot produce this, since it never receives the
   template's declared usages as input.** Add the function that combines them:
   ```ts
   export function resolveBufferUsage(
     declared: readonly BufferUsageFlag[],
     bindings: readonly ResourceBinding[]
   ): BufferUsageFlag[]; // union of `declared` and `inferBufferUsage(bindings)`, deduplicated
   ```
   `declared` (`TypeRef.buffer.usages` on the template — F1.5's existing field, now real thanks to
   step 1) is the primitive **author's declared minimum** — e.g. a primitive whose buffer shape
   intrinsically requires `'storage'` regardless of how it's wired declares that directly.
   `resolveBufferUsage` is the actual entry point callers use to get a materialized resource's full
   allocation usage set; `inferBufferUsage` (step 5) is the internal aggregation step it calls,
   exported separately because F2.1's own gate tests it in isolation. The relationship is additive,
   never contradictory, matching the same pattern this project already used for `dataType`/`type`
   (F1.5) and `wgsl`/`implementation` (F1.3): one is a declared floor, the other is the canonical,
   always-complete result — declaring a `usages` value never *removes* a flag inference would
   otherwise add.
7. Do **not** implement any GPU allocation, bind-group derivation, or execution here — this
   milestone is types and inference only. `ResourceTemplate`/`ResourceInstance` values are defined
   and validated, never realized. Realization is F2.2 (materialization/planning) and F2.3
   (allocation)'s job.

## Gate

1. A test proves `ResourceTemplate.shape` only accepts `TypeRef`'s `buffer`/`texture`/`sampler`
   kinds — a `scalar`/`vector`/etc. `TypeRef` must not type-check as a valid `shape` (`Extract<...>`
   should make this a compile-time guarantee; write a test that would fail to compile if this
   narrowing were removed, matching this project's exhaustiveness-test convention).
2. A test proves `ResourceTemplate` has no `id` field at the type level (e.g. assigning an object
   literal with an extra `id` property to a `ResourceTemplate`-typed variable should fail to
   compile under `excessPropertyChecks`), and that `ResourceInstance` requires one — the concrete
   proof the ownership-level fix actually holds, not just documentation saying it should.
3. A test proves two different `node.id`s produce two different `ResourceInstance.id`s for the
   same primitive template — the direct regression case for finding 1.
4. A test proves `ResourceLifetime`'s three variants round-trip through serialization unchanged
   (no information loss for `history`'s `slots` field specifically).
5. A test proves `inferBufferUsage` aggregates a `'write'` binding and a separate `'read'` binding
   (two distinct `ResourceBinding` entries referring to the same `resourceId`, different `access`)
   into `['storage']` — the concrete case Foundation 2's own buffer-feedback proof needs, and the
   direct regression case for finding 2 (no attempt to derive this from a `GraphDocument`).
6. A test proves `resolveBufferUsage(['uniform'], bindings)`, where `bindings` includes a `'write'`
   entry, produces a usage set containing **both** `'uniform'` and `'storage'` — proving the
   declared/inferred relationship is additive via the actual function that combines them, not just
   asserted in prose with no corresponding API.
7. A test proves `inferBufferUsage([])` (no bindings at all) returns `[]`, not a thrown error —
   deliberately not attempting exhaustive inference for a resource nothing references yet.
8. `check` **and** `test` green for `graph`, `runtime-webgpu`, full workspace (clear every
   `packages/*/dist` first, per `AGENTS.md`'s documented stale-dist issue).

## Out of scope

GPU resource allocation (F2.3). Bind-group layout derivation (F2.3). Materializing
`ResourceInstance` from a real `GraphDocument` (F2.2). Attaching `ResourceBinding[]` to `Pass`
(F2.2 — planner-level, independent of any kernel/command graph node shape; Foundation 3 later
translates kernel/command nodes into this shape, it does not define it, so F2.2 never waits on
Foundation 3). Wiring `ResourceLifetime.history` into `collectFeedbackTargets`'s actual detection
logic (F2.2). Full WebGPU usage-flag inference for every binding scenario (partially Foundation
3's concern, once kernels exist). N-deep (>2 slot) history. Any change to Foundation 1's
milestones or to this document's own Foundation 2 sequencing beyond this one milestone.

## Handoff

→ Resource identity lives at the correct level (id-less `ResourceTemplate` on the primitive,
`ResourceInstance` materialized per node — not yet wired to real documents, that's F2.2), and
usage resolution has a real, callable API (`resolveBufferUsage` combining a template's declared
floor with `inferBufferUsage`'s aggregation over explicit `ResourceAccess` bindings) rather than a
gate requirement with no function to satisfy it. `ResourceBinding` attaches at the `Pass`
(planner) level, not the kernel/command level, so F2.2 never depends on Foundation 3 existing
first. Once this lands: F2.2 (resource dependency planner — generalizing `frameGraph/types.ts` to
this milestone's resource shapes, materializing `ResourceInstance`s from real documents, attaching
`ResourceBinding[]` to `Pass`, updating `resolveTargetSizes` for resource-kind-aware sizing) is
next — request its own contract at that point.
