## Context

Second of five milestones freezing this project's "elemental" graph contracts (F1.1 → **F1.2** →
F1.5 → F1.3 → F1.4a). **This milestone is now frozen and approved for implementation** — do not
revisit F1.1, F1.3, F1.4a, or F1.5's design here; each gets its own contract when it becomes next.

`CoordinateSpace` is a closed union of 8 values, 100% planet/terrain-specific, hardcoded into the
generic `packages/graph` core. Only 30 port declarations across the whole codebase use a
non-default space (21 terrain, 6 surface, 2 generic). This milestone opens it to an arbitrary
string (`SpaceId`) and adds a separate, pluralizable `semantics: SemanticTag[]` field for
orthogonal properties (units, color space) that can co-occur with a space on the same port — the
parent review's own model requires multiple simultaneous tags (`"space:world"`, `"unit:m"`,
`"color:linear-srgb"`), which a single collapsed field cannot express.

**Additional research beyond the parent plan's own "verified current state," confirmed by direct
grep just before writing this brief** (the parent plan named 4 type locations and a handful of
consuming files; there are more):
- `packages/compiler/src/primitiveLoader.ts:57-65` has a **hardcoded `COORDINATE_SPACES` Set**
  (the same 8 values), used to **validate** the `space:` field when parsing a WGSL module's
  self-describing frontmatter YAML (`parsePortMetadata`, ~line 437-450) — throwing `Invalid
  coordinate space: ${value}` for anything not in the set. This is a **third closed-set enforcement
  point**, distinct from the `CoordinateSpace` type itself and `validate.ts`'s edge-compatibility
  check — a primitive authored via WGSL frontmatter declaring a novel `space:` value would be
  rejected here, at module-parse time, before it ever reaches `packages/graph`'s registration. This
  must be opened too, not just the type.
- `packages/graph-editor/src/types.ts:9-15`'s `PortBindingState` interface has its own
  `space?: CoordinateSpace` field — a **fifth** type location carrying this field, beyond the four
  the parent plan named (`Port`, `PortSpec`, `GroupInputMapping`, `GroupOutputMapping`).
- The exact node-port synchronization point is `instantiatePorts`
  (`packages/graph-editor/src/nodePortUtils.ts:3-12`), called by `instantiateNodeInputs`/
  `instantiateNodeOutputs` — it currently does `space: spec.space ?? 'none'` and has no `semantics`
  handling at all (the field doesn't exist yet). This is the one function that turns a `PortSpec`
  template into a real `Port` instance; get this one right and node creation is correct everywhere
  it's called from.
- `packages/graph/src/contract.ts:12-21`'s `formatPorts` builds the swap-family mechanical contract
  string as `` `${dataType}${space ? '@'+space : ''}` `` — confirms the "keep space in the contract
  string, preserve current swap behavior exactly" decision the parent plan already made; no `role`
  metadata bypass needed, just don't touch this function's `space` handling.
- `packages/compiler/src/groupCodegen.ts:457-465` also renders `space:` into generated group-module
  frontmatter text (`\n    space: ${i.space}` / `${o.space}`) — a compiler-side counterpart to
  `primitiveSources.ts`'s editor-side rendering; both need the same treatment (render `semantics`
  alongside `space`, and both already treat the value as an opaque string, so no logic change
  needed there beyond the field itself existing).

## Fix

1. In `packages/graph/src/types.ts` (currently defines `CoordinateSpace` at ~line 22-31): replace
   `CoordinateSpace` with:
   ```ts
   export type SpaceId = string;
   export type SemanticTag = string;
   ```
   Keep `CoordinateSpace` as a deprecated type alias (`export type CoordinateSpace = SpaceId;`) if
   anything still imports the name directly — check `graph-editor/src/types.ts`'s import and update
   it to `SpaceId`, don't leave two names for the same concept.
2. Add both fields to all **five** locations (not four): `Port` (`types.ts:33-41`), `PortSpec`
   (`primitive.ts:34-41`), `GroupInputMapping`, `GroupOutputMapping` (`types.ts:96-114`), and
   `PortBindingState` (`graph-editor/src/types.ts:9-15`):
   ```ts
   space?: SpaceId;
   semantics?: SemanticTag[];
   ```
3. In `packages/graph/src/validate.ts` (~line 98-102): keep the exact-match-or-`'none'` rule
   unchanged — it already operates on `space` as a plain string comparison, so no logic change
   needed, just confirm the type change doesn't require a cast anywhere.
4. Move the 8 existing planet-specific values (`'world_dir'`, `'body_dir'`, `'world_pos'`,
   `'body_pos'`, `'ideal_fragment_body_dir'`, `'height_meters'`, `'world_radius_meters'`,
   `'scale_ctx'`) out of `packages/graph`'s core into a library-registered constants module (e.g.
   `packages/graph/src/primitives/terrain/spaces.ts`, imported by the terrain primitives that use
   them) — `'none'` stays a core-recognized default, it's not planet-specific.
5. **Open `primitiveLoader.ts`'s `COORDINATE_SPACES` validation** (~line 57-65, 437-450) — this is
   the concrete fix that makes the type change actually load-bearing, not just cosmetic. Either
   remove the closed-set check entirely (accept any non-empty string for `space:` in WGSL
   frontmatter, matching the now-open `SpaceId`), or replace it with a call into whatever registers
   library space constants (step 4) if a "known spaces" concept is still useful for
   typo-catching — pick whichever is simpler; do not leave a hardcoded 8-value Set enforcing
   closure after the type itself is open.
6. In `nodePortUtils.ts`'s `instantiatePorts` (~line 3-12): copy `semantics` across alongside the
   existing `space` copy:
   ```ts
   space: spec.space ?? 'none',
   ...(spec.semantics !== undefined ? { semantics: dedupeCanonicalSemantics(spec.semantics) } : {}),
   ```
   Add a small `dedupeCanonicalSemantics(tags: SemanticTag[]): SemanticTag[]` helper (dedupe +
   lexicographic sort) in `packages/graph` (co-locate with `SemanticTag`'s definition or in
   `dataType.ts`) — call it from every write path that sets `semantics`: this function, node
   creation/resync in `irAdapter.ts` (`syncNodePortLists`, `createNode`), and deserialization. This
   is a general invariant of the field, not a one-time migration cleanup.
7. Update `packages/graph/src/contract.ts` — **no change** to `formatPorts`'s `space` handling
   (confirmed correct as-is); `semantics` does not participate in the swap-family contract string
   at all.
8. Update `packages/graph-editor/src/primitiveSources.ts` (~line 54-62) and
   `packages/compiler/src/groupCodegen.ts` (~line 457-465) to also render `semantics` into
   frontmatter YAML alongside `space`, matching each file's existing rendering style for `space`
   (both already treat it as an opaque string/array; this is display/documentation only, not
   validation logic).
9. Add a new test proving a non-terrain `space` value (e.g. a hypothetical audio-domain frame like
   `'stereo_field'`) registers and validates correctly from outside `packages/graph`'s core —
   the actual proof of openness.

## Gate

1. All 30 existing `space` port declarations produce identical validation results on existing
   graphs — run the full `graph`/`graph-editor` test suites before and after, diff for any changed
   outcome (parity, not just "doesn't crash").
2. A new test registers a non-terrain `space` value from outside `packages/graph`'s core (per fix
   step 9) and proves `validate.ts` accepts/rejects edges correctly without any core code knowing
   what the value means.
3. A new test attaches `semantics: ['color:linear-srgb']` to a port alongside a `space` value on
   the same port, proving the two are independent and co-occur cleanly.
4. **Round-trip gate:** a port's `space` and `semantics` survive `instantiatePorts` (`PortSpec` →
   `Port`), a full `serializeGraph`/`deserializeGraph` cycle, and `syncNodePortLists`'s
   node-repair/resync path (irAdapter.ts) — proving propagation through every location identified
   above, not just the type definitions in isolation.
5. A test sets `semantics` with duplicate and out-of-order entries (e.g. `['unit:m',
   'color:linear-srgb', 'unit:m']`) through each write path (`instantiatePorts`,
   `syncNodePortLists`/`createNode`, deserialization) and asserts the stored/serialized result is
   always deduplicated and canonically ordered.
6. A test proves `primitiveLoader.ts` no longer rejects a novel `space:` value in WGSL frontmatter
   with `Invalid coordinate space` — the concrete proof the third closed-set gate (beyond the type
   and `validate.ts`) is actually open, not just the type definition.
7. A test proves `contract.ts`'s swap-family contract string is byte-identical before and after for
   every existing primitive (confirms the deliberate "no change" to `formatPorts`).
8. `check` **and** `test` green for `graph`, `compiler`, `graph-editor`, and the full workspace
   (clear every `packages/*/dist` first, per `AGENTS.md`'s documented stale-dist issue).

## Out of scope

Validating `semantics` compatibility on edges (informational/display-only for now). Building a
general conversion graph between arbitrary spaces (only one conversion primitive,
`terrain.worldNormal`, exists today — speculative until a second domain needs one). Any change to
F1.1, F1.3, F1.5, or F1.4a's design.
