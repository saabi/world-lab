## Context

F3.5 is the proof milestone for the F3.1→F3.4 compute-kernel path. It does **not** design or start
F3.6's true pipeline-kernel work. Real graph-authored vertex displacement, dynamic per-document
fragment bindings, typed-varying dataflow through the existing `stage.vertex`/`stage.fragment`
render path, storage textures, command graph nodes, or a general resource-node authoring model are
all out of scope here.

The original sequencing plan described F3.5 as "a bundled, pickable, hardcoded sample proving a
real compute-dispatch kernel through F3.4's wiring." F3.4's final contract folded much of that
sample/UI work into F3.4 itself. That does not make this milestone useless; it changes its job:
F3.5 should **close the proof**, make it resistant to regression, and complete the visual gate that
F3.4 could not fully automate in this environment.

The proof target is deliberately narrow:

- A user can open a bundled sample from the graph editor's sample picker.
- The sample contains a real graph node registered as a `{ kind: 'kernel'; stage: 'compute' }`
  primitive with a fixed `KernelBindingTemplate[]`.
- `GraphFrameExecutor.execute` discovers that document shape, calls the dedicated compute-buffer
  executor, and returns a `Float32Array` result under `computeBuffers`.
- The shared editor preview loop does not early-exit for this buffer-only graph.
- The preview pane enumerates the result as a selectable buffer, routes it to a buffer renderer, and
  displays changing values without reusing the image/CPU scalar preview paths incorrectly.

If the final F3.4 revision already satisfies a step below, F3.5's implementation should preserve it
and add the missing proof tests/visual evidence rather than rewriting the same feature. If the final
F3.4 revision has intentionally backed out some editor/sample hooks, F3.5 is where they are restored
under this proof contract.

## Verified Current State

- `packages/runtime-webgpu/src/consumers/computeBufferTarget.ts` exists in the F3.4 implementation
  and owns the first purpose-built compute proof executor. It builds a one-buffer `PassGraph`, uses
  `ResourceRealizer` for persistent allocation, seeds `[1..N]` once, dispatches the generic
  `executeComputeKernel` path, and reads back a `Float32Array`.
- `ComputeBufferExecutor` is intentionally stateful even though F3.3's `executeComputeKernel` is a
  plain function. The proof depends on persistent buffer contents across frames: first dispatch
  doubles the seed, second dispatch doubles the first result.
- `GraphFrameExecuteResult.targets` is `Record<string, Uint8Array>` and is correct for image-like
  outputs only. Compute-buffer results must stay in a separate `computeBuffers?: Record<string,
  Float32Array>` field. Do not mix `Float32Array` values into `targets`.
- `EffectPreviewPanel` consumes `snapshot.targets[targetId]` as RGBA8 pixels. `CpuPreviewPanel`
  evaluates a scalar/vector `PortRef` through the CPU runtime. Neither is an acceptable renderer for
  raw storage-buffer readback.
- `PreviewFamily`/`PreviewRenderer` and `layoutStorage.ts` parser arrays are closed sets. Adding or
  retaining a `'buffer'` family/renderer requires updating all switches and parser arrays, with tests
  proving persisted selections survive reload.
- A current local dirty tree may be revising F3.4's editor/sample hooks. The executor of F3.5 must
  inspect the final accepted F3.4 tree before implementing and must not blindly assume that
  `foundation-compute-buffer`, `DataBufferPreviewPanel`, `deriveComputeBufferTarget` exports, or the
  `'buffer'` preview family are all still present. The gate below states the required final form.

## Fix Steps

1. **Confirm or restore the graph-level compute target surface.**
   - `target.computeBuffer` must be registered as a `{ kind: 'kernel'; stage: 'compute' }`
     primitive with one fixed read-write buffer binding named `values`.
   - `deriveComputeBufferTarget(doc)` must return at most one descriptor and must reject invalid
     `elementCount` values clearly.
   - `@world-lab/graph` must export the descriptor/discovery APIs used by runtime/editor packages.

2. **Add an end-to-end `GraphFrameExecutor` proof on a real device.**
   - Construct a minimal `GraphDocument` containing a `target.computeBuffer` node with
     `elementCount: 20`.
   - Call `new GraphFrameExecutor().execute(...)` twice on the same device.
   - Assert `result.targets` remains empty for the compute-only document.
   - Assert `result.computeBuffers?.[nodeId]` is `[2,4,...,40]` on the first frame and
     `[4,8,...,80]` on the second frame.
   - Use the package's existing `skipIf`/WebGPU test pattern, but make the test visibly fail if it
     runs and the result is not routed through `GraphFrameExecutor`.

3. **Confirm or restore the bundled sample.**
   - `computeBufferDoublingGraph()` must exist in `graphBuilders.ts`.
   - `GRAPH_SAMPLES` must include a pickable sample with id `foundation-compute-buffer` and label
     `Foundation — Compute buffer`.
   - The sample must validate with `validateGraphFull`.
   - The sample must enumerate exactly one preview buffer for the compute result unless future
     preview infrastructure intentionally adds more.

4. **Confirm or restore preview-loop dispatch for buffer-only compute documents.**
   - `createPreviewFrameLoop` must treat `deriveComputeBufferTarget(graph)` as a valid reason to
     execute, just as it already does for pipeline passes and `target.bufferFeedback`.
   - `PreviewFrameSnapshot` must carry `computeBuffers?: Readonly<Record<string, Float32Array>>`.
   - Successful frame emission must copy `result.computeBuffers` through to the snapshot.

5. **Confirm or restore buffer preview enumeration and renderer routing.**
   - `PreviewFamily` must include `'buffer'`.
   - `enumeratePreviewBuffers(computeBufferDoublingGraph())` must return a buffer keyed by the
     compute target node id, with `dataType: 'storageBuffer'`, `family: 'buffer'`, `inferred: true`,
     and `source: { sinkNode: nodeId }`.
   - `rendererForPreviewFamily('buffer')` must return `'buffer'`.
   - `resolvePreviewRenderer` must preserve explicit `rendererOverride: 'buffer'`.
   - `layoutStorage.ts` must parse and round-trip both `familyOverride: 'buffer'` and
     `rendererOverride: 'buffer'`.

6. **Confirm or restore visible buffer rendering.**
   - `PreviewZone.svelte` must route `previewRenderer === 'buffer'` to `DataBufferPreviewPanel`.
   - `DataBufferPreviewPanel` must subscribe to the shared `PreviewFrameLoop`, read
     `snapshot.computeBuffers?.[targetId]`, and render at least: element count, last value, and the
     first several values.
   - The panel must not depend on a non-null `PortRef`; compute targets are sink-backed, not
     field-output-backed.

7. **Add a browser visual gate artifact.**
   - Start/use the shared `apps/webgputoy` dev server on `http://localhost:5173`.
   - Load the `Foundation — Compute buffer` sample through the sample picker.
   - Verify the buffer preview tab/panel is visible and values change across frames (for the
     20-element sample, the first frame should show doubled seed values, and later frames should
     continue changing).
   - Capture a screenshot or paste a concise visual-gate note into the task board/PR context. Do
     not commit local screenshot files.

8. **Close F3.5 without broadening scope.**
   - Do not add a second compute primitive.
   - Do not generalize resource nodes.
   - Do not change the image pipeline.
   - Do not start F3.6's dynamic-binding/pipeline-kernel design.

## Gate

1. `target.computeBuffer` remains registered as a compute kernel with fixed binding
   `{ name:'values', binding:0, resourceKind:'buffer', access:'read-write', stages:['compute'] }`.
2. `deriveComputeBufferTarget` returns the descriptor for a single node, returns `null` without one,
   rejects multiple nodes, and rejects missing/zero/negative/fractional `elementCount`.
3. `ComputeBufferExecutor` still proves persistence: two consecutive calls on the same device and
   size produce second-frame values double the first-frame values.
4. `GraphFrameExecutor` real-device test proves a compute-only document returns `computeBuffers`
   through the top-level executor, not only through `ComputeBufferExecutor`.
5. `GraphFrameExecutor` compute-only result does not populate `targets` with a `Float32Array`.
6. `GraphFrameExecutor.dispose()` cascades into `ComputeBufferExecutor.dispose()`.
7. `computeBufferDoublingGraph()` validates with `validateGraphFull`.
8. `GRAPH_SAMPLES` includes `foundation-compute-buffer`.
9. `enumeratePreviewBuffers(computeBufferDoublingGraph())` returns the selectable buffer-family
   preview entry.
10. `inferDefaultPreviewBuffer(computeBufferDoublingGraph())` selects the compute-buffer preview.
11. `resolvePreviewRenderer` maps the compute-buffer preview to the `'buffer'` renderer.
12. `createPreviewFrameLoop` dispatches a compute-buffer-only graph instead of emitting
    `"No pipeline preview targets"`.
13. `PreviewFrameSnapshot` copies `computeBuffers` through from executor result to subscribers.
14. `DataBufferPreviewPanel` renders a summary from an emitted `Float32Array` snapshot.
15. `layoutStorage` round-trips saved pane selections using `familyOverride:'buffer'` and
    `rendererOverride:'buffer'`.
16. Existing ShaderToy image samples, cross-pass texture read, buffer feedback, and mesh samples keep
    their existing preview-family/renderer behavior unchanged.
17. Manual browser visual gate: bundled sample is pickable in `apps/webgputoy`, dispatches, and the
    buffer panel visibly updates.
18. Package gates pass:
    - `npm run check --workspace @world-lab/graph` if graph files are touched
    - `npm test --workspace @world-lab/graph` if graph files are touched
    - `npm run check --workspace @world-lab/runtime-webgpu`
    - `npm test --workspace @world-lab/runtime-webgpu`
    - `npm run check --workspace @world-lab/graph-editor`
    - `npm test --workspace @world-lab/graph-editor`
    - clear `packages/*/dist`, then full `npm run check`
    - full `npm test`

## Out of Scope

- Any change to `stage.vertex` or `stage.fragment`.
- Any change to `pipelineVertex.ts` or `fullscreenFragment.ts`.
- A generalized graph-authored compute-node system with arbitrary upstream resources.
- Resource-authoring primitives beyond the fixed proof target.
- Storage textures.
- Command graph nodes or first-class dispatch commands.
- Vertex-displacement proof samples. Those belong to F3.6 or a later pipeline-kernel milestone.

## Handoff

F3.5 should end with a visible, bundled, hard-to-regress compute proof and no new architecture. Once
it lands, Foundation 3's static compute-kernel path is closed. The next substantial design question
is F3.6: true pipeline kernels with dynamic document-derived bindings and typed varying dataflow
through render pipelines.
