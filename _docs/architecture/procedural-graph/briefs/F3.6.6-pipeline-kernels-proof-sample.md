## Context

The last milestone in the sequence, and the only one that touches the editor. F3.6.1-5 each proved
one real piece in isolation — stage discovery, execution branching, a real fragment kernel, the
vertex invocation model, a real paired varying — but **nothing today can render a
`stage.vertexKernel`+`stage.fragmentKernel` pipeline through the actual execution path a real
document uses.** Confirmed by direct investigation, not assumed: `PipelineGraphExecutor.execute`
already branches on the *fragment* stage's primitive kind (F3.6.2's work), but the *vertex* side has
no equivalent branch at all — `resolveVertexAssembly`/`assemblePipelineVertexWgsl` (the legacy,
hand-written vertex path) is called unconditionally by both `executeFullscreenFragment` and
`executeKernelFragment`, regardless of what vertex-stage primitive a document actually wires in.
F3.6.4's/F3.6.5's own real-device proofs deliberately stayed standalone (calling
`assembleVertexKernelPositionModuleAsync` directly, not through `PipelineGraphExecutor`) for exactly
this reason — the wiring didn't exist yet. This milestone builds it, then proves it the same way
every prior proof-milestone in this codebase has: a bundled, pickable sample, checked visually in a
real browser.

**Confirmed, not assumed, by direct investigation before drafting this contract:**

- `GraphFrameExecutor` and `planIndependentGraphFramePasses` never inspect vertex-stage kind at all
  — `derivePipelinePresentations` only walks `display → fragment(color) → field` edges
  (`packages/graph/src/pipeline.ts:30-40,103-111`), and `isPipelineStage(node, 'vertex')` already
  returns `true` for `stage.vertexKernel` today (it only checks `metadata.pipelineStageKind`, landed
  since F3.6.1). **Fixing `PipelineGraphExecutor.execute`'s vertex branch alone is sufficient** —
  zero changes needed to `graphFrameExecutor.ts`, `graphFramePlan.ts`, `previewFrameLoop.ts`, or any
  `GraphEditor.svelte` code. The full call chain (`GraphEditor.svelte` → `createPreviewFrameLoop` →
  `GraphFrameExecutor.execute` → `PipelineGraphExecutor.execute` per pass) already reaches this
  function unconditionally; picking a bundled sample that uses `stage.vertexKernel` will exercise the
  fixed branch with no other editor-side change.
- Bundled samples are plain TypeScript object literals, not JSON — a builder function in
  `packages/graph-editor/src/graphBuilders.ts` returning a `GraphDocument`, registered as
  `{id, label, build}` in the flat `GRAPH_SAMPLES` array (`packages/graph-editor/src/samples.ts:24-75`).
  `samples.test.ts` hardcodes `GRAPH_SAMPLES.length` and requires every sample to produce a
  `GraphArtifact` passing `validateGraphFull` (`samples.test.ts:19,203-208`) — both need updating for
  a new entry.
- The edge-discovery pattern for a new stage input is the existing, already-used-four-times-in-this-file
  local helper `incoming(doc, node, port)` (`pipelineGraph.ts:76-78`) — the same one
  `resolveVertexForFragment`/`findPipelineGeometrySource`/`planPipelineGraph` already use for
  `varyings`/`in`/`mesh`. No new discovery primitive needed for `stage.vertexKernel`'s `position`/`uv`
  inputs.
- `assembleVertexKernelPositionModuleAsync` (F3.6.4/F3.6.5) takes `output`/`uvOutput` as
  caller-supplied `PortRef`s — nothing discovers them from a document's edges today; every existing
  test hand-supplies them directly. This milestone's Fix step 1 closes that gap.
- `KernelFragmentInput`/`executeKernelFragment` (F3.6.3/F3.6.5) has no way to receive a
  separately-assembled vertex module — when `varyings` is supplied, `assembleKernelFragmentModuleAsync`
  already correctly returns fragment-only code (F3.6.5's fix), but `executeKernelFragment`'s own draw
  call still uses `assembly.vertexCount`, which is `0` in that case (nothing currently supplies a
  real vertex module to pair it with in a live execution path — F3.6.5's own tests concatenate and
  compile manually, outside `executeKernelFragment` entirely). This milestone's Fix step 2 closes
  that gap.
- No trig/wave primitive exists (`math.sin` does not exist — confirmed by grep across
  `packages/graph/src/primitives/`), but `noise.perlin3d` does
  (`packages/graph/src/primitives/perlin3d.ts:87-92`, `inputs:[{position:vec3f}],
  outputs:[{value:f32}]`) — the bundled sample's displacement uses noise, not a sine wave.
- `assembleVertexKernelPositionModuleAsync` does not support ShaderToy uniforms (`host.iTime` etc.)
  — confirmed, it never checks `wgslReferencesShaderToyUniform` or declares a `u` binding, unlike the
  fragment-kernel path. The bundled sample's displacement is therefore **static** (a fixed,
  position-derived noise pattern), not time-animated — extending vertex-kernel assembly for
  ShaderToy uniforms is a real, separate future need, out of scope here (see Out of scope).
- `validateGraphCompleteness`'s `unconnected-input` finding is a **warning**, not an error
  (`packages/graph/src/validate.ts:23-29`) — `stage.vertexKernel`'s `uv` input can be left
  unconnected in a document without failing `validateGraphFull`, though this milestone's own sample
  wires it anyway (a real, working varying is the entire point of the proof).

## Verified current state

- **`sliceGraph` requires named `doc.outputs` entries, confirmed the source of a real failure mode
  this contract's own first draft missed:** `packages/compiler/src/slice.ts:17-21` —
  `doc.outputs.find((o) => o.name === name)`, throwing `` `Unknown output: ${name}` `` if absent.
  `assembleVertexKernelPositionModuleAsync`'s `compileGraph` call requests `outputs: ['position',
  'uv']` unconditionally (`vertexKernelPosition.ts:84-88`) — a real pipeline document's own
  `doc.outputs` only ever names its display-consumed field (e.g. `'image'`), never `'position'`/
  `'uv'`. F3.6.5's own tests never hit this because their hand-built test graphs declared exactly
  those two names directly. See Fix step 3's correction.
- **`collectParamFields` collects every upstream node's own numeric params unconditionally, not
  just edge-driven ones** — `packages/runtime-webgpu/src/emitGraphEval.ts:170-199`: for each node in
  the slice, `if (bindings[paramName]?.kind === 'edge') continue;` only skips params **fed via an
  edge into a downstream node's promotable slot** — a node's *own* literal params (e.g.
  `constant.f32`'s `value`) are never edge-fed themselves and are always collected. Any
  `constant.f32`/`math.gain`/`math.bias`-shaped node anywhere in a vertex-kernel's upstream slice
  unavoidably produces a `GraphParams` uniform at `@binding(0)`
  (`vertexKernelPosition.ts:106`) that nothing in `executeKernelFragment`
  (`kernelFragment.ts:266`, fragment-params-only) can carry or bind. See Fix step 4's correction.
- **`PipelineGraphExecutor.execute`'s exact current fragment branch** (`pipelineGraph.ts:264-312`,
  landed through F3.6.3/F3.6.5): discovers `fragmentNode`/`fragmentImpl` via
  `getPrimitive(fragmentNode.primitive)?.implementation`, branches on `fragmentImpl?.kind ===
  'kernel' && fragmentImpl.stage === 'fragment'`, calls `executeKernelFragment` with `bindings:
  fragmentImpl.bindings`, `kernelBindings: input.kernelFragmentBindings ?? {emptyMaps}`,
  `channelTargets: input.channelTargets`. Falls through to `executeFullscreenFragment` (unchanged)
  otherwise. This milestone adds a **nested** check inside the kernel-fragment branch: is
  `plan.vertexStageNode`'s own primitive *also* a kernel?
- **`resolvePipelineGeometryParams`, exported, unchanged, directly reusable**
  (`packages/runtime-webgpu/src/pipelineVertex.ts:19-34`) — same function `resolveVertexAssembly`
  already calls internally for the legacy path; this milestone's new branch calls it the same way,
  from a real document's `plan.geometryNode`.
- **`KernelFragmentAssembly`/`assembleKernelFragmentModuleAsync`'s exact current shape** (F3.6.5-landed,
  `consumers/kernelFragment.ts`) — when `input.varyings` is supplied, returns fragment-stage-only
  `code` (no `struct VSOut`, no vertex WGSL) and `vertexCount: 0` (from the now-skipped
  `resolveVertexAssembly` call). `executeKernelFragment`'s own draw call
  (`pass.draw(assembly.vertexCount)`) inherits that `0` today — never exercised in a live path since
  nothing calls it with `varyings` outside F3.6.5's own standalone tests.
- **`assembleVertexKernelPositionModuleAsync`'s exact current signature** (F3.6.5-landed,
  `vertexKernelPosition.ts`): `{graph, output: PortRef, uvOutput?: PortRef, geo?:
  PipelineGeometryParams, resolver}` → `{code, vertexCount, varyings: VaryingDecl[]}`. Geometry
  params default to `DEFAULT_PIPELINE_GEOMETRY_PARAMS` when omitted — this milestone's new branch
  always supplies real ones via `resolvePipelineGeometryParams(input.graph, plan)`.
- **`stage.vertexKernel`'s exact registered port shape** (F3.6.5-landed,
  `packages/graph/src/primitives/pipeline/index.ts`): `inputs:[{mesh:geometry}, {position:vec3f},
  {uv:vec2f}], outputs:[{varyings:varyings}]`.
- **`cosinePaletteEffectGraph`'s exact structure** (`packages/graph-editor/src/graphBuilders.ts:191-249`),
  the direct template for this milestone's new sample builder — `snapshotNode(id, primitiveId, {x,y},
  params?)`/`portRef(nodeId, primitiveId, direction, index)` helpers, geometry→persist→vertex→fragment→display
  spine plus a field-computing subgraph feeding the fragment's `color` input. This milestone's sample
  mirrors this shape exactly, substituting `stage.vertexKernel`/`stage.fragmentKernel` for
  `stage.vertex`/`stage.fragment` and adding the position/uv-computing subgraph feeding the vertex
  kernel.

## Fix

### 1. Discover `stage.vertexKernel`'s `position`/`uv` inputs from the document

`packages/runtime-webgpu/src/pipelineGraph.ts`, inside `PipelineGraphExecutor.execute`, using the
existing `incoming` helper (no new discovery function needed):

```ts
const vertexNode = input.graph.nodes.find((node) => node.id === plan.vertexStageNode);
const vertexImpl = vertexNode ? getPrimitive(vertexNode.primitive)?.implementation : undefined;
```

placed alongside the existing `fragmentNode`/`fragmentImpl` lookup two lines above it.

### 2. `executeKernelFragment` accepts a pre-assembled vertex module

`packages/runtime-webgpu/src/consumers/kernelFragment.ts`. `KernelFragmentInput` gains:

```ts
export interface KernelFragmentVertexModule {
	code: string;
	vertexCount: number;
}

export interface KernelFragmentInput {
	// ...existing fields...
	varyings?: readonly VaryingDecl[];
	vertexModule?: KernelFragmentVertexModule;
}
```

`executeKernelFragment` passes `input.varyings` through to `assembleKernelFragmentModuleAsync`
(already accepts it, unused by any caller today — this is its first real caller), then:

```ts
const assembly = await assembleKernelFragmentModuleAsync({
	graph, output, bindings, wgslTypes: kernelBindings.wgslTypes, resolver: input.resolver,
	varyings: input.varyings
});
const code = input.vertexModule ? `${input.vertexModule.code}\n\n${assembly.code}` : assembly.code;
const vertexCount = input.vertexModule?.vertexCount ?? assembly.vertexCount;
```

use `code`/`vertexCount` in place of `assembly.code`/`assembly.vertexCount` for the rest of the
function (pipeline creation, `pass.draw(vertexCount)`). When `input.vertexModule` is omitted (every
existing caller, F3.6.2/F3.6.3's own tests), behavior is byte-identical to today — `assembly.code`/
`assembly.vertexCount` used exactly as now.

### 3. `PipelineGraphExecutor.execute`'s new nested branch

Inside the existing `if (fragmentImpl?.kind === 'kernel' && fragmentImpl.stage === 'fragment')`
block, before the existing `return executeKernelFragment({...})`:

```ts
if (vertexImpl?.kind === 'kernel' && vertexImpl.stage === 'vertex') {
	const positionEdge = incoming(input.graph, vertexNode!.id, 'position');
	if (!positionEdge) {
		throw new Error(
			`Pipeline vertex kernel ${plan.vertexStageNode} is missing its position input`
		);
	}
	const uvEdge = incoming(input.graph, vertexNode!.id, 'uv');
	const geo = resolvePipelineGeometryParams(input.graph, plan);
	const resolver = input.resolver ?? createStandardLibraryResolver();

	// Blocking correction, caught by a pre-routing review: assembleVertexKernelPositionModuleAsync
	// calls compileGraph with outputs: ['position', 'uv'] (vertexKernelPosition.ts:84-88), and
	// compileGraph -> sliceGraph looks up doc.outputs.find(o => o.name === 'position') — throwing
	// `Unknown output: position` if no such NAMED entry exists (packages/compiler/src/slice.ts:17-21).
	// A real pipeline document's own doc.outputs only ever names its DISPLAY-consumed field (e.g.
	// 'image') — never 'position'/'uv', which are purely internal to this stage's own inputs.
	// F3.6.5's own tests only worked because their hand-built test graphs happened to declare
	// doc.outputs: [{name:'position',...}, {name:'uv',...}] directly — a real document never will.
	// Fix: synthesize a temporary, augmented document with those two names added, mirroring this
	// codebase's own established idiom for exactly this situation — `effectiveGraphDocument`
	// (`packages/graph/src/pipeline.ts`, `{...doc, outputs: effectiveOutputs(doc)}`) already merges
	// structurally-derived outputs into a compile/preview view without mutating the real document.
	// Do NOT require the bundled sample (or any future vertex-kernel document) to declare these
	// names itself — that would leak an internal assembly-function implementation detail into
	// user-authored document structure.
	const augmentedGraph: GraphDocument = {
		...input.graph,
		outputs: [
			...input.graph.outputs,
			{ name: 'position', from: positionEdge.from },
			...(uvEdge ? [{ name: 'uv', from: uvEdge.from }] : [])
		]
	};
	const vertexAssembly = await assembleVertexKernelPositionModuleAsync({
		graph: augmentedGraph,
		output: positionEdge.from,
		uvOutput: uvEdge?.from,
		geo,
		resolver
	});
	return executeKernelFragment({
		device: input.device,
		graph: input.graph,
		output: plan.fieldOutput,
		bindings: fragmentImpl.bindings,
		resolver,
		width: input.width,
		height: input.height,
		host: input.host,
		target: input.target,
		kernelBindings: input.kernelFragmentBindings ?? {
			wgslTypes: new Map(), resourceIds: new Map(), resources: new Map()
		},
		channelTargets: input.channelTargets,
		varyings: vertexAssembly.varyings,
		vertexModule: { code: vertexAssembly.code, vertexCount: vertexAssembly.vertexCount }
	});
}
// existing return executeKernelFragment({...}) — unchanged, reached only when the vertex stage
// is NOT a kernel (i.e. plain stage.vertex, paired with stage.fragmentKernel — already works
// today via resolveVertexAssembly, unaffected by this branch).
```

The `position` input is treated as structurally required (throws a clear, specific error if
unwired) — a vertex kernel with no position input cannot produce anything meaningful.
`uv` stays optional (`uvEdge?.from`, `undefined` when unwired), matching F3.6.4/F3.6.5's own
position-only-is-valid precedent. Import `assembleVertexKernelPositionModuleAsync` from
`./vertexKernelPosition.js` and `resolvePipelineGeometryParams` from `./pipelineVertex.js` (both
already exported, neither previously imported in this file).

**Not touched, deliberately:** `executeFullscreenFragment`'s own call sites (`stage.vertexKernel`
paired with legacy `stage.fragment` is out of scope — see Out of scope). `resolveVertexForFragment`/
`planPipelineGraph`'s existing validation (unchanged — `isPipelineStage(vertex, 'vertex')` already
accepts `stage.vertexKernel` today, confirmed in Context).

### 4. Register the bundled sample

`packages/graph-editor/src/graphBuilders.ts`, a new builder function,
`vertexKernelDisplacementGraph()`, mirroring `cosinePaletteEffectGraph`'s exact structure and helper
usage. Node/edge summary (exact `{x,y}` layout positions and node ids at the implementer's
discretion, matching this file's existing style):

- **Spine:** `geometry.plane` → `buffer.persist` → `stage.vertexKernel` → `stage.fragmentKernel` →
  `target.display` (mirrors `cosinePaletteEffectGraph`'s own spine exactly, substituting the two
  kernel stages).
- **Vertex displacement subgraph, deliberately zero param-bearing nodes — see the correction
  below.** One `procedural.metricPosition` node; extract `x`/`y`/`z` via
  `vector.vec3f.x`/`.y`/`.z`; sample `noise.perlin3d` from the *undisplaced* `metricPosition`
  output; add the **raw, unscaled** noise value directly to the extracted `z` (via `math.add` —
  no amplitude/scale step); reconstruct the displaced position via `vector.vec3f` from `{x, y,
  displacedZ}`. Wire the reconstructed vec3f into `stage.vertexKernel`'s `position` input.
- **Correction, caught by a pre-routing review of an earlier draft: no `constant.f32` (or any other
  param-bearing node — `math.gain`/`math.bias` and similar have the identical problem) anywhere in
  this subgraph.** `assembleVertexKernelPositionModuleAsync` declares any collected `GraphParamField`s
  as a `GraphParams` uniform at `@binding(0)` unconditionally
  (`vertexKernelPosition.ts:106`) — but `executeKernelFragment` only ever allocates and binds the
  **fragment** assembly's own params buffer (`kernelFragment.ts:266`); it has no mechanism to also
  carry and bind a *vertex*-side params buffer, and `KernelFragmentVertexModule` (Fix step 2) is
  deliberately just `{code, vertexCount}` — no params plumbing. `collectParamFields`
  (`emitGraphEval.ts:170-199`) collects **every** node's own numeric params in the upstream slice,
  regardless of how its output is wired downstream — a `constant.f32` node feeding the amplitude
  multiply would unavoidably trigger this, not merely if wired a particular way. Extending
  `KernelFragmentVertexModule`/`executeKernelFragment` to carry and bind vertex-side params without
  colliding with the fragment's own `@binding(0)`/`@binding(N)` allocation is a real, separate
  design question (binding-index coordination across two independently-assembled modules) — not
  attempted here; this milestone's own sample simply has nothing to carry. If a future sample needs
  a real vertex-side param, that plumbing is new, scoped work, not assumed to already exist.
- **Varying subgraph**, feeding `stage.vertexKernel`'s `uv` input: `vector.vec2f` from the *same*
  extracted `x`/`y` (undisplaced) — a real, per-vertex, non-uniform value shared with the
  displacement subgraph's own upstream nodes (matching F3.6.5's own "shared upstream, deduped
  params" proof shape — a *shape* worth keeping even though this milestone's own graph has no
  literal params to dedupe).
- **Fragment coloring subgraph**, feeding `stage.fragmentKernel`'s `color` input: one
  `procedural.uv` node (reads the interpolated `uv` varying via F3.6.5's `uvExpr` mechanism,
  automatic — no explicit wiring needed for the override itself, only for `procedural.uv`'s node to
  exist and feed downstream), combined into a `vec4f` via `vector.combine.vec2f_f32_f32` (matching
  F3.6.5's own landed test fixture, `fragmentUvGraph`, exactly — reuse that shape, not a new one).

Register in `packages/graph-editor/src/samples.ts`:

```ts
{
	id: 'foundation-vertex-kernel-displacement',
	label: 'Foundation — Vertex kernel displacement',
	build: vertexKernelDisplacementGraph
}
```

`samples.test.ts` needs its hardcoded `GRAPH_SAMPLES.length` bumped by one, plus a
`getGraphSample('foundation-vertex-kernel-displacement')`-defined assertion matching this file's
existing per-sample test style.

## Gate

1. `PipelineGraphExecutor.execute` on a `stage.vertexKernel`+`stage.fragmentKernel` document with
   `position` unwired throws the specific `'is missing its position input'` error, before any GPU
   work — a GPU-free test, matching every prior F3.6 branch-routing proof's own style.
2. `PipelineGraphExecutor.execute` on a document using legacy `stage.vertex` (paired with
   `stage.fragmentKernel`, F3.6.3's own existing test shape) is completely unaffected — passes
   through the existing, unchanged branch, `executeFullscreenFragment`/plain `stage.vertex`+
   `stage.fragment` documents equally unaffected. Direct regression proof, not assumed.
3. **Real-device test, the core wiring proof:** a document shaped like Fix step 4's bundled sample
   (or a smaller, equivalent test-only graph exercising the same structure) executed through
   `PipelineGraphExecutor.execute` directly (not `GraphFrameExecutor`, matching this file's own
   existing test scope) renders successfully, producing a **non-uniform** image — reusing the same
   "sample multiple pixel colors, assert more than one distinct value" technique F3.6.5's own gate
   already established, now proven through the *real, document-driven execution path* rather than a
   standalone, hand-assembled one.
4. `executeKernelFragment` with `vertexModule` supplied draws `vertexModule.vertexCount` vertices,
   not `0` — a direct, minimal proof of Fix step 2's specific bug (the previously-inherited `0` from
   `assembly.vertexCount` when varyings are present with no external vertex module).
5. **The augmented-document fix (position/uv named outputs) verified directly, not just
   assumed to compile:** a real-device or assembly-only test proves `assembleVertexKernelPositionModuleAsync`
   succeeds when called with a document whose *only* declared `doc.outputs` entry is the display's
   own field (e.g. `'image'`, matching a real pipeline document's actual shape) plus the
   milestone's own synthesized `position`/`uv` entries — not a document that already happens to
   declare `position`/`uv` itself (F3.6.5's own tests all did this; this milestone's whole point is
   that a real document never will). A negative-path companion test — calling the assembly function
   with the *unaugmented* document directly — should throw `Unknown output: position`, confirming
   the augmentation step is actually load-bearing, not vestigial.
6. **Zero vertex-side params, verified, not assumed:** the bundled sample's vertex-kernel assembly
   (`vertexAssembly.code`, or the `emitted.params`/equivalent the assembly function computes
   internally) contains no `GraphParams` struct and no `@binding(0) var<uniform> params` declaration
   — direct proof the sample's displacement subgraph (Fix step 4's correction) genuinely has zero
   param-bearing nodes in its upstream slice, not merely described that way in prose.
7. `check` **and** `test` green for `graph`, `graph-editor`, `runtime-webgpu`, full workspace (clear
   every `packages/*/dist` first). Real-device tests independently confirmed to execute, not silently
   skip (`--reporter=verbose` scoped to the package).
8. **Manual/visual gate, required by standing project convention for any milestone with one:** load
   the bundled sample (`'foundation-vertex-kernel-displacement'`) in a real browser via the actual
   editor UI (`apps/scene-editor` or `apps/webgputoy`'s graph editor, whichever hosts
   `packages/graph-editor`'s sample picker), confirm it renders without console errors, and confirm
   the displayed surface visibly shows non-planar, noise-displaced geometry with color varying
   smoothly across it (the `uv`-driven fragment coloring) — not a flat plane, not a solid color, not
   a broken/black render. Paste a screenshot of the confirmed behavior, per this project's own
   visual-gate convention (`AGENTS.md`).

## Out of scope

`stage.vertexKernel` paired with legacy `stage.fragment` (the ShaderToy path) — `stage.fragment`
cannot consume a named varying at all (its own `varyings` input has always been purely structural);
pairing them would produce an unused varying output, technically harmless but not a meaningful
capability, and not needed for this milestone's proof. Extending
`assembleVertexKernelPositionModuleAsync` for ShaderToy host uniforms (`host.iTime` etc.) so a
vertex kernel's displacement could animate over time — confirmed not supported today, a real,
separate future need; this milestone's sample uses a static, position-derived displacement instead.
Wiring `stage.vertexKernel` through `executeFullscreenFragment` at all. Any change to
`GraphFrameExecutor`, `graphFramePlan.ts`, `previewFrameLoop.ts`, or `GraphEditor.svelte` — confirmed
unnecessary by this contract's own pre-drafting investigation (see Context). Any change to
`stage.vertex`/`stage.fragment`'s own behavior or any existing sample's rendering. Any change to
F3.6.1-5's landed contracts beyond the two explicitly-scoped, purely-additive extension points (Fix
steps 1-2).

## Handoff

→ `PipelineGraphExecutor.execute` now genuinely branches on vertex-stage kind, the same way it has
branched on fragment-stage kind since F3.6.2 — a document wiring `stage.vertexKernel` and
`stage.fragmentKernel` together renders correctly through the real, document-driven execution path
every other pipeline document already uses, with zero `GraphFrameExecutor`/editor-side changes
needed. A bundled, pickable sample demonstrates real, graph-authored, noise-based vertex
displacement with a real, shared, per-vertex `uv` varying driving fragment coloring — visually
confirmed in a real browser, closing out the entire F3.6 pipeline-kernels sequence this design
document's six sub-problems (A)-(F) opened. F3.6 is done after this. Any further generalization
(more varyings, varying types beyond `vec2f`, ShaderToy uniforms on the vertex side, a real
`u32`-consuming primitive for raw vertex/instance indices, `HostBinding.stages` enforcement) is a
separate, later, evidence-backed decision — not committed to here, not assumed to be needed.
