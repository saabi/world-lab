# Brief — F3.3: compute kernels and dispatch domains

**Type:** codegen fix + new runtime capability, third milestone of the Foundation 3 generic-kernels
sequence · **Package:** `@world-lab/compiler` (one-line codegen fix), `@world-lab/runtime-webgpu`
(new capability) · **Depends on:** F3.1, landed (`4c28431`) — `KernelBindingTemplate`/
`resolveKernelBindings`/`ResolvedKernelBinding` (`packages/graph/src/implementation.ts`,
`packages/graph/src/kernelBinding.ts`) and `bindingDeclKindForTemplate`
(`packages/compiler/src/stageEntry.ts`), all currently unconsumed by any real GPU code — this
milestone gives them their first real caller. **Not** dependent on F3.2 (`0c4b7d8`, typed
varyings) — unrelated concern, compute kernels have no vertex/fragment pairing. · **Design
authority:** [foundation-3-generic-kernels-plan.md](../foundation-3-generic-kernels-plan.md) ·
[elemental-webgpu-architecture-review.md, Foundation 3](../elemental-webgpu-architecture-review.md#foundation-3-generic-kernels)
· **Contract author:** Opus · **Recommended executor:** Cursor or Codex · **Status:** ready to
route

## Context

Third of Foundation 3's sequence (F3.1 → F3.2 → **F3.3** → F3.4 → F3.5). This milestone does two
things: fixes a real, live codegen bug in `assembleStageEntry`'s compute template, and gives F3.1's
binding-template model its first real GPU-facing consumer — a standalone function that assembles a
compute kernel's WGSL, builds a real `GPUBindGroup` from `ResolvedKernelBinding[]`, derives a
dispatch domain, and calls `dispatchWorkgroups`.

**This is not the first compute pipeline in `runtime-webgpu`.** Re-verifying the Foundation 3
plan's own claim while drafting this contract found it was wrong (see Verified current state,
first bullet, and the correction already applied to
[foundation-3-generic-kernels-plan.md](../foundation-3-generic-kernels-plan.md)): three hand-rolled
`GPUComputePipeline`/`dispatchWorkgroups` call sites already exist
(`planeScalarPreview.ts`/`meshGen.ts`/`vegetationCandidates.ts`). What is genuinely new here is a
compute pipeline built through the **generic** model — `KernelBindingTemplate` →
`resolveKernelBindings` → a real `GPUBindGroup`, and `assembleStageEntry`'s compute template →
a real `GPUShaderModule` — rather than each consumer hand-templating its own WGSL and hand-writing
its own bind group with no shared abstraction between them.

**This milestone does not wire a real graph-authored compute kernel into `GraphFrameExecutor` or
any document.** Its deliverable is a standalone function that takes an already-compiled
`ConsumerShader`, a `KernelBindingTemplate[]`, and already-realized GPU resources supplied directly
by the caller (not derived from a `GraphDocument`/`ResourceRealizer` walk) — mirroring F3.1's own
deferral of "deriving `resourceIds` from a real `GraphDocument`" and F3.2's "no `packages/graph`
changes, no real pipeline wiring." Real per-document kernel wiring, replacing `stage.vertex`'s
`wgsl-function` hardcode and giving `stage.compute` its first real primitive, is F3.4's job.

Because nothing in the real render/preview pipeline changes, this milestone has **no manual/visual
gate** and needs **no bundled sample** — same situation as F3.1 and F3.2. The real visual/compute
proof sample is F3.5's job once F3.4 wires a real kernel through this machinery.

## Verified current state

- **The plan document's claim "the first real `GPUComputePipeline` creation... today there are
  zero" was wrong, and has been corrected in
  [foundation-3-generic-kernels-plan.md](../foundation-3-generic-kernels-plan.md) directly (not
  just here).** Grep for `createComputePipeline`/`dispatchWorkgroups` across `packages/*/src`
  (excluding tests) finds three real, working call sites: `planeScalarPreview.ts:79-86` (helper
  `createComputePipeline`) + `:141` (`dispatchWorkgroups(Math.ceil(width/8), Math.ceil(height/8))`),
  `meshGen.ts:447` + `:488`, `vegetationCandidates.ts:394-401` + `:529`. Each hand-assembles its own
  WGSL (a manually-templated `@compute @workgroup_size(...)` entry, correctly spelled — confirmed by
  direct read of `planeScalarPreview.ts:61`), its own `GPUBuffer`s (via `createStorageBuffer`/
  `alignTo`, `packages/runtime-webgpu/src/buffers.ts`), and its own bind group — all with
  `layout: 'auto'` and zero shared abstraction between the three. None goes through
  `assembleStageEntry`'s compute branch or `KernelBindingTemplate`. This milestone's actual
  contribution is the first pipeline built through the *generic* model, not the first pipeline,
  full stop — the brief below is scoped and worded accordingly, and Fix step 6's device test proves
  the generic path end-to-end, not merely that *a* compute pipeline can be created (already proven
  three times over).
- **`assembleStageEntry`'s compute template still emits invalid WGSL today**
  (`packages/compiler/src/stageEntry.ts:215-217`, current landed state after F3.1+F3.2):
  ```ts
  } else if (shader.stage === 'compute') {
  	const [x, y, z] = opts.workgroupSize ?? [64, 1, 1];
  	entry = `@compute @workgroupSize(${x}, ${y}, ${z})\nfn cs_main(@builtin(global_invocation_id) gid: vec3u) {\n\t${fn}(${args});\n}`;
  }
  ```
  `@workgroupSize` (no underscore) is not a valid WGSL attribute — real WGSL requires
  `@workgroup_size`. `stageEntry.test.ts`'s own "wraps a compute consumer in @compute with a
  workgroup size" test asserts the wrong string today (`.toContain('@compute @workgroupSize(8, 8,
  1)')`), so the bug has stayed live with a green test suite. Confirmed via direct read this was
  **not** touched by F3.1 (which added `storage-read-write` to `BindingDecl.kind` and
  `bindingDeclKindForTemplate`, not the compute entry template) or F3.2 (which touched only the
  vertex/fragment branches). This milestone is the first to give the compute branch a real caller,
  so fixing the bug here — rather than as an isolated drive-by earlier — puts the fix next to its
  first real test coverage (Fix step 1, Gate items 1-2).
- **`KernelBindingTemplate`/`resolveKernelBindings`/`ResolvedKernelBinding`
  (`packages/graph/src/kernelBinding.ts`, landed F3.1) have zero consumers anywhere** — confirmed
  by grep, the only non-test references are within `kernelBinding.ts`/`implementation.ts`
  themselves and `bindingDeclKindForTemplate`'s type-only import. `resolveKernelBindings`'s exact
  signature: `resolveKernelBindings(templates: readonly KernelBindingTemplate[], ownerStage:
  ShaderStage, resourceIds: ReadonlyMap<string, string>): ResolvedKernelBinding[]` — the
  `resourceIds` map is name→resourceId, caller-supplied (F3.1 explicitly deferred deriving it from
  a real document). `ResolvedKernelBinding` carries `resourceId`/`access`/`name`/`binding`/
  `resourceKind`/`stages` — everything needed to build a `GPUBindGroupEntry` **except** a concrete
  GPU resource (`GPUBuffer`/`GPUTexture`/`GPUSampler`) and a WGSL type string, both still owed by
  whoever actually has real GPU resources — this milestone.
- **`bindingDeclKindForTemplate` (`packages/compiler/src/stageEntry.ts:61-79`, landed F3.1) also
  has zero consumers anywhere** — confirmed by grep. It takes `Pick<KernelBindingTemplate,
  'resourceKind' | 'access'>` and returns `BindingDecl['kind']`, but nothing yet builds the
  remaining `BindingDecl` fields (`group`/`binding`/`name`/`wgslType`) from a
  `KernelBindingTemplate` — F3.1's own brief explicitly deferred this ("turning a resolved
  `KernelBindingTemplate` into a fully-formed `BindingDecl`... is explicitly deferred"). This
  milestone is that deferred step (Fix step 3) — `bindingDeclKindForTemplate`'s first real caller.
  Per F3.1's own documented convention, every binding is implicitly `group: 0` (no multi-bind-group
  support anywhere in the codebase yet).
- **`KernelBindingTemplate` has no `wgslType` field** (`packages/graph/src/implementation.ts:64-70`)
  — only an abstract `resourceKind: 'buffer' | 'texture' | 'sampler'`. A concrete WGSL type string
  (`'array<f32>'`, `'texture_2d<f32>'`, `'sampler'`) is not derivable from `resourceKind` alone (a
  buffer could back `array<f32>`, `array<vec4f>`, a custom struct, etc.) — this is exactly the gap
  F3.1's brief flagged as deferred ("a real WGSL type derived from the bound resource's concrete
  shape"). This milestone's `buildKernelBindingDecls` (Fix step 3) requires the caller to supply a
  `wgslType` per binding name directly, rather than inventing a resourceKind→wgslType inference
  that would be wrong for anything but the simplest cases — the same "two separate,
  caller-correlated lists" pattern `fullscreenFragment.ts` already uses today (hand-built
  `BindingDecl[]` alongside a separately-tracked `channelBindings` map,
  `packages/runtime-webgpu/src/consumers/fullscreenFragment.ts:148-192`).
- **`Pass.bindings`/`ResourceBinding[]` (F2.2) still has exactly one consumer:
  `resolveBufferUsage`, for usage-flag inference only** — this milestone does not change that; it
  operates entirely outside the `Pass`/`PassGraph`/`ResourceRealizer` frame-graph abstraction (see
  next bullet), so it is not `Pass.bindings`'s second consumer. `resolveKernelBindings`
  (F3.1) already is a second consumer of the general `ResourceBinding` shape, just not through
  `Pass.bindings` specifically.
- **F2.2's `resolveBufferSizes`/`resolveTargetSizes`** (`packages/runtime-webgpu/src/frameGraph/order.ts:261-295`,
  confirmed by direct read) derive `[width,height]`/`elementCount` from a whole `PassGraph`'s
  `targets` list — a resource-kind-aware split (`isTextureResourceTarget`/`isBufferResourceTarget`
  type guards) this milestone's own `resolveDispatchDomain` (Fix step 4) mirrors in spirit, but
  cannot literally reuse: this milestone has no `PassGraph` (no real document wiring, see Context).
  `resolveDispatchDomain` instead takes a single caller-supplied resource-shape descriptor, not a
  whole graph — deliberately narrower, consistent with everything else in this milestone staying
  outside the frame-graph abstraction until F3.4.
- **`BufferFeedbackExecutor.execute`** (`packages/runtime-webgpu/src/consumers/bufferFeedback.ts:140-144`,
  `async execute(device: GPUDevice, gridWidth: number, gridHeight: number): Promise<{ pixels:
  Uint8Array }>`) is the plan's own cited precedent for "dispatch stays a plain runtime function
  call." It is a stateful class because it needs cross-frame history (ping-ponging two buffer
  slots via its own `ResourceRealizer` instance). This milestone's `executeComputeKernel` (Fix step
  6) has no such requirement — no persistent state, no history — so it is a **plain function**, not
  a class: simpler than its own precedent because the precedent's statefulness doesn't apply here.
- **Every existing GPU consumer in this codebase uses `layout: 'auto'`** (confirmed by grep across
  `fullscreenFragment.ts`, `bufferFeedback.ts`, `planeScalarPreview.ts`, `meshGen.ts`,
  `vegetationCandidates.ts` — zero explicit `GPUBindGroupLayoutEntry`/`GPUPipelineLayout`
  construction anywhere in the package). This milestone follows the same convention rather than
  inventing new bind-group-*layout* construction machinery — only `GPUBindGroupEntry[]` (the bind
  group's actual resource list, resolved against the auto-inferred layout) needs building from
  `ResolvedKernelBinding[]` (Fix step 2).
- **The readback convention** (`copyBufferToBuffer` → `mapAsync(GPUMapMode.READ)` →
  `getMappedRange()` → `unmap()`/`destroy()`) is established at
  `planeScalarPreview.ts:144-155` and reused verbatim by the device test below (Fix step 7) to
  prove a dispatched kernel actually mutated its output buffer, not just that it compiled.

### 2. `buildComputeBindGroupEntries` — resolved bindings → real `GPUBindGroupEntry[]`

New file `packages/runtime-webgpu/src/computeKernel.ts`:

```ts
import type { ResolvedKernelBinding } from '@world-lab/graph';

export type ComputeKernelResource =
	| { kind: 'buffer'; buffer: GPUBuffer }
	| { kind: 'texture'; texture: GPUTexture }
	| { kind: 'sampler'; sampler: GPUSampler };

export function buildComputeBindGroupEntries(
	resolved: readonly ResolvedKernelBinding[],
	resources: ReadonlyMap<string, ComputeKernelResource>
): GPUBindGroupEntry[] {
	return resolved.map((binding) => {
		const resource = resources.get(binding.resourceId);
		if (resource === undefined) {
			throw new Error(
				`Kernel binding "${binding.name}" resolves to resource id "${binding.resourceId}", ` +
					'which has no entry in the resources map'
			);
		}
		if (resource.kind !== binding.resourceKind) {
			throw new Error(
				`Kernel binding "${binding.name}" declares resourceKind:'${binding.resourceKind}' ` +
					`but its resolved resource is kind:'${resource.kind}'`
			);
		}
		if (resource.kind === 'buffer') {
			return { binding: binding.binding, resource: { buffer: resource.buffer } };
		}
		if (resource.kind === 'texture') {
			return { binding: binding.binding, resource: resource.texture.createView() };
		}
		return { binding: binding.binding, resource: resource.sampler };
	});
}
```

`ComputeKernelResource` is a tagged union rather than a plain `GPUBuffer | GPUTexture | GPUSampler`
deliberately — an `instanceof` discriminator on real WebGPU interface types is not a reliable
cross-environment check (browser vs. Node/Dawn), so the caller tags each resource explicitly, and
this function validates the tag against `binding.resourceKind` itself rather than trusting it
silently (the second `throw` above) — this is real, load-bearing validation, not a formality: it's
what catches a caller wiring the wrong resource to the wrong binding name.

### 3. `buildKernelBindingDecls` — `bindingDeclKindForTemplate`'s first real caller

Same file:

```ts
import { bindingDeclKindForTemplate, type BindingDecl } from '@world-lab/compiler';
import type { KernelBindingTemplate } from '@world-lab/graph';

export function buildKernelBindingDecls(
	templates: readonly KernelBindingTemplate[],
	wgslTypes: ReadonlyMap<string, string>
): BindingDecl[] {
	return templates.map((template) => {
		const wgslType = wgslTypes.get(template.name);
		if (wgslType === undefined) {
			throw new Error(`Kernel binding "${template.name}" has no declared WGSL type`);
		}
		return {
			group: 0,
			binding: template.binding,
			name: template.name,
			kind: bindingDeclKindForTemplate(template),
			wgslType
		};
	});
}
```

`wgslTypes` maps binding `name` → a concrete WGSL type string (e.g. `'array<f32>'`) — supplied by
the caller, not inferred from `resourceKind` (see Verified current state: not derivable in
general). Every binding lands at `group: 0`, matching every existing consumer's convention.

### 4. `DispatchDomain`/`resolveDispatchDomain` — resource-kind-aware dispatch sizing

Same file:

```ts
export type DispatchDomain =
	| readonly [number, number, number]
	| { kind: 'buffer'; elementCount: number }
	| { kind: 'texture'; width: number; height: number };

export function resolveDispatchDomain(
	domain: DispatchDomain,
	workgroupSize: readonly [number, number, number]
): [number, number, number] {
	if (Array.isArray(domain)) {
		return [domain[0], domain[1], domain[2]];
	}
	const [workgroupX, workgroupY] = workgroupSize;
	if (domain.kind === 'buffer') {
		return [Math.ceil(domain.elementCount / workgroupX), 1, 1];
	}
	return [Math.ceil(domain.width / workgroupX), Math.ceil(domain.height / workgroupY), 1];
}
```

An explicit `[x, y, z]` tuple passes straight through — the "or an explicit param" half of the
plan's own description. `'buffer'`/`'texture'` mirror F2.2's `resolveBufferSizes`/
`resolveTargetSizes` resource-kind-aware split in spirit (element-count-driven vs.
dimension-driven), scoped down to a single descriptor rather than a whole `PassGraph` (see Verified
current state).

### 5. `executeComputeKernel` — the plain runtime function call

Same file:

```ts
import { resolveKernelBindings, type KernelBindingTemplate } from '@world-lab/graph';
import { assembleStageEntry, type ConsumerShader } from '@world-lab/compiler';

export interface ComputeKernelInput {
	/** A compiled compute-stage function library (from `compileGraph`), same as
	 *  `assembleStageEntry` already consumes elsewhere. */
	shader: ConsumerShader;
	bindings: readonly KernelBindingTemplate[];
	/** Binding name -> concrete WGSL type string. */
	wgslTypes: ReadonlyMap<string, string>;
	/** Binding name -> resourceId (same shape `resolveKernelBindings` already expects). */
	resourceIds: ReadonlyMap<string, string>;
	/** resourceId -> the real, already-allocated GPU resource. */
	resources: ReadonlyMap<string, ComputeKernelResource>;
	/** output name -> generated fn name driving `cs_main`'s body. */
	outputFns: Record<string, string>;
	callArgs?: string[];
	workgroupSize: readonly [number, number, number];
	dispatch: DispatchDomain;
}

export async function executeComputeKernel(
	device: GPUDevice,
	input: ComputeKernelInput
): Promise<void> {
	const resolved = resolveKernelBindings(input.bindings, 'compute', input.resourceIds);
	const bindingDecls = buildKernelBindingDecls(input.bindings, input.wgslTypes);
	const stage = assembleStageEntry(input.shader, {
		bindings: bindingDecls,
		outputFns: input.outputFns,
		callArgs: input.callArgs,
		workgroupSize: [...input.workgroupSize] as [number, number, number]
	});

	const module = device.createShaderModule({ label: 'compute-kernel', code: stage.code });
	const pipeline = await device.createComputePipelineAsync({
		label: 'compute-kernel',
		layout: 'auto',
		compute: { module, entryPoint: 'cs_main' }
	});

	const entries = buildComputeBindGroupEntries(resolved, input.resources);
	const bindGroup = device.createBindGroup({ layout: pipeline.getBindGroupLayout(0), entries });
	const [x, y, z] = resolveDispatchDomain(input.dispatch, input.workgroupSize);

	const encoder = device.createCommandEncoder({ label: 'compute-kernel' });
	const pass = encoder.beginComputePass({ label: 'compute-kernel' });
	pass.setPipeline(pipeline);
	pass.setBindGroup(0, bindGroup);
	pass.dispatchWorkgroups(x, y, z);
	pass.end();
	device.queue.submit([encoder.finish()]);
}
```

A plain `async` function, not a class — unlike `BufferFeedbackExecutor`, there is no cross-frame
state to own (see Verified current state). It returns `void`: readback shape varies too much by use
case to standardize here (a scalar buffer vs. an RGBA texture vs. a struct array each want a
different readback API) — the caller reads back its own output resource afterward, the same way
every existing hand-rolled consumer already does. Do **not** add a readback helper, a `dispose()`,
or any caching/reuse of pipelines/bind groups across calls — out of scope (see below).

## Out of scope

Wiring a real graph-authored compute kernel into `GraphFrameExecutor`, `ResourceRealizer`, or any
document — `executeComputeKernel` takes already-realized GPU resources and an already-compiled
`ConsumerShader` supplied directly by the caller; deriving either from a real `GraphDocument` is
F3.4's job. A `stage.compute` primitive registration (`{ kind: 'kernel'; stage: 'compute' }` used by
an actual node in the primitive registry) — F3.4's job. Any readback helper, `dispose()`, or
pipeline/bind-group caching/reuse across calls — `executeComputeKernel` is a plain, stateless
function; adding caching without a real caller that needs it (F3.4) would be premature. Deriving a
`wgslType` automatically from `resourceKind` — not generally possible (see Verified current state);
callers keep supplying it explicitly, same as `fullscreenFragment.ts` already does for its own
`BindingDecl[]`. Multiple bind groups (`group: 0` only, matching every existing consumer). Storage
textures (`resourceKind: 'texture'` with `access !== 'read'` still rejected by
`validateKernelBindingTemplates`/`bindingDeclKindForTemplate`, per F2.3's/F3.1's own deferral —
untouched here). Migrating `planeScalarPreview.ts`/`meshGen.ts`/`vegetationCandidates.ts`'s
hand-rolled compute pipelines onto this generic path — not required by this milestone, ever, as a
consequence of it landing (same "existing paths keep working, nothing is forced to migrate"
discipline as every prior Foundation 2/3 milestone). Typed varyings (F3.2, unrelated — compute
kernels have no vertex/fragment pairing). `{ kind: 'command'; command: GpuCommandKind }`
(Foundation 4's seam). Any change to Foundation 1's, Foundation 2's, F3.1's, or F3.2's landed
contracts.

## Handoff

→ `assembleStageEntry`'s compute template finally emits valid WGSL. `KernelBindingTemplate`/
`resolveKernelBindings`/`bindingDeclKindForTemplate` (F3.1, previously zero real consumers) now
have a real, tested, end-to-end path from a graph-level binding template to an actual dispatched

executor F2.4 built, while today's ShaderToy/cross-pass-texture/buffer-feedback samples stay
regression-green throughout. Request F3.4's own contract when appropriate.
