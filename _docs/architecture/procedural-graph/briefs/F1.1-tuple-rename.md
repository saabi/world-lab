# Brief — F1.1: rename static `list<T>` to `tuple<T>`

**Type:** rename / type-system correction, first milestone of the Foundation 1 elemental-contracts
migration · **Package:** `@world-lab/graph`, `@world-lab/compiler`, `@world-lab/runtime-webgpu`,
`@world-lab/graph-editor` · **Depends on:** none — lowest-risk milestone in Foundation 1, can start
immediately · **Design authority:**
[foundation-1-elemental-contracts-plan.md, F1.1 section](../foundation-1-elemental-contracts-plan.md)
(revision 7, six rounds of independently-verified external review) · **Contract author:** Opus ·
**Recommended executor:** Codex (headless, algorithmic rename with a clean failing-test gate) ·
**Status:** ready to route

## Context

This is the first of five milestones freezing this project's "elemental" graph contracts (see the
parent plan for the full sequence: F1.1 → F1.2 → F1.5 → F1.3 → F1.4a). Per explicit instruction,
**this milestone is now frozen and approved for implementation** — do not revisit F1.2–F1.4a's
design here; those get their own contract when each becomes next. The remaining open questions in
the parent plan are all in F1.3/F1.4a, not this one.

`list<f32>`/`list<vec2f>`/`list<vec3f>`/`list<vec4f>` currently conflate two different codegen
strategies under one name, selected by *edge count* rather than an explicit type:
- Multiple edges into a `list<T>` port → static unroll into a WGSL `array<T, N>` constructor.
- A single edge from a `storageBuffer` port → a runtime `for` loop over `arrayLength(&buffer)`.

**Verified: zero registered production primitives declare a `list<T>` input port.** The only
current users are test fixtures. This makes the rename a pure, mechanical, behavior-preserving
change with no production callers to break.

## Fix

1. **Rename the type**, in `packages/graph/src/types.ts:18`:
   ```ts
   // before
   export type ListDataType = 'list<f32>' | 'list<vec2f>' | 'list<vec3f>' | 'list<vec4f>';
   // after
   export type ListDataType = 'tuple<f32>' | 'tuple<vec2f>' | 'tuple<vec3f>' | 'tuple<vec4f>';
   ```
   Keep the exported type name `ListDataType` as-is (it's a category name, not the string values) —
   only the four literal string values change from `list<...>` to `tuple<...>`. Do not rename to
   `staticList<T>`; the parent plan settled on `tuple<T>`.
2. **Update every file matching the string prefix `'list<'` to match `'tuple<'` instead.** Verified
   by direct grep — the complete file list, beyond `types.ts`:
   - `packages/graph/src/dataType.ts` — `canonicalDataType`'s recursive `list<...>` parsing
     (`normalized.startsWith('list<')`, `.slice(5, -1)`) becomes `tuple<...>` parsing (adjust the
     slice offset for the new 6-character prefix: `'tuple<'.length === 6`, not 5).
   - `packages/graph/src/ports.ts` — `compatibleDataTypes`'s `list<T>` compatibility rule
     (`toCanonical.startsWith('list<')`, `.slice(5, -1)`) — same slice-offset fix.
   - `packages/compiler/src/groupCodegen.ts` (~lines 328-354, 406-414) — static-unroll vs.
     storage-buffer-loop dispatch; update any `list<` string checks to `tuple<`.
   - `packages/runtime-webgpu/src/emitGraphEval.ts` (~lines 346-372, 411-424) — mirrors
     `groupCodegen.ts`'s dispatch; same update.
   - `packages/graph-editor/src/previewBuffers.ts` — the `ALL_DATA_TYPES` literal array (~line
     47-50) and the exhaustive `previewFamily()` switch (~line 147-150, `_exhaustive: never`
     pattern) both enumerate `'list<f32>'` etc. by name; update all four cases in both places.
   - `packages/graph-editor/src/irAdapter.ts` (~lines 322, 446) — `toPort.dataType.startsWith('list<')`
     checks controlling edge-validation UI; update to `'tuple<'`.
3. **Migrate every test fixture referencing `list<T>`** to `tuple<T>` — this is the entire
   compatibility surface, since no production primitive uses it. Confirmed fixture locations:
   `packages/graph/src/validateMultipleInputs.test.ts`, `packages/graph/src/ports.test.ts`,
   `packages/graph/src/dataType.test.ts`, `packages/runtime-webgpu/src/emitGraphEval.test.ts`,
   `packages/graph-editor/src/irAdapter.test.ts`, `packages/graph-editor/src/previewBuffers.test.ts`.
   Search each for the literal string `list<` to find every occurrence — do not assume the above
   line numbers are exhaustive within each file; grep the whole file.
4. **Do not touch the runtime storage-buffer loop path's behavior** — only its *name* changes where
   it currently mentions `list<`. The single-edge-from-`storageBuffer` → `for`-loop codegen strategy
   itself is unchanged, still untyped, still selected by edge count. Do not invent an interim
   `storageBuffer<T>` or `buffer<T>` type in this milestone — that belongs to a later milestone
   (F1.5 or Foundation 2), once `TypeRef` exists to give it a real element type. Adding one here
   would just be deleted later.
5. **Do not** build `flow.forEach`/`flow.map`/`flow.reduce` container nodes. Zero code for these
   exists today (verified); they are out of scope for this milestone entirely.

## Gate

1. `tuple<T>` compiles to byte-identical WGSL to today's `list<T>` static-unroll case — this is a
   pure rename, not a codegen behavior change. Diff the generated WGSL for the migrated test
   fixtures before/after and confirm no change beyond the type name appearing in comments/labels
   (if any).
2. A regression test pins today's exact dispatch behavior unchanged: a single edge from a
   `storageBuffer`-typed port still triggers the runtime loop; anything else (including multiple
   edges where one happens to be a `storageBuffer` source) still triggers static unroll. This test
   exists specifically so a later milestone can't silently change this dispatch rule without a
   failing test.
3. Grep the entire `packages/` tree for the literal string `list<` after the change — it should
   return zero matches outside of comments explicitly discussing the historical rename (if any are
   added) or unrelated string literals that happen to contain the substring (verify any such match
   individually, don't just suppress the check).
4. `check` **and** `test** green for `graph`, `compiler`, `runtime-webgpu`, `graph-editor`, and the
   full workspace (`npm run check --workspaces` / `npm run test --workspaces` from repo root, per
   `AGENTS.md`'s gate discipline — clear every `packages/*/dist` first, per the documented stale-dist
   issue with `customConditions`).

## Out of scope
