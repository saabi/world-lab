# F3.6.4 — vertex kernel invocation model: design discussion

**Status:** draft design discussion, **not a contract** — no Fix steps, no Gate items, nothing here
should be routed to an implementer as-is · **Parent:**

· [foundation-3-generic-kernels-plan.md, F3.6](./foundation-3-generic-kernels-plan.md) ·
**Depends on:** F3.6.1 (`29a3208`), F3.6.2 (`9183107`), both landed; F3.6.3
(fragment-kernel derived bindings), contract drafted and in flight, not yet landed at the time of
writing — this document does not depend on F3.6.3's own landing, only on its contract's scope

## Why this needs its own design pass

Sub-problem (F) from the parent design document, in full: `emitGraphEval`'s entire host-input
vocabulary is fragment/field-oriented — there is no defined way for a graph-authored vertex kernel
to read `vertex_index`/`instance_index`, or to read "this vertex's grid position" as a graph value.
F3.2 already built the *assembly*-layer capability to receive such values (`assembleStageEntry`'s
vertex branch passes `vid`/`iid` to generated functions via `callArgs`) — but nothing on the
*emission* side (`emitGraphEval`) can produce a graph value from them. This is not a one-line fix
like (C) was; it requires deciding what new host-input vocabulary exists, whether it reuses existing
mechanisms or needs new ones, and how a graph-authored "position" output becomes a valid clip-space
`vec4f`. F3.6.5 (the real vertex-kernel primitive) cannot be scoped correctly until these are
answered — discovering them partway through implementing F3.6.5 is exactly the risk the parent
document's own pre-drafting review flagged for (E) and (F) both.

## (G) What geometry actually exists to sample, precisely

**Confirmed by direct read, not assumed: there is exactly one geometry-generation model in the
pipeline graph today, and it is not a vertex-buffer read.** Two primitives declare
`role: 'pipelineGeometrySource'` — `geometry.plane` (`packages/graph/src/primitives/pipeline/plane.ts`)
and `geometry.fullscreenPlane` (a legacy alias resolving to the same primitive, per `plane.ts`'s own
help text: *"legacy geometry.fullscreenPlane graphs resolve to this same plane-grid primitive"*).
Both are backed by a single WGSL function, `plane_grid_position(vid, resU, resV, width, height,
rotationX, rotationY, rotationZ) -> vec3f`, called directly inside `assemblePipelineVertexWgsl`'s
hand-written `vs_main` template (`pipelineVertex.ts:59-71`) — a pure arithmetic, triangle-list-aware
mapping from `vid` to a clip-space-ready local position (confirmed by its exact CPU mirror,
`planeGridPosition` in `packages/graph/src/primitives/pipeline/planeGrid.ts:69-113` — quad/corner
indexing from `vid`, no buffer, no storage read anywhere). "Geometry-source sampling" for this
codebase's current pipeline model means *evaluating a pure function of `vid` and a handful of
scalar params*, not reading an arbitrary vertex attribute buffer. Whatever this document recommends
should fit that reality, not a more general vertex-buffer-sampling model nothing here needs yet.

## (H) Raw vertex-invocation builtins (`vertex_index`/`instance_index`)

**The gap.** No existing `HostBinding` represents these. `assembleStageEntry`'s vertex branch already
declares `@builtin(vertex_index) vid: u32, @builtin(instance_index) iid: u32` as the entry function's
own parameters (`packages/compiler/src/stageEntry.ts:237`) and threads them to every generated
output function via `opts.callArgs` (landed F3.2, confirmed unused by any real caller today — no
vertex-stage assembly function exists yet to supply `callArgs: ['vid', 'iid']`). The assembly layer
is ready; nothing emits a graph value from them.

**Recommendation: two new `host.*` primitives, mirroring `host.fragCoord`'s exact existing shape.**
`host.fragCoord` is `{kind:'host-input', binding:{context:'stage-builtin', key:'fragCoord',
stages:['fragment']}}`, emitting `position.xy` — trivial because `position` is already the fragment
entry's own parameter. The direct parallel:

```ts
// host.vertexIndex
outputs: [{ name: 'index', type: { kind: 'scalar', scalar: 'u32' } }],  // NOT dataType: 'u32' — see below
implementation: { kind: 'host-input', binding: { context: 'stage-builtin', key: 'vertexIndex', stages: ['vertex'] } }
// emitHostInput's new branch:
// if (binding.context === 'stage-builtin' && binding.key === 'vertexIndex') return `let ${variable}: u32 = vid;`;

// host.instanceIndex — identical shape, key: 'instanceIndex', emits `let ${variable}: u32 = iid;`
```

**Port typing correction, caught by a pre-routing review of an earlier draft of this document: the
output port must use `type: {kind:'scalar', scalar:'u32'}`, never `dataType: 'u32'`.** Confirmed by
direct read: the legacy `ValueDataType` union (`packages/graph/src/types.ts:5`) is exactly `'f32' |
'vec2f' | 'vec3f' | 'vec4f' | 'bool'` — no `u32` member — and `typeRefToDataType`'s own `scalar`
branch (`packages/graph/src/dataType.ts:48-52`) only converts `f32`/`bool` scalars to a legacy
`DataType`, explicitly returning `undefined` for `u32`/`i32`/`f16`. This is not an oversight to work
around; `Port.dataType`/`PortSpec.dataType` are both explicitly documented as `@deprecated
Compatibility/display alias... absent when no DataType equivalent exists`
(`packages/graph/src/types.ts:71-72`, `primitive.ts:45-46`) — `u32` is exactly the "no equivalent
exists" case the deprecation comment already anticipates. `type: TypeRef` is the canonical field
(`Port.type?: TypeRef`) and is what real WGSL codegen actually reads (`typeRefToWgsl`,
`dataType.ts:102-114`, returns `'u32'` for a `{kind:'scalar', scalar:'u32'}` type directly — no
issue there). **No existing primitive in this codebase currently declares a port via `type` instead
of `dataType`** (confirmed by grep) — `host.vertexIndex`/`host.instanceIndex` would be the first,
meaning F3.6.4's contract needs a real, direct test proving the whole chain still works end to end
for a `dataType`-less, `type`-only port: registration succeeds, `resolvePortDataType`/coercion
elsewhere in the graph don't silently choke on the missing `dataType`, and — the actual thing that
matters — `emitHostInput`'s new branch really does emit `let <var>: u32 = vid;` (or `iid`) for a
real graph wired through it. Not exhaustively traced here; flagging it as a real, first-of-its-kind
path rather than assuming existing `dataType`-based machinery generalizes for free.

Both trivial, both mechanical, both exactly the same size and risk as `host.fragCoord`'s own existing
implementation (modulo the typing correction above) — no new mechanism, no new `HostBinding` context
needed (`'stage-builtin'` already exists and already means "this WGSL identifier is a stage-provided
builtin already in scope," exactly the case here). `evalCPU` for both: no meaningful CPU-preview
equivalent exists (there is no "current vertex index" in a CPU field-evaluation context) — return a
stable placeholder (`0`) or omit CPU evaluation entirely, matching whatever this codebase's existing
convention is for GPU-only host
inputs (needs a quick check against `host.iFrame`/similar at contract time; not resolved here).

## (I) Geometry-source position — reuse `procedural.metricPosition`, don't invent a new primitive

**The apparent gap, and why it's smaller than it first looks.** Section (F) of the parent document
framed this as needing "a new primitive family" or extending `HostBinding`'s vocabulary. Closer
reading of `emitHostInput`'s *existing* `metricPosition` branch
(`packages/runtime-webgpu/src/emitGraphEval.ts:93-96`) found something that changes the shape of
this problem:

```ts
if (binding.context === 'invocation' && binding.key === 'metricPosition') {
	const expr = opts?.positionExpr ?? 'vec3<f32>(u, v, 0.0)';
	return `let ${variable}: vec3<f32> = ${expr};`;
}
```

`procedural.metricPosition` (`packages/graph/src/primitives/metricPosition.ts` — note the id prefix
is `procedural.`, not `host.`; it predates and is broader than F3's `host.*` family, already used
elsewhere in this codebase's procedural/terrain evaluation contexts per its help text, *"World/metric
position from the procedural evaluation context"*) **already supports an assembly-time override for
what "the current position" means**, via `EmitGraphEvalOptions.positionExpr` — the exact same
mechanism `fragCoordExpr`/`iResolutionExpr`/`iTimeExpr` already use for their own overrides. Nothing
about `metricPosition`'s registration, its `evalCPU`, or its `emitHostInput` branch needs to change.

**Recommendation: a vertex-kernel assembly call supplies `positionExpr:
'plane_grid_position(vid, ${resU}u, ${resV}u, ${width}, ${height}, ${rotationX}, ${rotationY},
${rotationZ})'`** (the exact same string-interpolated params `assemblePipelineVertexWgsl` already
builds today via `resolvePipelineGeometryParams` — nothing about *how geometry params flow in*
changes, only *where the resulting expression gets plugged in*). Any graph wired to
`procedural.metricPosition`'s output, evaluated in vertex-assembly context, transparently receives
the grid position instead of the fragment-side procedural-UV fallback. **This means F3.6.4 likely
does not need a new primitive for geometry sampling at all** — only a new call site supplying the
right `positionExpr`, decided at the point F3.6.5 builds the vertex-assembly function (out of scope
for a design document to write itself, but worth flagging now so F3.6.5's contract doesn't
independently reinvent a new mechanism for something that already exists).

**Correction, caught by a pre-routing review of an earlier draft — `positionExpr` alone is not
sufficient; the function it calls needs to actually be in scope.** `emitHostInput`'s `metricPosition`
branch only *substitutes* `positionExpr`'s text into the generated body (`let ${variable}: vec3<f32>
= ${expr};`) — it does not know or care what identifiers that expression string references, and
nothing about `emitGraphEval`/`compileGraph`'s own dependency walk would discover
`plane_grid_position`'s definition, because `geometry.plane` is never an upstream node of the
position-computing *expression graph* itself (it feeds `stage.vertex`'s separate `mesh` input, a
different, parallel path — confirmed in section (G)). Today, `plane_grid_position`'s WGSL definition
reaches the assembled module through one specific, explicit call:
`resolveVertexAssembly`(`fullscreenFragment.ts:92-107`) does `const planeModule = await
resolver.resolve('geometry.plane'); ...
assemblePipelineVertexWgsl(geo, planeModule.source)`, which prepends that resolved source verbatim
ahead of the hand-written `vs_main`. **Whatever F3.6.5's vertex-kernel assembly function turns out to
be must perform this exact same explicit resolve-and-prepend step itself** whenever it supplies a
`plane_grid_position(...)`-shaped `positionExpr` — reusing `resolveVertexAssembly` directly if its
vertex WGSL is still wanted verbatim, or at minimum resolving and including `geometry.plane`'s module
source by the same mechanism. Skipping this step doesn't fail loudly at graph-authoring time; it
fails at WGSL compile time with an undefined-function error, for the very first real graph anyone
wires through `procedural.metricPosition` in a vertex-kernel context — exactly the kind of
"prose-satisfied but not actually wired" gap this whole F3.6 sequence has been catching in review
after review. F3.6.5's contract needs an explicit Fix step (and a real, non-mocked test compiling
the result) for this, not an implicit assumption that supplying `positionExpr` is the whole story.

**Open risk worth flagging, not resolving here:** `procedural.metricPosition` is shared across
however many other evaluation contexts already use it (unaudited — this document did not trace every
existing caller). Confirm at F3.6.5 contract time that none of them assume `positionExpr` is *only*
ever the procedural-UV fallback in a way a vertex-context override would silently break; nothing
found during this investigation suggests a conflict (the option is per-call, not global), but it
wasn't exhaustively checked against every existing sample.

## (J) The vertex "position" output's type: `vec3f` graph value → `vec4f` clip position

**The gap.** `assembleStageEntry`'s vertex branch requires the position-producing function to return
exactly what `VSOut.position: vec4f` expects (`out.position = ${fn}(${args})`, no implicit wrap) —
confirmed, `varyingsStructWgsl` always declares `@builtin(position) position: vec4f` as a fixed
field (`stageEntry.ts:141`). But `procedural.metricPosition` (and `plane_grid_position` it would be
overridden to call) produces a `vec3f`. Today's hand-written template bridges this with a trivial,
hardcoded `vec4f(p, 1.0)` lift (`pipelineVertex.ts:69`) — there is no camera/view/projection matrix
anywhere in this pipeline model (confirmed: the whole `stage.fragment`/ShaderToy-style pipeline
draws a flat NDC-space plane and computes everything per-pixel; this is not a 3D-camera pipeline).

**Option J1 — the graph itself must produce a `vec4f`.** Whoever authors the vertex kernel's position
output wires a `vector.combine.vec3f_f32` (or similar existing combinator, if one exists) node
appending `w = 1.0` explicitly. No new WGSL-generation logic needed — `emitGraphVec4Eval` (already
built, already used by the fragment side) works unmodified. *Con:* every vertex-kernel graph has to
remember this ritual step; a forgotten one fails at `emitGraphVec4Eval`'s own output-type check with
a — hopefully clear — type-mismatch error, not silently.

**Option J2 — the vertex-assembly wrapper function does the lift.** Mirror
`buildGraphEvalFn`/`buildKernelGraphEvalFn`'s existing pattern (both hand-templated wrapper functions
built by the *caller*, not by `emitGraphEval` itself) with a vertex-specific version that always
appends `, 1.0)`: `fn graph_eval_position(vid: u32, iid: u32) -> vec4f { ...body...; return
vec4f(${resultExpr}, 1.0); }`, using `emitGraphVec3Eval` (already built, already exists, currently
unused by any real caller — this would be its first) instead of `emitGraphVec4Eval` for the position
output specifically. *Con:* forecloses ever wanting a graph-authored `w` (unlikely to matter for a
flat-plane pipeline with no real projection, but a real constraint nonetheless — worth someone
confirming at F3.6.5 contract time that `w` genuinely never needs to be anything but `1.0` for this
pipeline model).

**Recommendation: J2.** Matches this codebase's own established idiom exactly — every existing
wrapper (`buildGraphEvalFn` for fragment ShaderToy, `buildKernelGraphEvalFn` for fragment kernels,
both hand-templated by the caller, not generalized inside `emitGraphEval`) already does its own
small amount of caller-side wrapping around a raw `emitGraphEval` result. A vertex-position wrapper
doing the same `vec4f(..., 1.0)` lift is the same pattern, not a new one, and gives
`emitGraphVec3Eval` its first real caller — continuing this session's own repeated pattern of
generic-but-previously-unused F3.1/F3.2 machinery getting its first real consumer one milestone at a
time.

## (K) Varyings interaction — not this document's problem, but adjacent

F3.6.5 pairs a real vertex kernel with F3.6.3's real fragment kernel via F3.2's already-built
`assertVaryingsMatch` (landed, still unused by any real path — this would be its first real caller,
same pattern as (J)'s `emitGraphVec3Eval` point). A vertex kernel's *varying* outputs (as opposed to
its `position` output) face the exact same "what type does the graph need to produce, and how does
`vid`/`iid`/geometry-position flow into computing them" questions as `position` does — but a varying
is a per-vertex value handed to the fragment stage (e.g. a displaced normal, a computed UV), not
necessarily built from `procedural.metricPosition` the same way position is. This document does not
attempt to resolve the varying case in detail — (B)'s own B1 recommendation (call `emitGraphEval`
once per output) already covers the *mechanism*; this document's (H)/(I)/(J) findings (new
stage-builtin host inputs, `metricPosition` reuse, the `vec4f` lift pattern) are the *ingredients*
F3.6.5 combines, once per output it needs.

## (L) A pre-existing, unrelated gap this document deliberately does not fix

`HostBinding.stages` (already declared on every existing host-input primitive, including the two new
ones (H) recommends) is **not enforced anywhere** — confirmed by the parent document's own (A)
section and re-confirmed here: `emitGraphEval.ts` never reads `.stages`. Wiring `host.vertexIndex`
into a *fragment*-stage graph today would not be caught structurally; it would fail as a WGSL compile
error (undefined `vid` in `fs_main`) at the device layer, with a much less useful message than a
graph-validation error would give. This is a real, already-known gap, not created by this document's
recommendations — fixing it (making `validateGraph`/`validateGraphCompleteness` check a host-input
node's declared `stages` against the stage it's actually reachable from) is a reasonable, small,
separate improvement someone could scope on its own, but is out of scope here — F3.6.4's own contract
should not silently grow to include it unless explicitly asked for.

## Tentative shape of F3.6.4's contract, once scoped

Not a commitment — for whoever drafts F3.6.4's actual contract to accept, adjust, or discard:

1. Register `host.vertexIndex`/`host.instanceIndex` (section H) — two new, small, mechanical
   `host-input` primitives (output ports typed via `type: {kind:'scalar', scalar:'u32'}`, **not**
   `dataType: 'u32'` — no `DataType` equivalent exists), two new `emitHostInput` branches, plus a
   real test proving the whole `dataType`-less port chain works and `emitHostInput` really emits
   `let <var>: u32 = vid;`/`iid` — the first port in this codebase declared via `type` instead of
   `dataType`, not assumed to generalize for free. Lowest-risk slice otherwise, no interaction with
   any existing sample (nothing wires these yet).
2. Document (not necessarily code — may already just work) that `procedural.metricPosition` +
   `positionExpr` is the sanctioned mechanism for vertex-stage geometry-position sampling (section
   I) — confirm via a real, direct test that assembling with a `plane_grid_position`-shaped
   `positionExpr` produces correct, **compiling** WGSL for a graph wired to `procedural.metricPosition`,
   proving the reuse claim this document makes, not just asserting it. Critically, this means the
   test's assembly step must also resolve and prepend `geometry.plane`'s module source (mirroring
   `resolveVertexAssembly`'s existing pattern) — `positionExpr` alone produces a call to a function
   that doesn't exist yet in the assembled text otherwise; see section (I)'s correction.
3. Decide and prove the `vec4f` position lift (section J, recommending J2) with a real, minimal
   vertex-assembly proof — likely needs at least a thin vertical slice of what F3.6.5 will build in
   full, scoped down to "does the wrapper compile and produce a plausible clip-space position for a
   trivial graph," not the full vertex-kernel primitive itself.
4. Explicitly leave (K)'s varying-combination question and (L)'s `HostBinding.stages` enforcement
   gap to F3.6.5 and a separate, later, evidence-backed decision respectively — noted here so
   whoever scopes F3.6.4's real contract doesn't feel obligated to solve either.

## Open questions this document deliberately does not resolve

- **Does `host.vertexIndex`/`host.instanceIndex` need CPU-evaluation support at all**, or is
  omitting `evalCPU` (or returning an inert placeholder) an accepted pattern for GPU-only host
  inputs already? Needs a quick check against how other GPU-only-meaningful host inputs (if any)
  handle this — not checked exhaustively here.
- **Does anything today rely on `procedural.metricPosition`'s `positionExpr` always being the
  procedural-UV fallback** in a way a vertex-context override could silently change? Flagged in
  (I), not exhaustively audited.
- **Is `w = 1.0` really always correct for this pipeline's position output**, or does some future
  need (an actual camera/projection matrix, e.g. if this pipeline model is ever extended past
  flat-plane ShaderToy-style rendering) mean J1's graph-authored-`vec4f` flexibility is worth its
  extra ritual after all? Recommendation (J2) assumes not, based on the pipeline's current, entirely
  flat-plane scope — worth revisiting if that scope ever changes.
