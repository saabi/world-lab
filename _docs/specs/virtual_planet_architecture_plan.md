# Virtual Planet Renderer — Architecture and Implementation Plan

**Audience:** repository agents and implementation assistants  
**Repository:** `saabi/world-lab`  
**Status:** Proposed architecture for next-stage renderer

---

## 1. Purpose

This document defines an implementation direction for turning the current procedural planet prototype into a dynamic real-time planet renderer that supports:

- full-planet orbit views,
- near-surface flight,
- walking on the surface,
- scale-dependent procedural detail,
- GPU-assisted depth/picking queries,
- GPU-assisted local height/collision maps,
- no memory-heavy static planet mesh,
- a WebGPU-first architecture with a simpler WebGL/Three.js fallback.

The desired renderer should not depend on precomputed terrain tiles or large static meshes. It should generate transient patch geometry and terrain detail from deterministic procedural functions, with vertex density chosen from screen-space requirements.


Current renderer characteristics:

- Uses Three.js and `WebGLRenderer`.
- Uses `OrbitControls` for debug/orbit navigation.
- Builds one `PlaneGeometry`.
- Copies it into an `InstancedBufferGeometry`.
- Uses an `aIdx` instance attribute to generate patch positions.
- Uses shader-side procedural displacement to turn the patch field into a planet.
- Uses GLSL functions for hash, noise, fBM, Voronoi, height, erosion-like shaping, water, material color and lighting.
- Exposes many useful procedural planet parameters through the UI and presets.

The existing prototype is valuable because it already follows the correct general idea: the planet shape is mostly generated by shader functions, not by storing a large mesh.

The main limitations are:

1. The current grid is angular/polar and produces uneven density and pole convergence.
2. The same procedural sampling logic is duplicated between vertex and fragment shaders.
3. The renderer uses a single global coordinate scheme, which will not be stable near the surface.
4. It lacks a separated planet kernel usable by rendering, picking, collision and top-down height maps.
5. It lacks a real patch scheduler for near-constant screen-space density.
6. It does not yet support a WebGPU compute/indirect pipeline.

## 3. Core Design Principle

The renderer should be described as:

> **stateless procedural screen-space sampling**

Not:

> static mesh LOD

There will still be multiple resolutions, but they should be generated dynamically as transient patch descriptors. The important distinction is that no high-memory static mesh or terrain tile pyramid is required.

A better phrase than “no LOD” is:

> **No persistent planet LOD mesh. Use dynamic procedural patches selected by projected screen-space density.**

This means:

- no prebuilt planet mesh,
- no permanent tile mesh hierarchy,
- no baked terrain textures required for geometry,
- transient per-frame patch descriptors,
- procedural vertex displacement on the GPU,
- scale-gated terrain functions so distant patches do not evaluate small details.

## 5. Planet Kernel

Create a central procedural kernel. This should become the most important technical boundary in the project.

Conceptual API:

```ts
export interface PlanetSample {
  unitDir: Vec3;
  radiusMeters: number;
  heightMeters: number;
  waterHeightMeters: number;
  worldRadiusMeters: number;
  normal: Vec3;
  materialId: number;
  albedo: Vec3;
  roughness: number;
  biome: number;
}

export interface ScaleContext {
  cameraAltitudeMeters: number;
  distanceToCameraMeters: number;
  metersPerPixel: number;
  maxFeatureFrequency: number;
  mode: 'orbit' | 'flight' | 'surface';
}

export function samplePlanet(
  unitDir: Vec3,
  params: PlanetParameters,
  scale: ScaleContext
): PlanetSample;
```

Shader-side equivalent:

```wgsl
struct PlanetSample {
  unitDir: vec3f,
  heightMeters: f32,
  waterHeightMeters: f32,
  worldRadiusMeters: f32,
  normal: vec3f,
  materialId: u32,
  albedo: vec3f,
  roughness: f32,
  biome: f32,
};

struct ScaleContext {
  cameraAltitudeMeters: f32,
  distanceToCameraMeters: f32,
  metersPerPixel: f32,
  maxFeatureFrequency: f32,
  mode: u32,
};

fn sample_planet(unit_dir: vec3f, scale: ScaleContext) -> PlanetSample {
  // Macro terrain, continents, mountain belts, erosion, local detail.
}
```

The current GLSL `samplePlanet` function is the starting point, but it should be migrated out of `planet.vert` and `planet.frag` into shared shader modules.

## 6. Scale-Dependent Detail

The terrain kernel should be layered by physical feature scale. Small details should not even be evaluated when their projected size is below the pixel threshold.

Example layer stack:

```txt
Layer                  Always?      Approx. feature scale
----------------------------------------------------------
base sphere            yes          planet radius
continent masks        orbit+       100 km – 5000 km
tectonic cells         orbit+       50 km – 1000 km
mountain chains        orbit/flight 1 km – 100 km
river/erosion fields   flight+      10 m – 10 km
surface roughness      surface      0.1 m – 100 m
rocks/pebbles          walking      cm – m
grass/soil variation   walking      cm – m
```

Each procedural layer should have metadata:

```ts
export interface TerrainLayer {
  id: string;
  minFeatureMeters: number;
  maxFeatureMeters: number;
  maxDistanceMeters: number;
  minPixelsPerFeature: number;
  amplitudeMeters: number;
}
```

Shader usage:

```wgsl
fn should_eval_layer(feature_meters: f32, distance_meters: f32, meters_per_pixel: f32) -> bool {
  let projected_px = feature_meters / meters_per_pixel;
  return projected_px >= 0.5;
}
```

This prevents orbit views from paying for pebble noise and prevents subpixel terrain layers from shimmering.

## 7. Coordinate System

### 7.1 CPU coordinates

Use double precision on the CPU for global planet state.

Recommended global model:

```ts
export interface GeodeticPosition {
  latitudeRad: number;
  longitudeRad: number;
  altitudeMeters: number;
}

export interface EcefPosition {
  x: number;
  y: number;
  z: number;
}
```

Use ECEF or geodetic coordinates for persistent camera/player state. Do not store the player as a Three.js `Vector3` in global planet units once surface mode exists.

### 7.2 GPU coordinates

Use camera-local float32 coordinates on the GPU.

Each frame, compute a local tangent frame:

```ts
export interface LocalFrame {
  originEcef: Vec3d;      // double precision CPU-side origin
  east: Vec3d;
  north: Vec3d;
  up: Vec3d;
  planetCenterLocal: Vec3f;
  cameraLocal: Vec3f;
}
```

Near the surface, all rendered geometry should be transformed into this local frame before being sent to or computed on the GPU.

Conceptual conversion:

```ts
function ecefToLocal(p: Vec3d, frame: LocalFrame): Vec3f {
  const d = sub3d(p, frame.originEcef);
  return [
    dot3d(d, frame.east),
    dot3d(d, frame.north),
    dot3d(d, frame.up)
  ];
}
```

This avoids float precision jitter when walking on a planet-sized object.

### 7.3 Orbit vs surface

Use two coordinate regimes:

```txt
Orbit mode:
  normalized planet units or planet-radius-scaled units are acceptable.

Flight/surface mode:
  use meter units in a camera-local tangent frame.
```

The transition can be gradual, but the implementation should explicitly know which regime it is using.

## 8. Patch Strategy by View Mode

### 8.1 Orbit mode: cube-sphere patches

Replace the current polar/angular grid with cube-sphere patches.

Why:

- avoids pole singularities,
- more even area distribution,
- easier frustum culling,
- simple face/UV patch descriptors,
- natural stepping stone toward WebGPU patch generation.

Patch descriptor:

```ts
export interface CubeSpherePatch {
  face: 0 | 1 | 2 | 3 | 4 | 5;
  uvMin: [number, number];
  uvMax: [number, number];
  resolution: number;
  morph: number;
}
```

Vertex path:

```txt
face + uv
  -> cube position
  -> normalized unit sphere direction
  -> PlanetKernel.sample(unitDir)
  -> displaced world/local position
```

### 8.2 Surface mode: dynamic patch carpet

Surface rendering still needs a patch field. Use concentric procedural patch rings or a local quadtree around the camera.

Near the camera:

```txt
small patch footprint
high geometric detail
small terrain layers enabled
```

Toward the horizon:

```txt
larger patch footprint
same or lower internal vertex count
small terrain layers disabled
silhouette density preserved as needed
```

Surface patch descriptor:

```ts
export interface SurfacePatch {
  originLocalMeters: [number, number]; // tangent plane x/y
  sizeMeters: number;
  resolution: number;
  ring: number;
  maxFeatureMeters: number;
  morph: number;
}
```

The surface vertex shader maps:

```txt
local patch xy
  -> approximate geodetic offset / local tangent direction
  -> unit sphere direction
  -> PlanetKernel.sample(unitDir)
  -> camera-local float position
```

### 8.3 Flight mode

Flight mode is a bridge between orbit and walking.

Use:

- cube-sphere or large curved patches for far planet body,
- local surface patch carpet under/around camera,
- aggressive horizon culling,
- atmospheric depth/fog to hide far geometric limitations.

## 9. Surface Patch Scheduler

The scheduler should choose patch footprint and resolution from projected screen-space density.

Target:

```txt
projected vertex spacing ≈ target pixels
```

Approximation:

```ts
worldVertexSpacingMeters =
  targetVertexSpacingPx * distanceToPatchMeters / focalLengthPx;
```

For a patch of `resolution` vertices:

```ts
patchSizeMeters =
  worldVertexSpacingMeters * (resolution - 1);
```

A practical first version can use rings:

```txt
Ring 0: nearest ground around camera
Ring 1: local walking range
Ring 2: mid-distance terrain
Ring 3: far terrain
Ring 4: horizon band
```

Example, not fixed constants:

```txt
Ring       Distance          Patch size        Resolution
---------------------------------------------------------
0          0–32 m            4–8 m             32×32
1          32–128 m          16–32 m           32×32
2          128–512 m         64–128 m          24×24
3          512–2048 m        256–512 m         16×16
4          horizon           1–4 km            8×8 or 16×16
```

The horizon distance for a spherical planet is approximately:

```ts
function horizonDistance(radiusMeters: number, altitudeMeters: number): number {
  return Math.sqrt(2 * radiusMeters * altitudeMeters + altitudeMeters * altitudeMeters);
}
```

At walking height, this is modest. At flight altitude, it grows quickly.

### 9.1 Scheduler pseudocode

```ts
interface SurfacePatchSchedulerInput {
  radiusMeters: number;
  altitudeMeters: number;
  focalLengthPx: number;
  targetVertexSpacingPx: number;
  maxHorizonMultiplier: number;
  baseResolution: number;
}

function scheduleSurfacePatches(input: SurfacePatchSchedulerInput): SurfacePatch[] {
  const horizon = horizonDistance(input.radiusMeters, input.altitudeMeters);
  const maxDistance = horizon * input.maxHorizonMultiplier;

  const rings = buildDistanceBands(maxDistance);
  const patches: SurfacePatch[] = [];

  for (const ring of rings) {
    const distance = ring.midDistanceMeters;
    const worldSpacing =
      input.targetVertexSpacingPx * distance / input.focalLengthPx;

    const resolution = chooseResolution(ring, input.baseResolution);
    const patchSize = snapPow2(worldSpacing * (resolution - 1));

    const ringPatches = tileAnnulusWithPatches({
      innerRadius: ring.inner,
      outerRadius: ring.outer,
      patchSize,
      resolution,
      ringIndex: ring.index
    });

    patches.push(...ringPatches);
  }

  return cullPatchesAgainstFrustumAndHorizon(patches);
}
```

### 9.2 Seams

Use deterministic sampling from the same planet function, so there are no data seams. There can still be tessellation seams between different patch sizes.

Initial seam strategy:

1. snap patch origins/sizes to power-of-two grids,
2. add skirts along patch edges,
3. keep neighbor resolution ratios limited to 2:1,
4. later add morphing/stitching if needed.

For MVP, skirts are acceptable.

## 10. GPU Passes

### 10.1 WebGPU target pipeline

```txt
Frame
  1. Update camera/local frame uniforms
  2. Generate or upload patch descriptors
  3. Optional compute culling / patch expansion
  4. Terrain render pass
  5. Water render pass
  6. Atmosphere/sky pass
  7. Debug overlays
  8. Picking pass when requested
  9. Local heightfield/collision pass when needed
```

Advanced WebGPU path:

```txt
Compute pass:
  input camera, planet params
  generate visible patch descriptors
  write indirect draw records

Render pass:
  draw terrain patches via indirect draw
  sample PlanetKernel in vertex shader
  sample MaterialKernel in fragment shader
```

### 10.2 WebGL fallback

Keep it simpler:

```txt
CPU:
  schedule patches
  cull patches
  upload patch descriptors as instanced attributes/uniform texture

GPU:
  draw fixed-grid patch instances
  evaluate GLSL PlanetKernel in vertex/fragment
```

Do not try to force compute-like behavior into WebGL. The fallback can have fewer patches, lower collision resolution and no GPU-side patch generation.

## 11. Depth Hit Testing and Collision

### 11.1 Camera POV hit testing

Use a special picking/depth pass from the active camera.

Render to an offscreen texture containing at least:

```txt
linear depth
unitDir or local position
height
patch id or material id
```

Possible packed result:

```ts
export interface PickResult {
  hit: boolean;
  depthMeters: number;
  unitDir: [number, number, number];
  localPositionMeters: [number, number, number];
  heightMeters: number;
  materialId: number;
}
```

Readback should be async and delayed by one or more frames. For pointer hover, latency is acceptable. For immediate movement collision, use the local heightfield instead.

### 11.2 Local top-down heightfield

For walking, use a local orthographic height/normal/material map centered around the player or camera.

```ts
export interface LocalHeightFieldSettings {
  centerEcef: Vec3d;
  sizeMeters: number;
  resolution: number;
  updateThresholdMeters: number;
}
```

Texture contents:

```txt
R: height meters
GBA or separate texture: encoded normal
optional: material/biome id
```

This local map should be refreshed when the player moves far enough from its center.

Use cases:

- player grounding,
- slope detection,
- basic collision,
- foot placement,
- object placement,
- local physics queries.

This is better for walking than reading arbitrary camera depth.

Limitation:

- Heightfields do not support caves, overhangs, arches or vertical cliffs with multiple possible heights.
- If those are required later, add local SDFs or explicit procedural collision primitives.

### 12.2 Include strategy

#### WGSL

WGSL does not have native `#include`. Use a small build-time or runtime preprocessor.

Example source:

```wgsl
#include "../common/constants.wgsl"
#include "../common/math.wgsl"
#include "../noise/fbm.wgsl"
#include "../noise/voronoi.wgsl"
#include "../planet/types.wgsl"
#include "../planet/kernel.wgsl"

@vertex
fn main(input: VertexInput) -> VertexOutput {
  // ...
}
```

Implement a Vite plugin or helper:

```ts
export async function loadWgsl(path: string): Promise<string> {
  const source = await fetchShaderSource(path);
  return resolveIncludes(source, path);
}
```

For build-time bundling, prefer importing shaders as raw strings:

```ts
import terrainShader from './gpu/wgsl/terrain/surfacePatchVertex.wgsl?raw';
```

Then run `resolveIncludes(terrainShader, basePath)` before creating the shader module.

#### GLSL

The repo already uses `glslify`. Keep GLSL fallback compatible with that approach.

Example:

```glsl
#pragma glslify: fbm4 = require('../noise/fbm.glsl')
#pragma glslify: voronoi3 = require('../noise/voronoi.glsl')
#pragma glslify: samplePlanet = require('../planet/kernel.glsl')
```

or simple text includes if glslify becomes inconvenient.

### 12.3 Shared structs

Mirror TypeScript and GPU structs carefully. Prefer explicit packing.

```ts
export interface GpuPlanetParams {
  radiusMeters: number;
  waterLevel: number;
  voronoiScale: number;
  voronoiAmplitude: number;
  detailScale: number;
  detailAmplitude: number;
  erosion: number;
  sandCutoff: number;
  vegetationLevel: number;
  snowCover: number;
  polarScale: number;
  polarAmplitude: number;
}
```

For WebGPU, create uniform/storage buffer layout utilities:

```ts
export function writePlanetParamsToBuffer(
  params: PlanetParameters,
  target: Float32Array,
  offset = 0
): void {
  target[offset + 0] = params.radiusMeters;
  target[offset + 1] = params.waterLevel;
  target[offset + 2] = params.voronoiScale;
  target[offset + 3] = params.voronoiAmplitude;
  // ...
}
```

Avoid ad-hoc uniform dictionaries once moving away from the legacy Three.js shader prototype.

- rotating the mesh as the main planet movement model,
- any assumption that `resx/resy` defines planet quality globally,
- polar `angle` as the primary surface coverage model.

### Phase 4 — Surface patch scheduler

Goal: implement the near-surface patch carpet.

Tasks:

1. Create `SurfacePatch` descriptor.
2. Implement CPU ring scheduler.
3. Choose patch size from approximate projected screen-space density.
4. Snap patch origins and sizes to power-of-two grids.
5. Limit neighbor resolution ratio.
6. Add skirts to hide cracks.
7. Add debug colors per ring.
8. Add horizon distance limit.

Acceptance:

- dense patches near camera,
- larger patches toward horizon,
- roughly constant screen-space vertex density,
- no major cracks,
- no full-planet mesh required in surface mode.

Goal: separate navigation and rendering assumptions.

Tasks:

   - orbit,

3. Surface/walk uses local patch carpet.
4. Flight may blend both.
5. Add mode-specific camera controls.
6. Add altitude-based automatic transition option.

Acceptance:

- orbit view works,
- flight near terrain works,
- walk mode remains grounded to generated height,
- renderer mode is explicit and debuggable.

---

### Phase 6 — Camera POV depth/picking pass

Goal: support mouse hit-testing and terrain selection.

Tasks:

1. Add offscreen render target.
2. Render depth and encoded terrain data.
3. Implement async pixel readback.
4. Return `PickResult`.
5. Add debug cursor marker on terrain.
6. Add throttling/debouncing to avoid excessive readback.

Acceptance:

- clicking terrain returns stable unit direction/height/local point,
- marker appears where clicked,
- readback latency does not block rendering.

---

### Phase 7 — Local heightfield/collision pass

Goal: support walking and collision from the same terrain kernel.

Tasks:

1. Create top-down orthographic local height map.
2. Center it around player/camera.
3. Generate height, normal and material.
4. Refresh when player moves beyond threshold.
5. Implement bilinear height sampling on CPU or GPU.
6. Use it for walking mode grounding and slope checks.

Acceptance:

- walking camera follows terrain height,
- terrain visible and terrain collision agree,
- heightfield updates smoothly as player moves.

---

### Phase 8 — WebGPU backend

Goal: add the WebGPU-first path.

Tasks:

1. Create `RenderBackend` interface.
2. Implement `WebGLBackend` around current/near-term Three.js path.
3. Implement `WebGPUBackend`.
4. Add WGSL module/include loader.
5. Port PlanetKernel to WGSL.

## 16. Risks and Mitigations

### Risk: trying to avoid “LOD” too literally

Mitigation:

- use “screen-space procedural sampling” language,
- accept dynamic multi-scale patches,
- avoid stored mesh pyramids.

### Risk: WebGPU port too early

Mitigation:

- first clean kernel and patch abstractions,
- implement CPU-scheduled WebGL/Three.js patch descriptors,
- port to WebGPU after behavior is understood.

### Risk: coordinate precision bugs

Mitigation:

- introduce local frames early,
- keep global camera/player state in double precision,
- never send huge planet-scale positions as raw float32 near the surface.

### Risk: render/collision mismatch

Mitigation:

- all render and collision passes must use the same PlanetKernel,
- local heightfield should be generated by the same kernel,
- add visual debug comparing rendered terrain and collision height.

### Risk: shader module divergence

Mitigation:

- keep GLSL and WGSL kernels structurally similar,
- use generated tests or sample comparisons,
- isolate noise and terrain layers into small modules.

## 18. Summary

The project should evolve from the current single legacy Three.js shader prototype into a renderer built around four core abstractions:

```txt
PlanetKernel
  deterministic procedural terrain/material function

CoordinateFrame
  global double precision + GPU local float precision

PatchScheduler
  transient screen-space procedural patch descriptors

RenderBackend
  WebGPU-first, WebGL fallback
```

Orbit rendering should use cube-sphere procedural patches. Surface rendering should use a dynamic multi-scale patch carpet around the camera, with near patches small and detailed, far patches larger and functionally simplified, and the visible horizon covered by progressively coarser patches. Collision and picking should be GPU-assisted but must call the same planet kernel as rendering.

This approach preserves the original spirit of the project: no large static planet mesh, fully dynamic generation, shader-driven terrain, and scale-dependent detail from orbit down to walking on the surface.
